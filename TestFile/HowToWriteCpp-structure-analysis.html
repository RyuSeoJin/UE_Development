01 — 개념 이해

UE C++ vs 표준 C++

언리얼 엔진의 C++는 표준 C++와 근본적으로 다른 확장을 포함합니다. 리플렉션 시스템, 가비지 컬렉션, 매크로 기반 메타데이터가 핵심 차이점입니다.

💡

리플렉션 시스템: 표준 C++에는 런타임 타입 정보(RTTI)가 제한적이지만, 언리얼은 UCLASS, UPROPERTY, UFUNCTION 매크로를 통해 완전한 리플렉션을 지원합니다. 이를 통해 에디터 노출, 직렬화, 네트워크 리플리케이션, 가비지 컬렉션이 자동화됩니다.

02 — 네이밍 컨벤션

접두사 규칙과 명명 표준

언리얼 엔진은 엄격한 접두사 규칙을 따릅니다. 접두사만 보면 해당 타입의 역할과 상속 관계를 즉시 파악할 수 있습니다.

접두사의미대표 예시
AActor 파생 클래스AActor, ACharacter, APlayerController, AGameModeBase
UUObject 파생 클래스 (Actor 제외)UActorComponent, UUserWidget, USceneComponent
F구조체 / 일반 C++ 클래스FVector, FString, FTransform, FName, FText
E열거형 (Enum)ECollisionChannel, EMovementMode, ENetRole
T템플릿 클래스TArray, TMap, TSet, TSharedPtr, TWeakPtr
I인터페이스IInterface
bBoolean 변수 접두사bIsActive, bCanJump, bReplicates
⚠️

필수 규칙: .generated.h 파일은 반드시 마지막 #include여야 합니다. GENERATED_BODY()는 클래스 본문의 첫 번째 줄에 위치해야 합니다. 이 규칙을 어기면 UHT(Unreal Header Tool)가 컴파일 에러를 발생시킵니다.

MyCharacter.h — 올바른 헤더 구조
// 일반 헤더를 먼저 포함
#include "CoreMinimal.h"
#include "GameFramework/Character.h"
// .generated.h는 반드시 마지막!
#include "MyCharacter.generated.h"

UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()  // 반드시 첫 줄!

public:
    AMyCharacter();
};
03 — 핵심 매크로 시스템

UCLASS, UPROPERTY, UFUNCTION, USTRUCT, UENUM

언리얼의 매크로 시스템은 C++ 클래스를 엔진에 등록하고 에디터, 블루프린트, 네트워크, 직렬화 시스템과 연동하는 핵심 메커니즘입니다.

UCLASS() 지정자

클래스를 언리얼 리플렉션 시스템에 등록합니다. 지정자(Specifier)로 동작을 세밀하게 제어합니다.

지정자설명
Abstract인스턴스화 불가. 기반 클래스로만 사용
Blueprintable블루프린트에서 이 클래스를 상속할 수 있음
BlueprintType블루프린트 변수 타입으로 사용 가능
Placeable에디터에서 레벨에 직접 배치 가능
Config=Game설정 파일에서 프로퍼티 값 로드
Transient직렬화 대상에서 제외 (저장하지 않음)
EditInlineNew디테일 패널에서 인라인 편집 가능
DefaultToInstanced기본적으로 인스턴스화된 프로퍼티로 취급
MinimalAPI타입 정보만 내보내고, 함수는 내보내지 않음
meta=(BlueprintSpawnableComponent)블루프린트에서 AddComponent로 추가 가능
UCLASS 사용 예시
// Blueprintable: BP로 상속 가능 / BlueprintType: BP 변수 타입으로 사용 가능
// Abstract: 이 클래스 자체는 인스턴스화 금지 — 반드시 자식을 만들어 쓴다
UCLASS(Blueprintable, BlueprintType, Abstract)
class MYGAME_API UMyBaseAbility : public UObject  // MYGAME_API: 다른 모듈에 클래스 공개(DLL export)
{
    GENERATED_BODY()  // 리플렉션 코드가 들어갈 자리 — 클래스 본문 첫 줄 고정
public:
    // PURE_VIRTUAL: UObject는 순수 가상 함수(= 0)를 못 쓰므로 이 매크로로 대체
    // 자식이 오버라이드하지 않고 호출하면 런타임 에러를 낸다
    virtual void ActivateAbility() PURE_VIRTUAL(UMyBaseAbility::ActivateAbility,);
};

UPROPERTY 사용 예시
// 에디터 어디서나 편집 + BP 읽기/쓰기 가능, 값은 0~100으로 제한
UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Stats",
    meta=(ClampMin="0", ClampMax="100"))
float Health = 100.0f;

// 컴포넌트 포인터는 보통 VisibleAnywhere — 포인터 자체를 바꾸면 안 되니 편집 금지, 내용만 보이게
UPROPERTY(VisibleAnywhere, BlueprintReadOnly, Category="Components")
UStaticMeshComponent* MeshComp;

// Replicated: 서버 값이 클라이언트로 자동 복제 (GetLifetimeReplicatedProps 등록 필요)
UPROPERTY(Replicated, BlueprintReadOnly, Category="State")
bool bIsAlive = true;

// EditDefaultsOnly: BP 클래스 기본값에서만 편집(레벨 인스턴스에선 불가)
// TSubclassOf: AActor 계열 클래스만 선택 가능하도록 타입 제한
UPROPERTY(EditDefaultsOnly, BlueprintReadOnly, Category="Config",
    meta=(AllowPrivateAccess="true"))
TSubclassOf<AActor> ProjectileClass;

UFUNCTION 사용 예시
// 블루프린트에서 호출 가능한 함수
UFUNCTION(BlueprintCallable, Category="Combat")
void FireWeapon();

// 사이드 이펙트 없는 순수 함수
UFUNCTION(BlueprintPure, Category="Stats")
float GetHealthPercent() const;

// BP에서만 구현하는 이벤트
UFUNCTION(BlueprintImplementableEvent, Category="Events")
void OnLevelUp(int32 NewLevel);

// C++ 기본 구현 + BP 오버라이드
UFUNCTION(BlueprintNativeEvent, Category="Damage")
float TakeDamage(float Amount);
float TakeDamage_Implementation(float Amount);

// 서버 RPC
UFUNCTION(Server, Reliable, WithValidation)
void ServerRPC_Attack();

USTRUCT() & UENUM()

구조체와 열거형을 리플렉션 시스템에 등록합니다.

USTRUCT 예시
// BlueprintType: 이 구조체를 BP 변수로 사용 가능. 구조체 이름은 F 접두사
USTRUCT(BlueprintType)
struct FItemData
{
    GENERATED_BODY()  // 구조체에도 필수

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    FString ItemName;

    // 값 타입은 초기값을 꼭 지정 — 안 하면 쓰레기 값이 들어간다
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    int32 ItemCount = 0;

    // 포인터도 nullptr로 초기화 — UPROPERTY라서 GC 추적 대상이 된다
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    UTexture2D* Icon = nullptr;
};
UENUM 예시
// BP에 노출하는 enum은 반드시 uint8 기반 + E 접두사
UENUM(BlueprintType)
enum class EWeaponType : uint8
{
    // UMETA(DisplayName): 에디터/BP 드롭다운에 표시될 이름
    Sword     UMETA(DisplayName="검"),
    Bow       UMETA(DisplayName="활"),
    Staff     UMETA(DisplayName="지팡이"),
    Shield    UMETA(DisplayName="방패"),
};
04 — 액터 라이프사이클

Actor 생성에서 소멸까지

액터의 라이프사이클은 생성 경로에 따라 다르게 진행됩니다. 각 단계에서 허용되는 작업을 정확히 이해해야 안정적인 코드를 작성할 수 있습니다.

기본 라이프사이클 흐름

1
Constructor (생성자)
CreateDefaultSubobject()로 컴포넌트 생성, SetupAttachment()로 부착. 기본값 설정만! GetWorld(), SpawnActor() 등 월드 관련 호출 금지.
2
PostInitializeComponents
모든 컴포넌트의 초기화가 완료된 후 호출. 컴포넌트 간 상호 참조 설정에 적합.
3
BeginPlay
게임플레이 로직의 시작점. 월드가 완전히 초기화된 상태. 대부분의 초기화 로직은 여기에 작성.
4
Tick (매 프레임)
매 프레임 호출. PrimaryActorTick.bCanEverTick = false;로 비활성화 가능. 성능을 위해 가능하면 타이머나 이벤트 기반으로 대체.
5
EndPlay
액터 제거, 레벨 전환, 세션 종료 시 호출. 리소스 정리, 타이머 해제, 델리게이트 언바인드 수행.
6
Destroy → BeginDestroy → FinishDestroy → GC
소멸 체인. Destroy() 호출 후 GC가 메모리를 회수. UObject의 소멸은 GC에 의해 비동기적으로 처리.
⚠️

주의: Constructor에서 GetWorld()를 호출하면 nullptr 크래시가 발생합니다! Constructor에서는 BeginPlay 로직을 절대 실행하지 마세요. CDO(Class Default Object) 생성 시에는 월드가 존재하지 않습니다.

Spawning 흐름 vs Loading 흐름

🚀 Spawning 흐름 (런타임 생성)

코드에서 SpawnActor()로 액터를 동적 생성할 때의 호출 순서:

SpawnActor PostSpawnInitialize PostActorCreated ExecuteConstruction OnConstruction
PreInitializeComponents InitializeComponent PostInitializeComponents BeginPlay

📂 Loading 흐름 (레벨 로드)

레벨에 배치된 액터가 로드될 때의 호출 순서:

PostLoad InitializeActorsForPlay RouteActorInitialize BeginPlay
💡

Deferred Spawn: SpawnActorDeferred<T>()로 생성을 시작하면 추가 설정 후 FinishSpawning()을 호출하여 초기화를 완료합니다. 스폰 전에 UPROPERTY 값을 설정해야 할 때 유용합니다.

05 — 핵심 클래스 계층

주요 클래스와 역할

언리얼 엔진의 게임플레이 프레임워크를 구성하는 핵심 클래스들의 역할과 관계를 이해하면 올바른 상속 구조를 설계할 수 있습니다.

UObject
모든 UE 객체의 최상위 클래스
가비지 컬렉션 관리, 리플렉션 지원, 직렬화, CDO(Class Default Object) 패턴의 기반. 월드에 배치는 불가.
AActor
월드에 배치/스폰 가능한 객체
컴포넌트를 소유하고, Tick/BeginPlay 라이프사이클 보유. 네트워크 리플리케이션 지원. 트랜스폼은 RootComponent에 의존.
APawn
Controller에 의해 조종 가능한 Actor
플레이어 또는 AI Controller에 의해 Possess되어 입력을 받는 액터. MovementComponent 연동 가능.
ACharacter
APawn + 이동 + 충돌 + 메시
CharacterMovementComponent(걷기/점프/수영/비행), CapsuleComponent(충돌), SkeletalMeshComponent(애니메이션)를 기본 탑재.
APlayerController
플레이어 입력 → Pawn 조작
플레이어의 입력을 Pawn에 전달. HUD/UI 관리, RPC의 시작점. 각 플레이어당 1개 존재.
AGameModeBase
게임 규칙 정의, 서버 전용
게임 룰, 기본 Pawn/Controller 클래스 지정, 플레이어 로그인 처리. 서버에서만 존재하며 클라이언트에 리플리케이트되지 않음.
AGameStateBase
게임 전체 상태 관리
매치 시간, 점수 등 게임 전체 공유 데이터 저장. 모든 클라이언트에 자동 리플리케이트.
APlayerState
개별 플레이어 상태
플레이어 이름, 점수, 팀 등 개별 플레이어 정보. 모든 클라이언트에 리플리케이트. Pawn이 파괴되어도 유지.
UActorComponent
트랜스폼 없는 로직 컴포넌트
공간 정보 없이 순수 로직만 제공. InventoryComponent, HealthComponent 등 데이터/로직 모듈화에 활용.
USceneComponent
트랜스폼 있는 공간 컴포넌트
위치/회전/스케일 정보를 가지며 부착 계층 구조 지원. StaticMeshComponent, CameraComponent 등의 기반 클래스.
UUserWidget
UMG UI 위젯 기반 클래스
UMG(Unreal Motion Graphics) UI 시스템의 기반. 체력바, 인벤토리 화면, 메인 메뉴 등 모든 UI 요소의 부모 클래스.
06 — 필수 타입 & 컨테이너

언리얼 전용 타입과 컨테이너

언리얼 엔진은 STL 대신 자체 타입과 컨테이너를 사용합니다. 가비지 컬렉션, 리플렉션, 현지화와 통합되어 있어 표준 C++ 타입 대신 반드시 UE 타입을 사용해야 합니다.

문자열 타입

타입설명용도
FString가변 문자열. TEXT() 매크로로 생성일반적인 문자열 조작, 로깅
FName경량 식별자. 대소문자 무시, 해시 기반 빠른 비교에셋 이름, 소켓, 본 이름
FText현지화 지원 텍스트UI 표시용 텍스트, 다국어 지원

타입설명

타입설명

07 — 블루프린트 연동 패턴

C++와 블루프린트를 연결하는 5가지 패턴

언리얼의 핵심 철학은 "C++로 기반을 만들고, 블루프린트로 확장한다"입니다. 다음 5가지 패턴을 익히면 프로그래머와 디자이너 간 효율적인 협업이 가능합니다.

패턴 1: BlueprintCallable — C++ 함수를 BP에서 호출

가장 기본적인 연동. C++에서 구현한 함수를 블루프린트 그래프에서 노드로 호출합니다.

BlueprintCallable
UFUNCTION(BlueprintCallable, Category="Inventory")
bool AddItem(FName ItemID, int32 Count);
// BP에서 "Add Item" 노드로 사용 가능

패턴 2: BlueprintPure — 사이드 이펙트 없는 계산

실행 핀 없이 데이터만 반환하는 순수 함수. 상태를 변경하지 않는 getter/계산 함수에 사용합니다.

BlueprintPure
UFUNCTION(BlueprintPure, Category="Stats")
float GetHealthPercent() const
{
    return CurrentHealth / MaxHealth;
}
// BP에서 실행 핀 없이 값만 가져옴

패턴 3: BlueprintImplementableEvent — BP에서만 구현

C++에서 선언만 하고 구현은 블루프린트에서 합니다. C++ 본문을 작성하지 않습니다.

BlueprintImplementableEvent
// 헤더에 선언만 (구현 작성 금지!)
UFUNCTION(BlueprintImplementableEvent, Category="Events")
void OnItemCollected(FName ItemID);

// C++에서 호출하면 BP 구현이 실행됨
OnItemCollected("Potion_HP");

패턴 4: BlueprintNativeEvent — C++ 기본 + BP 오버라이드

C++에서 기본 구현을 제공하고, 블루프린트에서 선택적으로 오버라이드합니다. _Implementation 접미사로 기본 구현을 작성합니다.

BlueprintNativeEvent
// 헤더
UFUNCTION(BlueprintNativeEvent, Category="Damage")
float CalculateDamage(float BaseDamage);

// cpp — _Implementation 접미사 필수!
float AMyCharacter::CalculateDamage_Implementation(float BaseDamage)
{
    return BaseDamage * DamageMultiplier;
}
// BP에서 오버라이드하지 않으면 C++ 구현이 실행됨

패턴 5: BlueprintAssignable — 이벤트 디스패처 BP 노출

C++의 멀티캐스트 델리게이트를 블루프린트에서 바인딩할 수 있게 노출합니다.

BlueprintAssignable
// 델리게이트 타입 선언
DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(
    FOnHealthChanged, float, NewHealth);

// UPROPERTY로 노출
UPROPERTY(BlueprintAssignable, Category="Events")
FOnHealthChanged OnHealthChanged;

// C++에서 브로드캐스트
OnHealthChanged.Broadcast(CurrentHealth);
// BP에서 "Bind Event" 또는 "Assign" 노드로 수신
08 — 팁 & 주의사항

자주 발생하는 실수와 모범 사례

UE C++ 개발에서 자주 발생하는 실수와 이를 방지하기 위한 모범 사례를 정리합니다.

흔한 실수 (Pitfalls)

⚠️
Constructor에서 GetWorld() 호출 — CDO 생성 시 월드가 없으므로 nullptr 크래시 발생. 월드 의존 로직은 반드시 BeginPlay() 이후에 작성하세요.
⚠️
.generated.h가 마지막 #include가 아닌 경우 — UHT(Unreal Header Tool)가 생성한 코드와 충돌하여 컴파일 에러 발생. 항상 마지막 #include로 배치하세요.
⚠️
GENERATED_BODY() 누락 — 리플렉션 코드가 생성되지 않아 컴파일 에러. UCLASS/USTRUCT 선언 시 반드시 본문 첫 줄에 포함하세요.
⚠️
UPROPERTY 없는 UObject* — 가비지 컬렉터가 참조를 추적하지 못해 예고 없이 메모리를 해제합니다. UObject 포인터는 반드시 UPROPERTY()로 마킹하세요.
⚠️
Hot Reload 사용 (UE5) — UE5에서 Hot Reload는 불안정합니다. 대신 Live Coding (Ctrl+Alt+F11)을 사용하세요. 헤더 변경 시에는 에디터 재시작이 필요할 수 있습니다.
⚠️
헤더 변경 후 에디터 미재시작 — UPROPERTY, UFUNCTION 시그니처 변경 후에는 Live Coding으로 반영되지 않는 경우가 있습니다. 이런 경우 에디터를 완전히 재시작해야 합니다.

Do / Don't

✅ Do (권장)

  • UObject 포인터는 항상 UPROPERTY()로 마킹
  • 컴포넌트는 Constructor에서 CreateDefaultSubobject로 생성
  • Blueprint에 노출할 함수에는 Category를 명시
  • Tick 대신 타이머(FTimerManager)나 이벤트 사용
  • TEXT() 매크로로 문자열 리터럴 감싸기
  • IsValid() 또는 nullptr 체크 후 포인터 사용
  • C++ 기반 클래스 + BP 파생 클래스 구조 활용
  • 네트워크 함수에 WithValidation 추가로 보안 강화

❌ Don't (금지)

  • Constructor에서 GetWorld(), SpawnActor() 호출
  • new 키워드로 UObject 생성 (반드시 NewObject 사용)
  • UObject에 TSharedPtr 사용 (GC 충돌)
  • std::string, std::vector 등 STL 컨테이너 사용
  • Hot Reload 의존 (UE5에서는 Live Coding 사용)
  • .generated.h를 마지막 #include 이외의 위치에 배치
  • GENERATED_BODY() 없이 UCLASS 선언
  • UPROPERTY 없이 UObject* 멤버 변수 선언

자주 묻는 질문 (FAQ)

질문답변
BlueprintCallable과 BlueprintPure의 차이는?BlueprintCallable은 실행 핀이 있고 사이드 이펙트를 가질 수 있습니다. BlueprintPure는 실행 핀 없이 값만 반환하며 상태를 변경하지 않습니다.
BlueprintImplementableEvent vs BlueprintNativeEvent?ImplementableEvent는 C++에 구현이 없고 BP에서만 구현합니다. NativeEvent는 C++에 _Implementation 기본 구현을 제공하고 BP에서 선택적으로 오버라이드합니다.
EditAnywhere vs EditDefaultsOnly?EditAnywhere는 BP 기본값과 레벨 인스턴스 모두에서 편집 가능. EditDefaultsOnly는 BP 에디터의 기본값만 편집 가능하고, 레벨에 배치된 인스턴스에서는 편집 불가.
TSharedPtr를 UObject에 쓰면 안 되는 이유?UObject는 가비지 컬렉터(GC)가 관리합니다. TSharedPtr는 참조 카운팅 기반이므로 GC와 충돌하여 이중 해제(double free) 또는 댕글링 포인터가 발생합니다.
FName vs FString vs FText는 언제 사용?FName: 식별자, 비교 연산이 잦은 경우. FString: 문자열 조작이 필요한 경우. FText: UI에 표시하거나 현지화가 필요한 경우.
Server RPC에 WithValidation이 왜 필요한가?클라이언트가 보내는 RPC는 치트 가능성이 있습니다. WithValidation은 _Validate 함수에서 입력값을 검증하여 부정 행위를 방지합니다.
