가챠 모집 스쿼드 편성 시너지
01 — 개념 이해

길드 시스템이란

길드 시스템은 생산성 앱(포모도로·퀘스트)에 게임화 요소를 더하는 캐릭터 수집·편성 시스템입니다. 사용자는 Gold(앱 사용 중 적립되는 인게임 화폐)를 소비해 가챠(랜덤 카드 뽑기)를 돌리고, 모집한 길드원을 스쿼드(출전 편성)에 배치합니다. 특정 길드원 조합이 갖춰지면 시너지(조합 보너스)가 활성화됩니다.

이 시스템의 모든 런타임 로직은 UGuildSubsystem(UGameInstanceSubsystem을 상속하는 서브시스템, 즉 프로그램이 실행되는 동안 자동으로 살아있는 서비스 객체)에 모여 있으며, 마스터 데이터(길드원 스탯·시너지 정의)는 DataTable(언리얼 에디터에서 표 형태로 편집하는 데이터 시트)로 관리합니다.

핵심 용어

📚

길드원(Guild Member) — 모집 가능한 캐릭터. 12명으로 구성된 고정 풀에서 가챠로 획득합니다.
가챠(Gacha) — Gold 1,000을 소비해 랜덤 3장의 카드를 제시받고 1명을 선택하는 모집 방식.
스쿼드(Squad) — 현재 출전 중인 길드원 편성. 슬롯(초기 2개, 최대 4개)에 길드원을 배치합니다.
프리셋(Preset) — 스쿼드 편성 상태를 저장해두는 슬롯. 초기 1개, 해금으로 최대 3개.
시너지(Synergy) — 특정 길드원 조합이 스쿼드에 모두 편성되면 활성화되는 보너스.
희귀도(Rarity) — Normal / Rare / Epic / Legendary 4등급. 가챠 확률 가중치와 연동됩니다.

💡

길드원은 실제 게임플레이에 영향을 주지 않지만, 수집·편성의 재미와 시너지 보너스로 앱 사용 동기를 높입니다.

02 — 길드원 데이터 구조

마스터 데이터 & 런타임 보유 목록

길드원의 정보는 두 계층으로 나뉩니다. 마스터 데이터(이름·스탯·스토리 등 불변 정보)는 DataTable로 에디터에서 편집하고, 런타임 보유 목록(어떤 길드원을 소유하고 있는지)은 TArray<FString> OwnedMemberIds로 SaveGame에 저장합니다.

FGuildMemberRow — DataTable 행 구조체

USTRUCT(BlueprintType)
struct FGuildMemberRow : public FTableRowBase
{
    GENERATED_BODY()

    // 고유 식별자 (예: "member_01")
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FString Id;

    // 길드원 이름 (UI에 표시)
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FString Name;

    // 희귀도 등급
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    EGuildRarity Rarity;

    // 캐릭터 이미지 경로 (UTexture2D 소프트 레퍼런스)
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    TSoftObjectPtr<UTexture2D> ImagePath;

    // 스탯 (ATK / DEF / SPD / SPL)
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FGuildStats Stats;

    // 캐릭터 배경 스토리
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FString Story;

    // 이 길드원이 참여하는 시너지 ID 목록
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    TArray<FString> Synergies;
};

필드타입설명

런타임 보유 목록은 TArray<FString> OwnedMemberIds로, 가챠에서 선택한 길드원의 Id를 저장합니다. 이 배열은 USaveGame에 직렬화되어 앱을 껐다 켜도 유지됩니다.

03 — 가챠 시스템

모집 흐름

가챠(앞서 설명한 랜덤 모집 시스템)는 Gold 1,000을 소비합니다. 한 번의 모집에서 3명의 후보가 카드로 제시되며, 사용자는 그중 1명을 선택하거나 건너뛸 수 있습니다.

모집 절차

1
Gold 잔액 확인
UEconomySubsystem(Gold/XP를 관리하는 경제 서브시스템)에서 현재 Gold를 조회합니다. 1,000 미만이면 토스트 알림("Gold가 부족합니다")을 띄우고 중단합니다.
2
Gold 1,000 차감
UEconomySubsystem::AddGold(-1000)을 호출하여 잔액을 깎습니다.
3
희귀도 가중치 기반 랜덤 3명 선정
현재 길드 레벨에 따른 확률 테이블(아래 참조)로 희귀도를 먼저 결정하고, 해당 희귀도 풀에서 랜덤으로 선택합니다. 이미 보유한 길드원과 같은 모집 안에서 이미 뽑힌 길드원은 후보에서 제외(중복 불가)합니다.
4
카드 3장 UI 제시
3명의 후보를 카드 형태로 사용자에게 보여줍니다. 각 카드에는 이름·이미지·희귀도·스탯이 표시됩니다.
5
사용자 1명 선택 또는 건너뛰기
사용자가 카드를 탭하면 해당 길드원을 획득합니다. "건너뛰기"를 누르면 아무도 획득하지 않습니다(Gold는 반환되지 않습니다).
6
OwnedMemberIds에 추가
선택 시 OwnedMemberIds.Add(SelectedId)로 보유 목록에 추가하고, SaveGame을 동기화합니다.

희귀도 확률 가중치 (길드 레벨별)

레벨 범위NormalRareEpicLegendary
1 ~ 570%20%8%2%
6 ~ 1060%25%11%4%
11 ~ 1550%28%15%7%
16 ~ 2040%30%20%10%

위 표의 각 행에서 네 확률의 합은 반드시 100%여야 합니다. 가중치를 변경할 때 합이 100이 아니면 비정상적인 확률 분포가 됩니다.

보유 상한: 8명. 이미 보유한 캐릭터는 가챠 후보에서 자동 제외됩니다. 따라서 보유 수가 많아질수록 미보유 풀이 좁아져 원하는 캐릭터를 얻을 확률이 올라갑니다.

⚠️

전원 보유 시 모집 불가. OwnedMemberIds.Num() >= 8이면 가챠 버튼을 UI에서 비활성화(회색 처리)하고, 탭 시 "모든 길드원을 보유 중입니다" 토스트를 띄웁니다.

배반(Betrayal) 이벤트

극히 낮은 확률로 발생하는 특별 연출 이벤트입니다. 가챠 카드 제시 시 드래곤 등장 등의 특수 애니메이션이 재생됩니다. 이 이벤트의 구현은 순수 UI 레이어에서 처리하며, UGuildSubsystem의 로직에는 영향을 주지 않습니다.

04 — 스쿼드 편성

편성 규칙 & 프리셋

스쿼드(앞서 설명한 출전 편성)는 보유 중인 길드원을 슬롯에 배치하는 것입니다. 초기에는 2개 슬롯만 사용할 수 있으며, 해금 시스템(Gold를 소비해 기능을 여는 구조, 05 경제 문서 참조)을 통해 3번째·4번째 슬롯을 열 수 있습니다.

편성 규칙

규칙설명
한 캐릭터는 한 슬롯에만같은 길드원을 두 슬롯에 중복 편성할 수 없습니다.
빈 슬롯 허용반드시 모든 슬롯을 채울 필요 없습니다. 1명만 편성해도 됩니다.
미보유 캐릭터 편성 불가OwnedMemberIds에 존재하는 길드원만 배치할 수 있습니다.

프리셋 시스템

프리셋(앞서 설명한 편성 저장 슬롯)은 현재 스쿼드 편성 상태를 저장하고 나중에 불러오는 기능입니다. 초기 1개 프리셋을 사용할 수 있으며, 해금으로 최대 3개까지 확장됩니다.

데이터 구조는 TArray<FSquadPreset>이며, 각 프리셋은 TArray<FString>(슬롯별 길드원 ID)입니다. 빈 슬롯은 빈 문자열 ""로 표현합니다.

편성 변경 코드 골격

편성 변경 의사 코드 
// 슬롯에 길드원 배치
void SetSquadSlot(int32 SlotIndex, FString MemberId)
  1. SlotIndex 범위 확인 (0 ~ MaxSlots-1)
  2. MemberId가 OwnedMemberIds에 있는지 확인
  3. MemberId가 다른 슬롯에 이미 있으면 제거
  4. SquadSlots[SlotIndex] = MemberId
  5. 시너지 재계산 (RecalcSynergies)
  6. SaveGame 동기화

// 슬롯에서 길드원 제거
void ClearSquadSlot(int32 SlotIndex)
  1. SlotIndex 범위 확인
  2. SquadSlots[SlotIndex] = ""
  3. 시너지 재계산
  4. SaveGame 동기화
05 — 시너지 시스템

조합 보너스

시너지(앞서 설명한 조합 보너스)는 특정 길드원들이 모두 스쿼드에 편성되었을 때 활성화됩니다. 시너지 정의는 DataTable로 관리하며, 편성이 변경될 때마다 재계산합니다.

시너지 마스터 데이터

USTRUCT(BlueprintType)
struct FSynergyRow : public FTableRowBase
{
    GENERATED_BODY()

    // 시너지 고유 ID (예: "synergy_fire_duo")
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FString Id;

    // 시너지 이름 (UI 표시용)
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FString Name;

    // 활성화에 필요한 길드원 ID 목록
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    TArray<FString> RequiredMemberIds;

    // 보너스 설명 텍스트
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FString BonusDescription;
};

시너지 상태

상태조건UI 표시
Active모든 필수 멤버가 현재 스쿼드에 편성됨시너지 이름 + 보너스 설명을 밝게 표시
Inactive필수 멤버를 모두 보유하고 있지만, 일부가 스쿼드에 미편성시너지 이름을 반투명으로 표시 + "편성 필요" 힌트
Missing필수 멤버 중 미보유 길드원이 있음시너지 이름을 잠금 아이콘과 함께 표시

시너지 계산 흐름

시너지 재계산은 편성 변경 시마다(스쿼드 슬롯 배치/제거, 프리셋 로드 시) 실행됩니다.

1
현재 스쿼드 멤버 ID 목록 가져오기
SquadSlots 배열에서 빈 문자열을 제외한 유효 ID를 수집합니다.
2
모든 시너지 DataTable 순회
시너지 DataTable의 모든 행(FSynergyRow)을 하나씩 검사합니다.
3
RequiredMemberIds가 모두 스쿼드에 포함?
필수 멤버 ID가 전부 현재 스쿼드에 있으면 → Active.
4
일부 미편성이지만 보유?
필수 멤버를 모두 OwnedMemberIds에 갖고 있지만 스쿼드에 빠진 멤버가 있으면 → Inactive.
5
미보유 멤버 있음?
필수 멤버 중 보유하지 않은 길드원이 하나라도 있으면 → Missing.
💡

시너지는 순수 장식(보너스 표시)이므로 실제 능력치 계산은 필요 없습니다. UI에 "시너지 활성!" 텍스트와 보너스 설명을 표시하는 것이 전부입니다.

06 — 구현 스켈레톤

UGuildSubsystem 최소 코드

아래는 UGuildSubsystem의 최소 동작 뼈대입니다. 실제 프로젝트에서는 에러 핸들링·UI 연동·저장 동기화를 추가로 구현합니다.

#pragma once

#include "CoreMinimal.h"
#include "Subsystems/GameInstanceSubsystem.h"
#include "Engine/DataTable.h"
#include "GuildTypes.h"
#include "GuildSubsystem.generated.h"

/** 시너지 상태 열거형 */
UENUM(BlueprintType)
enum class ESynergyStatus : uint8
{
    Active,
    Inactive,
    Missing
};

/** 길드원 모집·편성·시너지를 관리하는 GameInstanceSubsystem */
UCLASS()
class RECORDTALES_API UGuildSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()

public:
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;

    /** 가챠: Gold 소비 후 3명 후보 반환 */
    UFUNCTION(BlueprintCallable, Category = "Guild")
    TArray<FString> HireCandidates();

    /** 후보 중 1명 선택 → 보유 목록에 추가 */
    UFUNCTION(BlueprintCallable, Category = "Guild")
    bool HireMember(const FString& MemberId);

    /** 스쿼드 슬롯에 길드원 배치 */
    UFUNCTION(BlueprintCallable, Category = "Guild")
    bool SetSquadSlot(int32 SlotIndex, const FString& MemberId);

    /** 스쿼드 슬롯에서 길드원 제거 */
    UFUNCTION(BlueprintCallable, Category = "Guild")
    void ClearSquadSlot(int32 SlotIndex);

    /** 특정 시너지의 현재 상태 조회 */
    UFUNCTION(BlueprintCallable, Category = "Guild")
    ESynergyStatus GetSynergyStatus(const FString& SynergyId) const;

    /** 프리셋 저장 */
    UFUNCTION(BlueprintCallable, Category = "Guild")
    void SavePreset(int32 PresetIndex);

    /** 프리셋 불러오기 */
    UFUNCTION(BlueprintCallable, Category = "Guild")
    void LoadPreset(int32 PresetIndex);

private:
    /** DataTable 레퍼런스 (길드원 마스터) */
    UPROPERTY()
    UDataTable* MemberTable = nullptr;

    /** DataTable 레퍼런스 (시너지 마스터) */
    UPROPERTY()
    UDataTable* SynergyTable = nullptr;

    /** 보유 길드원 ID 목록 */
    TArray<FString> OwnedMemberIds;

    /** 현재 스쿼드 슬롯 (인덱스 = 슬롯 번호) */
    TArray<FString> SquadSlots;

    /** 프리셋 저장 */
    TArray<TArray<FString>> Presets;

    /** 최대 슬롯 수 (해금에 따라 변동) */
    int32 MaxSlots = 2;

    /** 시너지 재계산 (편성 변경 시 호출) */
    void RecalcSynergies();
};
#include "GuildSubsystem.h"
#include "Kismet/GameplayStatics.h"

void UGuildSubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
    Super::Initialize(Collection);

    // DataTable 로드 (에셋 경로는 프로젝트에 맞게 수정)
    static ConstructorHelpers::FObjectFinder<UDataTable> MemberDT(
        TEXT("/Game/Data/DT_GuildMembers"));
    if (MemberDT.Succeeded()) MemberTable = MemberDT.Object;

    // 스쿼드 슬롯 초기화 (초기 2개)
    SquadSlots.SetNum(MaxSlots);
    for (auto& Slot : SquadSlots) Slot = TEXT("");

    // 프리셋 초기화 (초기 1개)
    Presets.SetNum(1);
}

TArray<FString> UGuildSubsystem::HireCandidates()
{
    TArray<FString> Candidates;

    // 1. Gold 잔액 확인
    auto* Economy = GetGameInstance()->GetSubsystem<UEconomySubsystem>();
    if (!Economy || Economy->GetGold() < 1000)
    {
        // Gold 부족 — 빈 배열 반환, UI에서 토스트 처리
        return Candidates;
    }

    // 2. Gold 차감
    Economy->AddGold(-1000);

    // 3. 미보유 멤버 풀 구성
    TArray<FGuildMemberRow*> Pool;
    if (MemberTable)
    {
        TArray<FGuildMemberRow*> AllRows;
        MemberTable->GetAllRows(TEXT("HireCandidates"), AllRows);
        for (auto* Row : AllRows)
        {
            if (!OwnedMemberIds.Contains(Row->Id))
                Pool.Add(Row);
        }
    }

    // 4. 가중치 기반 3명 선정 (간략화)
    const int32 Count = FMath::Min(3, Pool.Num());
    for (int32 i = 0; i < Count; ++i)
    {
        // TODO: 길드 레벨별 Rarity 가중치 적용
        int32 Idx = FMath::RandRange(0, Pool.Num() - 1);
        Candidates.Add(Pool[Idx]->Id);
        Pool.RemoveAt(Idx);
    }

    return Candidates;
}

bool UGuildSubsystem::SetSquadSlot(int32 SlotIndex, const FString& MemberId)
{
    if (SlotIndex < 0 || SlotIndex >= MaxSlots) return false;
    if (!OwnedMemberIds.Contains(MemberId)) return false;

    // 다른 슬롯에서 제거 (중복 방지)
    for (auto& Slot : SquadSlots)
    {
        if (Slot == MemberId) Slot = TEXT("");
    }

    SquadSlots[SlotIndex] = MemberId;
    RecalcSynergies();
    return true;
}

void UGuildSubsystem::ClearSquadSlot(int32 SlotIndex)
{
    if (SlotIndex < 0 || SlotIndex >= MaxSlots) return;
    SquadSlots[SlotIndex] = TEXT("");
    RecalcSynergies();
}

07 — 확장 & 주의

Pitfall & Do/Don’t

⚠️
가챠 확률 가중치 합이 100이 아니면 비정상 분포. 희귀도별 확률(Normal + Rare + Epic + Legendary)의 합은 반드시 100이어야 합니다. 에디터에서 가중치를 수정할 때는 합계를 검증하는 로직을 넣거나, DataTable Validate 콜백에서 경고를 출력하세요.
⚠️
보유 상한 도달 시 OwnedMemberIds.Num() 체크 필수. HireMember()와 HireCandidates() 모두에서 보유 수를 확인해야 합니다. 후보 생성 시 미보유 풀이 0이면 빈 배열을 반환하고, UI에서 가챠 버튼을 비활성화하세요.
⚠️
편성에서 제거한 캐릭터가 시너지에 영향 → 즉시 재계산. ClearSquadSlot()과 SetSquadSlot() 모두 마지막에 RecalcSynergies()를 호출해야 합니다. 프리셋 로드(LoadPreset()) 시에도 마찬가지입니다. 재계산을 빼먹으면 UI에 잘못된 시너지 상태가 표시됩니다.

✓ 이렇게

  • 마스터 데이터는 DataTable로 에디터에서 관리
  • 가챠 후보 생성 전 보유 상한·잔액 검증
  • 편성 변경 시 항상 시너지 재계산
  • SaveGame 동기화를 편성·모집 후 즉시 수행
  • 빈 슬롯은 빈 문자열로 명시적 표현

✗ 이건 피하기

  • 확률 가중치를 하드코딩 (DataTable이나 TMap으로)
  • 시너지 재계산 없이 슬롯 변경
  • 보유 상한 체크 없이 HireMember 호출
  • 프리셋 로드 후 슬롯 수 보정 누락
  • Gold 차감 전에 후보 생성 시작
