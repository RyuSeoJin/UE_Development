마스터-인스턴스 반복 규칙 상태 머신
01 — concept

개념 이해

퀘스트 시스템은 사용자가 해야 할 일(할 일, Task)을 생성·조회·수정·삭제(CRUD)하는 시스템입니다. 단발성 할 일("이번 주까지 리포트 제출")뿐 아니라, 반복 할 일("매일 운동", "매주 월요일 회의", "매월 1일 정산")도 지원합니다.

설계의 핵심은 마스터-인스턴스 구조입니다. 할 일의 원본 정보를 마스터 레코드라 부르고, 특정 날짜에 실제로 발생하는 할 일을 인스턴스라 부릅니다. 단발성 퀘스트는 마스터 1개 = 인스턴스 1개지만, 반복 퀘스트는 마스터 1개에서 날짜마다 인스턴스 N개가 생성됩니다.

핵심 용어

📖

마스터 레코드(Master Record) — 퀘스트의 원본 데이터. 제목, 설명, 카테고리, 반복 유형, 시작일, 마감일 등 모든 메타 정보를 가지고 있습니다. 데이터베이스(SaveGame)에 저장되는 단위입니다.

인스턴스(Instance / Occurrence) — 마스터에서 파생되어 "특정 날짜"에 존재하는 퀘스트 복제본. 반복 퀘스트의 경우, 한 마스터에서 여러 날짜의 인스턴스가 생성됩니다. 각 인스턴스는 독립적으로 완료/보류 상태를 가질 수 있습니다.

반복 규칙(Recurrence Rule) — 마스터에 설정된 반복 패턴. None(단발성), Daily(매일), Weekly(선택 요일), Monthly(선택 날짜) 4종류입니다. 반복 규칙에 따라 인스턴스가 자동 생성되는 날짜가 결정됩니다.

자정 롤오버(Midnight Rollover) — 매일 0시(자정)에 날짜가 바뀌면 새 날짜에 해당하는 반복 인스턴스를 준비하는 절차. 날짜 변경을 감지하고 ID 카운터를 초기화합니다.

D-Day — 마감일까지 남은 날짜를 표시하는 방식. D-3(3일 남음), D-DAY(오늘 마감), D+2(2일 초과) 형태로 보여줍니다.

💡

마스터는 할 일의 원본, 인스턴스는 특정 날짜에 발생하는 복제본. 반복 퀘스트는 마스터 1개에서 인스턴스 N개가 생성됩니다. 단발성 퀘스트는 마스터 자체가 곧 유일한 인스턴스입니다.

02 — data structure

데이터 구조

퀘스트 시스템의 데이터는 크게 3개의 구조체와 2개의 열거형으로 구성됩니다. 모든 구조체는 USTRUCT(BlueprintType)으로 선언하여 블루프린트에서도 접근할 수 있게 합니다.

FQuestMasterData — 마스터 레코드 구조체

USTRUCT(BlueprintType)
struct FQuestMasterData
{
    GENERATED_BODY()

    // 퀘스트 고유 ID: YYYYMMDD + 5자리 시퀀스 (예: 2026070200001)
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    FString Id;

    // 퀘스트 제목
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    FString Title;

    // 퀘스트 상세 설명
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    FString Description;

    // 카테고리 (사용자 정의 분류, 예: "공부", "운동")
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    FString Category;

    // 반복 유형: None(단발성), Daily, Weekly, Monthly
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    EQuestRecurrence Recurrence;

    // 시작일 (YYYY.MM.DD 형식)
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    FString StartDate;

    // 마감일 — 단발성 퀘스트의 절대 마감 날짜 (YYYY.MM.DD)
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    FString Deadline;

    // 마감 오프셋 — 반복 퀘스트 전용: 발생일로부터 N일 후가 마감
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    int32 DeadlineOffset;

    // 반복 종료일 (기본값: 시작일 + 1년). 이 날짜 이후엔 인스턴스 생성 안 함
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    FString RecEndDate;

    // 날짜별 인스턴스 상태 맵 (키: "YYYY.MM.DD", 값: 해당 날짜의 상태)
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    TMap<FString, FQuestOccurrenceData> RecurData;

    // 즐겨찾기 여부
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    bool bFavorite;
};

FQuestOccurrenceData, EQuestRecurrence, EQuestStatus

// 특정 날짜의 인스턴스 상태를 저장하는 구조체
USTRUCT(BlueprintType)
struct FQuestOccurrenceData
{
    GENERATED_BODY()

    // 완료 시각 (빈 문자열이면 미완료)
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    FString DoneAt;

    // 보류(일시정지) 시각 (빈 문자열이면 보류 아님)
    UPROPERTY(BlueprintReadWrite, Category = "Quest")
    FString SuspendedAt;
};

// 반복 유형 열거형
UENUM(BlueprintType)
enum class EQuestRecurrence : uint8
{
    None     UMETA(DisplayName = "단발성"),
    Daily    UMETA(DisplayName = "매일"),
    Weekly   UMETA(DisplayName = "매주"),
    Monthly  UMETA(DisplayName = "매월")
};

// 퀘스트 상태 열거형 (계산용 — DB에 저장하지 않음)
UENUM(BlueprintType)
enum class EQuestStatus : uint8
{
    Overdue    UMETA(DisplayName = "기한 초과"),
    Active     UMETA(DisplayName = "진행 중"),
    Upcoming   UMETA(DisplayName = "예정"),
    Done       UMETA(DisplayName = "완료"),
    Suspended  UMETA(DisplayName = "보류")
};

ID 생성 규칙

퀘스트 ID는 YYYYMMDD + 5자리 시퀀스로 구성됩니다. 같은 날 생성된 퀘스트는 시퀀스 번호가 1씩 증가합니다. 시퀀스 카운터는 TMap<FString, int32> IdCounters로 관리하며, 키는 "YYYYMMDD" 형식의 날짜 문자열입니다.

ID 생성 예시
// 2026년 7월 2일에 3번째로 생성된 퀘스트
날짜 부분: 20260702
시퀀스:    00003
최종 ID:   2026070200003

// IdCounters 맵
{ "20260702" : 3, "20260701" : 12 }
03 — recurrence engine

반복 규칙 엔진

반복 규칙 엔진은 마스터의 Recurrence 필드를 기반으로, "특정 날짜에 이 퀘스트의 인스턴스가 존재하는가"를 판정하는 로직입니다. 4가지 반복 유형마다 인스턴스 발생 조건이 다릅니다.

4가지 반복 유형 비교

유형발생 패턴마감일 계산추가 설정
None단발성 — 시작일에 1회만 존재Deadline 필드 (절대 날짜)없음
Daily매일 — 시작일~종료일 사이 모든 날발생일 + DeadlineOffset일없음
Weekly선택 요일 — 지정한 요일에만 발생발생일 + DeadlineOffset일요일 배열 (예: [Mon, Wed, Fri])
Monthly선택 날짜 — 매월 지정 날짜에 발생발생일 + DeadlineOffset일날짜 배열 (예: [1, 15])

마감일 계산

단발성 퀘스트의 마감일은 Deadline 필드에 저장된 절대 날짜를 그대로 사용합니다. 반복 퀘스트의 마감일은 각 인스턴스의 발생일에 DeadlineOffset(일 단위)을 더하여 계산합니다. 예를 들어 DeadlineOffset이 0이면 당일 마감, 1이면 다음 날 마감입니다.

반복 종료

반복 퀘스트는 영원히 반복하지 않습니다. RecEndDate 필드에 종료일을 지정하며, 기본값은 시작일로부터 1년 후입니다. 사용자가 원하면 더 짧거나 긴 기간으로 변경할 수 있습니다.

특정 날짜의 인스턴스 존재 판정 로직

1
Recurrence 확인
마스터의 Recurrence 값을 확인합니다. None이면 시작일과 대상 날짜가 일치하는지만 비교합니다.
2
날짜 범위 검증
대상 날짜가 StartDate 이상, RecEndDate 이하 범위 안에 있는지 확인합니다. 범위 밖이면 인스턴스가 존재하지 않습니다.
3
요일/날짜 매칭
Weekly이면 대상 날짜의 요일이 마스터의 요일 배열에 포함되는지 확인합니다. Monthly이면 대상 날짜의 일(day)이 날짜 배열에 포함되는지 확인합니다. Daily이면 이 단계를 건너뜁니다(매일 발생).
4
RecurData 조회
RecurData TMap에서 해당 날짜 키("YYYY.MM.DD")를 조회합니다. 키가 존재하면 저장된 상태(완료/보류)를 사용하고, 키가 없으면 기본 상태(미완료, 미보류)로 처리합니다.

단발성 vs 반복 퀘스트 비교

📄 단발성 (None)

마스터 1개 = 인스턴스 1개. 시작일에만 존재합니다. 마감일은 Deadline 필드의 절대 날짜.

삭제 시 마스터 자체를 제거합니다.

마스터 = 인스턴스 절대 마감일

🔄 반복 (Daily/Weekly/Monthly)

마스터 1개에서 날짜마다 인스턴스가 생성됩니다. 각 인스턴스의 마감일은 발생일 + DeadlineOffset.

삭제 시 "이후 모든 인스턴스 중단"(RecEndDate 변경) 또는 마스터 전체 삭제를 선택합니다.

마스터 1 : N 인스턴스 상대 마감일

반복 제거 (Recurring Remove)

사용자가 반복 퀘스트를 "삭제"하면, 단순히 마스터를 지우는 것이 아니라 선택한 날짜 이후의 인스턴스 생성을 중단합니다. 삭제 시점에 따라 3가지 시나리오가 있습니다.

시나리오조건동작
시작일 이전 삭제삭제 날짜 < StartDate아직 인스턴스가 하나도 발생하지 않았으므로 마스터 자체를 완전히 삭제합니다.
오늘 이전 또는 오늘 삭제삭제 날짜 ≤ 오늘RecEndDate를 삭제 날짜의 전날로 변경합니다. 이미 지난 인스턴스는 기록이 유지되고, 내일부터 새 인스턴스가 생기지 않습니다.
미래 날짜 삭제삭제 날짜 > 오늘해당 날짜를 RecEndDate로 설정합니다. 그 날까지의 인스턴스는 유지되고, 그 이후는 중단됩니다.
04 — state machine

상태 머신

퀘스트의 상태는 5가지(Overdue, Active, Upcoming, Done, Suspended)이며, 핵심 규칙은 상태를 DB에 저장하지 않는다는 것입니다. 상태는 현재 날짜와 인스턴스 데이터를 기반으로 매번 실시간으로 계산합니다.

상태 전환 다이어그램

INPUT
현재 날짜 + 인스턴스 데이터
오늘 날짜, 시작일, 마감일, DoneAt, SuspendedAt
→
CALCULATE
GetQuestStatus()
우선순위에 따라 Suspended → Done → Upcoming → Active → Overdue 순으로 판정
→
OUTPUT
EQuestStatus
UI에 표시할 상태 열거값 반환

상태 계산 로직

상태판정 조건우선순위
SuspendedSuspendedAt이 비어 있지 않음 (보류 시각이 기록됨)1 (최우선)
DoneDoneAt이 비어 있지 않음 (완료 시각이 기록됨)2
Upcoming시작일 > 오늘 (아직 시작되지 않음)3
Active시작일 ≤ 오늘 ≤ 마감일 (진행 기간 내)4
Overdue마감일 < 오늘 (마감일 지남)5
⚠️
상태는 DB에 저장하지 않습니다. 현재 날짜 기준으로 매번 실시간 계산합니다. 왜냐하면 날짜가 바뀌면 동일 퀘스트의 상태가 자동으로 변하기 때문입니다. 예를 들어, 어제 Active였던 퀘스트가 오늘 마감일이 지나면 자동으로 Overdue가 됩니다. 상태를 저장하면 매일 모든 퀘스트를 순회하여 갱신해야 하므로 불필요한 복잡성이 생깁니다.

D-Day 표시

D-Day는 (오늘 - 마감일)을 계산하여 사용자에게 직관적으로 남은 기간을 보여줍니다.

표시의미계산
D-3마감일까지 3일 남음오늘 < 마감일, 차이 = 3
D-DAY오늘이 마감일오늘 = 마감일
D+2마감일에서 2일 초과오늘 > 마감일, 차이 = 2

정렬 규칙

UI 목록에서 퀘스트를 표시할 때, 상태 그룹 순서로 먼저 정렬하고, 각 그룹 안에서 날짜 기준으로 재정렬합니다.

순서상태 그룹그룹 내 정렬
1Overdue (기한 초과)마감일 오름차순 (오래된 것이 위)
2Active (진행 중)마감일 오름차순 (급한 것이 위)
3Upcoming (예정)시작일 오름차순 (가까운 것이 위)
4Done (완료)완료 시각 내림차순 (최근 완료가 위)
5Suspended (보류)보류 시각 내림차순 (최근 보류가 위)
05 — midnight rollover

자정 롤오버

자정 롤오버(Midnight Rollover)는 매일 0시(자정)에 날짜가 바뀌면 새 날짜에 대한 ID 카운터를 초기화하고, 시스템이 새 날짜를 인식하도록 갱신하는 절차입니다. 반복 퀘스트의 인스턴스는 날짜 기반으로 계산되므로, 날짜 변경을 정확히 감지하는 것이 핵심입니다.

롤오버 흐름

1
날짜 변경 감지
Tick 함수 또는 FTimerHandle을 사용하여 주기적으로 현재 날짜를 확인합니다. 매 프레임 검사가 부담스러우면 1분~5분 간격의 타이머를 사용해도 됩니다.
2
LastRolloverDate 비교
시스템이 마지막으로 롤오버를 수행한 날짜(LastRolloverDate)와 현재 날짜를 비교합니다. 같으면 아무것도 하지 않습니다.
3
ID 카운터 초기화
날짜가 변경되었으면, 새 날짜에 대한 IdCounters 항목을 0으로 초기화합니다. 이전 날짜의 카운터는 그대로 유지합니다(이미 생성된 ID에 영향 없음).
4
LastRolloverDate 갱신
LastRolloverDate를 현재 날짜로 업데이트하고, SaveGame에 저장합니다. 이후 동일 날짜에 중복 롤오버가 발생하지 않습니다.

백그라운드 복귀

앱이 비활성 상태(백그라운드, 절전 모드)에서 다시 활성화될 때도 날짜 재확인이 필요합니다. 예를 들어 밤 11시에 앱을 최소화했다가 새벽 1시에 다시 열면, Tick이 멈춰 있던 동안 자정이 지나갔을 수 있습니다. FCoreDelegates::ApplicationHasEnteredForegroundDelegate에 바인딩하여 복귀 시 즉시 날짜를 재확인합니다.

⚠️

자정 롤오버를 놓치면 반복 인스턴스의 ID 카운터가 꼬이거나, 상태 계산이 이전 날짜 기준으로 돌아갈 수 있습니다. FocusSubsystem의 Tick에서도 날짜 체크를 추가하여 이중 안전망을 두세요. 또한 앱 복귀 시점에도 반드시 날짜를 재확인합니다.

06 — implementation skeleton

구현 스켈레톤

UQuestSubsystem은 UGameInstanceSubsystem(GameInstance에 붙는 서브시스템)으로 구현합니다. GameInstance의 수명(앱 시작~종료)과 동일하게 유지되므로, 레벨 전환에도 퀘스트 데이터가 사라지지 않습니다. 

#pragma once

#include "CoreMinimal.h"
#include "Subsystems/GameInstanceSubsystem.h"
#include "QuestTypes.h"
#include "QuestSubsystem.generated.h"

UCLASS()
class RECORDTALES_API UQuestSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()

public:
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;
    virtual void Deinitialize() override;

    // ── CRUD ──

    /** 새 퀘스트를 생성하고 ID를 반환한다. */
    UFUNCTION(BlueprintCallable, Category = "Quest")
    FString CreateQuest(const FQuestMasterData& InData);

    /** 기존 퀘스트를 수정한다. */
    UFUNCTION(BlueprintCallable, Category = "Quest")
    void UpdateQuest(const FString& QuestId, const FQuestMasterData& InData);

    /** 퀘스트를 삭제한다 (반복은 RecEndDate 변경). */
    UFUNCTION(BlueprintCallable, Category = "Quest")
    void DeleteQuest(const FString& QuestId, const FString& FromDate);

    // ── 조회 ──

    /** 특정 날짜에 존재하는 인스턴스 목록을 반환한다. */
    UFUNCTION(BlueprintCallable, Category = "Quest")
    TArray<FQuestMasterData> GetInstancesForDate(const FString& Date) const;

    /** 인스턴스의 현재 상태를 실시간 계산하여 반환한다. */
    UFUNCTION(BlueprintCallable, Category = "Quest")
    EQuestStatus GetQuestStatus(const FQuestMasterData& Master,
                                 const FString& Date) const;

    // ── 자정 롤오버 ──

    /** 날짜 변경을 감지하고 ID 카운터를 초기화한다. */
    void CheckMidnightRollover();

private:
    // 전체 마스터 레코드 배열
    UPROPERTY()
    TArray<FQuestMasterData> Quests;

    // 날짜별 ID 시퀀스 카운터 (키: "YYYYMMDD")
    TMap<FString, int32> IdCounters;

    // 마지막 롤오버 수행 날짜
    FString LastRolloverDate;

    // ID 생성 헬퍼
    FString GenerateId();

    // 주어진 날짜에 인스턴스가 존재하는지 판정
    bool HasInstanceOnDate(const FQuestMasterData& Master,
                          const FString& Date) const;
};

EQuestStatus UQuestSubsystem::GetQuestStatus(
    const FQuestMasterData& Master,
    const FString& Date) const
{
    // 1. RecurData에서 해당 날짜의 상태 조회
    const FQuestOccurrenceData* Occ = Master.RecurData.Find(Date);

    // 2. 우선순위 1: Suspended
    if (Occ && !Occ->SuspendedAt.IsEmpty())
    {
        return EQuestStatus::Suspended;
    }

    // 3. 우선순위 2: Done
    if (Occ && !Occ->DoneAt.IsEmpty())
    {
        return EQuestStatus::Done;
    }

    // 4. 현재 날짜 가져오기
    FString Today = FDateTime::Now().ToString(TEXT("%Y.%m.%d"));

    // 5. 마감일 계산
    FString EffectiveDeadline;
    if (Master.Recurrence == EQuestRecurrence::None)
    {
        EffectiveDeadline = Master.Deadline;
    }
    else
    {
        // 반복: 발생일(Date) + DeadlineOffset일
        // TODO: FString → FDateTime 변환 후 AddDays
        EffectiveDeadline = Date; // 임시: 오프셋 0일 경우
    }

    // 6. 우선순위 3: Upcoming (시작일 > 오늘)
    if (Master.StartDate > Today)
    {
        return EQuestStatus::Upcoming;
    }

    // 7. 우선순위 4: Active (시작일 ≤ 오늘 ≤ 마감일)
    if (Master.StartDate <= Today && Today <= EffectiveDeadline)
    {
        return EQuestStatus::Active;
    }

    // 8. 우선순위 5: Overdue (마감일 < 오늘)
    return EQuestStatus::Overdue;
}

void UQuestSubsystem::CheckMidnightRollover()
{
    FString Today = FDateTime::Now().ToString(TEXT("%Y%m%d"));

    if (Today != LastRolloverDate)
    {
        // 새 날짜에 대한 ID 카운터 초기화
        IdCounters.Add(Today, 0);
        LastRolloverDate = Today;

        // TODO: SaveGame에 LastRolloverDate 저장
        UE_LOG(LogTemp, Log,
            TEXT("QuestSubsystem: 자정 롤오버 완료 — %s"), *Today);
    }
}

FString UQuestSubsystem::GenerateId()
{
    FString Today = FDateTime::Now().ToString(TEXT("%Y%m%d"));
    int32& Counter = IdCounters.FindOrAdd(Today);
    Counter++;

    // YYYYMMDD + 5자리 시퀀스 (예: 2026070200001)
    return FString::Printf(TEXT("%s%05d"), *Today, Counter);
}

07 — extension & pitfalls

확장 & 주의

⚠️
RecurData TMap 키는 반드시 "YYYY.MM.DD" 형식으로 통일하세요. "2026.7.2"와 "2026.07.02"가 다른 키로 인식되어 인스턴스 상태가 누락됩니다. 날짜 문자열을 생성하는 모든 위치에서 FDateTime::ToString(TEXT("%Y.%m.%d"))를 사용하여 형식을 보장합니다.
⚠️
반복 종료일을 과거로 설정하면 기존 인스턴스도 비활성화됩니다. RecEndDate를 오늘보다 이전 날짜로 바꾸면, 오늘자 인스턴스도 HasInstanceOnDate()에서 범위 밖으로 판정되어 사라집니다. 사용자에게 "이미 완료/보류한 인스턴스 기록은 RecurData에 남아 있지만, 목록에서는 보이지 않게 됩니다"라고 안내하세요.
⚠️
즐겨찾기 "복제"는 새 마스터를 생성합니다(참조가 아닌 값 복사). 즐겨찾기를 통해 퀘스트를 복제하면 원본과 완전히 독립된 새 마스터가 만들어집니다. 원본을 수정해도 복제본에는 영향이 없고, 복제본을 삭제해도 원본은 유지됩니다. 이는 의도된 동작이며, 참조 방식이 아닙니다.

DO

  • 날짜 문자열은 항상 YYYY.MM.DD 형식 함수를 통해 생성
  • 상태 계산은 GetQuestStatus() 한 곳에 집중
  • 반복 퀘스트 삭제 시 RecEndDate 변경으로 "소프트 삭제"
  • 자정 롤오버를 Tick + 앱 복귀 두 곳에서 이중 체크
  • ID 카운터를 SaveGame에 영속하여 재시작 시 충돌 방지

DON'T

  • 상태(EQuestStatus)를 DB에 저장하지 말 것 — 날짜 변경 시 불일치 발생
  • 날짜 키를 직접 문자열 조합하지 말 것 — 형식 불일치 버그 원인
  • 반복 퀘스트를 하드 삭제하지 말 것 — 과거 기록이 사라짐
  • 자정 롤오버 없이 ID를 생성하지 말 것 — 날짜 넘어가면 카운터 충돌
  • 즐겨찾기 복제를 참조로 구현하지 말 것 — 원본 변경 시 부작용

퀘스트 시스템의 데이터 (실측 기준)
