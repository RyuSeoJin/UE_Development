데이터 설계 명세
DataTable · SaveGame · Settings

웹 프로토타입(Grimhollow — Guild Master v14)에 흩어져 있는 모든 데이터를 전수 조사하여, 언리얼 엔진의 데이터 자산(DataTable · Settings · SaveGame)으로 번역하는 단일 기준 문서입니다. 모든 수치는 프로토타입 코드에서 실측한 값입니다.

01 — 요구사항 · 데이터 인벤토리

프로토타입 데이터 전수 조사

프로토타입은 단일 HTML 파일 안의 자바스크립트 상수와 전역 상태 객체(A)에 모든 데이터를 담고 있습니다. 언리얼로 옮길 때는 이 데이터를 성격에 따라 3계층으로 나눕니다. 이 분류가 이 문서 전체의 뼈대입니다.

정적
기획 데이터 — 에디터에서 편집
동적
플레이어 상태 — SaveGame 저장
휘발
세션 상태 — 저장하지 않음
1곳
데이터당 보관처는 하나

핵심 용어

📚

데이터 3계층 (정적/동적/휘발) — 데이터를 "누가, 언제 바꾸는가"로 나누는 기준. 정적은 기획자가 에디터에서 밸런싱하는 값(캐릭터 스탯, 해금 비용), 동적은 플레이어마다 달라져 저장해야 하는 값(보유 골드, 완료한 퀘스트), 휘발은 세션 중에만 의미 있어 저장하면 오히려 버그가 되는 값(남은 타이머 초, 열려 있는 패널)입니다.
DataTable (데이터테이블) — 언리얼에서 표 형태의 정적 데이터를 담는 에셋. 각 행은 FTableRowBase를 상속한 구조체이며, CSV 파일로 임포트/재임포트할 수 있어 스프레드시트로 밸런싱할 수 있습니다.
FTableRowBase — DataTable의 행이 되기 위해 반드시 상속해야 하는 기본 구조체. 이걸 상속해야 에디터의 Data Table 생성 목록에 나타납니다.
RowName (행 이름) — DataTable에서 각 행을 식별하는 고유 이름(FName 타입). 다른 데이터가 이 행을 가리킬 때는 데이터를 복사하지 않고 RowName만 저장합니다(참조 방식).
USaveGame (세이브게임) — 플레이어 상태를 디스크에 저장/로드하는 언리얼 기본 클래스. UPROPERTY()가 붙은 멤버만 직렬화(파일로 변환)됩니다.
UDeveloperSettings (개발자 설정) — Project Settings 화면에 자동 노출되는 설정 클래스. 테이블로 만들기엔 낱개인 전역 상수(가챠 비용, 골드 틱 주기 등)의 홈입니다.
TSoftObjectPtr (소프트 참조) — 에셋을 즉시 메모리에 올리지 않고 경로만 들고 있다가 필요할 때 로드하는 참조. 텍스처·사운드처럼 큰 에셋을 테이블에 연결할 때 사용합니다.

전수 인벤토리 트리

프로토타입의 심볼 이름을 그대로 병기했습니다. 아래 트리에 없는 데이터는 이 프로젝트에 존재하지 않습니다.

정적 — 기획 데이터 (에디터 편집 대상) ├─ POOL — 고정 캐릭터 12명 (등급별 3명 × 4등급: 이름·종족·직업·스탯·스토리·관계) ├─ SYNERGIES — 시너지 7종 (페어 5 + 트리오 2) ├─ UNLOCKS_V2 — 기능 해금 17종 (프리셋 2 · 편성 슬롯 2 · 보상 강화 5×3계열) ├─ CONTENT_UNLOCKS — 콘텐츠 해금 12종 (곡 4 + 배경 8) ├─ levelUpXp / levelUpGoldCost — 레벨업 공식 (100×1.4^(Lv-1) / 500×Lv) ├─ GUILD_TITLES_V2 — 길드 칭호 (Lv1~20 구간별 10종) ├─ RARITY_BY_LEVEL_V2 — 레벨 구간별 가챠 확률 7브래킷 ├─ TRACKS / TRACK_GENRE_GROUPS — BGM 6곡 · 장르 그룹 5종 ├─ SOUNDS — 앰비언트 사운드 5종 (Rain·Wind·Fire·Forest·Ocean) ├─ STAT_POOL — 능력치 종류 9종 (모두 % 단위) ├─ categories 초기값 — 기본 카테고리 6종 (공부·프로젝트·업무·독서·운동·기타) └─ 낱개 상수 — HIRE_COST(1,000) · 골드 틱(6초당 +10) · questGoldDailyCap(2,000) · guildMemberCap(8) · MAX_LEVEL(99) · MAX_DESIGN_LEVEL(20) · 타이머 기본값(25/5/15/5분 · 반복 4회) · XP 보상값 동적 — 플레이어 상태 (SaveGame 저장 대상) ├─ A.records[] — 집중 기록 (카테고리·시간·모드·메모·시각) ├─ A.todos[] + idCounters + lastRolloverDate — 퀘스트 마스터 + 일일 ID 카운터 + 롤오버 기준일 ├─ A.guildMembers[] / squadPresets / activePreset / hireHistory — 보유 길드원·편성·모집 이력 ├─ A.gold / guildLevel / guildXp / guildTotalXp / unlockedIds / questGoldToday — 경제·진행도 ├─ A.focusStreakCount / currentBgId — 연속 집중 카운트 · 적용 중 배경 ├─ A.focusDuration ~ autoRestart — 타이머 사용자 설정 └─ A.categories / catColors — 사용자 편집 카테고리 목록·색상 휘발 — 세션 상태 (저장 금지) ├─ 타이머 진행: timerState · remainingTime · cycleIndex · accFocusTime · goldTickCounter ├─ 스톱워치 진행: swElapsed · swRunning · swPaused ├─ 가챠 진행: hireState · hireCandidates(3장) · hirePreselectedIdx ├─ 재생 상태: MP(곡 인덱스·재생 위치) · PREVIEW(미리듣기) └─ UI 상태: currentPanel · questsView · statusFilter · 캘린더 연/월 · 필터류 전부
💡

왜 "인스턴스 캐시(A.instances)"가 동적 목록에 없나요? 프로토타입의 반복 퀘스트 인스턴스는 마스터 레코드에서 날짜를 넣으면 언제든 다시 계산되는 파생 데이터입니다(마스터의 recurData 맵에 날짜별 완료 상태만 저장). 파생 데이터는 저장하지 않고 필요할 때 재계산하는 것이 원칙입니다 — 저장하면 원본(마스터)과 어긋날 위험만 생깁니다.

02 — 설계 결정 · 데이터 보관처

데이터 보관처 결정표

각 데이터 그룹을 어디에, 왜 두는지가 이 문서의 가장 중요한 결정입니다. 판단 기준은 세 가지 질문입니다.

1
기획자가 밸런싱하며 바꾸는 값인가?
그렇다면 코드가 아니라 에디터에서 편집돼야 합니다 → 행이 여러 개면 DataTable, 낱개 상수면 UDeveloperSettings.
2
플레이어마다 값이 다른가?
그렇다면 디스크에 남아야 합니다 → USaveGame. 정적 데이터를 가리킬 때는 복사하지 말고 RowName(행 이름)만 저장합니다.
3
앱을 껐다 켜면 사라져도 되는가?
그렇다면 저장하지 않습니다 → 각 Subsystem의 멤버 변수(휘발). 실수로 저장하면 "재시작했는데 타이머가 돌고 있는" 류의 버그가 됩니다.
데이터 그룹 (프로토타입 심볼)보관처근거
캐릭터 12명 (POOL)DT_GuildMembers (DataTable)등급·스탯·스토리는 밸런싱 대상. 가챠는 테이블 행 전체를 풀로 순회하므로 행 단위 관리가 자연스러움
시너지 7종 (SYNERGIES)DT_Synergies (DataTable)멤버 조합·보너스 수치가 밸런싱 대상. 멤버는 DT_GuildMembers의 RowName 배열로 참조
기능 해금 17종 (UNLOCKS_V2)DT_Unlocks (DataTable)비용·선행 조건 체인이 표 구조 그 자체. Requires 열이 같은 테이블의 RowName을 가리킴
콘텐츠 해금 (CONTENT_UNLOCKS)DT_MusicTracks · DT_Backgrounds에 통합정규화 결정: 프로토타입은 콘텐츠(TRACKS)와 해금 정보(CONTENT_UNLOCKS)를 별도 배열 + 인덱스(trackIdx)로 연결하지만, 둘은 1:1 관계이므로 UE에서는 콘텐츠 테이블에 Cost(0 = 기본 제공) 열로 흡수. 인덱스 연결은 행 삽입 시 깨지기 쉬움
레벨 진행 (공식·칭호·가챠 확률)DT_GuildLevels (DataTable, Lv1~20 행)정규화 결정: 공식(1.4^n)·칭호 배열·확률 브래킷 3곳에 흩어진 레벨 데이터를 레벨당 1행으로 통합. 기획자가 "Lv7에서 뭐가 열리지?"를 한 행에서 봄. Lv21+는 공식 폴백(§04-5)
BGM 6곡 (TRACKS)DT_MusicTracks (DataTable)곡별 메타(제목·장르·길이) + TSoftObjectPtr<USoundBase>로 실제 오디오 에셋 연결
앰비언트 5종 (SOUNDS)DT_AmbientSounds (DataTable)루프 사운드 에셋 연결 + 이름·아이콘. 볼륨은 동적(플레이어 설정)
배경 8종 (CONTENT_UNLOCKS의 bg)DT_Backgrounds (DataTable)프로토타입의 CSS 그라데이션은 UE에서 텍스처/머티리얼 에셋으로 대체 → 소프트 참조 열
낱개 상수 (가챠 비용·골드 틱·상한류·XP 보상·타이머 기본값·기본 카테고리)URecordTalesSettings (UDeveloperSettings)행이 하나뿐인 값을 테이블로 만들면 과설계. Project Settings ▸ Game에 자동 노출되어 재컴파일 없이 조정 가능(Config=Game)
플레이어 상태 전부 (기록·퀘스트·길드원·경제·설정)URecordTalesSaveGame (USaveGame)플레이어마다 다르고 재시작 후에도 남아야 함. 구성은 §05
세션·UI 상태각 Subsystem 멤버 변수 (저장 안 함)타이머 진행·가챠 카드·열린 패널은 세션 안에서만 유효. UPROPERTY(Transient)로 직렬화 제외 명시

프로토타입 실측값 vs 정의만 있는 값

⚠️
프로토타입에는 "선언만 되고 배선되지 않은" 상수가 있습니다. 코드를 그대로 믿고 옮기면 실제 동작과 다른 밸런스가 들어갑니다. 이 문서의 모든 테이블은 실제 호출부에서 실측한 값을 기본으로 하고, 미배선 값은 아래처럼 따로 표시합니다.
· XP_REWARDS 상수(스톱워치 분당 10 · 타이머 20 · 퀘스트 50) — 정의만 존재. 실제 지급 코드는 타이머 사이클 완료 +15, 3연속 보너스 +20, 스톱워치(25분 이상) +15, 퀘스트 마스터 완료 +10, 반복 인스턴스 완료 +5.
· getRewardMultiplier()(해금 티어당 +20% 보상 배율) — 함수는 있으나 지급 코드에 곱해지지 않음. UE 구현에서는 배선하는 것을 전제로 Settings에 필드를 마련합니다.
· 퀘스트 골드 보상 — questGoldDailyCap(일 2,000)은 있으나 지급 로직 자체가 미구현. 상한 필드만 예약하고 지급량은 기획 확정 필요.
03 — 공통 타입 · UENUM

열거형과 공용 구조체

프로토타입은 상태를 문자열('timer', 'daily'…)로 구분합니다. 문자열은 오타가 곧 버그이므로, UE에서는 전부 UENUM(열거형: 허용된 값만 담을 수 있는 타입)으로 바꿉니다. 이름은 기존 01~07 문서에서 이미 쓰던 것을 그대로 따릅니다(한 개념 한 용어).

UENUM값프로토타입 문자열주 사용처
ERecordSourceTimer, Stopwatch, Quest, Manual'timer' / 'sw' (2종을 4종 출처로 확장)기록 출처 구분 — 아카이브 필터 축 ① (구 EFocusMode 대체, v1.1)
EFocusStateIdle, Focusing, Resting, LastResting, PausedtimerState타이머 상태 머신 (휘발)
EQuestRecurrenceNone, Daily, Weekly, Monthlyrecurrence반복 규칙
EQuestStatusUpcoming, Active, Overdue, Done, Suspended파생값 (저장 안 함)날짜·완료 여부에서 계산되는 상태 — 저장하면 자정이 지날 때 어긋남. 임의 날짜 D 기준으로 일반화(GetStatusForDate(D)) — 캘린더 날짜별 표기 구간 규칙은 03 문서 "퀘스트 상태 타임라인 표기 규칙" 참조
EGuildRarityNormal, Rare, Epic, Legendary'normal'~'legendary'캐릭터 등급
EMemberStatFocusTime, GoldGain, QuestReward, StopwatchReward, AccuracyBonus, XpBonus, StreakBonus, AllReward, XpBoostSTAT_POOL 9종캐릭터 스탯·시너지 보너스 종류
EUnlockCategoryPreset, Squad, Stopwatch, Timer, QuestUNLOCKS_V2.cat기능 해금 분류
EPeriodFilterDaily, Weekly, MonthlyarchPeriodFilter아카이브 조회 (휘발)
EPanelFocus, Quests, ArchivecurrentPanelUI 패널 (휘발)
// 공통 타입 — 모든 서브시스템/테이블이 공유. 기존 문서(01~07)의 타입명과 동일.
#pragma once
#include "CoreMinimal.h"
#include "RecordTalesTypes.generated.h"

// UENUM(BlueprintType): 블루프린트 변수·핀으로 노출. enum은 uint8 기반이라야 BlueprintType 가능
UENUM(BlueprintType)
enum class ERecordSource : uint8 { Timer, Stopwatch, Quest, Manual };  // 구 EFocusMode 대체 (v1.1)

UENUM(BlueprintType)
enum class EQuestRecurrence : uint8 { None, Daily, Weekly, Monthly };

UENUM(BlueprintType)
enum class EGuildRarity : uint8 { Normal, Rare, Epic, Legendary };

// 능력치 종류 9종 — 프로토타입 STAT_POOL 순서 그대로. 값은 전부 % 단위.
UENUM(BlueprintType)
enum class EMemberStat : uint8
{
    FocusTime,        // 집중 시간
    GoldGain,         // 골드 획득
    QuestReward,      // 퀘스트 보상
    StopwatchReward,  // 스톱워치 보상
    AccuracyBonus,    // 명중 보너스
    XpBonus,          // 경험치 보너스
    StreakBonus,      // 연속 보상
    AllReward,        // 모든 보상
    XpBoost           // 경험치 부스트
};

UENUM(BlueprintType)
enum class EUnlockCategory : uint8 { Preset, Squad, Stopwatch, Timer, Quest };

// 스탯 1개 = (종류, 수치%). 캐릭터 스탯·시너지 보너스가 공용으로 사용.
// USTRUCT(BlueprintType): 이 구조체를 블루프린트에서도 변수로 사용 가능하게 노출
USTRUCT(BlueprintType)
struct FStatBonus
{
    GENERATED_BODY()   // 리플렉션 코드 삽입점 — USTRUCT/UCLASS 본문 맨 위에 필수 (없으면 컴파일 에러)

    // EditAnywhere: 에디터 디테일 패널에서 편집 허용 / BlueprintReadOnly: BP에서 읽기만(쓰기 금지)
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    EMemberStat Stat = EMemberStat::FocusTime;   // 초기값 지정 — 미지정 시 enum 첫 값이 아닌 쓰레기값 위험

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    float Value = 0.f;   // % 단위 (4 = +4%)
};
💡

EQuestStatus만 저장하지 않는 이유. Upcoming(시작 전)/Active(진행 중)/Overdue(기한 초과)는 "오늘 날짜"와 비교해야 정해지는 값입니다. 어제 저장한 Active가 오늘은 Overdue일 수 있으므로, 저장은 원인(시작일·마감일·완료 시각)만 하고 상태는 조회 시점에 계산합니다.

04-4 · DT_GuildLevels — 레벨 진행 (Lv1~20)

필요 XP는 floor(100 × 1.4^(Lv-1)), 레벨업 골드는 500 × Lv의 계산 결과를 행으로 굳힌 것입니다. 가챠 확률은 프로토타입의 7개 브래킷(1·3·5·8·12·16·20레벨 이상)을 레벨별로 전개했습니다.

struct FGuildLevelRow : public FTableRowBase

    GENERATED_BODY()

    UPROPERTY(EditAnywhere, BlueprintReadOnly)

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    int32 RequiredGold = 0;

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FText Title;                // 이 레벨의 칭호

    // 가챠 확률 (합계 100). 순서: Normal / Rare / Epic / Legendary
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    int32 WeightNormal = 0;
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    int32 WeightRare = 0;
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    int32 WeightEpic = 0;
    UPROPERTY(EditAnywhere, BlueprintReadOnly)

};

05-1 · FArchiveRecord — 통합 아카이브 기록 (구 FFocusRecord)

프로토타입은 한 기록에 날짜를 4가지 형태(date, dateLong, clock, ts)로 중복 저장합니다. UE에서는 FDateTime(날짜+시각을 담는 언리얼 표준 타입) 하나만(CreatedAt) 저장하고 나머지는 표시 시점에 파생시킵니다 — 같은 사실을 두 곳에 저장하면 언젠가 서로 어긋납니다. v1.1에서 통합 기록 ID 설계에 따라 모든 출처(타이머·스톱워치·퀘스트 완료·직접 작성)의 기록이 이 구조 하나로 통합되었습니다. 필드별 근거·ID 발급 규칙·마이그레이션의 단일 소스는 11 통합 기록 ID 문서입니다.

// 아카이브 "사건" 1건의 통합 기록. 프로토타입 A.records[] 의 한 항목을 확장.
USTRUCT(BlueprintType)
struct FArchiveRecord
{
    GENERATED_BODY()

    // 지정자 없는 UPROPERTY(): 에디터엔 안 보여도 SaveGame 직렬화 대상이 되게 표시 (붙지 않으면 저장이 안 됨)
    UPROPERTY()
    FString Id;                     // 통합 기록 ID "YYYYMMDD+5자리" — 정렬·식별 전용, 불변

    UPROPERTY()
    ERecordSource Source = ERecordSource::Timer;   // 출처 — 필터 축 ①

    UPROPERTY()
    FString Category;               // 사용자 카테고리 (예: 공부) — 필터 축 ②

    UPROPERTY()
    FDateTime CreatedAt;            // 사건 발생 시각 — 필터 축 ③, date/clock 등은 여기서 파생

    UPROPERTY()
    int32 DurationSeconds = 0;     // 집중한 시간(초) — Timer/Stopwatch만 의미

    UPROPERTY()
    FString Title;                  // 퀘스트 제목 스냅샷 / 직접 작성 제목

    UPROPERTY()
    FString Notes;                  // 세션 메모 (빈 문자열 허용)

    UPROPERTY()
    FString SourceRefId;            // Source==Quest: 원본 퀘스트 마스터 ID

    UPROPERTY()
    FDateTime DueDate;              // Source==Quest: 목표 완료 일자 — 가변이라 ID 밖
};

05-2 · FQuestMasterData — 퀘스트 마스터 + ID 규칙

퀘스트는 마스터(원본) + 날짜별 인스턴스 상태 구조입니다. 마스터에 반복 규칙이 있고, 반복 퀘스트의 날짜별 완료·중단 상태는 마스터 안의 맵(RecurData)에만 기록합니다.

// 반복 인스턴스의 날짜별 상태. 키는 날짜 문자열 "YYYY.MM.DD".
USTRUCT(BlueprintType)
struct FQuestInstanceData
{
    GENERATED_BODY()

    UPROPERTY() bool bDone = false;
    UPROPERTY() FDateTime DoneAt;
    UPROPERTY() bool bSuspended = false;
    UPROPERTY() FDateTime SuspendedAt;   // 중단 시각 — 캘린더 "중단일까지 표기" 규칙의 기준 (03 타임라인, v1.2 신규)
    UPROPERTY() FString SuspendReason;
};

// 퀘스트 마스터(원본). 프로토타입 A.todos[] 의 한 항목.
USTRUCT(BlueprintType)
struct FQuestMasterData
{
    GENERATED_BODY()

    UPROPERTY()
    FString Id;                     // "YYYYMMDD" + 5자리 시퀀스 (아래 규칙)

    UPROPERTY() FString Title;
    UPROPERTY() FString Description;
    UPROPERTY() FString Category;

    UPROPERTY()
    EQuestRecurrence Recurrence = EQuestRecurrence::None;

    UPROPERTY() FDateTime StartDate;            // 필수

    UPROPERTY() bool bHasDeadline = false;
    UPROPERTY() FDateTime Deadline;
    UPROPERTY() int32 DeadlineOffset = 1;      // 반복 퀘스트: 발생일 + N일 마감

    UPROPERTY() bool bHasRecurrenceEnd = false;
    UPROPERTY() FDateTime RecurrenceEnd;        // 미설정 시 시작일 + 1년

    // 단발 퀘스트의 완료/중단 (반복은 RecurData가 담당)
    UPROPERTY() FDateTime DoneAt;
    UPROPERTY() FDateTime SuspendedAt;
    UPROPERTY() FString SuspendReason;

    // 날짜키("YYYY.MM.DD") → 그 날짜 인스턴스의 상태
    UPROPERTY()
    TMap<FString, FQuestInstanceData> RecurData;

    UPROPERTY() bool bIsFavorite = false;
};
규칙내용 (프로토타입 실측)
ID 형식YYYYMMDD + 5자리 시퀀스 (예: 2026070200001). 날짜별 카운터(IdCounters)가 1부터 증가
발생 판정None: 시작일 당일만 · Daily: 매일 · Weekly: 시작일과 같은 요일 · Monthly: 시작일과 같은 일(日)
반복 종료종료일 미설정 시 시작일 + 1년까지만 인스턴스 생성
자정 롤오버날짜가 바뀌면(LastRolloverDate < 오늘) 지난 날짜부터 순차 처리. 앱을 며칠 껐다 켜도 밀린 날짜를 모두 따라잡음
상태 파생Upcoming/Active/Overdue/Done/Suspended는 저장하지 않고 시작일·마감·완료 시각과 오늘 날짜로 계산 (§03)

05-3 · 길드 상태 — 보유·편성·모집 이력

// 보유 길드원 1명. 스탯은 저장하지 않는다 — DT_GuildMembers를 RowName으로 참조.
USTRUCT(BlueprintType)
struct FOwnedGuildMember
{
    GENERATED_BODY()

    UPROPERTY()
    FGuid InstanceId;               // 보유 단위 식별자 (프로토타입 Date.now() 대체)

    UPROPERTY()
    FName MemberRow;                // DT_GuildMembers RowName (예: "Cael")

    UPROPERTY()
    int32 Level = 1;               // 성장 여지용 — 프로토타입은 항상 1

    UPROPERTY()
    FDateTime HiredAt;
};

// 가챠 이력 1건 — 뽑고 선택했는지 / 패스했는지.
USTRUCT(BlueprintType)
struct FHireHistoryEntry
{
    GENERATED_BODY()

    UPROPERTY() FName MemberRow;
    UPROPERTY() bool bSelected = false;   // false = 패스
    UPROPERTY() FDateTime At;
};

// 스쿼드 프리셋 1개 — 편성된 길드원의 InstanceId 목록 (슬롯 순서 유지).
USTRUCT(BlueprintType)
struct FSquadPreset
{
    GENERATED_BODY()

    UPROPERTY()
    TArray<FGuid> MemberInstanceIds;
};
⚠️
중복 보유 정책은 기획 확정이 필요합니다. 프로토타입 가챠는 같은 캐릭터가 후보로 중복 등장할 수 있고, 이미 보유한 캐릭터를 또 뽑는 것을 막는 코드가 없습니다. 그런데 시너지·편성 로직은 이름 기반이라 중복 보유는 아무 이득이 없습니다. 권장: 뽑기 확정 시 이미 보유한 캐릭터면 거부(또는 골드 환급) — 어느 쪽이든 결정을 데이터 규칙으로 명문화하고 UGuildSubsystem에서 강제하세요.

05-4 · 통합 세이브 클래스

// 슬롯 이름 "RecordTalesSave", UserIndex 0. 변경 즉시 저장이 원칙.
// USaveGame 상속: 이 객체를 통째로 디스크에 저장/로드할 수 있게 함(SaveGameToSlot/LoadGameFromSlot 대상).
// 여기 담긴 UPROPERTY() 필드만 직렬화된다 — 지정자 없는 일반 멤버는 저장에서 누락됨.
UCLASS()
class URecordTalesSaveGame : public USaveGame
{
    GENERATED_BODY()
public:
    // ── 마이그레이션 대비 ──
    UPROPERTY() int32 SaveVersion = 2;   // 구조 변경 시 +1, 로드 시 분기 (v2: 통합 기록 ID)

    // ── 기록 (03 아카이브 · 11 통합 기록 ID) ──
    UPROPERTY() TArray<FArchiveRecord> ArchiveRecords;     // 항상 Id 오름차순(=시간순)
    UPROPERTY() TMap<FString, int32> RecordIdCounters;    // "YYYYMMDD" → 다음 시퀀스 (퀘스트 IdCounters와 별도)
    UPROPERTY() TArray<FFocusRecord> FocusRecords;         // [구버전 잔재] v1 세이브 마이그레이션용 — 이전 후 Empty (11번 문서 05)

    // ── 퀘스트 (02) ──
    UPROPERTY() TArray<FQuestMasterData> QuestMasters;
    UPROPERTY() TMap<FString, int32> IdCounters;    // "YYYYMMDD" → 다음 시퀀스
    UPROPERTY() FDateTime LastRolloverDate;           // 마지막 롤오버 처리일

    // ── 길드 (04) ──
    UPROPERTY() TArray<FOwnedGuildMember> OwnedMembers;   // 상한 8 (Settings)
    UPROPERTY() TArray<FSquadPreset> SquadPresets;        // 해금에 따라 1~3개
    UPROPERTY() int32 ActivePresetIndex = 0;
    UPROPERTY() TArray<FHireHistoryEntry> HireHistory;

    // ── 경제 · 진행도 (05) ──
    UPROPERTY() int64 Gold = 5000;         // int64 — 상한 없는 재화는 넉넉하게
    UPROPERTY() int32 GuildLevel = 1;
    UPROPERTY() int32 GuildXp = 0;          // 현재 레벨에서 모은 XP
    UPROPERTY() int64 GuildTotalXp = 0;     // 누적 통계용
    UPROPERTY() TArray<FName> UnlockedIds;    // DT_Unlocks·콘텐츠 RowName
    UPROPERTY() int32 QuestGoldToday = 0;
    UPROPERTY() FDateTime QuestGoldDate;      // 날짜가 바뀌면 QuestGoldToday 리셋
    UPROPERTY() int32 FocusStreakCount = 0;

    // ── 표시 설정 (06) ──
    UPROPERTY() FName CurrentBackground = "default";

    // ── 타이머 사용자 설정 (01) — 기본값은 Settings에서 복사 ──
    UPROPERTY() int32 FocusMinutes = 25;
    UPROPERTY() int32 RestMinutes = 5;
    UPROPERTY() int32 LastRestMinutes = 15;
    UPROPERTY() int32 SingleRestMinutes = 5;
    UPROPERTY() int32 RepeatCount = 4;
    UPROPERTY() bool bIsRoutine = false;
    UPROPERTY() bool bAutoRestart = false;

    // ── 카테고리 — 사용자가 추가/삭제 가능하므로 동적 ──
    UPROPERTY() TArray<FString> Categories;   // 첫 실행 시 Settings 기본값 복사
    UPROPERTY() TMap<FString, FLinearColor> CategoryColors;
};

05-5 · 저장하지 않는 것 (휘발 상태 명세)

아래는 SaveGame에 넣으면 안 되는 상태의 전체 목록입니다. 각 Subsystem이 UPROPERTY(Transient) 멤버로 들고, 앱 시작 시 초기값으로 리셋됩니다.

상태보관 위치저장하지 않는 이유
타이머 진행 (EFocusState, 남은 초, 사이클 번호, 누적 집중 초, 골드 틱 카운터)UFocusSubsystem재시작 후 "돌고 있던 타이머"를 복원하면 실제 흐른 시간과 어긋남. 프로토타입도 미저장
스톱워치 진행 (경과 초, 실행/일시정지)UFocusSubsystem동일
가챠 진행 (상태, 후보 3장, 선택 인덱스)UGuildSubsystem세션 한정 연출 상태. 단, 아래 Pitfall 참조
음악/앰비언트 재생 위치·볼륨 슬라이더UAudioSubsystem재생 위치 복원은 가치 대비 복잡도만 높음
UI 상태 (열린 패널, 리스트/작성 뷰, 상태 필터, 캘린더 연/월, 더보기 카운트)UUIManagerSubsystem화면 상태는 매 세션 초기 화면에서 시작
⚠️
가챠 도중 종료 = 골드 증발. 프로토타입은 모집(1,000 Gold 차감) 직후 후보 3장을 휘발 상태로만 들고 있어, 이 시점에 앱이 꺼지면 골드만 사라집니다. UE 구현에서는 차감과 동시에 후보 3장(RowName 3개)을 SaveGame에 임시 필드로 기록하고, 선택/패스가 끝나면 비우는 것을 권장합니다. 이 필드(PendingHireCandidates)는 v1 스키마에 예약만 해 두어도 됩니다.
06 — 엔진 등록 절차

에디터에서 데이터 자산 만들기

구조체 헤더가 컴파일된 상태를 전제로, 에디터에서 실제 에셋을 만드는 순서입니다.

콘텐츠 폴더 구조

Content/ └─ RecordTales/ ├─ Data/ │ ├─ DT_GuildMembers ← GuildMemberRow │ ├─ DT_Synergies ← SynergyRow │ ├─ DT_Unlocks ← UnlockItemRow │ ├─ DT_GuildLevels ← GuildLevelRow │ ├─ DT_MusicTracks ← MusicTrackRow │ ├─ DT_AmbientSounds ← AmbientSoundRow │ └─ DT_Backgrounds ← BackgroundRow ├─ Audio/ (USoundBase 에셋 — 트랙 6 + 앰비언트 5) └─ Textures/ (초상화 12 + 배경 9)
1
DataTable 에셋 생성
Content Browser에서 Content/RecordTales/Data 폴더를 연 뒤 우클릭 ▸ Miscellaneous ▸ Data Table. "Pick Row Structure" 창에서 해당 행 구조체(예: GuildMemberRow)를 선택합니다. 구조체가 목록에 없다면 FTableRowBase 상속 누락이거나 컴파일 전입니다.
2
CSV 임포트
§04의 표를 CSV로 옮긴 파일(첫 열 = RowName)을 준비하고, DataTable 에셋을 열어 Reimport 또는 Content Browser Import로 불러옵니다. 이후 밸런싱은 CSV 수정 → Reimport 반복으로 진행합니다.
3
Settings 확인
URecordTalesSettings가 컴파일되면 Edit ▸ Project Settings ▸ Game ▸ Record Tales 섹션이 자동 생성됩니다. 값을 바꾸면 Config/DefaultGame.ini에 기록됩니다.
4
테이블 참조 연결
각 Subsystem이 자기 테이블을 TSoftObjectPtr<UDataTable> 프로퍼티로 들고, Initialize()에서 로드합니다. 하드코딩 경로 대신 Settings에 테이블 경로 프로퍼티를 두면 재배치에 안전합니다.
5
SaveGame 로드/생성
게임 시작 시 UGameplayStatics::DoesSaveGameExist("RecordTalesSave", 0) 확인 → 있으면 LoadGameFromSlot, 없으면 CreateSaveGameObject(URecordTalesSaveGame::StaticClass())로 새로 만들고 Settings의 초기값(시작 골드 5,000 · 기본 카테고리 6종 · 타이머 기본값)을 채워 첫 저장합니다.
6
저장 시점
Gold/XP 변동 · 퀘스트/기록 생성·완료 · 가챠 확정 · 해금 구매 · 설정 변경 직후 즉시 SaveGameToSlot. 프로토타입은 저장이 아예 없으므로(새로고침 = 전체 초기화), 저장 시점 설계는 UE 구현에서 새로 갖는 책임입니다.
07 — 검증

데이터 무결성 체크리스트

테이블을 다 채운 뒤, 아래를 순서대로 확인합니다. 특히 참조 무결성(RowName이 실제로 존재하는가)은 에디터가 자동으로 검사해 주지 않으므로 직접 확인해야 합니다.

#확인 항목기대값
1DT_GuildMembers 행 수12 (등급별 3명 × 4)
2DT_Synergies 행 수 / 멤버 참조7행. 모든 Members 값이 DT_GuildMembers의 RowName에 존재
3DT_Unlocks 행 수 / 선행 체인17행. 모든 Requires가 같은 테이블의 RowName (순환 없음)
4DT_GuildLevels 행 수 / 확률 합20행. 모든 행에서 N+R+E+L = 100
5콘텐츠 테이블 행 수곡 6 (기본 2 + 해금 4) · 앰비언트 5 · 배경 9 (기본 1 + 해금 8)
6세이브 라운드트립저장 → 앱 재시작 → 로드 후 Gold·퀘스트·보유 길드원·설정이 동일
7자정 롤오버기기 날짜를 +2일로 바꾸고 실행 → 밀린 2일치 반복 인스턴스가 모두 생성
8파생 상태 재현마감 지난 퀘스트가 로드 직후 Overdue로 표시 (저장된 상태가 아니라 계산 결과)

2~4번은 손으로 훑는 대신, 에디터 전용 검증 함수를 만들어 두는 것을 권장합니다. 예: UGuildSubsystem에 시너지 테이블을 순회하며 DT_GuildMembers->FindRow()가 nullptr이면 경고 로그를 남기는 ValidateTables()를 두고 Initialize()에서 개발 빌드에만 호출합니다.

✓ 이렇게

  • 데이터당 보관처는 한 곳 — 정적은 테이블, 동적은 세이브, 파생값은 계산
  • 정적 데이터 참조는 항상 RowName(FName)으로
  • 세이브에 SaveVersion을 처음부터 넣고 시작
  • 실측값을 기본으로, 미배선 상수는 표시해 두고 기획 확정 후 배선
  • 테이블 검증 함수를 개발 빌드에서 자동 실행

✗ 이건 피하기

  • 가챠 시점에 캐릭터 스탯을 세이브로 복사 (밸런스 패치가 안 먹힘)
  • 날짜를 문자열 4종으로 중복 저장 (프로토타입의 date/dateLong/clock/ts)
  • 퀘스트 상태(Overdue 등)를 저장 (자정이 지나면 거짓이 됨)
