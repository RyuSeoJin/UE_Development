01 — 지도를 읽는 법

기호·색·용어를 먼저 고정합니다

핵심 용어 — 이 여섯 개면 끝까지 읽힙니다

📖

소유(Ownership) — 변수를 선언하고 값 갱신을 책임지는 파일. 이 문서에서 변수의 주인은 항상 하나입니다. 예: Gold의 주인은 EconomySubsystem — 다른 시스템은 AddGold()로 "요청"만 합니다.

참조(Reference) — 남의 변수를 읽거나, 함수를 통해 바꿔달라고 요청하는 것. 값을 직접 만지지 않습니다.

선언(.h) / 정의(.cpp) — .h는 "이 파일에 무엇이 있다"는 목록(변수·함수 선언), .cpp는 "그것이 어떻게 동작한다"는 구현. 변수가 "산다"고 할 때 집 주소는 .h, 값을 바꾸는 손은 .cpp입니다.

델리게이트(Delegate) — "값이 바뀌었다"는 방송. 방송하는 쪽은 누가 듣는지 모르고, 구독한 쪽만 반응합니다. 시스템을 서로 모르는 채로 잇는 핵심 장치입니다.

직렬화(Serialization) — 메모리 속 변수 값을 디스크 파일(.sav)로 바꿔 쓰는 것. RecordTales에서는 URecordTalesSaveGame이 유일한 직렬화 지점입니다.

배선(Wiring) — 각 서브시스템의 Initialize()에서 델리게이트 구독과 의존성 선언으로 시스템을 잇는 작업. 프로젝트가 "유기적으로" 도는 이유는 전부 이 배선에 있습니다(05에서 실제 코드로 확인).

수명 배지 — 변수가 어디까지 살아남는가

파일 지도(02)의 모든 변수 행에는 배지가 하나씩 붙습니다. 배지는 "앱을 껐다 켰을 때 이 값이 어디서 오는가"를 뜻합니다.

03 — 소유 결정

왜 그 파일에 두는가

02의 배치는 우연이 아닙니다. 각 변수군의 소유 파일은 아래 세 가지 질문으로 결정됐습니다: ① 언제까지 살아야 하나(수명), ② 누가 값을 바꾸나(갱신 책임), ③ 몇 시스템이 쓰나(공유 범위).

04 — 연결 지도

변수는 시스템 사이를 어떻게 흘러가는가

이제 집(파일)들을 도로로 잇습니다. 연결은 딱 네 가지 통로(01의 범례)뿐이고, 세 단계로 봅니다: 전체 관계망(조감) → 저장 전도(값이 디스크로 가는 길) → 보상 전도(값이 만들어지는 길).

전체 관계망 — 옵시디언 그래프 뷰 스타일

먼저 하늘에서 내려다봅니다. 점 하나가 파일(시스템) 하나, 선 하나가 아래 전수 표의 연결 하나입니다. 점이 클수록 연결이 많은 파일이라, Economy와 SaveGame이 이 프로젝트의 두 중심임이 한눈에 보입니다.

05 — 흐름 추적

시나리오 3편 — 변수 하나를 끝까지 따라가기

지도가 맞는지 실제 사건으로 검증합니다. 세 편이면 생성(① 보상) · 영속(② 복원) · 소비(③ 지출)의 전체 순환이 덮입니다. 각 단계의 칩은 그 일이 일어나는 파일입니다.

여정 ① — 포모도로 1사이클 완주가 XP·레벨·기록이 되기까지

FocusSubsystem.cpp25분 집중 만료OnTimerExpired()
CurrentState: Focus → Rest
상태 머신이 다음 단계로 전환하며 완주를 확정합니다. 보상 규칙은 여기 없습니다 — Focus는 시간만 압니다.
FocusSubsystem.cpp완주 방송OnFocusCycleCompleted.Broadcast(1500, "공부")
신호: (DurationSeconds=1500, Category="공부")
누가 듣는지 모른 채 방송합니다. 구독자는 Economy와 Record 둘 — 배선은 각자의 Initialize()에서 이미 끝났습니다.
EconomySubsystem.cppXP 지급 + 스트릭HandleFocusCycleCompleted() → AddXP(15)
GuildXp: 40 → 55 · FocusStreakCount: 2 → 3 (+보너스 XP 20)

배선의 실제 코드 — 이 연결이 "만들어지는" 지점

세 여정 모두 "구독은 이미 돼 있다"고 전제했습니다. 그 구독이 실제로 일어나는 곳이 각 서브시스템의 Initialize()입니다. Economy를 예로 배선의 전체 형태를 보입니다.

// Economy/EconomySubsystem.h — 신호를 받아 보상으로 바꾸는 허브
#pragma once
#include "CoreMinimal.h"
#include "Subsystems/GameInstanceSubsystem.h"
#include "EconomySubsystem.generated.h"

DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(FOnGoldChanged, int64, NewGold);

UCLASS()
class RECORDTALES_API UEconomySubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()
public:
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;

    UFUNCTION(BlueprintCallable, Category = "Economy")
    void AddGold(int32 Amount);        // Gold의 유일한 입금 창구
    UFUNCTION(BlueprintCallable, Category = "Economy")
    bool TrySpendGold(int32 Amount);   // 유일한 출금 창구 — 잔액 판정 포함

    UPROPERTY(BlueprintAssignable)      // UI가 구독하는 방송 출구
    FOnGoldChanged OnGoldChanged;

private:
    // Focus의 방송을 받는 콜백 — 델리게이트 배선의 도착점
    UFUNCTION()
    void HandleFocusCycleCompleted(int32 DurationSeconds, const FString& Category);

    int64 Gold = 0;                    // 소유: 이 파일이 유일한 주인 (SAVE 사본으로 영속)
    int32 GuildXp = 0;
    int32 FocusStreakCount = 0;
};
// Economy/EconomySubsystem.cpp — 시스템 간 배선은 Initialize()에서 한 번만
#include "Economy/EconomySubsystem.h"
#include "Focus/FocusSubsystem.h"
#include "Data/RecordTalesSettings.h"

void UEconomySubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
    Super::Initialize(Collection);

    // ① 의존성 선언 — Focus가 나보다 먼저 초기화되도록 보장
    Collection.InitializeDependency(UFocusSubsystem::StaticClass());

    // ② 배선 — Focus의 "사이클 완주" 방송을 구독한다.
    //    Focus는 Economy를 모른다. 화살표는 Economy → Focus 한 방향뿐.
    UFocusSubsystem* Focus = GetGameInstance()->GetSubsystem<UFocusSubsystem>();
    Focus->OnFocusCycleCompleted.AddDynamic(
        this, &UEconomySubsystem::HandleFocusCycleCompleted);
}

void UEconomySubsystem::HandleFocusCycleCompleted(int32 DurationSeconds, const FString& Category)
{
    const URecordTalesSettings* S = GetDefault<URecordTalesSettings>();

    // 보상 수치는 전부 CONFIG 상수 — 코드 수정 없이 에디터에서 조정
    AddXP(S->XpTimerCycleComplete);                 // +15

    ++FocusStreakCount;                              // 연속 사이클 집계
    if (FocusStreakCount % S->StreakInterval == 0)  // 3연속마다
    {
        AddXP(S->XpStreakBonus);                     // +20 보너스
    }
    // 저장은 AddXP() 내부에서 GameInstance->SaveNow()로 위임
}
💡
배선을 읽는 요령 — ① InitializeDependency가 있으면 "누가 먼저 살아야 하는가"의 답이고, ② AddDynamic이 있으면 "누가 누구의 방송을 듣는가"의 답입니다. 새 시스템을 붙일 때도 이 두 줄만 추가하면 기존 코드는 한 글자도 바뀌지 않습니다.
06 — 확장 규칙

새 변수가 생기면 어디에 두는가

이 지도가 앞으로도 맞으려면, 새 변수를 추가할 때 아래 다섯 질문을 순서대로 통과시키면 됩니다. 각 질문의 답이 곧 파일 주소입니다.

1
기획자가 에디터에서 조정할 밸런스 값인가?
그렇다 → RecordTalesSettings.h에 CONFIG 상수로. 예: 새 보상 배율, 새 비용.
2
표로 관리되는 콘텐츠(여러 행)인가?
그렇다 → 행 구조체를 RecordTalesTypes.h에 정의하고 Content/Data/DT_*에 ROW로. 예: 새 캐릭터, 새 트랙.
3
어느 도메인의 규칙이 이 값을 바꾸는가?
값을 갱신하는 규칙이 있는 서브시스템이 주인입니다. 두 도메인이 다 바꾸고 싶다면 → 한쪽을 주인으로 정하고 다른 쪽은 요청 함수를 통해 접근.
4
앱을 껐다 켜도 남아야 하는가?
그렇다 → RecordTalesSaveGame.h에 SAVE 필드 추가 + 주인 서브시스템이 밀어넣기·복원 코드 작성 + 구조가 바뀌었으면 SaveVersion +1. 아니다 → RUNTIME으로 끝.
5
UI에 보여야 하는가?
그렇다 → 주인 서브시스템에 OnXXXChanged 델리게이트 추가, 위젯은 구독해서 그리기만. UI에 변수를 만들지 않습니다.

결정 플로차트 — 다섯 질문 한 장 요약 (Mermaid 스타일)

위 다섯 단계를 분기 그림으로 접으면 이렇게 됩니다. 마름모가 질문, 색 상자가 도착지(배치 파일)입니다.

flowchart TD
    Q1{"기획자가 조정할 밸런스 값인가?"} -- 예 --> S["RecordTalesSettings.h — CONFIG 상수"]
    Q1 -- 아니오 --> Q2{"표로 관리되는 콘텐츠인가?"}
    Q2 -- 예 --> D["RecordTalesTypes.h 행 구조체 + Content/Data/DT_* (ROW)"]
    Q2 -- 아니오 --> Q3["값을 바꾸는 규칙이 있는 서브시스템 = 주인 파일"]
    Q3 --> Q4{"껐다 켜도 남아야 하는가?"}
    Q4 -- 예 --> SG["RecordTalesSaveGame.h 필드 + SaveVersion +1"]
    Q4 -- "아니오 = RUNTIME" --> Q5{"UI에 보여야 하는가?"}
    SG --> Q5
    Q5 -- 예 --> DG["주인에 OnXXXChanged 델리게이트 — 위젯은 구독해 그리기만"]
    Q5 -- 아니오 --> END["배치 완료"]

✓ 이렇게

  • 변수의 주인 파일은 하나 — 나머지는 요청 함수로
  • 시스템 간 통지는 델리게이트로 (아는 방향 한쪽만)
  • 배선·의존성은 Initialize()에 모아 한눈에 보이게
  • 파생 가능한 값(TitleMap·SquadSlots)은 저장하지 않고 재계산
  • SaveGame 구조를 바꾸면 SaveVersion +1과 마이그레이션 동시 작성

✗ 이건 피하기

  • 서브시스템 A와 B가 서로를 구독 (양방향 = 순환 의존)
  • 위젯에 게임 값 사본 저장 — 표시는 소유가 아님
  • SaveGame 필드를 서브시스템 경유 없이 직접 수정
  • Settings 상수를 코드에 하드코딩으로 중복
  • 이 문서에 없는 시스템 간 연결을 문서 갱신 없이 추가
⚠️
순환 의존 경보 — "Economy도 Focus를 알고 Focus도 Economy를 알아야 할 것 같다"면 설계를 다시 봐야 합니다. 대부분 방송(델리게이트)을 거꾸로 건 것이 원인입니다: 값이 바뀌는 쪽이 방송하고, 반응하는 쪽이 구독합니다. 그래도 안 풀리면 둘이 공유하는 데이터를 셋째 시스템(또는 Data/의 공용 타입)으로 내리는 게 정석입니다.
