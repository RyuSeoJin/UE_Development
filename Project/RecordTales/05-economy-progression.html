01 — 개념 이해

경제 시스템이란

경제 시스템은 사용자의 집중 활동(포모도로·스톱워치)을 Gold/XP로 보상하고, 이를 소비해 길드 레벨업과 콘텐츠 해금으로 이어지는 순환 구조입니다. 모든 경제 로직은 UEconomySubsystem(UGameInstanceSubsystem을 상속하는 서브시스템, 즉 프로그램이 실행되는 동안 자동으로 살아있는 서비스 객체)에 모여 있습니다.

핵심 용어

📚

Gold — 인게임 화폐. 집중 중 초당 +1 틱으로 적립되며, 가챠·레벨업·해금에 소비됩니다.
XP (경험치) — 길드 레벨업에 필요한 경험치. 포모도로 완료·스톱워치 사용 등 행동으로 적립됩니다.
길드 레벨 (Guild Level) — 1~99 범위의 사용자 레벨. XP와 Gold를 모두 소비해 올립니다.
칭호 (Title) — 길드 레벨 구간에 따라 부여되는 명예 타이틀 (예: 견습 길드마스터).
해금 트리 (Unlock Tree) — Gold를 소비해 기능(편성 슬롯·프리셋)이나 콘텐츠(배경·BGM)를 여는 구조.

Gold & XP 획득·소비

Gold 획득

소스수량조건

퀘스트 완료+200일일 상한 2,000 Gold (한 날에 퀘스트 보상으로 받을 수 있는 최대치)

Gold 소비

Gold는 다음 용도로 소비됩니다.

용도비용

XP 획득

소스수량조건

연속 3사이클 보너스+20한 세션에서 3회 연속 포모도로를 완료했을 때 추가 보너스

💡

Gold는 현재 잔액만 저장. XP는 현재 레벨 내 잔여 XP(GuildXp)와 누적 총 XP(GuildTotalXp) 두 값을 저장합니다. 잔여 XP는 레벨업 시 차감되고, 총 XP는 차감 없이 계속 누적됩니다.

03 — 길드 레벨업

레벨업 조건 & 흐름

길드 레벨의 범위는 1~99이며, 디자인 캡(현실적으로 도달 가능한 상한)은 20입니다. 레벨업에는 XP와 Gold 모두 필요합니다.

레벨업 비용 공식

항목공식예시 (Lv5 → 6)
필요 XP100 × 1.4^(Lv-1)100 × 1.4^4 ≈ 384
필요 Gold500 × Lv500 × 5 = 2,500

XP 공식에서 1.4^(Lv-1)은 레벨이 올라갈수록 필요 XP가 지수적으로 증가한다는 뜻입니다. 레벨 1에서는 100 XP만 필요하지만, 레벨 15에서는 약 11,000 XP가 필요합니다.

레벨업 흐름

1
레벨업 버튼 클릭
UI에서 사용자가 레벨업 버튼을 탭합니다.
2
XP ≥ 필요XP 확인
GuildXp >= GetRequiredXP(GuildLevel)을 검사합니다.
3
Gold ≥ 필요Gold 확인
Gold >= GetRequiredGold(GuildLevel)을 검사합니다.
4
양쪽 차감
XP와 Gold를 각각 필요량만큼 차감합니다. GuildXp -= RequiredXP; Gold -= RequiredGold;
5
GuildLevel++
길드 레벨을 1 올립니다.
6
칭호 변경 확인
새 레벨이 칭호 구간 경계를 넘었는지 확인합니다(아래 칭호 시스템 참조).
7
가챠 확률 가중치 갱신
새 레벨에 따라 가챠의 희귀도 확률 테이블이 변경됩니다(04 길드 문서 참조).
8
SaveGame 동기화
변경된 레벨·XP·Gold를 SaveGame에 저장합니다.
⚠️

XP와 Gold 중 하나라도 부족하면 레벨업 불가. UI에서 부족한 자원을 빨간색으로 표시하고, 레벨업 버튼을 비활성화합니다.

04 — 해금 트리

기능 & 콘텐츠 해금

해금 아이템(앞서 설명한 해금 트리의 개별 항목)은 DataTable로 관리합니다. 각 항목은 FUnlockItemRow 구조체로 정의됩니다.

UENUM(BlueprintType)
enum class EUnlockType : uint8
{
    Feature    UMETA(DisplayName = "Feature"),
    Background UMETA(DisplayName = "Background"),
    Music      UMETA(DisplayName = "Music")
};

USTRUCT(BlueprintType)
struct FUnlockItemRow : public FTableRowBase
{
    GENERATED_BODY()

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FString Id;

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    EUnlockType Type;

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FString DisplayName;

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    int32 Cost;

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    FString Description;

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    bool bUnlocked = false;
};

해금 상태는 TArray<FString> UnlockedIds로 관리합니다. 특정 항목이 해금되었는지 확인할 때는 UnlockedIds.Contains(Id)를 사용합니다. 이 배열은 SaveGame에 저장되어 앱을 재실행해도 유지됩니다.

05 — 칭호 시스템

레벨 구간별 칭호

칭호(앞서 설명한 길드 레벨에 따른 명예 타이틀)는 레벨 구간에 매핑됩니다. 레벨업 시 새 구간에 진입하면 칭호가 자동으로 변경되며, UI 상단의 길드 정보 영역에 표시됩니다.

레벨칭호
1 ~ 3견습 길드마스터
4 ~ 6정식 길드마스터
7 ~ 9숙련 길드마스터
10 ~ 12정예 길드마스터
13 ~ 15전설의 길드마스터
16 ~ 18신화의 길드마스터
19 ~ 20만고불변의 길드마스터

구현 방식은 DataTable 또는 하드코딩(TMap<int32, FString>) 중 선택합니다. 칭호 수가 적고(7개) 변경 빈도가 낮으므로, 프로젝트 초기에는 하드코딩도 합리적입니다. 추후 칭호 수가 늘어나면 DataTable로 이전하세요.

06 — 구현 스켈레톤

UEconomySubsystem 최소 코드

아래는 UEconomySubsystem의 최소 동작 뼈대입니다. 실제 프로젝트에서는 일일 상한 검증·해금 연동·UI 이벤트 브로드캐스트를 추가로 구현합니다.

UCLASS()
class RECORDTALES_API UEconomySubsystem : public UGameInstanceSubsystem
{

public:
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;

    /** Gold 추가 (음수 = 차감) */

    void AddXP(int32 Amount);

    bool TryLevelUp();

    /** 해금 아이템 구매 */
    UFUNCTION(BlueprintCallable, Category = "Economy")
    bool PurchaseUnlock(const FString& UnlockId);

    /** 현재 레벨에서 필요한 XP */
    UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Economy")
    int32 GetRequiredXP(int32 Level) const;

    /** 현재 레벨에서 필요한 Gold */
    UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Economy")
    int32 GetRequiredGold(int32 Level) const;

    /** 현재 레벨에 해당하는 칭호 */
    UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Economy")
    FString GetTitle() const;

    /** Gold 잔액 조회 */
    UFUNCTION(BlueprintCallable, BlueprintPure, Category = "Economy")

{

    const int32 ReqXP   = GetRequiredXP(GuildLevel);
    const int32 ReqGold = GetRequiredGold(GuildLevel);

    if (GuildXp < ReqXP || Gold < ReqGold)
    {
        // 자원 부족 — UI에서 부족 자원을 빨간색 표시

    }

    // 차감

    Gold    -= ReqGold;

bool UEconomySubsystem::PurchaseUnlock(const FString& UnlockId)
{
    if (UnlockedIds.Contains(UnlockId)) return false; // 이미 해금

    // TODO: DataTable에서 해당 UnlockId의 Cost 조회
    int32 Cost = 0; // placeholder
    if (Gold < Cost) return false;

    Gold -= Cost;
    UnlockedIds.Add(UnlockId);

int32 UEconomySubsystem::GetRequiredXP(int32 Level) const
{
    // 공식: 100 * 1.4^(Lv-1)
    // double 사용으로 int32 오버플로 방지
    const double Base = 100.0 * FMath::Pow(1.4, static_cast<double>(Level - 1));
    return static_cast<int32>(FMath::CeilToDouble(Base));
}

int32 UEconomySubsystem::GetRequiredGold(int32 Level) const
{
    // 공식: 500 * Lv
    return 500 * Level;
}

FString UEconomySubsystem::GetTitle() const
{
    FString Result = TEXT("길드마스터");
    for (const auto& Pair : TitleMap)
    {
        if (GuildLevel >= Pair.Key)
            Result = Pair.Value;
    }
    return Result;
}

07 — 확장 & 주의

Pitfall & Do/Don’t

⚠️
1.4^(Lv-1) 계산에서 int32 오버플로 주의. 레벨 50 이상에서 100 * 1.4^49는 약 6.4 × 10^8로 int32 범위(약 21억) 안이지만, 레벨이 더 올라가면 넘칠 수 있습니다. 반드시 double로 계산한 뒤 int32로 캐스팅하세요. 위 스켈레톤 코드처럼 FMath::Pow의 인자를 double로 전달합니다.
⚠️
Gold 틱이 Tick에 직접 있으면 프레임 의존적. Tick(DeltaTime)에서 DeltaTime을 누적해 초당 +1을 구현하면, 프레임률에 따라 정확도가 달라집니다. FTimerHandle로 정확히 1초 주기 타이머를 설정하는 것이 권장됩니다. GetWorld()->GetTimerManager().SetTimer(Handle, this, &ThisClass::GoldTick, 1.0f, true) 형태.
⚠️
해금 아이템 ID 오타 시 Contains() 실패. UnlockedIds.Contains("sqaud3")처럼 오타가 있으면 해금 여부가 항상 false로 판정됩니다. ID를 static const FString으로 상수 정의하고, 코드 전체에서 상수를 참조하세요. 예: static const FString UNLOCK_SQUAD3 = TEXT("squad3");

✓ 이렇게

  • XP 공식은 double로 계산 후 int32 캐스팅
  • Gold 틱은 FTimerHandle 1초 주기로 정확하게
  • 해금 ID는 상수(static const FString)로 관리
  • 레벨업 전 XP + Gold 양쪽 모두 검증
  • SaveGame 동기화를 Gold/XP 변경 후 즉시 수행

✗ 이건 피하기

  • int32로 지수 계산 (오버플로 위험)
  • Tick(DeltaTime)에서 직접 Gold 누적 (프레임 의존)
  • 해금 ID를 문자열 리터럴로 여기저기 흩뿌리기
  • XP만 확인하고 Gold 검증 누락
  • 칭호 구간을 하드코딩한 뒤 칭호 추가 시 누락
