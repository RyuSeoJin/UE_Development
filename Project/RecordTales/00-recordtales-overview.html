포모도로 집중 관리와 길드 게임화를 결합한 생산성 앱을 언리얼 엔진으로 구현하기 위한 전체 설계 개요입니다.

01 — 프로젝트 개요

RecordTales란?

RecordTales(별칭 Grimhollow Guild Master)는 포모도로(Pomodoro) 기반 집중 관리와 길드 게임화(Gamification)를 결합한 생산성 앱입니다. 사용자는 25분 집중 타이머(포모도로)나 자유 시간 스톱워치로 작업하고, 그 결과로 Gold와 XP를 얻어 길드원을 모집하고 스쿼드를 편성합니다. 반복 퀘스트(일일/주간/월간)를 통해 습관을 추적하고, 아카이브에서 과거 기록을 캘린더로 되돌아봅니다. 음악과 앰비언트 사운드로 몰입감을 높이며, 모든 데이터는 단일 SaveGame 슬롯에 영속됩니다.

이 앱의 UE 구현은 GameInstanceSubsystem(게임 인스턴스에 묶이는 서브시스템) 패턴을 중심으로, 기능을 7개 서브시스템으로 분리합니다. 서브시스템(Subsystem)이란 엔진이 자동으로 생성하고 관리하는 서비스 객체로, 특정 수명(여기서는 GameInstance 수명)에 묶여 레벨이 바뀌어도 유지됩니다.

03 — 핵심 클래스 계층

UObject 기반 클래스 트리

RecordTales의 모든 C++ 클래스는 언리얼의 UObject(리플렉션, 가비지 컬렉션, 직렬화를 제공하는 최상위 부모 클래스)를 뿌리로 합니다. 아래 트리에서 엔진이 제공하는 부모 클래스와 프로젝트가 만드는 자식 클래스의 상속 관계를 확인할 수 있습니다.

// 엔진 부모 클래스 → RecordTales 프로젝트 클래스 // 전역 데이터 UGameInstance └── URecordTalesGameInstance ← 전역 데이터 (보유 목록, 설정) // 7개 서브시스템 (모두 GameInstanceSubsystem) UGameInstanceSubsystem ├── UFocusSubsystem ← 타이머 / 스톱워치 ├── UQuestSubsystem ← 퀘스트 CRUD + 반복 ├── URecordSubsystem ← 집중 기록 저장/조회 ├── UGuildSubsystem ← 길드원 + 편성 ├── UEconomySubsystem ← Gold / XP / 레벨 ├── UAudioSubsystem ← 앰비언트 / 뮤직 └── UUIManagerSubsystem ← 패널 / 모달 / 토스트 // 디스크 영속 USaveGame └── URecordTalesSaveGame ← 디스크 직렬화 // UI 위젯 (대표 예시) UUserWidget ├── UFocusPanelWidget ← 포커스 패널 ├── UQuestPanelWidget ← 퀘스트 패널 ├── UArchivePanelWidget ← 아카이브 패널 └── UGuildModalWidget ← 길드 모달
GameInstance Subsystem SaveGame UserWidget
🧭

GameInstanceSubsystem이란? UGameInstanceSubsystem을 상속하면 엔진이 자동으로 인스턴스를 생성하고 GameInstance 수명에 맞춰 파괴합니다. 별도 등록 코드가 필요 없고, GetSubsystem<T>()로 어디서든 접근할 수 있습니다. 레벨 전환에도 유지되므로 타이머/퀘스트/경제 같은 세션 전역 로직에 적합합니다.

04 — 책임 배치 결정표

어떤 기능을 어디에?

"이 기능/데이터를 어느 클래스에 둘지"를 결정하는 표입니다. 각 행의 근거 열을 읽으면 왜 그 위치에 배치했는지를 이해할 수 있습니다. 이 결정이 아키텍처의 뼈대이므로, 새 기능을 추가할 때도 이 표를 기준으로 배치 위치를 정합니다.

이 데이터 / 기능둘 곳근거
게임 규칙 (경제 공식, XP 테이블) UEconomySubsystem 경제 규칙은 런타임에만 존재하면 되고, 서브시스템에 캡슐화하면 독립 테스트가 가능합니다.
타이머 상태 머신 (Idle/Focusing/Resting) UFocusSubsystem 타이머 로직은 레벨 전환에 영향받지 않아야 합니다. GameInstance 수명의 서브시스템이 적합합니다.
퀘스트 CRUD + 반복 규칙 UQuestSubsystem 퀘스트 목록과 상태 계산을 한곳에 모아 일관된 CRUD를 보장합니다.
집중 기록 저장/조회 URecordSubsystem 기록 데이터의 필터/정렬/캘린더 뷰 로직을 격리해 UI와 분리합니다.
길드원 관리 (가챠/편성/시너지) UGuildSubsystem 가챠 확률, 편성 슬롯, 시너지 계산은 독립된 도메인이므로 전용 서브시스템으로 분리합니다.
오디오 재생 (앰비언트/뮤직) UAudioSubsystem 사운드 상태(현재 트랙, 볼륨)를 세션 내내 유지하려면 GameInstance 수명에 묶어야 합니다.
UI 패널/모달/토스트 관리 UUIManagerSubsystem 패널 스택과 모달 큐를 중앙에서 관리해 UI 충돌(두 모달 동시 오픈 등)을 방지합니다.
전역 데이터 (보유 길드원 목록, 앱 설정) URecordTalesGameInstance 세션 전역이면서 서브시스템들이 공통으로 참조할 데이터의 허브입니다. 레벨 전환에도 파괴되지 않습니다.
디스크 직렬화 (모든 영속 데이터) URecordTalesSaveGame 앱을 종료하고 다시 열어도 남아야 할 데이터는 SaveGameToSlot()으로 디스크에 기록합니다.
🧭

모든 서브시스템은 GameInstance 수명에 묶여 레벨 전환에도 유지됩니다. 따라서 사용자가 앱 내에서 화면을 전환해도 타이머가 초기화되거나 퀘스트 목록이 사라지는 일이 없습니다.

05 — 데이터 흐름

입력에서 디스크까지

사용자의 조작이 서브시스템을 거쳐 SaveGame에 영속되기까지의 전체 흐름입니다. 이 흐름을 이해하면 "데이터가 어디에 있고, 언제 저장되는지"를 명확히 알 수 있습니다.

저장/로드 흐름 (상세)

1
사용자 조작 → 서브시스템 상태 변경
사용자가 UI에서 버튼을 누르면 해당 서브시스템의 함수가 호출되어 메모리 상의 상태가 변경됩니다. 예: 타이머 시작 버튼 → UFocusSubsystem::StartFocus() → 상태가 Idle에서 Focusing으로 전환.
2
서브시스템 → SaveGame에 복사
주기적으로(예: 포모도로 1사이클 완료 시, 퀘스트 상태 변경 시) 또는 앱 종료 직전에, 각 서브시스템의 런타임 상태를 URecordTalesSaveGame 인스턴스의 UPROPERTY 필드에 복사합니다.
3
SaveGame → 디스크 직렬화
UGameplayStatics::SaveGameToSlot(SaveGameInstance, SlotName, UserIndex)를 호출하면 엔진이 UPROPERTY로 마킹된 필드를 자동으로 바이너리로 변환해 디스크에 저장합니다.
4
앱 시작 시 로드 → 서브시스템에 배포
앱을 다시 시작하면 UGameplayStatics::LoadGameFromSlot()으로 디스크에서 SaveGame을 복원하고, 각 서브시스템의 Initialize() 단계에서 필요한 데이터를 가져가 런타임 상태를 재구축합니다.
⚠️

저장 시점 주의: 포모도로 진행 중 강제 종료되면 해당 사이클의 보상이 유실될 수 있습니다. 중요한 상태 변경(사이클 완료, 퀘스트 완료, 가챠 결과)마다 즉시 저장을 트리거하는 것을 권장합니다.

06 — 프로젝트 폴더 구조

소스 코드 배치

C++ 소스 파일은 기능 단위(도메인)로 폴더를 나눕니다. 한 폴더 안에 해당 서브시스템의 헤더(.h)와 소스(.cpp), 그리고 관련 USTRUCT/UENUM 정의를 함께 둡니다. 이렇게 하면 "이 기능에 관한 파일이 어디 있지?"라는 질문에 폴더 하나만 열면 답할 수 있습니다.

Source/RecordTales/ ├── Core/ ← 전역 진입점 │ ├── RecordTalesGameInstance.h/.cpp │ └── RecordTalesSaveGame.h/.cpp ├── Focus/ ← 포커스(타이머/스톱워치) │ ├── FocusSubsystem.h/.cpp │ └── FocusTypes.h ← EFocusState, FFocusSettings ├── Quests/ ← 퀘스트 │ ├── QuestSubsystem.h/.cpp │ └── QuestTypes.h ← EQuestState, FQuestData ├── Archive/ ← 아카이브(기록 관리) │ └── RecordSubsystem.h/.cpp ├── Guild/ ← 길드(가챠/편성/시너지) │ ├── GuildSubsystem.h/.cpp │ └── GuildTypes.h ← FGuildMember, ESquadSlot ├── Economy/ ← 경제(Gold/XP/레벨/해금) │ ├── EconomySubsystem.h/.cpp │ └── EconomyTypes.h ← FEconomyState, 레벨 테이블 ├── Audio/ ← 오디오(앰비언트/뮤직) │ └── AudioSubsystem.h/.cpp ├── UI/ ← 위젯/패널/모달/공통 컴포넌트 │ ├── UIManagerSubsystem.h/.cpp │ ├── FocusPanelWidget.h/.cpp │ ├── QuestPanelWidget.h/.cpp │ ├── ArchivePanelWidget.h/.cpp │ └── GuildModalWidget.h/.cpp └── Data/ ← DataTable, USTRUCT 공용 정의 └── RecordTalesDataTypes.h ← 여러 모듈이 공유하는 구조체
💡

폴더 = 도메인 원칙을 지키면, 나중에 팀원이 합류하거나 특정 기능을 리팩터링할 때 영향 범위를 폴더 단위로 파악할 수 있습니다. 서브시스템 간 공유 타입은 Data/ 폴더에 모아 순환 의존을 방지합니다.
