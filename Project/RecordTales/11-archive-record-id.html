타이머·스톱워치·퀘스트 완료·직접 작성 — RecordTales에서 발생하는 모든 기록에 13자리 통합 ID(YYYYMMDD + 5자리 시퀀스)를 부여하고, 나중에 아카이브를 몰아볼 때 ID 순서(=발생 순서)로 정렬하고 출처·기간·카테고리로 필터링할 수 있는 저장 구조를 명세합니다.

FArchiveRecord URecordSubsystem SaveVersion 1 → 2
01 — 요구사항

무엇을 만들기로 했나

아카이브의 목표는 두 가지입니다. 첫째, 모든 기록을 한 줄로 세워 몰아보기 — 어떤 종류의 기록이든 "생성된 순서"라는 하나의 축으로 정렬되어야 합니다. 둘째, 원하는 조각만 걸러 보기 — 출처(타이머/스톱워치/퀘스트/직접 작성), 기간(일/주/월), 카테고리로 필터링할 수 있어야 합니다.

기존 구조(03 아카이브 시스템 v1.1)의 FFocusRecord로는 두 목표 모두 한계가 있습니다: ID가 없어서 기록을 영구적으로 가리킬 방법이 배열 위치뿐이고(수정·삭제·상세보기에 취약), 출처 필드가 없어서 타이머/스톱워치(Mode)까지만 구분될 뿐 "퀘스트에서 온 기록"과 "직접 작성한 기록"을 담을 자리가 없습니다.

퀘스트의 세 시점 문제

퀘스트에는 시점이 셋 있습니다 — 생성한 시점, 목표 완료 일자(DueDate), 완료한 시점. "퀘스트 기록의 ID에는 어느 날짜를 쓰나?"가 이 설계의 핵심 질문이었고, 답은 ID에는 불변의 한 시점만 담는다입니다:

시점성격ID에 넣을 수 있나
생성한 시점불변✅ 퀘스트 마스터 ID의 날짜부 (기존 그대로)
목표 완료 일자 (DueDate)가변 — 사용자가 미룰 수 있음❌ 금지. ID는 불변인데 목표일이 바뀌면 ID와 실제가 어긋남 → 일반 필드로 저장
완료한 시점불변이지만 미래에 발생하는 별개 사건✅ 완료 순간 생성되는 아카이브 기록 ID의 날짜부
🔑

핵심 용어 — 통합 기록 ID: 모든 아카이브 기록에 부여하는 13자리 식별자 YYYYMMDD(사건 발생일) + 5자리 시퀀스(그날 몇 번째 기록인지, 00001부터). 퀘스트 마스터 ID와 같은 포맷이지만 카운터는 별도. · 정의 vs 사건: 퀘스트 마스터는 할 일의 "정의", 아카이브 기록은 발생한 "사건" — 퀘스트를 완료하는 순간 사건 기록이 새로 생기고 자기 ID를 받습니다. · 사건 발생 시각: 기록이 저장되는 순간의 시각. ID 날짜부와 CreatedAt의 기준.

02 — 설계 결정

어떤 데이터를 어디에, 왜 두는가

02-B의 핵심인 책임 배치 결정표입니다. 각 행은 "이 정보를 여기 두는 이유"까지 포함합니다.

⚖️

ID 불변 원칙 — ID에는 부여 순간 확정되고 영원히 안 바뀌는 정보만 넣습니다(발생일 + 그날의 순번). 바뀔 수 있는 것(목표일·제목·카테고리)은 전부 필드로. 이 원칙 하나가 "ID로 정렬하면 항상 발생 순서"라는 보증을 만듭니다.

💡

왜 14자리 신규 체계가 아니라 기존 13자리인가 — 퀘스트 시스템(02)이 이미 YYYYMMDD+5자리 체계와 일자별 카운터·자정 롤오버 패턴을 검증해 두었습니다. 같은 포맷을 쓰면 규칙이 하나로 통일되고, 하루 99,999건 한도는 개인 기록 앱에서 사실상 무한입니다.

04 — 파일별 코드

복붙 가능한 뼈대

전제: 모듈 이름 RECORDTALES_API는 자기 프로젝트 모듈로 바꾸세요. 파일 위치는 Source/RecordTales/Archive/ 기준입니다.

.h ArchiveRecord.h — 기록 구조체(FArchiveRecord) & 출처 enum(ERecordSource)

#pragma once

#include "CoreMinimal.h"
#include "ArchiveRecord.generated.h"  // 반드시 마지막 #include

// 기록 출처 — 필터 축 ①. 캘린더 도트 색도 여기서 파생된다.
UENUM(BlueprintType)
enum class ERecordSource : uint8
{
    Timer      UMETA(DisplayName = "타이머"),
    Stopwatch  UMETA(DisplayName = "스톱워치"),
    Quest      UMETA(DisplayName = "퀘스트"),
    Manual     UMETA(DisplayName = "직접 작성")
};

// 통합 아카이브 기록 — 모든 "사건"이 이 구조 하나로 저장된다.
// (구 FFocusRecord를 대체. 마이그레이션은 05 섹션 참조)
USTRUCT(BlueprintType)
struct FArchiveRecord
{
    GENERATED_BODY()

    // ── 정렬·식별 (불변) ──────────────────────────────
    UPROPERTY(BlueprintReadOnly)
    FString Id;                // "2026070600001" — YYYYMMDD + 5자리 시퀀스

    // ── 필터 축 3종 ──────────────────────────────────
    UPROPERTY(BlueprintReadOnly)
    ERecordSource Source = ERecordSource::Timer;   // ① 출처

    UPROPERTY(BlueprintReadWrite)
    FString Category;          // ② 카테고리 (기존 체계 그대로)

    UPROPERTY(BlueprintReadOnly)
    FDateTime CreatedAt;       // ③ 기간 — 사건 발생(저장) 시각

    // ── 내용물 ───────────────────────────────────────
    UPROPERTY(BlueprintReadWrite)
    int32 DurationSeconds = 0; // Timer/Stopwatch만 의미 있음

    UPROPERTY(BlueprintReadWrite)
    FString Title;             // 퀘스트 제목 "스냅샷" / 직접 작성 제목

    UPROPERTY(BlueprintReadWrite)
    FString Notes;

    // ── 퀘스트 전용 (Source == Quest일 때만 채움) ─────
    UPROPERTY(BlueprintReadOnly)
    FString SourceRefId;       // 원본 퀘스트 마스터 ID (역추적용)

    UPROPERTY(BlueprintReadOnly)
    FDateTime DueDate;         // 목표 완료 일자 — 가변이므로 ID에 안 넣고 여기에
};

.h .cpp RecordSubsystem — ID 발급·저장·필터

#pragma once

#include "CoreMinimal.h"
#include "Subsystems/GameInstanceSubsystem.h"
#include "ArchiveRecord.h"
#include "RecordSubsystem.generated.h"

// 기록 저장 시 방송할 델리게이트 선언 — 여러 UI가 동시에 구독 가능(멀티캐스트)
DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(FOnRecordSaved, const FArchiveRecord&, Record);

// GameInstanceSubsystem — 세션 내내 살아 있어 기록·카운터를 한 곳에서 관리
UCLASS()
class RECORDTALES_API URecordSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()

public:
    // 저장 — 호출자는 내용물만 채운다. Id·CreatedAt은 여기서 확정.
    UFUNCTION(BlueprintCallable)
    FString SaveRecord(FArchiveRecord Record);   // 반환: 발급된 Id

    // 조회 — 필터 축 3종. Source는 다중 선택(빈 배열 = 전체).
    UFUNCTION(BlueprintCallable)
    TArray<FArchiveRecord> GetRecordsFiltered(
        FDateTime RangeStart, FDateTime RangeEnd,
        const TArray<ERecordSource>& Sources,
        const FString& Category) const;

    UPROPERTY(BlueprintAssignable)
    FOnRecordSaved OnRecordSaved;   // UI 갱신용 방송

private:
    // ID 발급의 유일한 통로 — 카운터 증가 + 조립이 항상 함께.
    FString MakeRecordId(const FDateTime& When);

    // SaveGame과 동기화되는 상태 (Initialize에서 로드, 변경 즉시 flush)
    TArray<FArchiveRecord> ArchiveRecords;   // 항상 Id 오름차순(=시간순)
    TMap<FString, int32> RecordIdCounters;  // "YYYYMMDD" → 다음 시퀀스
};
#include "RecordSubsystem.h"

FString URecordSubsystem::MakeRecordId(const FDateTime& When)
{
    // 날짜부: 사건 발생일. 자정에 걸친 세션도 "저장 시각" 기준 — 규칙이 단순해진다.
    const FString DateKey = When.ToString(TEXT("%Y%m%d"));

    // 그날의 다음 시퀀스. 키가 없으면 0에서 시작해 첫 발급이 00001.
    int32& Counter = RecordIdCounters.FindOrAdd(DateKey);
    ++Counter;

    // 5자리 제로 패딩 — 고정 자릿수라서 문자열 사전순 정렬 == 시간순 정렬.
    return FString::Printf(TEXT("%s%05d"), *DateKey, Counter);
}

FString URecordSubsystem::SaveRecord(FArchiveRecord Record)
{
    // (1) 사건 발생 시각과 ID를 저장 순간에 확정 — 호출자는 관여 못 함
    Record.CreatedAt = FDateTime::Now();
    Record.Id        = MakeRecordId(Record.CreatedAt);

    // (2) 항상 끝에 append — 발급 순서가 곧 배열 순서이므로 정렬 불변식 유지
    ArchiveRecords.Add(Record);

    // (3) SaveGame flush + UI 방송 (기존 "변경 즉시 저장" 관례 그대로)
    FlushToSaveGame();
    OnRecordSaved.Broadcast(Record);
    return Record.Id;
}

TArray<FArchiveRecord> URecordSubsystem::GetRecordsFiltered(
    FDateTime RangeStart, FDateTime RangeEnd,
    const TArray<ERecordSource>& Sources,
    const FString& Category) const
{
    TArray<FArchiveRecord> Result;
    for (const FArchiveRecord& R : ArchiveRecords)
    {
        // 필터는 전부 "필드"로 — ID를 파싱하지 않는다
        if (R.CreatedAt < RangeStart || R.CreatedAt > RangeEnd) continue;
        if (Sources.Num() > 0 && !Sources.Contains(R.Source))   continue;
        if (!Category.IsEmpty() && R.Category != Category)      continue;
        Result.Add(R);
    }
    // 원본이 Id 오름차순이므로 Result도 시간순. 최신순 표시는 UI에서 역순 순회.
    return Result;
}

.cpp 퀘스트 완료 훅 — UQuestSubsystem에 추가

// 퀘스트 완료 처리 끝에 추가 — 완료라는 "사건"을 아카이브에 남긴다.
void UQuestSubsystem::CompleteQuest(const FString& MasterId)
{
    // ... 기존 완료 처리 (상태 갱신, 보상 지급 등) ...

    const FQuestMasterData* Master = FindMaster(MasterId);
    if (Master == nullptr) { return; }

    FArchiveRecord Record;
    Record.Source      = ERecordSource::Quest;
    Record.Title       = Master->Title;        // 스냅샷(복사) — 참조 아님!
    Record.Category    = Master->Category;
    Record.SourceRefId = Master->Id;           // 생성일은 이 ID의 날짜부에 이미 있음
    Record.DueDate     = Master->DueDate;      // 목표일은 필드로 — ID에 안 들어감

    // 완료일은 SaveRecord가 찍는 CreatedAt·ID 날짜부가 담당한다.
    GetGameInstance()->GetSubsystem<URecordSubsystem>()->SaveRecord(Record);
}
05 — 등록 · 마이그레이션

SaveGame에 편입하고 기존 데이터를 살리는 순서

전제: 기존 세이브에는 TArray<FFocusRecord> FocusRecords(ID 없음)가 들어 있습니다. 아래 절차가 끝나면 기존 기록도 전부 ID를 갖고 새 배열로 이사합니다.

01
SaveGame에 새 필드 추가
URecordTalesSaveGame에 TArray<FArchiveRecord> ArchiveRecords와 TMap<FString,int32> RecordIdCounters를 추가합니다. 기존 FocusRecords 필드는 아직 지우지 않습니다 — 구버전 세이브를 읽으려면 남아 있어야 합니다.
02
SaveVersion 2로 승격
기본값을 SaveVersion = 2로 올립니다. 로드 시 SaveVersion < 2이면 아래 03 마이그레이션을 실행한 뒤 버전을 2로 올려 저장합니다 — 마이그레이션은 세이브당 정확히 1회만 돕니다.
03
기존 기록에 ID 소급 부여
구 FocusRecords를 Timestamp 오름차순으로 정렬해서 순회하며(구 배열은 최신순이었음), 각각 FArchiveRecord로 변환해 MakeRecordId(Timestamp)로 ID를 부여하고 ArchiveRecords에 append합니다. Mode → Source 매핑: Timer→Timer, Stopwatch→Stopwatch. 이렇게 하면 과거 기록도 "그날 몇 번째"가 실제 순서대로 매겨집니다.
04
구 배열 비우기 + 저장
변환이 끝나면 FocusRecords.Empty()로 비우고(필드 자체는 다음 대청소 때까지 유지), SaveGameToSlot("RecordTalesSave", 0)으로 즉시 저장합니다. 이후 앱이 죽어도 마이그레이션이 다시 돌지 않습니다.
// 로드 직후 1회 실행 — SaveVersion 1 → 2
void URecordSubsystem::MigrateV1ToV2(URecordTalesSaveGame* Save)
{
    if (Save->SaveVersion >= 2) { return; }   // 이미 새 버전이면 아무것도 안 함

    // 구 배열은 최신순(index 0이 최신)이었으므로, 시간 오름차순으로 정렬해 순회
    TArray<FFocusRecord> Old = Save->FocusRecords;
    Old.Sort([](const FFocusRecord& A, const FFocusRecord& B)
             { return A.Timestamp < B.Timestamp; });

    for (const FFocusRecord& OldRec : Old)
    {
        FArchiveRecord NewRec;
        NewRec.CreatedAt       = OldRec.Timestamp;   // 사건 시각 승계
        NewRec.Id              = MakeRecordId(OldRec.Timestamp);  // 소급 발급
        NewRec.Source          = (OldRec.Mode == EFocusMode::Timer)
                               ? ERecordSource::Timer : ERecordSource::Stopwatch;
        NewRec.Category        = OldRec.Category;
        NewRec.DurationSeconds = OldRec.DurationSeconds;
        NewRec.Notes           = OldRec.Notes;
        ArchiveRecords.Add(NewRec);                  // 오름차순 순회라 append로 정렬 유지
    }

    Save->FocusRecords.Empty();   // 이사 완료 — 구 배열 비움 (필드는 유지)
    Save->SaveVersion = 2;        // 재실행 방지 도장
    FlushToSaveGame();
}
⚠️
마이그레이션 전에 카운터가 비어 있어야 합니다 — RecordIdCounters는 새 필드라 구세이브에선 자연히 비어 있지만, 개발 중 세이브를 오가며 테스트할 때는 꼬일 수 있습니다. 테스트 시 세이브 슬롯을 지우고(DeleteGameInSlot) 처음부터 검증하세요.

#시나리오기대 결과

V3완료 후 퀘스트 마스터의 제목 변경/삭제기록의 Title은 완료 당시 그대로 (스냅샷 검증)
V4기록 1건 삭제 후 새 기록 저장삭제된 번호는 재사용되지 않음 — 카운터는 계속 전진
V523:59에 시작한 세션을 00:01에 저장ID 날짜부 = 저장한 날(다음 날). 시작일이 아님
V6출처 필터 [Quest, Manual] + 주간 범위 + 카테고리 "공부"세 조건의 AND 결과만 반환, 반환 순서는 시간순

⚠️
ID로 필터링하고 싶은 유혹 — "ID 앞 8자리로 날짜 필터하면 되잖아?"는 날짜 버킷까지만 허용입니다(날짜부는 불변이므로 안전). 시각 범위·주간 경계 같은 실제 기간 연산은 반드시 CreatedAt으로 하세요. ID 파싱이 늘어나는 순간 ID 포맷이 "바꿀 수 없는 API"가 되어 버립니다.
⚠️
기기 시계 역행 — 사용자가 시계를 과거로 돌리면 이미 카운터가 큰 날짜에 다시 기록이 붙어도 ID는 여전히 유일합니다(카운터가 날짜별로 이어지므로). 다만 CreatedAt 순서와 ID 순서가 일시적으로 어긋날 수 있음을 알아두세요 — 개인 기록 앱에서는 허용 가능한 트레이드오프입니다.
