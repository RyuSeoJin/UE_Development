아카이브 시스템
기록 저장 & 캘린더

집중 세션 기록을 저장하고, 캘린더와 필터를 통해 조회하는 아카이브 시스템입니다. 기간·타입·카테고리별 필터링과 퀘스트 상태 연동까지 설계합니다.

기록 저장 캘린더 필터링
01 — concept

개념 이해

아카이브 시스템은 포커스 세션(집중 타이머 또는 스톱워치) 종료 후 사용자가 기록을 저장하고, 날짜·타입·카테고리별로 과거 기록을 조회하는 시스템입니다. "오늘 얼마나 집중했는지", "이번 주 공부에 얼마나 시간을 썼는지", "이번 달 운동 기록은 몇 번인지"를 한눈에 확인할 수 있게 합니다.

아카이브는 URecordSubsystem(GameInstanceSubsystem)이 담당하며, 기록 데이터는 URecordTalesSaveGame을 통해 디스크에 영속됩니다. 캘린더 위젯은 날짜별로 어떤 활동이 있었는지 도트(점)로 요약해 보여주며, RecordTales 전체에서 재사용되는 공통 UI 컴포넌트입니다.

핵심 용어

📖

FArchiveRecord — 아카이브에 저장되는 통합 기록 데이터(구 FFocusRecord 대체). 13자리 통합 기록 ID(YYYYMMDD+5자리 시퀀스), 출처(Source: 타이머/스톱워치/퀘스트/직접 작성), 카테고리, 집중 시간, 메모, 사건 발생 시각(CreatedAt)을 포함합니다. 구조체 정의·ID 발급 규칙의 단일 소스는 11 통합 기록 ID 문서입니다.

카테고리(Category) — 기록을 분류하는 사용자 정의 태그. "공부", "운동", "독서" 등을 자유롭게 추가할 수 있으며, 각 카테고리마다 고유 색상이 지정됩니다.

기간 필터(Period Filter) — 기록을 조회할 시간 범위를 결정하는 필터. Daily(하루), Weekly(월~일 한 주), Monthly(한 달) 3가지가 있습니다.

캘린더 도트(Calendar Dot) — 캘린더에서 날짜 아래에 표시되는 작은 점. F(주황, Timer 기록 있음), S(초록, Stopwatch 기록 있음), T(노랑, 퀘스트 있음) 3종류가 있습니다. 한 날짜에 여러 도트가 동시에 표시될 수 있습니다.

02 — data structure

데이터 구조

아카이브 시스템의 핵심 데이터는 FArchiveRecord 구조체입니다(v1.2에서 구 FFocusRecord를 대체). 집중 세션 종료·퀘스트 완료·직접 작성 — 어떤 출처든 하나의 "사건"이 하나의 레코드가 되어, 13자리 통합 기록 ID를 부여받고 배열 끝에 추가됩니다. 아래는 요약이며, 필드별 근거·ID 발급 규칙·마이그레이션의 단일 소스는 11 통합 기록 ID & 아카이브 저장 설계 문서입니다.

FArchiveRecord — 통합 기록 구조체 (요약)

// 기록 출처 — 필터 축 ①. 캘린더 도트 색도 여기서 파생. (구 EFocusMode 대체)
UENUM(BlueprintType)
enum class ERecordSource : uint8
{
    Timer      UMETA(DisplayName = "타이머"),
    Stopwatch  UMETA(DisplayName = "스톱워치"),
    Quest      UMETA(DisplayName = "퀘스트"),
    Manual     UMETA(DisplayName = "직접 작성")
};

// 통합 아카이브 기록 (요약 — 전체 정의·필드별 근거는 11번 문서가 단일 소스)
USTRUCT(BlueprintType)
struct FArchiveRecord
{
    GENERATED_BODY()

    // 통합 기록 ID "YYYYMMDD+5자리" — 정렬·식별 전용, 불변
    UPROPERTY(BlueprintReadOnly)
    FString Id;

    // 필터 축 3종: 출처 · 카테고리 · 사건 발생 시각
    UPROPERTY(BlueprintReadOnly)
    ERecordSource Source;

    UPROPERTY(BlueprintReadWrite)
    FString Category;

    UPROPERTY(BlueprintReadOnly)
    FDateTime CreatedAt;

    // 내용물
    UPROPERTY(BlueprintReadWrite)
    int32 DurationSeconds;      // Timer/Stopwatch만 의미

    UPROPERTY(BlueprintReadWrite)
    FString Title;              // 퀘스트 제목 스냅샷 / 직접 작성 제목

    UPROPERTY(BlueprintReadWrite)
    FString Notes;

    // 퀘스트 전용 (Source == Quest)
    UPROPERTY(BlueprintReadOnly)
    FString SourceRefId;        // 원본 퀘스트 마스터 ID

    UPROPERTY(BlueprintReadOnly)
    FDateTime DueDate;          // 목표 완료 일자 — 가변이므로 ID 밖
};

필드 설명 테이블

필드타입설명
IdFString통합 기록 ID(YYYYMMDD+5자리 시퀀스). 고정 자릿수라 문자열 사전순 정렬 = 시간순. 정렬·식별 전용 — 필터링에 파싱하지 않습니다.

CategoryFString기록의 분류. RecordSubsystem의 카테고리 목록에서 선택합니다. 필터 축 ②.
CreatedAtFDateTime사건 발생(저장) 시각. 기간 필터(축 ③)와 시각 표시는 전부 이 필드로 — 날짜·시각 파생 표기는 표시 시점에 생성합니다.

TitleFString퀘스트 제목 스냅샷(참조 아님) 또는 직접 작성 기록의 제목.

SourceRefIdFStringSource가 Quest일 때 원본 퀘스트 마스터 ID(13자리). 역추적용.
DueDateFDateTimeSource가 Quest일 때 목표 완료 일자. 가변 정보라 ID에 넣지 않고 필드로 둡니다.

카테고리 시스템

카테고리는 두 가지 데이터로 관리됩니다. TArray<FString> Categories는 카테고리 이름 목록이고, TMap<FString, FLinearColor> CatColors는 각 카테고리의 고유 색상을 저장합니다. 사용자가 새 카테고리를 추가할 때 이름과 색상을 함께 지정합니다.

카테고리 추가 예시
// 카테고리 추가 로직
void AddCategory(FString Name, FLinearColor Color)
{
    // 1. 중복 검사
    if (Categories.Contains(Name)) return;

    // 2. 이름 목록에 추가
    Categories.Add(Name);

    // 3. 색상 맵에 등록
    CatColors.Add(Name, Color);

    // 4. SaveGame에 동기화
    SaveToSlot();
}

// 기본 카테고리 예시:
//   "공부"  → (0.35, 0.56, 0.98, 1.0)  파랑
//   "운동"  → (0.22, 0.85, 0.70, 1.0)  틸
//   "독서"  → (0.61, 0.43, 1.0,  1.0)  퍼플
03 — save flow

기록 저장 흐름

기록 저장은 포커스 세션이 끝난 직후에 발생합니다. UFocusSubsystem(포커스 타이머/스톱워치를 관리하는 서브시스템)이 세션 종료를 알리면, UI가 기록 모달을 띄우고, 사용자가 카테고리를 선택하고 메모를 입력한 뒤 "저장"을 누르면 URecordSubsystem에 기록이 추가됩니다.

1
세션 종료 → 델리게이트 브로드캐스트
UFocusSubsystem에서 타이머 완료 또는 스톱워치 중지 시 OnSessionCompleted 델리게이트(콜백 시스템)를 브로드캐스트합니다. 이 델리게이트에는 집중 시간(초)과 모드(Timer/Stopwatch) 정보가 포함됩니다.
2
Record Modal 표시
UI 레이어가 델리게이트를 수신하여 기록 모달(팝업 대화상자)을 화면에 띄웁니다. 모달에는 카테고리 선택 드롭다운과 메모 입력란이 있습니다.
3
사용자 입력
사용자가 카테고리를 선택하고, 선택적으로 메모를 입력합니다. 카테고리는 필수이며, 메모는 빈 문자열이 허용됩니다.
4
SaveRecord 호출
사용자가 "저장" 버튼을 누르면, UI가 URecordSubsystem::SaveRecord(FArchiveRecord)를 호출합니다. 호출자는 내용물(출처·카테고리·시간·메모)만 채우며, Id(통합 기록 ID)와 CreatedAt(사건 발생 시각)은 이 시점에 SaveRecord가 확정합니다 — 발급 통로는 이 한 곳뿐입니다.
5
배열 끝에 Append (ID 오름차순 유지)
새 기록을 TArray<FArchiveRecord> ArchiveRecords 배열의 맨 끝에 추가합니다. ID는 발급 순서대로 커지므로 append만으로 배열이 항상 ID 오름차순(=시간순)을 유지합니다. 최신순 화면은 UI에서 역순으로 순회해 표시합니다. (v1.1까지의 "인덱스 0 삽입" 관례를 대체 — 근거는 11번 문서)
6
SaveGame에 동기화
Records 배열을 URecordTalesSaveGame에 저장하여 디스크에 영속합니다. 앱을 재시작해도 기록이 유지됩니다.
💡

기록 저장은 Optional. 사용자가 "건너뛰기"를 누르면 기록하지 않습니다. 집중 세션 자체는 완료되지만, 아카이브에 기록이 남지 않습니다. 이는 사용자가 기록을 원하지 않는 가벼운 세션(테스트, 짧은 타이머 등)을 위한 것입니다.

04 — calendar

캘린더

캘린더 위젯은 월별 탐색과 날짜 선택 기능을 제공합니다. 좌우 화살표로 이전/다음 달을 탐색하고, 특정 날짜를 탭하면 해당 날짜의 기록 목록이 표시됩니다. 각 날짜 셀 아래에는 도트(점)가 표시되어, 그 날짜에 어떤 종류의 활동이 있었는지를 색상으로 요약해 보여줍니다.

날짜별 도트 표시

도트색상데이터 소스판정 조건
F주황RecordSubsystem해당 날짜에 Source = Timer인 기록이 1개 이상 존재
S초록RecordSubsystem해당 날짜에 Source = Stopwatch인 기록이 1개 이상 존재
T노랑QuestSubsystem아래 타임라인 표기 규칙에 따라 해당 날짜에 표기되는 퀘스트가 1개 이상 존재

하나의 날짜에 여러 도트가 동시에 표시될 수 있습니다. 예를 들어 2026년 7월 2일에 타이머 기록과 퀘스트가 모두 있다면, F와 T 도트가 나란히 표시됩니다.

퀘스트 상태 타임라인 표기 규칙

"퀘스트가 캘린더의 어느 날짜부터 어느 날짜까지, 어떤 상태로 보이는가"의 규칙입니다. 핵심 원칙은 두 가지입니다: ① 표기 상태는 저장하지 않습니다 — 저장된 사실(생성일·시작일·마감일·완료일·중단일)에서 날짜 D를 넣으면 계산되는 파생값입니다(02 문서의 GetStatusForDate(D) 개념 — "오늘"뿐 아니라 과거 날짜 조회에도 같은 함수 사용). ② 가변 정보(목표일 변경 등)가 과거 표기를 바꾸지 않도록, 판정은 항상 기록된 사실 필드로만 합니다.

상태날짜 D에 표기되는 구간판정에 쓰는 저장 사실
진행 예정 (Upcoming)마스터 생성일 ≤ D < 시작일 — 마스터 단위 (단발·반복 공통. 반복 인스턴스에는 예정 구간이 없음)마스터 ID 날짜부(생성일), StartDate
진행 중 (Active)시작일(반복: 인스턴스 발생일) ≤ D ≤ 마감일, 완료·중단 전StartDate/발생일, Deadline(반복: 발생일 + DeadlineOffset)
완료 (Done)완료일까지 표기 후 종료 — 완료라는 "사건"은 Source=Quest 아카이브 기록이 영구 보존하므로 캘린더 중복 표기 불필요bDone, DoneAt
지연 (Overdue)마감일 < D. 단발성: 완료/중단 전까지 계속 표기(처리를 유도하는 동기 장치). 반복: 다음 인스턴스 발생 전날까지만Deadline, bDone/bSuspended, 반복 규칙(다음 발생일)
중단 (Suspended)중단일까지 표기 후 종료bSuspended, SuspendedAt
🔄

반복 퀘스트(Weekly/Monthly)는 "기간 지속"이 아니라 "지정 요일/일자에 발생하는 인스턴스"입니다. 각 인스턴스의 표기 창 = 발생일 ~ 발생일+DeadlineOffset. 지연을 "다음 발생 전날까지"로 자르는 이유: 매일 반복을 일주일 놓치면 지연 인스턴스가 7개 쌓여 캘린더가 지연 표기로 도배되고 다음 회차와 이중으로 보이기 때문입니다. 놓친 회차는 다음 회차가 올 때까지만 보이고, 자리를 넘겨줍니다.

⚠️
표기 상태를 저장하고 싶은 유혹 — "매일 밤 그날의 상태를 스냅샷으로 저장하면 되지 않나?"는 함정입니다. 상태를 저장하는 순간 목표일 변경·완료·중단이 일어날 때마다 과거 스냅샷과 어긋나고, 이를 맞추려면 매일 전체 퀘스트를 순회해야 합니다. 파생 계산이면 그런 동기화 문제 자체가 없습니다. (같은 원리: 02 문서 "상태는 DB에 저장하지 않습니다")
📅

캘린더는 RecordTales 전체에서 재사용되는 공통 위젯입니다. 아카이브 화면에서는 기록 조회용으로, 퀘스트 생성/수정 화면에서는 시작일·마감일·반복 종료일(앞서 설명한 RecEndDate) 선택에도 동일한 캘린더 위젯을 사용합니다. 위젯을 하나만 만들고 용도별로 모드를 전환하는 구조입니다.

05 — filtering

필터링 시스템

필터링은 3축으로 구성됩니다: 기간(어느 범위의 날짜를 볼 것인가) × 타입(Timer/Stopwatch/전체) × 카테고리(특정 분류 또는 전체). 3축을 조합하면 "이번 주 공부 카테고리의 타이머 기록만"처럼 세밀한 조회가 가능합니다.

기간 필터 범위

기간 필터범위예시 (기준일: 2026.07.02 수요일)
Daily선택한 날짜 1일2026.07.02 하루
Weekly선택한 날짜가 속한 월요일~일요일2026.06.29(월) ~ 2026.07.05(일)
Monthly선택한 날짜가 속한 월의 1일~말일2026.07.01 ~ 2026.07.31

타입 필터

All(전체), Timer(포모도로 타이머만), Stopwatch(스톱워치만) 3가지 중 하나를 선택합니다. 기본값은 All입니다.

카테고리 필터

전체 카테고리를 보거나, 특정 카테고리 하나를 선택할 수 있습니다. 카테고리 목록은 RecordSubsystem의 Categories 배열에서 가져옵니다.

필터 적용 로직

1
기간으로 날짜 범위 계산
선택된 기간 필터(Daily/Weekly/Monthly)와 기준 날짜를 입력받아 시작일·종료일을 계산합니다. Weekly는 월요일 시작 기준입니다.
2
범위 내 기록 필터
Records 배열에서 Date 필드가 계산된 범위(시작일 ≤ Date ≤ 종료일)에 해당하는 기록만 추출합니다.
3
타입 필터 적용
All이 아닌 경우, Mode 필드가 선택된 타입(Timer 또는 Stopwatch)과 일치하는 기록만 남깁니다.
4
카테고리 필터 적용
전체가 아닌 특정 카테고리가 선택된 경우, Category 필드가 일치하는 기록만 남깁니다.
5
날짜별 그룹핑 + 누적 시간 계산
필터링된 기록을 CreatedAt에서 파생한 날짜 기준으로 그룹핑하고, 각 그룹(날짜)의 DurationSeconds를 합산하여 누적 시간을 계산합니다. 전체 기간의 총 누적 시간도 함께 계산합니다.

퀘스트 표시 연동

아카이브 화면에서는 집중 기록뿐 아니라, 같은 기간 내의 퀘스트 인스턴스도 함께 표시합니다. UQuestSubsystem::GetInstancesForDate()를 기간 내 각 날짜에 대해 호출하여 퀘스트를 가져옵니다.

TArray<FArchiveRecord> URecordSubsystem::GetRecordsForDateRange(
    const FString& StartDate,
    const FString& EndDate) const
{
    TArray<FArchiveRecord> Result;

    for (const FArchiveRecord& Rec : ArchiveRecords)
    {
        // 기간 판정은 CreatedAt(사건 발생 시각)에서 파생 — ID를 파싱하지 않는다
        const FString RecDate = Rec.CreatedAt.ToString(TEXT("%Y.%m.%d"));
        if (RecDate >= StartDate && RecDate <= EndDate)
        {
            Result.Add(Rec);
        }
    }

    return Result;
}

TArray<FArchiveRecord> URecordSubsystem::GetRecordsFiltered(
    const FString& BaseDate,
    EPeriodFilter Period,
    const TArray<ERecordSource>& Sources,
    const FString& CategoryFilter) const
{
    // 1단계: 기간으로 날짜 범위 계산
    FString RangeStart, RangeEnd;
    CalcDateRange(BaseDate, Period, RangeStart, RangeEnd);

    // 2단계: 범위 내 기록 가져오기 (필터는 전부 필드로 — ID 파싱 금지)
    TArray<FArchiveRecord> Filtered = GetRecordsForDateRange(RangeStart, RangeEnd);

    // 3단계: 출처 필터 — 다중 선택, 빈 배열이면 전체 통과
    if (Sources.Num() > 0)
    {
        Filtered.RemoveAll([&](const FArchiveRecord& R) {
            return !Sources.Contains(R.Source);
        });
    }

    // 4단계: 카테고리 필터
    if (!CategoryFilter.IsEmpty())
    {
        Filtered.RemoveAll([&](const FArchiveRecord& R) {
            return R.Category != CategoryFilter;
        });
    }

    // 원본이 Id 오름차순이므로 결과도 시간순 — 최신순 표시는 UI에서 역순 순회
    return Filtered;
}

07 — extension & pitfalls

확장 & 주의

⚠️
기록 배열이 커지면 날짜별 TMap 인덱싱을 고려하세요. TArray<FArchiveRecord>를 순차 탐색하는 GetRecordsForDateRange()는 기록이 수백~수천 개가 되면 느려질 수 있습니다. 이때 TMap<FString, TArray<int32>>(날짜 → 배열 인덱스)로 날짜별 인덱스를 별도로 유지하면 O(날짜 수)로 조회할 수 있습니다. 단, 삽입 시 두 곳(배열 + 맵)을 동시에 갱신해야 하며, 저장 구조 자체는 바꿀 필요가 없습니다.
⚠️
주간 범위 계산에서 월요일 시작 기준을 통일하세요. FDateTime::GetDayOfWeek()는 0=월요일, 6=일요일을 반환합니다. 주의 시작을 일요일로 바꾸면 캘린더 위젯과 필터링의 "주" 범위가 어긋나 다른 기록이 보입니다. RecordTales 전체에서 월요일 시작으로 통일합니다.

DO

  • 기록 삽입 시 항상 배열 끝에 Add(append)하여 ID 오름차순 유지 — 최신순은 표시 시 역순
  • Id·CreatedAt은 SaveRecord 안에서만 확정 — 호출자가 직접 채우지 말 것
  • 날짜 문자열은 YYYY.MM.DD 형식 함수를 통해 생성
  • 캘린더 위젯은 하나만 만들고 모드별로 재사용
  • 카테고리 색상은 FLinearColor로 저장하여 UI에서 직접 사용
  • 통계 대시보드 확장 시 이 필터링 로직을 기반으로 구축

DON'T

  • 인덱스 0에 Insert하지 말 것 — ID 오름차순 불변식이 깨짐 (v1.1까지의 구 관례)
  • 필터링·기간 연산에 ID를 파싱하지 말 것 — 필터 축은 Source·CreatedAt·Category 필드로
  • 주간 범위 시작일을 일요일로 바꾸지 말 것 — 캘린더와 불일치
  • 카테고리를 하드코딩하지 말 것 — 사용자가 자유롭게 추가 가능해야 함
  • 날짜별 인덱싱 없이 수천 개 기록을 매번 전체 탐색하지 말 것
  • DeleteRecord 시 배열만 수정하고 SaveGame 동기화를 빼먹지 말 것
💡

통계 대시보드(일별/주별/월별 집중 시간 차트)는 이 데이터를 기반으로 확장 가능합니다. GetRecordsFiltered()의 결과를 날짜별로 그룹핑하고 DurationSeconds를 합산하면 차트 데이터가 됩니다. 카테고리별 비율 파이 차트, 주간 추이 라인 차트 등으로 확장할 수 있습니다.

아카이브 시스템의 데이터 (실측 기준)

이 섹션은 프로토타입 v14(TestFile/RecordTales.html) 코드에서 실측한 데이터 기준입니다. 본문의 예시 수치와 다를 경우 이 섹션이 우선하며, 전 시스템 통합 스키마는 프로젝트 허브의 08 데이터 설계 명세 문서에 있습니다.

FArchiveRecord — 저장 필드 (날짜 정규화 결정 + v1.2 통합 ID 확장)
