02 — 설계 결정

왜 이렇게 만드나

결정내용근거
DB 문지기 분리DatabaseSubsystem이 열닫·CREATE TABLE·트랜잭션 창구를 전담 — 기능 시스템(Economy·Archive·이후 Gacha·Estate)은 대등하게 문지기에만 의존아카이브 "기능" 시스템이 전체 DB의 문지기를 겸하면 모든 시스템이 그 클래스에 숨은 의존을 갖게 됨 — 09는 스파이크라 임시 겸직이었고, 테이블이 6종으로 늘기 전에 분리
상태 = 1행 UPDATEplayer는 항상 1행(CHECK(Id=1)) — 값이 바뀌면 그 행을 덮어씀기록(archives, INSERT로 쌓임)과 반대 성격. 계정은 하나니 행도 하나
쓰기 주인 하나EconomySubsystem만 player를 씀 — UI·다른 시스템은 AddGold() 호출만같은 값을 두 곳에서 고치면 정합성이 무너짐(10 소유 결정표)
레벨은 저장 안 함레벨 = AccountFocusSeconds / 3600 — 매번 계산정직한 레벨: 파생 가능한 값을 저장하면 시간과 어긋날 여지가 생김
보호 범위 3종Gold·Shards·AccountFocusSeconds만 Secured 블롭으로 봉인치트 유인이 큰 값만 — 전부 감싸면 비용만 늘고 실익 적음
저장 시점 = 변화 즉시AddGold() 등이 값 변경 직후 SavePlayer() 호출데스크톱 로컬 SQLite라 UPDATE 1행 비용이 미미 — 유실 창을 없애는 쪽이 이득
🧭

서브시스템은 "기능을 쓸 때" 열리는 게 아닙니다. GameInstanceSubsystem의 Initialize()는 게임(앱)이 시작되는 순간 엔진이 자동 호출합니다 — 플레이어가 아카이브 UI를 여는 시점과 무관합니다. 그래서 문지기가 게임 시작 즉시 DB를 열고, InitializeDependency가 "문지기 먼저"를 보장하므로 Economy·캐릭터 시스템은 깨어나는 순간 이미 DB가 준비돼 있습니다.

💡

Secured 원리 요약(설계 배경은 03 데이터 설계 06 상태 보호). 보호 값 3종을 암호화한 사본(Secured BLOB)으로 같은 행에 함께 저장합니다. 이 블롭이 정본이고 평문 컬럼은 편의 사본 — 로드 때 둘을 비교해 다르면(=누가 평문을 편집) 블롭의 진짜 값으로 자동 복원합니다. 블롭 자체를 건드리면 복호화 검사(MAGIC·CRC)에서 들통납니다.

DB 문지기 UDatabaseSubsystem 만들기
열닫·PRAGMA·CREATE TABLE의 단일 창구 — 09의 DB 코드가 이리로 온다

클래스 위저드에서 부모 GameInstanceSubsystem으로 생성합니다. 역할은 딱 셋 — 열고, 스키마를 만들고, 닫는다. 게임 기능은 하나도 없습니다(그래서 어떤 기능 시스템과도 대등한 토대가 됩니다).

#pragma once
#include "Subsystems/GameInstanceSubsystem.h"
#include "SQLiteDatabase.h"          // 값 멤버로 품으므로 전방 선언이 아니라 실제 포함이 필요
#include "DatabaseSubsystem.generated.h"

// DB 문지기 — Archive.db의 열닫·스키마 생성·(이후) 트랜잭션 창구를 전담.
// 게임 기능 없음: Economy·Archive 같은 기능 시스템들이 이 클래스에만 의존한다
UCLASS()
class UDatabaseSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()
public:
    // 게임 시작 시 엔진이 1회 호출 — 여기서 DB를 열고 테이블을 보장한다
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;
    // 게임 종료 시 — 열었던 곳에서 닫는다(수명 대칭)
    virtual void Deinitialize() override;

    // 기능 시스템들에게 핸들을 "빌려준다" — 소유·수명 관리는 오직 여기
    FSQLiteDatabase* GetDb() { return &Database; }

private:
    void CreateTables();   // CREATE TABLE IF NOT EXISTS 전부 — 스키마의 단일 관리 지점(FK 순서 보장)
    FSQLiteDatabase Database;
};
#include "DatabaseSubsystem.h"

void UDatabaseSubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
    Super::Initialize(Collection);

    // ── 09의 4~5단계에서 ArchiveSubsystem에 있던 "열기" 코드를 그대로 이 자리로 옮긴다 ──
    const FString DbPath = FPaths::ProjectSavedDir() / TEXT("Archive.db");
    if (!Database.Open(*DbPath, ESQLiteDatabaseOpenMode::ReadWriteCreate))
    {
        // 열기 실패 시 크래시 대신 로그 — 이후 GetDb()->IsValid()가 false라 기능 시스템들이 안전하게 우회
        UE_LOG(LogTemp, Error, TEXT("[DB] 열기 실패: %s"), *DbPath);
        return;
    }
    Database.Execute(TEXT("PRAGMA foreign_keys = ON;"));   // FK 강제 — 연결(세션)마다 켜야 함
    CreateTables();
    UE_LOG(LogTemp, Log, TEXT("[DB] 준비 완료: %s"), *DbPath);
}

void UDatabaseSubsystem::Deinitialize()
{
    if (Database.IsValid()) Database.Close();   // 열었던 곳에서 닫는다
    Super::Deinitialize();
}

void UDatabaseSubsystem::CreateTables()
{
    // ① archives — 09 5단계의 CREATE TABLE SQL을 "그대로 잘라" 이 자리로 옮긴다(내용 무변경).
    //    Database.Execute(TEXT("CREATE TABLE IF NOT EXISTS archives ( ... 09 그대로 ... );"));

    // ② player — 이 문서 신규(스키마 정본: 05 문서와 컬럼 동일).
    //    CHECK (Id = 1): Id가 1이 아닌 행은 DB가 거부 → "계정은 1행" 규칙을 스키마가 강제
    Database.Execute(TEXT(
        "CREATE TABLE IF NOT EXISTS player ("
        "  Id                  INTEGER PRIMARY KEY CHECK (Id = 1),"
        "  Gold                INTEGER NOT NULL DEFAULT 0,"
        "  Shards              INTEGER NOT NULL DEFAULT 0,"
        "  AccountFocusSeconds INTEGER NOT NULL DEFAULT 0,"
        "  NextInstanceId      INTEGER NOT NULL DEFAULT 1,"
        "  NextGuildId         INTEGER NOT NULL DEFAULT 1,"
        "  Secured             BLOB"          // 보호 값 3종의 암호화 사본(8단계에서 채움)
        ");"));

    // 기본 행 보장 — 이미 1행이 있으면 OR IGNORE가 조용히 무시(첫 실행에만 삽입됨)
    Database.Execute(TEXT("INSERT OR IGNORE INTO player (Id) VALUES (1);"));
}
💡

왜 테이블 생성이 문지기의 일인가 — 테이블 만들기는 "값 쓰기"(각 주인의 일)가 아니라 그릇 준비(DB 인프라)입니다. 열기 직후 한곳에서 전부 만들면 ① 기능 시스템들이 깨어날 땐 테이블 존재가 보장되고 ② FK로 엮인 테이블(예: owned_abilities→owned_characters)의 생성 순서를 한 파일에서 관리하며 ③ 이후 PRAGMA user_version 마이그레이션도 이곳 하나만 고치면 됩니다.

ArchiveSubsystem 수술 — DB 소유를 내려놓기
09의 임시 겸직 해소: 이제 archives 기능만 전담

09에서 UArchiveSubsystem이 갖고 있던 DB 열닫 코드를 제거하고, 문지기에게 핸들을 빌리도록 바꿉니다. 바뀌는 곳은 세 군데뿐입니다.

// [제거] FSQLiteDatabase Database;      ← 더는 DB를 소유하지 않는다
// [추가] 문지기에게 빌린 핸들 — 소유 아님, 널 체크 후 사용
FSQLiteDatabase* Db = nullptr;

// (SQLiteDatabase.h 포함은 전방 선언 class FSQLiteDatabase; 로 바꿔도 됨 — 포인터만 들고 있으므로)
void UArchiveSubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
    Super::Initialize(Collection);

    // [제거] Open / PRAGMA / CREATE TABLE 코드 전부 — 1단계의 문지기로 이관 완료

    // [추가] 의존성 선언 — 문지기를 "먼저" 초기화시키고 핸들을 빌린다.
    //        이게 없으면 초기화 순서 복불복 → DB가 안 열린 채 이 시스템이 돌 수 있다
    UDatabaseSubsystem* Gate = Collection.InitializeDependency<UDatabaseSubsystem>();
    Db = Gate ? Gate->GetDb() : nullptr;
}

// Deinitialize()의 Database.Close()도 [제거] — 닫기는 문지기의 일.
// 나머지 함수(CreateArchive 등)의 Database. 를 Db-> 로 일괄 치환하고,
// 함수 첫머리에 if (!Db || !Db->IsValid()) return; 안전 체크를 넣는다.

수술 후 빌드 → 실행해서 09의 archives 기능(기록 넣고 읽기)이 그대로 동작하는지 먼저 확인하세요 — 구조만 바뀌고 행동은 같아야 정상입니다.

EconomySubsystem 뼈대 만들기
상태의 쓰기 주인 — 문지기 의존은 Archive와 똑같은 한 줄

클래스 위저드에서 부모 GameInstanceSubsystem으로 생성합니다(절차는 『C++ 클래스 생성 · 빌드 워크플로우』). 2단계의 Archive와 완전히 같은 방식으로 문지기에 의존합니다 — 이 대칭이 분리의 이점입니다.

#pragma once
#include "Subsystems/GameInstanceSubsystem.h"
#include "EconomySubsystem.generated.h"

class FSQLiteDatabase;   // 전방 선언 — 포인터만 들므로 헤더 포함은 .cpp에서(컴파일 시간 절약)

UCLASS()
class UEconomySubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()
public:
    // 엔진이 게임 시작 시 1회 호출 — 여기서 DB 핸들 확보 + 로드
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;

    // 상태 변경 창구 — 다른 시스템·BP는 이 함수들만 부른다(직접 쓰기 금지)
    UFUNCTION(BlueprintCallable) void AddGold(int64 Delta);          // 증감 겸용(음수 = 차감)
    UFUNCTION(BlueprintCallable) void AddFocusSeconds(int32 Seconds); // 세션 종료 시 공부 시간 적립

    // 조회 — BlueprintPure: 실행 핀 없이 값만 읽는 노드가 됨
    UFUNCTION(BlueprintPure) int64 GetGold() const { return Gold; }
    UFUNCTION(BlueprintPure) int32 GetAccountLevel() const
    { return (int32)(AccountFocusSeconds / 3600); }  // 레벨 = 파생값. 저장 안 함(정직한 레벨)

private:
    void LoadPlayer();   // 시작 시 1회 — DB → 멤버(8단계에서 검증·복원 추가)
    void SavePlayer();   // 값 변경 직후 — 멤버 → DB(UPDATE 1행)

    // 메모리 캐시 — 진실은 DB, 이건 세션 동안의 사본(RUNTIME)
    int64 Gold = 0;
    int32 Shards = 0;
    int64 AccountFocusSeconds = 0;

    FSQLiteDatabase* Db = nullptr;   // 문지기에게 빌린 핸들(소유 아님)
};
#include "EconomySubsystem.h"
#include "DatabaseSubsystem.h"         // 문지기 — GetDb()
#include "SQLiteDatabase.h"            // FSQLiteDatabase 실체(전방 선언 해소)

void UEconomySubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
    Super::Initialize(Collection);

    // 1) 의존성 선언 — 문지기를 "먼저" 초기화시키고 받아온다(2단계 Archive와 동일 패턴)
    UDatabaseSubsystem* Gate = Collection.InitializeDependency<UDatabaseSubsystem>();

    // 2) 핸들 확보 + 첫 로드. IsValid 체크: DB 열기 실패 시 크래시 대신 0값으로 시작
    Db = Gate ? Gate->GetDb() : nullptr;
    if (Db && Db->IsValid()) LoadPlayer();
}
4
LoadPlayer — 값 읽어오기 (1차: 평문만)
SELECT 1행 → 멤버. Secured 검증은 8단계에서 얹는다

먼저 평문만 읽는 버전으로 배관을 뚫습니다(09의 리듬 — 쉬운 것부터 왕복 성공). 봉인 검증은 8단계에서 이 함수를 고쳐 완성합니다.

#include "SQLitePreparedStatement.h"   // 빠지면 컴파일 실패(09 트러블슈팅과 동일)

void UEconomySubsystem::LoadPlayer()
{
    // 준비된 문(Prepared Statement): SQL을 미리 컴파일해 두고 실행 — 09에서 쓴 것과 같은 도구
    FSQLitePreparedStatement Stmt;
    Stmt.Create(*Db, TEXT("SELECT Gold, Shards, AccountFocusSeconds FROM player WHERE Id = 1;"));

    // Step() == Row: 결과 행이 하나 나왔다는 뜻. player는 1행이라 while 불필요
    if (Stmt.Step() == ESQLitePreparedStatementStepResult::Row)
    {
        Stmt.GetColumnValueByIndex(0, Gold);                 // 컬럼 인덱스는 SELECT 나열 순서(0부터)
        Stmt.GetColumnValueByIndex(1, Shards);
        Stmt.GetColumnValueByIndex(2, AccountFocusSeconds);
    }
    Stmt.Destroy();   // 준비된 문은 명시적으로 정리(핸들 누수 방지)

    UE_LOG(LogTemp, Log, TEXT("[Economy] Load: Gold=%lld Focus=%llds Lv=%d"),
           Gold, AccountFocusSeconds, GetAccountLevel());   // 출력 로그로 로드 확인용
}
5
상태 변경 + SavePlayer — UPDATE 배관
값이 바뀌는 즉시 1행을 덮어쓴다
void UEconomySubsystem::AddGold(int64 Delta)
{
    // 음수 방지: 차감이 잔고를 넘으면 0에서 멈춘다(빚 없음 — 정직한 경제)
    Gold = FMath::Max<int64>(0, Gold + Delta);
    SavePlayer();                     // 변화 즉시 저장 — 유실 창 제거(02 설계 결정)
}

void UEconomySubsystem::AddFocusSeconds(int32 Seconds)
{
    if (Seconds <= 0) return;         // 시간은 앞으로만 흐른다 — 음수 적립은 버그 신호
    AccountFocusSeconds += Seconds;   // 레벨은 이 값에서 파생될 뿐, 따로 저장하지 않는다
    SavePlayer();
}

void UEconomySubsystem::SavePlayer()
{
    if (!Db || !Db->IsValid()) return;

    FSQLitePreparedStatement Stmt;
    // ?1 ?2 ?3 = 바인딩 자리 — 문자열 조립 대신 바인딩을 쓰면 이스케이프 사고가 원천 차단된다
    Stmt.Create(*Db, TEXT(
        "UPDATE player SET Gold = ?1, Shards = ?2, AccountFocusSeconds = ?3 WHERE Id = 1;"));
    Stmt.SetBindingValueByIndex(1, Gold);
    Stmt.SetBindingValueByIndex(2, Shards);
    Stmt.SetBindingValueByIndex(3, AccountFocusSeconds);
    Stmt.Execute();                   // UPDATE는 결과 행이 없으니 Execute 한 번이면 끝
    Stmt.Destroy();
}

7
RTSecured — 봉인/개봉 함수 만들기
FAES 암호화 + MAGIC/CRC 무결성 — 05 명세와 동일 코드

보호 값 3종을 봉인(암호화 BLOB로)하고 개봉(복호화+진위 검사)하는 자유 함수 2개입니다. GKey의 32바이트는 직접 무작위로 채우세요 — 이 키가 유출되면 방어가 뚫립니다.

#pragma once
#include "CoreMinimal.h"

// player 행의 치트 유인 값 3개를 하나로 묶은 그릇 — 이 값들만 Secured로 보호한다
struct FSecuredAccount
{
    int64 Gold = 0;
    int32 Shards = 0;
    int64 AccountFocusSeconds = 0;
};

// 봉인: 값 → (MAGIC+값+CRC) 평문 → AES 암호화 → BLOB. player.Secured에 저장
TArray<uint8> SealAccount(const FSecuredAccount& In);

// 개봉: BLOB → AES 복호화 → MAGIC·CRC 검사. 성공=true(Out 채움), 실패=false(변조/손상)
bool OpenAccount(const TArray<uint8>& Blob, FSecuredAccount& Out);
#include "RTSecured.h"
#include "Misc/AES.h"        // FAES::EncryptData/DecryptData (AES-256) — Core 모듈, Build.cs 추가 불필요
#include "Misc/Crc.h"        // FCrc::MemCrc32 — 무결성 검사
#include "Serialization/BufferArchive.h"
#include "Serialization/MemoryReader.h"

// 빌드에 심는 32바이트(256비트) 대칭키 — 반드시 무작위 값으로 직접 채울 것.
// 키가 클라에 있으므로 "완전 방어"는 아니다(서버 없는 싱글의 적정선 — 03 문서 참조)
static const uint8 GKey[32] = { /* 32개 무작위 바이트 */ };
static constexpr uint32 SEAL_MAGIC = 0x52544131;   // 'RTA1' — 복호화 성공 판정 서명

static void MakeKey(FAES::FAESKey& K) { FMemory::Memcpy(K.Key, GKey, 32); }

TArray<uint8> SealAccount(const FSecuredAccount& In)
{
    // 1) 평문 페이로드 = MAGIC | Gold | Shards | Seconds | CRC32(앞 24바이트)
    FBufferArchive Ar;
    uint32 Magic = SEAL_MAGIC;
    int64 G = In.Gold; int32 S = In.Shards; int64 F = In.AccountFocusSeconds;
    Ar << Magic << G << S << F;                       // 4+8+4+8 = 24바이트
    uint32 Crc = FCrc::MemCrc32(Ar.GetData(), Ar.Num());
    Ar << Crc;                                        // 무결성 태그(변조 감지)를 뒤에 붙임

    // 2) AES는 16바이트 블록 단위 — 0으로 채워 배수를 맞춘 뒤 제자리 암호화
    const int32 Pad = (16 - (Ar.Num() % 16)) % 16;
    for (int32 i = 0; i < Pad; ++i) { uint8 Z = 0; Ar << Z; }
    FAES::FAESKey K; MakeKey(K);
    FAES::EncryptData(Ar.GetData(), Ar.Num(), K);

    TArray<uint8> Out; Out.Append(Ar.GetData(), Ar.Num());
    return Out;                                       // → player.Secured BLOB
}

bool OpenAccount(const TArray<uint8>& Blob, FSecuredAccount& Out)
{
    // 블롭이 없거나 블록 배수가 아니면 손상 — 개봉 실패
    if (Blob.Num() == 0 || Blob.Num() % 16 != 0) return false;

    TArray<uint8> Buf = Blob;                        // 원본 보존을 위해 복사본을 복호화
    FAES::FAESKey K; MakeKey(K);
    FAES::DecryptData(Buf.GetData(), Buf.Num(), K); // 키가 틀리면 쓰레기 바이트가 나온다

    FMemoryReader Ar(Buf);
    uint32 Magic = 0; Ar << Magic;
    if (Magic != SEAL_MAGIC) return false;           // 서명 불일치 = 키 틀림/블롭 변조

    int64 G = 0; int32 S = 0; int64 F = 0;
    Ar << G << S << F;
    uint32 StoredCrc = 0; Ar << StoredCrc;
    const int32 PayloadBytes = 24;                    // MAGIC(4)+G(8)+S(4)+F(8) — 봉인 때와 동일 범위
    if (FCrc::MemCrc32(Buf.GetData(), PayloadBytes) != StoredCrc) return false; // 무결성 실패

    Out.Gold = G; Out.Shards = S; Out.AccountFocusSeconds = F;
    return true;                                     // 진짜 값 복원 성공
}
8
저장·로드에 봉인 연결 — 자동 복원 완성
저장은 평문+블롭 동시, 로드는 3갈래 판정

4·5단계의 LoadPlayer/SavePlayer를 최종본으로 고칩니다. 핵심: 저장은 평문과 블롭을 같은 UPDATE로(어긋난 채 남지 않게), 로드는 개봉 → 비교 → 3갈래.

#include "RTSecured.h"   // SealAccount / OpenAccount (7단계)

void UEconomySubsystem::SavePlayer()
{
    if (!Db || !Db->IsValid()) return;

    // 현재 값 3종을 봉인 — 평문과 블롭이 "같은 순간의 값"이 되도록 저장 직전에 만든다
    const TArray<uint8> Sealed = SealAccount({ Gold, Shards, AccountFocusSeconds });

    FSQLitePreparedStatement Stmt;
    Stmt.Create(*Db, TEXT(
        "UPDATE player SET Gold = ?1, Shards = ?2, AccountFocusSeconds = ?3, Secured = ?4 WHERE Id = 1;"));
    Stmt.SetBindingValueByIndex(1, Gold);
    Stmt.SetBindingValueByIndex(2, Shards);
    Stmt.SetBindingValueByIndex(3, AccountFocusSeconds);
    Stmt.SetBindingValueByIndex(4, Sealed);          // TArray<uint8> → BLOB 바인딩
    Stmt.Execute();
    Stmt.Destroy();
}

void UEconomySubsystem::LoadPlayer()
{
    int64 PlainGold = 0; int32 PlainShards = 0; int64 PlainFocus = 0;
    TArray<uint8> SecuredBlob;

    FSQLitePreparedStatement Stmt;
    Stmt.Create(*Db, TEXT(
        "SELECT Gold, Shards, AccountFocusSeconds, Secured FROM player WHERE Id = 1;"));
    if (Stmt.Step() == ESQLitePreparedStatementStepResult::Row)
    {
        Stmt.GetColumnValueByIndex(0, PlainGold);
        Stmt.GetColumnValueByIndex(1, PlainShards);
        Stmt.GetColumnValueByIndex(2, PlainFocus);
        Stmt.GetColumnValueByIndex(3, SecuredBlob);   // BLOB → TArray<uint8>
    }
    Stmt.Destroy();

    // [갈래 0] 블롭이 아직 없음(첫 실행 — 한 번도 SavePlayer 안 함) → 평문 채택 후 즉시 봉인 생성
    if (SecuredBlob.Num() == 0)
    {
        Gold = PlainGold; Shards = PlainShards; AccountFocusSeconds = PlainFocus;
        SavePlayer();                                 // 지금 값으로 첫 봉인을 만들어 둔다
        return;
    }

    FSecuredAccount Real;
    if (!OpenAccount(SecuredBlob, Real))
    {
        // [갈래 1] 블롭 자체가 변조/손상 → 진짜 값을 알 수 없음 → 복원 불가.
        //          여기선 0으로 초기화 + 경고(제품에선 안내 팝업/백업 복구 UI로 확장)
        UE_LOG(LogTemp, Warning, TEXT("[Economy] Secured 손상 — 상태를 초기화합니다"));
        Gold = 0; Shards = 0; AccountFocusSeconds = 0;
        SavePlayer();
        return;
    }

    if (Real.Gold != PlainGold || Real.Shards != PlainShards || Real.AccountFocusSeconds != PlainFocus)
    {
        // [갈래 2] 평문만 편집됨(DB Browser 치트) → 블롭의 진짜 값 채택 = 자동 복원 + 알림
        UE_LOG(LogTemp, Warning, TEXT("[Economy] 비정상 데이터 감지 — 마지막 정상 값으로 복구"));
        Gold = Real.Gold; Shards = Real.Shards; AccountFocusSeconds = Real.AccountFocusSeconds;
        SavePlayer();                                 // 평문을 즉시 재기록해 정합성 회복
        return;
    }

    // [갈래 3] 일치 → 정상. 평문 그대로 채택
    Gold = PlainGold; Shards = PlainShards; AccountFocusSeconds = PlainFocus;
}
⚠️
빼먹기 쉬운 함정 — 앞으로 player에 보호 값을 새로 추가하면(예: 다른 재화) 봉인 페이로드에도 같이 넣어야 합니다. 평문만 갱신하고 Secured를 빼먹으면, 다음 로드에서 자동 복원이 옛 값으로 되돌려버립니다(갈래 2가 "정상 저장"을 치트로 오인).
04 — 검증

치트 시뮬레이션까지 눈으로 확인

1
DB Browser로 값·블롭 확인
게임을 한 번 실행 후 종료 → DB Browser for SQLite로 [프로젝트]/Saved/Archive.db 열기 → player 테이블에 1행, Gold 값과 Secured 칸의 BLOB(바이너리 표시)이 보여야 합니다. archives 테이블도 그대로 있어야 합니다(2단계 수술이 행동을 안 바꿨다는 증거).
2
치트 시뮬레이션 — 직접 위조해 본다
① 게임 종료 상태에서 DB Browser로 Gold를 999999로 수정 → Write Changes(저장) ② 게임 실행 ③ 출력 로그에 비정상 데이터 감지 — 마지막 정상 값으로 복구가 찍히고 Get Gold가 원래 값이면 성공. DB Browser로 다시 열면 Gold도 원래 값으로 돌아와 있습니다.
3
블롭 파괴 갈래도 확인 (선택)
Secured 칸을 NULL로 지우고 실행하면 [갈래 0](첫 실행 취급 — 평문 채택 후 재봉인)으로, 블롭 바이트를 일부 수정하고 실행하면 [갈래 1](손상 — 초기화)로 흐릅니다. 각 갈래의 로그를 확인하세요.
