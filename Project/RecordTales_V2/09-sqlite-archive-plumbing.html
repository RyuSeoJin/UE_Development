01 — 목표 & 시작하기

왜 "한 컬럼"부터인가

SQLite를 붙일 때 초보자가 가장 많이 막히는 지점은 스키마가 아니라 배관입니다 — 플러그인이 켜졌는지, #include가 통과하는지, 빌드가 반영됐는지, 블루프린트에서 함수가 보이는지. 이 네 가지가 한 번에 얽히면 어디가 잘못됐는지 알 수 없습니다.

그래서 데이터는 title 문자열 하나로 최소화하고, "넣고 → 다시 읽어온다"는 왕복만 통과시킵니다. 이 왕복이 되면 배관은 뚫린 것이고, 이후 컬럼을 늘리는 일은 SQL CREATE TABLE·INSERT 문자열만 고치면 됩니다.

🚀

① 전제 조건
· C++ 프로젝트여야 합니다(블루프린트 전용이면 0단계에서 전환). C++ 클래스 생성·Build.cs·컴파일의 공통 절차는 중앙 문서 「C++ 클래스 워크플로우」(ue-cpp-class-workflow)를 따릅니다.
· 이 문서의 서브시스템은 UGameInstanceSubsystem 파생입니다 — 수명·자동 호출 개념은 중앙 문서 「GameInstance 서브시스템」(ue-gameinstance-subsystem)에 있습니다.
· 검증용으로 DB Browser for SQLite(무료)를 설치해 두면 데이터를 눈으로 확인할 수 있습니다(7단계).

② 만들 파일 — 서브시스템 클래스 1쌍만 추가합니다. 3단계의 C++ 클래스 위저드로 생성하면 .h .cpp가 함께 만들어집니다(분리 레이아웃이면 각각 Public/·Private/ 폴더로 — 3단계 참고).

파일부모 클래스위저드 검색어역할
Archive/ArchiveSubsystem.h/.cppUGameInstanceSubsystemGameInstanceSubsystem블루프린트와 SQLite 사이의 유일한 다리 — DB 열기/닫기 + CreateArchive/GetArchives

③ 코드 외 작업 — 코드만으로는 끝나지 않습니다. 아래 두 가지는 반드시 에디터/설정에서 해야 합니다(각각 1·2단계).

작업어디서왜 필요한가
플러그인 SQLiteSupport 활성화편집 > 플러그인이것만 켜면 SQLiteCore 플러그인도 자동으로 함께 켜짐(.uproject엔 이거면 충분)
Build.cs에 SQLiteCore(+SQLiteSupport) 모듈 추가소스 편집헤더 include만이 아니라 링크까지 되려면 SQLiteCore 필수
⚠️

④ 컴파일 주의 — Build.cs 수정과 새 UCLASS/USTRUCT 추가는 라이브 코딩으로 반영되지 않습니다. 에디터를 완전히 닫고 Visual Studio에서 빌드하세요. 이걸 모르면 "코드를 고쳤는데 아무 일도 안 일어나는" 상태에 빠집니다(6단계에서 다시 강조).

프로젝트 폴더에 Source/ 폴더와 .sln 파일이 있나요?

✅

있다 — 이미 C++ 프로젝트입니다. 1단계로 넘어가세요.

🔧

없다 — 블루프린트 전용입니다. 아래로 먼저 전환하세요.

Visual Studio 2022 설치 시 "C++를 사용한 게임 개발" 워크로드를 반드시 체크해야 합니다. 없으면 빌드가 되지 않습니다.

그다음 에디터에서 도구 > 새로운 C++ 클래스 → None → 아무 이름 → 생성. 이 순간 프로젝트가 C++로 전환되고 Source/와 .sln이 생깁니다. 되돌릴 수는 없지만, 기존 블루프린트는 그대로 다 씁니다.

🔎

확인 — 프로젝트 루트에 [프로젝트명].sln이 보이고, 더블클릭하면 Visual Studio가 열린다.

1
플러그인 켜기
설치가 아니라 스위치입니다 — 이미 엔진에 있습니다

에디터에서 편집 > 플러그인 → 검색창에 sqlite → SQLite Support를 체크 → 에디터 재시작. 이것 하나만 켜면 됩니다 — SQLite Support가 저수준 SQLiteCore를 의존성으로 함께 끌어옵니다.

🧭

플러그인은 SQLiteSupport 하나면 됩니다 — 이걸 켜면 SQLiteCore 플러그인도 의존성으로 자동 활성화되어, .uproject엔 SQLiteSupport만 있어도 됩니다. 단, C++에서 직접 쓰려면 Build.cs에 SQLiteCore 모듈을 넣어야 링크됩니다(2단계). "플러그인 활성화"와 "Build.cs 모듈 의존"은 별개예요.

[프로젝트명].uproject를 메모장으로 열면 Plugins에 아래 항목이 들어가 있어야 합니다.

"Plugins": [
    { "Name": "SQLiteSupport", "Enabled": true }
]
🔎

확인 — 플러그인 창에서 SQLite Support가 체크돼 있고, 재시작 후 위 항목이 .uproject의 Plugins에 보인다.

2
Build.cs에 모듈 추가
이게 있어야 #include가 통과합니다

Source/[프로젝트명]/[프로젝트명].Build.cs를 열고 의존 모듈 목록에 "SQLiteCore"와 "SQLiteSupport"를 더합니다.

// PublicDependencyModuleNames = 이 모듈이 공개적으로 의존하는 엔진 모듈 목록.
// 여기에 이름을 넣어야 해당 모듈의 public 헤더를 #include 할 수 있다.
PublicDependencyModuleNames.AddRange(new string[]
{
    "Core", "CoreUObject", "Engine", "InputCore",
    "SQLiteCore", "SQLiteSupport"      // ← 이번에 추가하는 두 모듈
});
🧭

왜 SQLiteCore가 필요한가 — 컴파일 vs 링크
· 컴파일(헤더 찾기): SQLiteSupport만 있어도 SQLiteDatabase.h·SQLitePreparedStatement.h가 인클루드 경로에 올라와 #include는 통과합니다.
· 링크(구현 찾기): 하지만 FSQLiteDatabase·FSQLitePreparedStatement의 실제 구현은 SQLiteCore 모듈에 들어 있어, 이 모듈을 안 넣으면 unresolved external symbol(LNK2019)이 납니다. 그래서 SQLiteCore를 반드시 넣습니다(둘 다 넣으면 확실).

⚠️
Build.cs를 고치면 라이브 코딩으로는 반영되지 않습니다. 에디터를 완전히 닫고 Visual Studio에서 빌드해야 모듈 링크가 갱신됩니다. 이걸 모르면 다음 단계에서 Cannot open include file: 'SQLiteDatabase.h'가 계속 납니다.
3
서브시스템 클래스 만들기
블루프린트와 SQL 사이의 유일한 다리

에디터에서 도구 > 새로운 C++ 클래스 → 모든 클래스 탭 → GameInstanceSubsystem 검색해서 부모로 선택 → 이름 ArchiveSubsystem.

UGameInstanceSubsystem이라서 게임 시작 시 Initialize가, 종료 시 Deinitialize가 자동으로 불립니다. 덕분에 DB를 언제 열고 닫을지 직접 관리할 필요가 없습니다 — 서브시스템의 수명이 곧 DB의 수명입니다.

🗂️

왜 .h는 Public/, .cpp는 Private/에 생기나 — 지금 모듈이 Public/Private 분리 레이아웃이면 위저드가 자동으로 나눠 넣습니다. .h(선언 = 남이 봐야 하는 "인터페이스")는 Source/[프로젝트명]/Public/에, .cpp(구현 = 모듈 내부 사정)는 Source/[프로젝트명]/Private/에 갑니다. 언리얼 빌드 툴(UBT)이 Public/은 다른 모듈까지, Private/는 이 모듈만 인클루드 경로에 올려 주므로, .cpp에서 #include "ArchiveSubsystem.h"는 폴더 경로 없이도 통과합니다. Public/·Private/ 폴더가 아예 없는 "플랫 레이아웃"이면 둘 다 Source/[프로젝트명]/에 나란히 생깁니다.

🔎

확인 — ArchiveSubsystem.h와 ArchiveSubsystem.cpp가 둘 다 생겼다. 분리 레이아웃이면 각각 Public/·Private/에, 플랫이면 같은 폴더에. 헤더만 보이면 Private/ 폴더를 열어 .cpp를 확인하세요 — 사라진 게 아니라 형제 폴더에 있습니다.

4
헤더 작성
블루프린트가 보게 될 것들

블루프린트에 노출할 구조체 1개(FArchiveRow)와 함수 2개(CreateArchive·GetArchives)를 선언합니다. 아래를 ArchiveSubsystem.h(분리 레이아웃이면 Public/의 것)에 붙여 넣으세요.

// ArchiveSubsystem.h — 블루프린트와 SQLite 사이의 유일한 다리
#pragma once
#include "CoreMinimal.h"
#include "Subsystems/GameInstanceSubsystem.h"   // UGameInstanceSubsystem 선언
#include "SQLiteDatabase.h"                      // FSQLiteDatabase — 1·2단계로 켠 플러그인 헤더
#include "ArchiveSubsystem.generated.h"          // UHT 생성 — 항상 맨 마지막 include

// 아카이브 1건 = 한 행. BlueprintType이라 이 구조체를 BP에서 변수·핀으로 쓸 수 있다.
USTRUCT(BlueprintType)
struct FArchiveRow
{
    GENERATED_BODY()

    // BlueprintReadOnly = BP에서 읽기만(쓰기 금지). Category = 디테일 패널 그룹 이름.
    UPROPERTY(BlueprintReadOnly, Category="Archive")
    int32 ArchiveId = 0;          // archives 테이블의 PRIMARY KEY 값

    UPROPERTY(BlueprintReadOnly, Category="Archive")
    FString Title;                // title 컬럼 — 지금 단계의 유일한 데이터
};

UCLASS()
class RECORDTALES_API UArchiveSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()

public:
    // 게임 시작 시 엔진이 자동 호출 → 여기서 DB를 연다.
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;
    // 게임 종료 시 자동 호출 → 여기서 DB를 닫는다.
    virtual void Deinitialize() override;

    // 아카이브 1건 추가. 반환값 = 방금 만들어진 archive_id (실패 시 -1).
    UFUNCTION(BlueprintCallable, Category="Archive")
    int32 CreateArchive(const FString& Title);

    // 저장된 아카이브 전체를 최신순으로 반환.
    UFUNCTION(BlueprintCallable, Category="Archive")
    TArray<FArchiveRow> GetArchives();

private:
    FSQLiteDatabase Db;   // 이 서브시스템이 소유하는 DB 핸들 (수명 = 게임 세션)
};
⚠️
가장 흔한 컴파일 에러 — RECORDTALES_API. 이 매크로는 본인 모듈 이름을 대문자로 바꾸고 _API를 붙인 것이어야 합니다. 모듈이 MyGame이면 MYGAME_API, Proj_RecordTales면 PROJ_RECORDTALES_API입니다. 3단계에서 생성된 .h에 이미 올바른 매크로가 적혀 있으니 그걸 그대로 쓰세요(위 예시의 RECORDTALES_API를 무작정 복사하지 말 것).
5
구현 작성
SQL이 사는 유일한 곳

DB 열기/닫기와 INSERT/SELECT를 구현합니다. SQL 문자열이 등장하는 곳은 이 .cpp 하나뿐입니다. 아래를 ArchiveSubsystem.cpp(분리 레이아웃이면 Private/의 것)에 붙여 넣으세요. 헤더(4단계)와 달리 여기선 쿼리를 실행하므로 SQLitePreparedStatement.h도 include합니다(빠지면 FSQLitePreparedStatement를 못 찾아 컴파일 실패).

// ArchiveSubsystem.cpp — SQL이 사는 유일한 파일
#include "ArchiveSubsystem.h"
#include "SQLitePreparedStatement.h"   // FSQLitePreparedStatement / ESQLitePreparedStatementStepResult
#include "Misc/Paths.h"                // FPaths::ProjectSavedDir()

void UArchiveSubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
    Super::Initialize(Collection);   // 부모 초기화 먼저 — 관례상 항상 맨 위

    // DB 파일은 반드시 Saved/ 아래에 둔다 (아래 주의 박스 참고).
    const FString Path = FPaths::ProjectSavedDir() / TEXT("Archive.db");
    UE_LOG(LogTemp, Log, TEXT("DB 경로: %s"), *Path);   // 로그로 실제 경로 확인

    // ReadWriteCreate = 없으면 새로 만들고, 있으면 읽기/쓰기로 연다.
    if (!Db.Open(*Path, ESQLiteDatabaseOpenMode::ReadWriteCreate))
    {
        UE_LOG(LogTemp, Error, TEXT("DB 열기 실패: %s"), *Db.GetLastError());
        return;                       // 열기 실패 시 이후 함수는 IsValid()==false로 안전하게 빠진다
    }

    Db.Execute(TEXT("PRAGMA foreign_keys = ON;"));   // 외래키 강제 (지금은 없지만 습관)

    // IF NOT EXISTS = 이미 있으면 그대로 둔다 → 컬럼을 늘려도 기존 테이블은 안 바뀜(3섹션 참고).
    Db.Execute(TEXT(
        "CREATE TABLE IF NOT EXISTS archives ("
        "  archive_id INTEGER PRIMARY KEY AUTOINCREMENT,"   // 자동 증가 번호
        "  title      TEXT NOT NULL"                          // 지금 단계의 유일한 컬럼
        ");"
    ));

    UE_LOG(LogTemp, Log, TEXT("DB 준비 완료"));
}

void UArchiveSubsystem::Deinitialize()
{
    Db.Close();                     // 세션 종료 시 파일 핸들 정리
    Super::Deinitialize();
}

int32 UArchiveSubsystem::CreateArchive(const FString& Title)
{
    if (!Db.IsValid()) return -1;   // DB가 안 열렸으면 -1

    // PreparedStatement = SQL 틀을 미리 컴파일하고 값만 바인딩 → SQL 인젝션 안전.
    FSQLitePreparedStatement Stmt;
    if (!Stmt.Create(Db, TEXT("INSERT INTO archives (title) VALUES (?1);")))
    {
        UE_LOG(LogTemp, Error, TEXT("준비 실패: %s"), *Db.GetLastError());
        return -1;
    }

    Stmt.SetBindingValueByIndex(1, Title);   // ?1 자리에 Title을 끼워 넣음
    Stmt.Execute();                          // INSERT 실행

    return (int32)Db.GetLastInsertRowId();   // AUTOINCREMENT가 방금 준 번호
}

TArray<FArchiveRow> UArchiveSubsystem::GetArchives()
{
    TArray<FArchiveRow> Out;
    if (!Db.IsValid()) return Out;   // 빈 배열 반환

    FSQLitePreparedStatement Stmt;
    Stmt.Create(Db, TEXT(
        "SELECT archive_id, title FROM archives ORDER BY archive_id DESC;"));  // 최신순

    // Step()이 Row를 돌려주는 동안 = 결과 행이 남아 있는 동안 반복.
    while (Stmt.Step() == ESQLitePreparedStatementStepResult::Row)
    {
        FArchiveRow Row;
        Stmt.GetColumnValueByIndex(0, Row.ArchiveId);   // 0번 컬럼 = archive_id
        Stmt.GetColumnValueByIndex(1, Row.Title);       // 1번 컬럼 = title
        Out.Add(Row);
    }
    return Out;
}
🚫

평소엔 "빌드(Build)", "다시 빌드(Rebuild)"는 피하세요. Rebuild는 프로젝트를 통째로 지웠다 다시 만들어 오래 걸리고, 당신 코드와 무관한 모듈·플러그인까지 재컴파일하다 곁가지 에러를 부를 수 있습니다. 코드나 Build.cs를 고쳤을 때도 보통 Build로 충분합니다.
7
블루프린트에서 호출
여기서 SQL은 한 글자도 안 보입니다

아무 블루프린트(레벨 BP도 됩니다)에서 다음처럼 연결합니다.

핀 색: ▷ 흰=실행 순서 · ● 파랑=객체(서브시스템·구조체) · ● 청록=정수 · ● 분홍=문자열. Target 두 개는 Get Game Instance Subsystem(Class=Archive Subsystem) 하나로 채웁니다. Array Element(Archive Row)를 Break Archive Row로 풀어 Title만 Print String에 연결.

🔬

최종 확인 — 눈으로 보기. DB Browser for SQLite(무료)를 설치하고 [프로젝트]/Saved/Archive.db를 열어보세요. archives 테이블에 방금 넣은 행이 실제로 보입니다. 이 도구는 앞으로 계속 씁니다 — 코드가 의심스러울 때 데이터를 직접 눈으로 확인하는 게 디버깅 속도를 완전히 바꿉니다.

스키마를 바꿨을 때 — DB 초기화(개발 중)

왜 필요한가 — CREATE TABLE IF NOT EXISTS는 테이블이 이미 있으면 아무것도 하지 않습니다. 게다가 DB 파일은 껐다 켜도 디스크에 그대로 남으므로, CREATE 문에 새 컬럼을 넣어도 기존 테이블(옛 스키마)이 그대로라 새 컬럼이 안 생깁니다. 그래서 파일을 지워 "테이블이 아예 없는" 상태로 만든 뒤 다시 실행하면, CREATE 문이 새 스키마로 처음부터 테이블을 만듭니다.

순서할 일
1게임/에디터를 완전히 종료 (실행 중이면 파일이 잠겨 있고, DB는 Initialize에서만 열려서 껐다 켜야 반영됨)
2파일 탐색기로 [프로젝트]/Saved/ 폴더로 이동
3Saved/의 .db 파일 삭제(이 문서 예제는 Archive.db, 이름을 바꿨다면 그 파일). 옆에 *.db-wal·*.db-shm·*.db-journal이 있으면 함께 삭제 — 안 그러면 옛 데이터가 딸려올 수 있음
4게임을 다시 실행 → 파일이 없으니 CREATE 문이 새 스키마로 테이블 생성
5DB Browser로 열어 새 컬럼이 생겼는지 확인
⚠️

파일을 지우면 그 안의 데이터도 전부 사라집니다. 개발 중 테스트 데이터라 괜찮은 것이지, 실제 사용자 데이터엔 절대 이렇게 하면 안 됩니다. 파일을 안 지우고 그 테이블만 비우려면 DB Browser의 SQL 실행 탭에서 DROP TABLE archives; 실행 후 게임을 재실행해도 됩니다.

🧭

출시 후에는 파일 삭제 금지 — 사용자 데이터를 지울 수 없으니 마이그레이션을 합니다: PRAGMA user_version으로 스키마 버전을 관리하고, ALTER TABLE archives ADD COLUMN …으로 기존 데이터를 지키면서 컬럼을 추가합니다. "개발 중 = 파일 삭제, 출시 후 = ALTER TABLE" 두 갈래이고, 지금 단계에선 파일 삭제가 정답입니다.

🧭

이 배관이 본 채택으로 승격됨(2026-07-20) — 처음엔 실현 가능성을 증명하는 스파이크였지만, 이후 풀-DB 저장 전략이 확정되어 이 배관이 V2 영속 저장의 본체가 됐습니다: 기록(archives)뿐 아니라 상태(player·guilds·owned_*·buildings)까지 전부 이 DB의 테이블로 확장합니다. 전체 테이블 설계·확장 순서는 10 코드 & 변수 연결 지도가 기준입니다(07·03·05의 SaveGame 기반 기술은 후속 개정 대상).

🚪

구조 업데이트(2026-07-20) — 이 문서에서 UArchiveSubsystem이 맡던 DB 열닫·CREATE TABLE은 이후 11(플레이어 상태 저장) 1~2단계에서 전용 문지기 UDatabaseSubsystem으로 이관됐습니다. 이 문서의 절차(처음 배관 뚫기)는 그대로 유효하며, 최종 구조에선 "열닫·스키마 = 문지기 / archives 기능 = ArchiveSubsystem"으로 나뉩니다.
