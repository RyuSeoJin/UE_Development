데이터 3계층
구현 해설

ROW(밸런스 표) · DB(영속 상태) · RUNTIME(메모리 캐시)은 구현체가 서로 다릅니다 — ROW는 행 구조체 + DataTable 에셋 + CSV, DB는 SQLite 플러그인 + 문지기 서브시스템, RUNTIME은 서브시스템의 멤버 변수. 이 문서는 세 계층이 각각 무엇으로, 어떤 절차로 구현되는지를 한눈에 잇습니다. 계층별 상세 정본은 05 · 09 · 13 · 14 문서가 담당합니다.

ROW · DB · RUNTIME 구현 통합 해설 02-A 아키텍처 패턴
01 — 개념

계층마다 구현체가 다르다

세 계층은 "어디에 저장하느냐"만 다른 게 아니라 무엇으로 만들어지느냐가 다릅니다. ROW는 코드(구조체)와 에디터 에셋과 시트가 함께 만들고, DB는 플러그인과 서브시스템 한 쌍이 만들며, RUNTIME은 별도 저장 장치 없이 서브시스템 멤버로만 삽니다. 어떤 데이터를 어디 둘지 헷갈리면 "누가 이 값을 소유하는가" 하나로 판별합니다 — 기획이 소유하면 ROW, 플레이어의 진행이 소유하면 DB, 아무도 소유하지 않으면(매번 계산하거나 버리면) RUNTIME입니다.

계층구현체실체(파일 · 에셋)
ROW — 밸런스 표 행 구조체(FTableRowBase 상속) + DataTable 에셋 + CSV RTTables.h · 콘텐츠의 DT_* 에셋 11종 · 구글 시트
DB — 영속 상태 SQLite 플러그인 + DB 문지기 서브시스템 DatabaseSubsystem.h/.cpp · Saved/Archive.db(7테이블)
RUNTIME — 메모리 캐시 서브시스템 멤버 변수 + 파생 함수 RTTypes.h의 캐시 struct 6종 · 각 기능 서브시스템
🧭

핵심 용어. FTableRowBase = 이 구조체를 상속해야 에디터가 "DataTable의 행 타입"으로 인정하는 엔진 기반 구조체. DataTable 에셋(DT_*) = 행 구조체를 스키마 삼아 값(행)을 담는 에디터 에셋 — 값은 CSV 임포트로 채움. DB 문지기 = DB의 열기/닫기 · 스키마 생성 · 트랜잭션 창구만 전담하는 인프라 서브시스템(UDatabaseSubsystem, 게임 기능 0개). InitializeDependency = "저 서브시스템을 나보다 먼저 초기화해 달라"는 엔진 API — 기능 시스템이 깨어날 때 DB가 이미 준비돼 있음을 보장. 파생값 = 저장하지 않고 매번 계산하는 값(예: 레벨 = FocusSeconds / 3600) — 원천과 어긋날 여지를 없앰.

💡

이전 판과의 차이. V2는 USaveGame(.sav)과 UDeveloperSettings(CONFIG)를 폐기했습니다. 상태는 전부 DB 테이블 행으로, 밸런스는 전부 ROW 표로 갔습니다 — 저장소를 단일화해 이중 저장 정합성 걱정을 없애고, 여러 테이블을 고치는 작업을 트랜잭션 하나로 묶기 위해서입니다.

03 — 책임

누가 만들고, 누가 쓰나

같은 값을 두 곳에서 고치면 지도가 무너집니다. 계층마다 만드는 주체와 쓰는 주체를 하나로 고정합니다 — 나머지는 읽기만 합니다.

대상주인(생성 · 쓰기)근거
밸런스 값 (ROW 표의 모든 수치) 기획 — 구글 시트 코드는 읽기만. 튜닝 = 시트 수정 → CSV 재임포트, 컴파일 · DB 변경 없음.
행 구조체(F*Row) · enum · 캐시 struct 코드 — RTTables.h · RTTypes.h 표의 스키마는 코드가 정의. 프로퍼티명 = CSV 헤더 철자 일치가 임포트 계약.
Archive.db 열닫 · 스키마 · 트랜잭션 UDatabaseSubsystem (문지기) 열닫 주체가 둘이면 "닫힌 핸들에 쿼리" 사고 — 수명 관리 지점을 하나로.
테이블별 행 쓰기 (INSERT · UPDATE) 각 주인 서브시스템 player=Economy · guilds · 소분류=Guild · owned_*=Gacha · buildings=Estate · archives=Archive. 쓰기 주인은 테이블당 하나.
RUNTIME 캐시 · 파생값 각 서브시스템 멤버 · 함수 저장 장치 없음. 값이 바뀌는 함수 안에서 멤버 갱신 + DB 반영을 함께(어긋날 틈 제거). 파생값은 함수로만.
04 — 구현 절차

계층별 3단계 파이프라인

ROW — 구조체 → 에셋 → 캐시

1
행 구조체 선언 — RTTables.h
FTableRowBase를 상속한 USTRUCT(BlueprintType) 11종. 프로퍼티는 UPROPERTY(EditAnywhere)로 열어 에디터 · CSV가 채울 수 있게 합니다. 프로퍼티 이름은 CSV 헤더와 철자까지 동일해야 임포트가 매칭됩니다.
2
에셋 생성 + CSV 임포트
콘텐츠 브라우저 우클릭 → Miscellaneous ▸ Data Table → 행 타입으로 F*Row 선택 → DT_* 에셋 생성. 값은 구글 시트 → CSV 내보내기 → 에셋에 재임포트로 채웁니다(전제: 1단계 구조체가 컴파일된 상태. 결과: 에셋을 열면 표에 행이 보임).
3
로드 캐시 — TMap<FName, TArray<Row>>
서브시스템 Initialize에서 표 전체를 읽어 그룹 키(AbilityId · BuildingId)로 묶고 인덱스(StepIndex · Level)로 정렬해 캐시합니다. RowName 문자열 파싱에 의존하지 않는 것이 규약입니다.

DB — 배관 → 문지기 → 2줄 관례

1
배관 (1회) — 플러그인 + Build.cs
편집 > 플러그인에서 SQLiteSupport 체크(엔진 내장 스위치 — SQLiteCore는 의존성으로 자동 활성화) → 에디터 재시작. 그리고 Build.cs에 "SQLiteCore", "SQLiteSupport" 모듈 추가. 배관 왕복 검증 절차는 09 문서(한 컬럼 왕복) 담당.
2
문지기 — UDatabaseSubsystem
UGameInstanceSubsystem 파생이라 앱 시작 · 종료 시 Initialize/Deinitialize가 자동 호출됩니다 — 서브시스템 수명 = DB 수명. 열기(Saved/Archive.db) → PRAGMA foreign_keys = ON → CREATE TABLE IF NOT EXISTS 7종(FK 순서) → 트랜잭션 3함수 제공. 통합 정본은 13 문서.
3
기능 시스템 연결 — 2줄 관례
각 기능 서브시스템 Initialize에서 ① Collection.InitializeDependency<UDatabaseSubsystem>()(문지기 먼저 보장) ② GetDb()(핸들 빌리기). 이후 자기 소유 테이블만 prepared statement로 INSERT/UPDATE. 핸들 소유 · Close()는 금지.

RUNTIME — 그릇 → 로드 → 쓰기 규칙

1
그릇 — RTTypes.h의 캐시 struct
USTRUCT(BlueprintType) 6종(FOwnedCharacter · FGuildData 등). FTableRowBase를 상속하지 않으며, 직렬화가 아니라 블루프린트 노출 · 복사 편의가 목적이라 BlueprintReadOnly로 둡니다.
2
로드 — Initialize에서 SELECT 1회
각 서브시스템이 자기 테이블을 읽어 TArray<FGuildData> 같은 멤버로 담습니다. player는 1행이라 struct 없이 EconomySubsystem의 멤버 변수(Gold · Shards · AccountFocusSeconds)로 직접 듭니다. 이후 UI · 로직은 메모리만 읽습니다.
3
쓰기 규칙 + 파생 함수
값이 바뀌는 함수 안에서 멤버 갱신과 DB UPDATE를 함께 수행합니다(캐시 · DB가 어긋날 틈 제거). 레벨처럼 계산 가능한 값은 저장하지 않고 GetLevel()로 매번 계산하며, 세션 타이머 · 가챠 후보 3장 같은 과정값은 결과만 DB에 남기고 버립니다.
⚠️

가장 많이 막히는 세 가지. ① "플러그인 활성화"와 "Build.cs 모듈 의존"은 별개 — 플러그인만 켜면 #include는 통과해도 링크(LNK2019)가 실패합니다. ② DB 파일은 반드시 Saved/ 아래 — Content/는 패키징 후 쓰기 금지 구역입니다. ③ Build.cs 수정과 새 UCLASS/USTRUCT 헤더는 Live Coding으로 반영되지 않습니다 — 에디터 종료 → IDE 빌드.

🧭

착수 순서. 처음부터 만든다면 09(한 컬럼 왕복으로 배관 검증) → 13(문지기 + 7테이블) → 기능 서브시스템 확장(11 Economy → 12 Guild → Gacha · Estate), ROW 표는 필요해지는 시점에 구조체 + CSV로 추가. 현재 리포 기준 실작업 정본은 14 저장 계층 재구축(8단계)이며, 전체 코드 완성본은 마스터 페이지가 담고 있습니다.

ROW — 행 구조체와 표 캐시

// ROW 행 구조체 — FTableRowBase를 상속해야 에디터가 "DataTable의 행 타입"으로 인정한다
// (11종 전부 이 형태 — 여기서는 DT_Ability_Step 하나만 발췌. 전체 정본: 05 문서)
USTRUCT(BlueprintType)
struct FAbilityStepRow : public FTableRowBase
{
    GENERATED_BODY()

    // 프로퍼티 이름은 CSV 헤더와 철자까지 같아야 임포트가 자동 매칭된다(계약)
    UPROPERTY(EditAnywhere) FName  AbilityId;      // 어느 능력치인가(그룹 키) — RowName 파싱 대신 이 컬럼으로 묶는다
    UPROPERTY(EditAnywhere) int32  StepIndex = 0;  // 몇 단계인가(정렬 키)
    UPROPERTY(EditAnywhere) float  Value = 0.f;    // 그 단계 효과값 — 자동 등분이 아니라 표에서 직접 지정
    UPROPERTY(EditAnywhere) int32  RollWeight = 0; // 뽑기 시 이 단계로 시작할 가중치
};
// 표 캐시 — Initialize에서 1회. RowName("FocusMult_Legendary_1") 문자열을 파싱하지 않는 것이 핵심:
// 컬럼(AbilityId·StepIndex)이 정본이라 RowName 형식이 바뀌어도 코드가 안 깨진다
void UGachaSubsystem::CacheStepTable(UDataTable* StepTable)
{
    TArray<FAbilityStepRow*> Rows;
    StepTable->GetAllRows(TEXT("CacheStep"), Rows);   // 1) 표 전체를 한 번에 읽는다 — 이후 표 접근 없음

    for (const FAbilityStepRow* Row : Rows)
    {
        StepsByAbility.FindOrAdd(Row->AbilityId).Add(*Row);   // 2) 능력치별로 그룹 (TMap<FName, TArray<FAbilityStepRow>>)
    }
    for (auto& Pair : StepsByAbility)
    {
        // 3) 단계 순 정렬 — 이후 "현재 Step의 값"은 배열 인덱스 조회 한 번이면 된다
        Pair.Value.Sort([](const FAbilityStepRow& A, const FAbilityStepRow& B){ return A.StepIndex < B.StepIndex; });
    }
}

DB — 문지기의 열닫과 스키마 보장

// DB 문지기 — 게임 기능 0개. 열고, 스키마를 만들고, 닫고, 트랜잭션 창구를 빌려준다.
// 기능 시스템들은 이 클래스"에만" 의존한다 (복붙 정본: 13 문서)
UCLASS()
class RECORDTALES_API UDatabaseSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()
public:
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;  // 앱 시작 시 엔진이 자동 1회 호출
    virtual void Deinitialize() override;                                // 앱 종료 시 자동 — 연 곳에서 닫는다(수명 대칭)

    // 핸들 "대여" — 소유·수명 관리는 오직 여기. 빌린 쪽은 쓰기 전 IsValid() 확인
    FSQLiteDatabase* GetDb() { return &Database; }

    // 다중 테이블 변경(추출·강화)의 단일 창구 — 값 쓰기는 각 주인, 묶음의 시작/끝은 문지기
    bool BeginTransaction();     // BEGIN — 이미 열려 있으면 false(SQLite는 중첩 불가)
    bool CommitTransaction();    // COMMIT — 묶은 변경을 한 번에 확정
    bool RollbackTransaction();  // ROLLBACK — 중간 실패 시 전부 원상 복귀
private:
    void CreateTables();         // CREATE TABLE IF NOT EXISTS 7종 — FK 순서 보장(DDL 정본: 05)
    FSQLiteDatabase Database;     // DB 핸들 실체 — 이 서브시스템만 소유(수명 = 앱 세션)
    bool bInTransaction = false;  // 중첩 BEGIN 방지 플래그
};
void UDatabaseSubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
    Super::Initialize(Collection);   // 부모 초기화 먼저 — 관례상 항상 맨 위

    // 1) DB 파일은 반드시 Saved/ 아래 — Content/는 패키징 후 쓰기 금지 구역(disk I/O error 함정)
    const FString DbPath = FPaths::ProjectSavedDir() / TEXT("Archive.db");

    // 2) ReadWriteCreate = 없으면 새로 만들고, 있으면 읽기/쓰기로 연다.
    //    실패 시 크래시 대신 로그 — 기능 시스템들은 IsValid()==false로 안전하게 우회
    if (!Database.Open(*DbPath, ESQLiteDatabaseOpenMode::ReadWriteCreate))
    {
        UE_LOG(LogTemp, Error, TEXT("[DB] 열기 실패: %s"), *DbPath);
        return;
    }

    Database.Execute(TEXT("PRAGMA foreign_keys = ON;"));   // 3) FK 강제 — 연결(세션)마다 다시 켜야 함
    CreateTables();                                        // 4) 7테이블 보장 — 기능 시스템이 깨기 전에 스키마 완성
}

void UDatabaseSubsystem::Deinitialize()
{
    Database.Close();        // 연 곳에서 닫는다 — 기능 시스템은 절대 Close() 하지 않는다
    Super::Deinitialize();
}

RUNTIME — 캐시 struct와 2줄 관례, 쓰기 규칙

// RUNTIME 캐시 struct — DB 행(guilds)을 담는 그릇. FTableRowBase 상속 없음.
// 직렬화가 목적이 아니라 블루프린트 노출·복사 편의 목적이라 BlueprintReadOnly로 둔다
USTRUCT(BlueprintType)
struct FGuildData
{
    GENERATED_BODY()

    UPROPERTY(BlueprintReadOnly) int32   GuildId = 0;      // guilds.GuildId(AUTOINCREMENT) 그대로 — 재사용 없음
    UPROPERTY(BlueprintReadOnly) FString GuildName;
    UPROPERTY(BlueprintReadOnly) int32   FocusSeconds = 0; // 레벨의 원천 — 레벨 자체는 저장하지 않는다

    // 파생값은 함수로 — 값이 두 곳(시간·레벨)에 살면 어긋날 수 있어 "계산"만 허용
    int32 GetLevel() const { return FocusSeconds / 3600; }   // 60분(3600초) = 1레벨(정직한 레벨)
};
// 기능 서브시스템 연결 — "2줄 관례". 모든 기능 시스템(Economy·Gacha·Estate·Archive)이 동일한 형태
void UGuildSubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
    Super::Initialize(Collection);

    // 1줄째: 문지기를 "나보다 먼저" 초기화해 달라(엔진 보장) — 깨어나면 DB가 이미 준비돼 있다
    UDatabaseSubsystem* Gate = Collection.InitializeDependency<UDatabaseSubsystem>();

    // 2줄째: 핸들을 "빌린다" — 소유·Close()는 금지(열닫 주체는 문지기 하나)
    Db = Gate ? Gate->GetDb() : nullptr;

    LoadGuilds();   // DB → RUNTIME 멤버(TArray<FGuildData>)로 1회 로드 — 이후 UI는 메모리만 읽는다
}

// 쓰기 규칙 — 멤버 갱신과 DB UPDATE를 같은 함수에서 함께(캐시·DB가 어긋날 틈 제거)
void UGuildSubsystem::AddFocusSeconds(int32 GuildId, int32 Seconds)
{
    if (!Db || !Db->IsValid()) return;   // 빌린 핸들은 쓰기 전 유효성 확인(관례) — DB 열기 실패 시 안전 우회

    FGuildData* G = FindGuild(GuildId);
    if (!G) return;

    G->FocusSeconds += Seconds;   // 1) RUNTIME 캐시 갱신

    // 2) 같은 자리에서 DB 반영 — UPDATE guilds SET FocusSeconds=? WHERE GuildId=?
    //    (레벨 컬럼은 없다 — GetLevel()이 매번 계산. 실 SQL 바인딩은 12 문서 정본)
}
06 — 확장

새 데이터를 더할 때

계층 구조가 무너지는 건 대부분 확장 시점입니다. 새 값 하나를 더할 때마다 01의 판별 질문("누가 이 값을 소유하는가")으로 계층을 정하고, 아래 규칙을 지킵니다.

✓ 이렇게

  • 새 밸런스 값은 ROW 컬럼으로 추가하고 시트 → CSV로 튜닝 (코드는 읽기만)
  • 새 영속 상태는 "어느 테이블 + 쓰기 주인은 누구"부터 결정 (소유 정본: 10 문서)
  • 여러 테이블을 고치는 작업은 문지기의 트랜잭션 창구(Begin/Commit/Rollback)로 묶기
  • 계산 가능한 값은 파생 함수로만 (레벨 = FocusSeconds / 3600)
  • 출시 후 스키마 확장은 PRAGMA user_version 분기 + ALTER TABLE (개발 중엔 DB 파일 삭제가 빠름)

✗ 이건 피하기

  • RowName 문자열 파싱으로 표를 묶기 (그룹 키 컬럼이 정본)
  • 기능 시스템이 DB를 직접 Open/Close (열닫 주체는 문지기 하나)
  • 능력치 효과값을 DB에 복사 (Step만 저장, 값은 표 조회)
  • 레벨을 컬럼으로 저장 (원천 시간과 어긋나는 순간 정직함이 깨짐)
  • player 저장 시 Secured 갱신 누락 (로드 때 자동 복원이 옛 값으로 되돌림)
