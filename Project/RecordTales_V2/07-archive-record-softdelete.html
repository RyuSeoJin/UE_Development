아카이브 · 기록 ID
· 소프트 삭제

모든 활동은 SQLite Archive.db의 archives 테이블 하나에 기록되고, 각 기록 행은 (길드 → 카테고리 → 소스) 컬럼으로 태그됩니다. 길드별로 나눠 보거나 전체를 모아 볼 수 있고, 삭제해도 휴지통에서 되살릴 수 있습니다.

archives 테이블 + 태그 컬럼 밀리초 ID 소프트 삭제 · DeletedAt
01 — 개념

통합 저장, 분리 조회

기록은 길드마다 따로 저장하지 않고 archives 테이블 하나(계정 단위 통합 원장, Archive.db 안)에 모읍니다. 대신 각 행에 GuildId · 카테고리 · 소스 컬럼을 새깁니다. 그래서 특정 길드에 들어가면 그 길드 행만 WHERE로 필터해 보이고(분리), 기념비에선 전 길드를 합쳐 볼 수 있습니다(전체 조망).

🏷️

카테고리는 2축입니다(12 문서 확정). 기록 행의 Category는 소분류 — 계정 공유 풀(archive_categories)의 이름 태그라 어느 길드의 기록에든 같은 소분류를 쓸 수 있습니다(예: '애니메이션'). 대분류는 기록이 아니라 길드의 속성(guilds.Category, 길드당 1개)이라, 대분류 필터는 guilds를 JOIN해 거릅니다. 두 축은 조합 필터도 가능합니다(상세 SQL: 12 문서 4단계).

🧭

왜 통합인가. 길드별로 사일로화하면 "올해 다 합쳐 몇 시간 했나" 같은 전체 조망을 영영 못 만듭니다. 테이블 하나 + 태그 컬럼이면 per-guild 뷰와 전체 뷰가 같은 데이터의 조회 조건 차이일 뿐이라 둘 다 공짜로 나옵니다. RecordTales의 이름값(내 기록의 이야기)이 여기 있습니다.

🗄️

저장소. 기록은 DB — 세션이 끝날 때 행 하나가 INSERT로 남고, 이후엔 소프트 삭제(DeletedAt)만 UPDATE됩니다. 테이블 스키마 전문은 05(데이터 스키마 명세), 배관 실습은 09(SQLite 배관)가 담당합니다. SaveGame(.sav)은 쓰지 않습니다(풀-DB 확정).

02 — 기록 ID

밀리초 + 길드번호

기록 ID는 YYYYMMDDhhmmssSSS-{GUILDID} 형식이며, 이 문자열이 그대로 archives.Id 컬럼(TEXT PRIMARY KEY)이 됩니다.

부분예역할
YYYYMMDDhhmmssSSS20260712143005123밀리초까지 — 고정폭 0-패딩이라 문자열 정렬해도 시간순 보장·충돌 방지
{GUILDID}000001생성순 길드번호. 최소 6자리 0-패딩이되 초과 시 자릿수 확장(→1000000). 재사용 없음
⚠️

정렬은 문자열 ID가 아니라 컬럼으로. ID의 길드 접미사는 가독용일 뿐, 실제 정렬·필터는 별도 컬럼 CreatedAt·정수 GuildId로 합니다(ORDER BY CreatedAt · WHERE GuildId=?). 그래야 길드번호가 자릿수 확장돼도(가변 폭) 정렬이 안 깨집니다.

03 — 뷰

어디서 무엇을 보나

모든 뷰는 같은 archives 테이블에 대한 조회 조건 차이입니다. 살아있는 기록만 보이도록 공통으로 DeletedAt IS NULL이 붙습니다.

1
홈 = "오늘 할 일"
앱을 열면 전 길드 통합의 오늘 할 일(행동)이 먼저. 즉흥 집중 시작 경로도 짧게. — 조회: 오늘 날짜 조건 + DeletedAt IS NULL
2
길드 패널 = 그 길드 아카이브
길드 건물에 들어가면 그 길드로 필터된 기록(분야별). — 조회: WHERE GuildId=? AND DeletedAt IS NULL
3
기념비 = 전체 기록
기념비형 오브젝트를 선택하면 전 길드를 모은 회고 뷰(전체 조망). — 조회: GuildId 조건 없음 + DeletedAt IS NULL
4
캘린더 도트
집중/스톱워치/퀘스트 소스별 점. per-guild / 전체 두 모드. — 조회: Source·기간 조건 조합
04 — 소프트 삭제 · 휴지통

지워도 되살린다

길드·기록 행에 DeletedAt 컬럼(삭제 시각)을 둡니다 — NULL이면 살아있음, 시각이 있으면 휴지통. 행을 지우지 않으므로 무기한 보존됩니다.

1
삭제 = UPDATE 한 줄
삭제는 실제 제거가 아니라 UPDATE archives SET DeletedAt=지금 WHERE Id=? 한 줄. 일반 화면은 DeletedAt IS NULL만 조회하므로 그 즉시 목록에서 사라져 보입니다.
2
휴지통 = 복원
DeletedAt IS NOT NULL만 모아 보여주고, 복원은 SET DeletedAt=NULL로 되돌립니다. 휴지통 조회는 휴지통을 열 때만 실행(평소 화면 속도 유지 — DB라 애초에 필요한 행만 읽습니다).
3
완전 삭제 · Cascade
완전 삭제만 실제 DELETE. 길드 소프트 삭제 → 그 길드 기록도 함께 숨김(guilds.DeletedAt 기준으로 조회에서 제외, 복원 시 함께 복원). 길드를 완전 삭제할 땐 그 길드 기록 행도 함께 DELETE(한 트랜잭션).
🧭

용량 걱정 없음. 기록 하나는 숫자·짧은 텍스트라 ~100~200바이트. 수년 heavy 사용도 수 MB 수준이라 무기한 보존이 문제 되지 않습니다. DB 조회는 조건에 맞는 행만 읽으므로, SaveGame 시절 고민이던 "삭제 항목 지연 로드"도 WHERE 조건 하나로 끝납니다.

💡

메모리 표현. DB에선 DeletedAt 하나로 관리하고, 코드가 화면에 쓸 때만 "삭제됐나?"를 참/거짓(bDeleted 파생)으로 변환해 노출합니다 — 진실은 항상 DB의 DeletedAt입니다(05 스키마 명세와 동일 규약).

05 — 스켈레톤

뼈대 코드

클래스는 09 배관에서 만든 UArchiveSubsystem(실코드명 — 초기 설계의 URecordSubsystem 역할을 이 클래스가 맡습니다)입니다. 아래는 배열 조작이 아니라 SQL 연산의 뼈대입니다 — DB 열닫·프리페어드 스테이트먼트 실물은 09가 담당합니다.

UCLASS()
class UArchiveSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()
public:
    // 활동 완료 시 기록 생성 — 현재 길드·카테고리·소스를 컬럼으로 새겨 archives에 INSERT
    void AddRecord(int32 GuildId, const FString& Category,
                   ERecordSource Source, int32 DurationSeconds);

    // 소프트 삭제 / 복원 / 완전 삭제 (휴지통) — UPDATE·DELETE 한 줄씩
    void SoftDelete(const FString& RecordId);
    void Restore(const FString& RecordId);
    void Purge(const FString& RecordId);

    // 뷰: 살아있는 것(DeletedAt IS NULL)만 SELECT. 정렬은 CreatedAt 컬럼 기준
    TArray<FArchiveRecord> GetGuildRecords(int32 GuildId) const;
    TArray<FArchiveRecord> GetAllRecords() const;
    TArray<FArchiveRecord> GetTrash() const;   // 휴지통(IS NOT NULL) — 열 때만 호출
};
// 기록 ID = 밀리초 타임스탬프 + 생성순 길드번호. 정렬은 이 문자열이 아니라 CreatedAt 컬럼으로.
FString MakeRecordId(const FDateTime& Now, int32 GuildId)
{
    // YYYYMMDDhhmmss(초까지) + 밀리초 3자리 = 고정폭·유일
    FString Ts = Now.ToString(TEXT("%Y%m%d%H%M%S"));
    Ts += FString::Printf(TEXT("%03d"), Now.GetMillisecond());
    // 길드번호는 최소 6자리 0-패딩(넘으면 자동 확장). 접미사는 가독용일 뿐
    return FString::Printf(TEXT("%s-%06d"), *Ts, GuildId);
}

06 — 확장

더 얹을 때

✅

이렇게: 스키마가 바뀌면(컬럼 추가 등) PRAGMA user_version으로 버전을 읽어 분기하고 ALTER TABLE archives ADD COLUMN …으로 넓힌다. 구버전 데이터에 GuildId가 없으면 마이그레이션에서 "기본 길드"로 백필(UPDATE … WHERE GuildId IS NULL)한다. 개발 중엔 DB 파일 삭제(09 초기화 절차)가 빠르다.

⚠️

이건 피하기: 문자열 ID로 정렬하지 말 것(CreatedAt 컬럼으로). 삭제한 길드의 GuildId를 재사용하지 말 것(과거 기록 태그가 엉킨다). 소프트 삭제 없이 곧바로 DELETE하지 말 것(실수 복구 불가). 길드 완전 삭제 시 기록 정리를 빼먹지 말 것(한 트랜잭션으로 함께).
