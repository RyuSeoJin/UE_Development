RecordTales V2 · v0.1 · Estate Economy

영지 · 경영 건물
경제

뽑은 캐릭터를 경영 건물에 배치해 Gold와 조각을 버는 계정 단위 경제입니다. 산출은 캐릭터 능력치로 커지고, 집중 중일 때 N배가 되어 "게임을 최대로 돌리려면 실제로 집중해야" 하도록 설계됩니다.

2 건물 · 2 자원 유휴 1배 + 집중 N배 02-A 아키텍처 패턴
01 — 개념

영지가 하는 일

영지(Estate)는 계정 단위의 홈 화면이자, 뽑은 캐릭터가 "일하는" 공간입니다. 중복 허용·무제한 보유로 캐릭터가 많이 쌓이는데, 경영 건물이 비주력 캐릭터에게 일자리를 주어 놀지 않게 하고 자원을 순환시킵니다.

건물은 두 종류입니다. 골드 건물은 캐릭터를 배치하면 Gold를 산출하고, 기억 거래소는 캐릭터를 분해해 조각을 얻습니다. 골드 건물의 산출은 집중(타이머 가동) 중일 때 N배가 됩니다.

💡

게임이 집중을 유도한다. 유휴(집중 안 함) 상태에선 1배로만 벌리고 오프라인 누적에 상한이 있습니다. 반면 집중 중에는 N배. 그래서 "경영으로 많이 벌려면 실제 집중이 필요"해, 게임 요소가 생산성을 방해하지 않고 오히려 당깁니다.

02 — 구성

건물 · 슬롯 · 자원

경영 경제는 2건물·2자원으로 단순합니다(복잡할수록 일정 관리 앱에서 멀어짐). 캐릭터 배치처는 골드 건물 하나이고, 기억 거래소는 분해 전용입니다.

건물산출배치비고
골드 건물 (음식점·상점 등 테마별) Gold 슬롯(≤3, 성장 해금)에 캐릭터 배치 산출은 아래 공식. 테마 매칭 캐릭터면 보너스.
기억 거래소 조각 배치 없음 — 캐릭터를 넣어 분해(교환) 등급별 고정 조각 지급.
🧭

자원 2종. Gold(가챠·건물 업그레이드·해금에 씀)와 조각(캐릭터 능력치 단계 강화에 씀). 건물 업그레이드로 기본 산출이 오르고, 슬롯이 열립니다(레벨별, DT_Building_Upgrade).

03 — 산출 공식

얼마나 버는가

골드 건물의 시간당 산출은 더하는 부분(flat)과 곱하는 부분(상태 배율)로 나뉩니다.

🧮

골드/시간 = (건물 기본 산출 + Σ 캐릭터 flat 보너스) × 상태 배율
• 일반(집중 안 함) 배율 = 1 + Σ(캐릭터 일반 배율 보너스)
• 집중 배율 = 기본 집중 배율 N(영지 규칙) + Σ(캐릭터 집중 배율 보너스)
• flat 보너스는 상태와 무관하게 항상 더해집니다.

워크드 예시

기본 산출 10인 음식점에 세 캐릭터를 배치했다고 합시다(수치는 예시).

캐릭터flat일반 배율 보너스집중 배율 보너스
A+3——
B+2—×2
C—×2×2
합+51+(2−1)=2N+(2−1)+(2−1)=N+2
상태계산시간당 Gold
일반(유휴)(10 + 5) × 230 (오프라인 누적 캡 적용)
집중 중(10 + 5) × (N + 2)15 × (N+2)

※ 배율 보너스는 "각 배율 − 1"을 더합니다(예: ×2 → +1). N(영지 기본 집중 배율)·건물 기본 산출·캐릭터 능력치 값은 모두 DataTable 튜닝 대상입니다.

04 — 흐름

배치에서 수집까지

1
배치
골드 건물 슬롯에 캐릭터를 넣습니다(FBuildingState.DeployedInstanceIds). 그 캐릭터의 능력치가 산출에 반영됩니다.
2
산출 (상태 배율)
유휴면 일반 배율(1배 계열), 집중 중이면 집중 배율(기본 N배 + 캐릭터 보너스). 시간당 산출이 공식으로 정해집니다.
3
유휴 누적 (오프라인 캡)
앱을 닫아도 유휴 산출이 쌓이되, 오프라인 누적 캡까지만(돌아왔을 때 보너스 수준). 무한정 쌓이지 않아 "집중 안 해도 됨"을 막습니다.
4
수집
LastCollectTime과 신뢰 시간(조작 방지)으로 "지난 시간 × 산출"을 계산해 Gold 지갑에 넣습니다.

FBuildingState — 내 건물 하나

// 계정 세이브의 Estate 배열 원소. 영지는 계정 단위 하나.
USTRUCT()
struct FBuildingState
{
    GENERATED_BODY()
    UPROPERTY() FName   BuildingId;                // → DT_Building (어떤 건물인지)
    UPROPERTY() int32   Level = 1;               // 업그레이드 레벨 → DT_Building_Upgrade(기본 산출·슬롯)
    UPROPERTY() TArray<int32> DeployedInstanceIds; // 배치된 캐릭터 InstanceId(열린 슬롯 수만큼)
    UPROPERTY() FDateTime LastCollectTime;         // 유휴 누적 계산 기준(오프라인 캡)
};

UEstateSubsystem — 산출 계산 · 수집

UCLASS()
class UEstateSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()
public:
    // 한 건물의 시간당 산출 = (기본 + Σflat) × 상태 배율
    // bFocusing=true면 집중 배율, false면 일반 배율을 쓴다
    float ComputeOutputPerHour(const FBuildingState& B, bool bFocusing) const;

    // 유휴 누적 수집: (지금 − LastCollectTime) × 산출, 단 오프라인 캡 적용
    int64 Collect(FBuildingState& B);
};

ComputeOutputPerHour — 공식의 코드 표현

float UEstateSubsystem::ComputeOutputPerHour(const FBuildingState& B, bool bFocusing) const
{
    // 1) 건물 기본 산출 (레벨별) — DT_Building_Upgrade에서 조회
    float Base = LookupBaseOutput(B.BuildingId, B.Level);

    // 2) 배치 캐릭터들을 순회하며 flat은 더하고, 상태별 배율 보너스는 모은다
    float FlatSum = 0.f, MultBonus = 0.f;
    for (int32 Id : B.DeployedInstanceIds)
        AccumulateAbilities(Id, bFocusing, FlatSum, MultBonus);  // 배율은 (값−1)씩 누적

    // 3) 상태 배율 = (집중이면 영지 기본 N배, 아니면 1) + 캐릭터 배율 보너스
    float StateMult = (bFocusing ? EstateFocusBaseNx : 1.f) + MultBonus;

    return (Base + FlatSum) * StateMult;  // (기본 + Σflat) × 상태 배율
}
🧭

수집에 신뢰 시간을 쓰는 이유. 기기 시계를 임의로 앞당겨 유휴 보상을 부풀리는 것을 막으려면, 로컬 시계 대신 신뢰 시간(trusted time)으로 경과를 계산해야 합니다. 오프라인 캡과 함께, 유휴 수입이 남용되지 않게 이중으로 막습니다.

06 — 확장

더 얹을 때

✅

이렇게: 새 골드 건물 테마는 DT_Building에 행을 더하고, 레벨별 산출·슬롯·비용은 DT_Building_Upgrade(건물별 개별 곡선)에 넣는다. 캐릭터의 테마 매칭 보너스(캐릭터 테마 == 건물 테마면 추가 보너스)는 산출에 얹되, 비매칭 캐릭터도 배치되어 능력치는 작동한다.

⚠️

이건 피하기: 유휴 산출에 캡을 빼지 말 것 — 방치만으로 게임이 굴러가 집중 유인이 사라진다. 산출 배율을 곱셈으로 무한 스택하지 말 것 — "각 배율 −1을 더한다"는 합산 규칙을 지켜 폭주를 막는다. 수집을 로컬 시계로만 계산하지 말 것 — 시계 조작에 뚫린다.

캐릭터 간 조합/건물 시너지는 나중 고려 항목입니다.
