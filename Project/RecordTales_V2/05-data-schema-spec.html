01 — 요구사항 / 범위

이 문서가 담는 것

Phase 2 게임 레이어의 데이터 스키마 전체입니다. 밸런스 값은 넣지 않고 구조(컬럼)만 정의합니다 — 실제 수치는 DataTable/CSV에서 튜닝합니다. 영속 상태는 SaveGame이 아니라 SQLite Archive.db에 담습니다.

🧭

두 계층 + 휘발: ROW DataTable 행 구조체(밸런스, FTableRowBase 상속) · DB SQLite 테이블(내 상태·기록) · RUNTIME 메모리 캐시(휘발). 능력치는 현재 단계만 저장하고 값은 표에서 조회합니다.

🗑️

폐기. 이전 판의 USaveGame(.sav)과 UDeveloperSettings(CONFIG)는 풀-DB 전환으로 더 이상 쓰지 않습니다. 상태는 전부 DB 테이블 행으로, 밸런스는 전부 표(ROW)로 갔습니다. (배경: 03 데이터 설계 · 10 코드/변수 지도)

03 — enum · 공용 타입

열거형

모든 등급·종류는 uint8 기반 UENUM(BlueprintType)으로(리플렉션·BP 노출 제약). 값 순서 = 등급 오름차순.

// 캐릭터 등급 — 뽑을 때 롤(인스턴스). 개화 슬롯 수·능력치 등급 분포를 결정
// RowName은 시트 접두 형식(Character_Rarity_Normal / Ability_Rarity_Normal) —
// 표 조회 시 "Character_"/"Ability_" 접두사 + enum 이름(Rarity_Normal…)을 조합해 FName을 만든다
UENUM(BlueprintType)
enum class ECharacterRarity : uint8
{
    Rarity_Normal    UMETA(DisplayName = "노말"),      // RowName "Character_Rarity_Normal"/"Ability_Rarity_Normal"의 뒷부분. UMETA DisplayName = 에디터/블프 표시용 한글
    Rarity_Rare      UMETA(DisplayName = "레어"),
    Rarity_Epic      UMETA(DisplayName = "에픽"),
    Rarity_Legendary UMETA(DisplayName = "레전더리")
};

// 능력치 등급 — 각 능력치 자체의 등급. 효과 크기(계수 tier)를 결정. 캐릭터 등급과 별개 축이라 타입 분리
UENUM(BlueprintType)
enum class EAbilityRarity : uint8
{
    Rarity_Normal    UMETA(DisplayName = "노말"),
    Rarity_Rare      UMETA(DisplayName = "레어"),
    Rarity_Epic      UMETA(DisplayName = "에픽"),
    Rarity_Legendary UMETA(DisplayName = "레전더리")
};

// 능력치 효과 종류 — 전부 골드 건물 배치 시 적용
UENUM(BlueprintType)
enum class EEffectType : uint8 { FlatGold, IdleMult, FocusMult };

// 기록 출처 — 아카이브 태그의 소스
UENUM(BlueprintType)
enum class ERecordSource : uint8 { Timer, Stopwatch, Quest };
04 — DataTable 행 구조체 10종

밸런스 표 (ROW)

모두 FTableRowBase를 상속해야 DataTable의 행 타입으로 지정할 수 있습니다. 값은 에디터/CSV에서 채웁니다.

💡

코드 상자 표기 규칙: 아래 10종은 실제로 단일 헤더 RTTables.h 하나에 다 들어갑니다 — 한 상자에 담기엔 너무 길어서 표시 편의상 3개 상자(능력치·캐릭터/가챠·건물)로 나눈 것일 뿐입니다. 탭 이름이 파일명.h — 부제처럼 대시(—)로 이어지면 "같은 파일의 일부 구간", 파일A.h / 파일B.h처럼 슬래시(/)로 이어지면 "실제로 다른 두 파일"이라는 뜻입니다.

// DT_Ability_Type — 능력치 종류(효과×등급). RowName 예: FocusMult_Legendary
USTRUCT(BlueprintType)
struct FAbilityTypeRow : public FTableRowBase
{
    GENERATED_BODY()
    UPROPERTY(EditAnywhere) FText DisplayName;                        // UI 표시 이름
    UPROPERTY(EditAnywhere) EAbilityRarity AbilityRarity = EAbilityRarity::Rarity_Normal; // 이 능력치의 등급 = 효과 tier
    UPROPERTY(EditAnywhere) EEffectType EffectType = EEffectType::FlatGold;  // flat/유휴/집중 중 무엇
    UPROPERTY(EditAnywhere) int32 SelectWeight = 1;                     // 같은 등급 내에서 뽑힐 가중치
};

// DT_Ability_Step — 능력치 단계별 값·시작 가중치. RowName 예: FocusMult_Legendary_1
// 단계마다 한 행. 등급 높을수록 행(단계) 수가 많다. 강화 비용은 DT_Ability_Enhance로 분리했다.
USTRUCT(BlueprintType)
struct FAbilityStepRow : public FTableRowBase
{
    GENERATED_BODY()
    UPROPERTY(EditAnywhere) FName AbilityId;              // 어느 능력치(그룹 키) — 필드로도 둬 조회·정렬
    UPROPERTY(EditAnywhere) int32 StepIndex = 0;        // 몇 단계(정렬 키)
    UPROPERTY(EditAnywhere) float Value = 0.f;           // 그 단계 효과값(직접 지정, 자동 등분 아님)
    UPROPERTY(EditAnywhere) int32 RollWeight = 0;       // 뽑기 시 이 단계로 시작할 가중치
};

// DT_Ability_Enhance — 능력치 등급 → 한 단계 올리는 조각 비용(단계 무관 고정). RowName = 능력치 등급(Ability_Rarity_Normal…)
// 단계별 보정치(후반 단계 가중)는 추후 별도 도입 예정. 지금은 등급당 고정값 하나.
USTRUCT(BlueprintType)
struct FAbilityEnhanceRow : public FTableRowBase
{
    GENERATED_BODY()
    // 컬럼명은 CSV 헤더와 철자 동일해야 함(Extract_ShardAmount와 대칭)
    UPROPERTY(EditAnywhere) int32 Upgrade_ShardAmount = 0;  // 이 등급 능력치 한 단계 강화 비용(조각)
};

봉인 방식 — 평문 페이로드를 AES로 감싼다

Secured 블롭 = 암호화(MAGIC │ Gold │ Shards │ Seconds │ CRC32). 복호화했을 때 MAGIC과 CRC가 맞아야 "진짜"로 인정합니다 — 블롭을 건드리면 이 검사에서 걸립니다. AES는 UE 내장 FAES(AES-256)를 씁니다.

// player 행의 치트 유인 값 3개를 하나로 묶은 그릇. 이 값들만 Secured로 보호한다
struct FSecuredAccount
{
    int64 Gold = 0;
    int32 Shards = 0;
    int64 AccountFocusSeconds = 0;
};

// 봉인: 값 → (MAGIC+값+CRC) 평문 → AES 암호화 → BLOB. player.Secured에 바인딩해 저장
TArray<uint8> SealAccount(const FSecuredAccount& In);

// 개봉: BLOB → AES 복호화 → MAGIC·CRC 검사. 성공=true(Out 채움), 실패=false(변조/손상)
bool OpenAccount(const TArray<uint8>& Blob, FSecuredAccount& Out);
#include "RTSecured.h"
#include "Misc/AES.h"        // FAES::EncryptData/DecryptData (AES-256) — Core 모듈
#include "Misc/Crc.h"        // FCrc::MemCrc32 — 무결성 검사용
#include "Serialization/BufferArchive.h"
#include "Serialization/MemoryReader.h"

// 빌드에 심는 32바이트(256비트) 대칭키. 유출되면 방어가 뚫리므로 눈에 안 띄게 둔다
// (완전 방어는 서버뿐 — 이건 "캐주얼 변조를 자동 복원"이 목표).
static const uint8 GKey[32] = { /* 32개 무작위 바이트로 채운다 */ };
// 복호화 성공 판정용 서명. 복호화 후 이 값이 안 나오면 키 불일치/변조
static constexpr uint32 SEAL_MAGIC = 0x52544131; // 'RTA1'

static void MakeKey(FAES::FAESKey& K) { FMemory::Memcpy(K.Key, GKey, 32); }

TArray<uint8> SealAccount(const FSecuredAccount& In)
{
    // 1) 평문 페이로드 = MAGIC | Gold | Shards | Seconds | CRC32(앞 24바이트)
    FBufferArchive Ar;                       // 값들을 바이트로 직렬화하는 버퍼
    uint32 Magic = SEAL_MAGIC;               // 개봉 때 이 서명으로 진짜 여부 판정
    int64 Gold = In.Gold; int32 Shards = In.Shards; int64 Sec = In.AccountFocusSeconds;
    Ar << Magic << Gold << Shards << Sec;    // 4+8+4+8 = 24바이트
    uint32 Crc = FCrc::MemCrc32(Ar.GetData(), Ar.Num()); // 지금까지 바이트의 체크섬
    Ar << Crc;                               // 무결성 태그를 뒤에 붙임(변조 감지용)

    // 2) AES 블록(16바이트) 배수로 0 패딩 후 제자리 암호화
    const int32 Pad = (16 - (Ar.Num() % 16)) % 16;
    for (int32 i = 0; i < Pad; ++i) { uint8 Z = 0; Ar << Z; }
    FAES::FAESKey K; MakeKey(K);
    FAES::EncryptData(Ar.GetData(), Ar.Num(), K);   // 평문 → 암호문(같은 버퍼)

    TArray<uint8> Out; Out.Append(Ar.GetData(), Ar.Num());
    return Out;                              // → player.Secured BLOB에 저장
}

bool OpenAccount(const TArray<uint8>& Blob, FSecuredAccount& Out)
{
    // 블롭이 없거나 블록 배수가 아니면 손상 — 개봉 실패
    if (Blob.Num() == 0 || Blob.Num() % 16 != 0) return false;
    TArray<uint8> Buf = Blob;             // 제자리 복호화할 복사본
    FAES::FAESKey K; MakeKey(K);
    FAES::DecryptData(Buf.GetData(), Buf.Num(), K); // 암호문 → 평문(키 틀리면 쓰레기)

    FMemoryReader Ar(Buf);
    uint32 Magic = 0; Ar << Magic;
    if (Magic != SEAL_MAGIC) return false;   // 서명 불일치 → 키 틀림/블롭 변조
    int64 Gold = 0; int32 Shards = 0; int64 Sec = 0;
    Ar << Gold << Shards << Sec;
    uint32 StoredCrc = 0; Ar << StoredCrc;
    // CRC는 MAGIC~Sec까지 24바이트에 대해 계산했었다 — 그대로 재계산해 비교
    const int32 PayloadBytes = sizeof(uint32) + sizeof(int64) + sizeof(int32) + sizeof(int64); // 24
    if (FCrc::MemCrc32(Buf.GetData(), PayloadBytes) != StoredCrc) return false; // 무결성 실패

    Out.Gold = Gold; Out.Shards = Shards; Out.AccountFocusSeconds = Sec;
    return true;                            // 진짜 값 복원 성공
}

저장 · 로드 — 자동 복원이 붙는 곳

저장은 평문 컬럼과 Secured를 같은 트랜잭션에서 함께 씁니다. 로드는 OpenAccount로 진짜 값을 얻어 평문과 비교하고, 다르면 진짜 값으로 덮어(자동 복원) + 사용자에게 알립니다.

// 저장 — 재화가 바뀔 때. 평문 3칸 + Secured를 한 UPDATE(한 트랜잭션)로 커밋
void UEconomySubsystem::SavePlayer()

    //   ?1..3 = 정수 바인딩, ?4 = Sealed를 BLOB로 바인딩(FSQLitePreparedStatement::SetBindingValue)

}

// 로드 — 앱 시작 시 1회. 평문과 Secured를 함께 읽어 검증·복원
void UEconomySubsystem::LoadPlayer()
{
    // SELECT Gold, Shards, AccountFocusSeconds, Secured FROM player WHERE Id=1;
    int64 PlainGold; int32 PlainShards; int64 PlainSec; TArray<uint8> SecuredBlob;
    // … 위 컬럼들을 읽어 담았다고 하자 …

    FSecuredAccount Real;
    if (!OpenAccount(SecuredBlob, Real))
    {
        // 1) 블롭 자체가 변조/손상 → 진짜 값을 알 수 없음 → 복원 불가
        //    경고 후 새 게임 유도(또는 별도 백업 슬롯에서 복구).
        HandleSecuredBroken();
        return;
    }
    if (Real.Gold != PlainGold || Real.Shards != PlainShards || Real.AccountFocusSeconds != PlainSec)
    {
        // 2) 평문만 편집됨 → 진짜(복호화) 값을 채택 = 자동 복원
        Gold = Real.Gold; Shards = Real.Shards; AccountFocusSeconds = Real.AccountFocusSeconds;
        SavePlayer();                       // 평문을 즉시 재기록해 정합성 회복
        NotifyRestored();                   // "비정상 데이터를 복구했습니다" 안내(알리고 복원)
    }
    else
    {
        // 3) 일치 → 정상. 평문을 그대로 채택
        Gold = PlainGold; Shards = PlainShards; AccountFocusSeconds = PlainSec;
    }
}

한계와 강화안. ① 키가 클라이언트에 있음 — 키를 추출한 소수는 블롭까지 다시 만들 수 있어 완전 방어는 아닙니다(그건 서버뿐). ② 평문+블롭을 둘 다 파괴하면 복원 불가 → 새 게임. ③ 더 강한 보증이 필요하면 CRC 대신 HMAC-SHA256(encrypt-then-MAC)로 교체합니다. AES-GCM은 UE 기본 API에 없어 외부 라이브러리가 필요하므로 지양합니다. 방치형 싱글에는 이 선(FAES + MAGIC/CRC)이 적정합니다.

표 등록 · DB 초기화 · 확인

1
DataTable 에셋 생성 · 캐시
각 F*Row를 행 타입으로 지정해 DT_* 애셋을 만들고(에디터: Miscellaneous ▸ Data Table), 값은 시트→CSV로 채운다. 단계·업그레이드처럼 (키+인덱스) 행은 서브시스템 Initialize에서 TMap<FName, TArray<Row>>로 그룹·정렬해 캐시(RowName 조합에 의존하지 않음).
2
DB 초기화
DB 문지기 UDatabaseSubsystem(11에서 분리)의 Initialize에서 Archive.db를 열고 PRAGMA foreign_keys = ON → 위 CREATE TABLE IF NOT EXISTS 6종을 순서대로 실행(있으면 건너뜀). 기능 서브시스템들은 InitializeDependency로 문지기에 의존해 핸들을 빌린다. 09 배관의 리듬대로 한 테이블씩 넓힌다.
3
스키마 마이그레이션
출시 후 컬럼을 더할 땐 PRAGMA user_version으로 현재 버전을 읽어 분기하고 ALTER TABLE … ADD COLUMN으로 넓힌 뒤 user_version을 올린다(SaveGame의 SaveVersion을 대체). 개발 중엔 DB 파일을 지우는 게 빠르다(09 초기화 절차).
