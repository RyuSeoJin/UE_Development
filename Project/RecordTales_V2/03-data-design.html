데이터 설계
DataTable · SQLite DB

V2의 데이터가 어디에 살고(정적 표 vs 영속 DB), 어떻게 서로 연결되는지의 지도입니다. 핵심 원칙은 "밸런스 값은 표에서 조회하고, 내 상태는 하나의 SQLite DB에 담는다"입니다. 영속 데이터는 SaveGame(.sav)이 아니라 풀-DB(SQLite)에 저장합니다.

01 — 지도 읽는 법

수명 배지 먼저

이 문서의 모든 데이터에는 수명 배지가 붙습니다. "이게 어디 살고 언제 사라지는가"를 배지 색으로 먼저 구분하세요. V2는 풀-DB 전환으로 영속 저장을 SQLite 하나로 통일했습니다 — 그래서 배지 세계가 단순해졌습니다.

배지의미V2 예시
ROWDataTable 행 — 손으로 채우는 밸런스 데이터(에디터/CSV, 코드는 읽기만)DT_Ability_Step, DT_GachaRateByLevel
DBSQLite Archive.db — 껐다 켜도 남는 영속 데이터의 유일한 저장소(상태=UPDATE, 기록=INSERT)Gold, owned_characters, archives
RUNTIME런타임 휘발 — 계산값, 세션 종료와 함께 소멸(저장 안 함)세션 타이머, 가챠 후보 3장
SAVE미사용 — SaveGame(.sav)은 풀-DB 전환으로 폐기(상태는 전부 DB)—
CONFIG미사용 — DeveloperSettings 상수도 폐기(밸런스는 전부 ROW)—
🧭

한 문장 원칙. ROW는 밸런스(모두에게 같은 규칙, 시트→CSV로 채움), DB는 내 상태(나에게만 있는 진행). 상태는 표에 있는 값을 복사하지 않고 RowName으로 참조합니다 — 그래서 수치 튜닝은 표만 고치면 되고, 이미 저장된 내 데이터는 건드릴 필요가 없습니다.

🗑️

폐기된 두 계층. SaveGame(.sav)과 DeveloperSettings(CONFIG)는 풀-DB 전환으로 더 이상 쓰지 않습니다. 이유 — ① 저장소 단일화(이중 저장 정합성 걱정 제거) ② 트랜잭션으로 여러 테이블 변경을 하나로 묶음 ③ DB Browser로 모든 데이터를 눈으로 확인. (파일명: 담는 범위는 계정 전체지만, 다른 데이터베이스와 이름이 헷갈리지 않도록 파일명은 Archive.db로 둡니다.)

03 — 소유 결정

표에서 조회? DB에 저장? 그리고 어느 축?

새 데이터를 추가할 때 정하는 두 가지 — ①어디 두나(표 ROW vs DB), ②누구 것인가(계정 공유 vs 길드별 축). 파생 가능하면 저장하지 않고 계산하며, 같은 값의 쓰기 주인은 언제나 하나입니다(나머지는 읽기만).

04 — 연결 지도

DB의 상태 행은 값을 복사하지 않고 RowName(키)으로 DataTable을 참조합니다. 아래는 그 참조 연결입니다(왼쪽 DB 행 → 오른쪽 ROW 표).

왜 참조인가. 값을 DB 행에 복사해두면, 밸런스를 패치할 때 이미 저장된 수천 개의 값이 옛 수치로 남습니다. RowName만 저장하면 시트→CSV로 표를 고치는 순간 모든 인스턴스가 새 밸런스를 즉시 반영합니다 — 유저의 Archive.db는 건드리지 않고요.

05 — 흐름 추적

시나리오: 뽑기 한 번

캐릭터 하나를 만드는 데 표들이 어떤 순서로 쓰이는지 따라가 봅니다(이 과정을 3번 반복해 3장을 보여주고 1택).

1
캐릭터 등급 롤
DT_GachaRateByLevel[계정 레벨] → 캐릭터 등급(노말~레전더리). 계정 레벨은 player.AccountXp에서 파생.
2
개화 슬롯 수
DT_Character_Ability_OpenChance[등급] → 4슬롯 각각의 개화 확률(계단형).
3
슬롯마다 능력치 결정
개화된 슬롯마다 DT_Character_Ability_GradeChance[등급] → 능력치 등급 → DT_Ability_Type에서 능력치 선택 → DT_Ability_Step의 RollWeight로 시작 단계 롤.
4
1택 → INSERT (한 트랜잭션)
3장 후보 중 1장을 고르는 순간에만 BEGIN → owned_characters 1행 + owned_abilities N행 INSERT → COMMIT. 효과값은 저장 안 함(표에서 조회). 후보 3장(RUNTIME)은 소멸.

시나리오: 추출 → 강화 (조각의 왕복)

5
추출 — 캐릭터 제거 + 조각 적립 (한 트랜잭션)
DT_Character_Extract[등급] → 지급 조각. BEGIN → DELETE owned_characters(1행) → UPDATE player.Shards → COMMIT. 캐릭터 소멸과 조각 적립이 한 트랜잭션이라 중간 상태가 불가능합니다.
6
강화 — 조각 차감 + 단계 상승 (한 트랜잭션)
개화된 비-최대 능력치 중 가중 랜덤으로 하나 선택 → 그 능력치 등급의 DT_Ability_Enhance.Upgrade_ShardAmount만큼. BEGIN → UPDATE player.Shards + UPDATE owned_abilities.Step → COMMIT.
🔒

왜 트랜잭션인가. 추출·강화는 두 테이블을 함께 고칩니다. BEGIN…COMMIT으로 묶으면 크래시가 나도 "조각만 날아가고 캐릭터·단계는 그대로" 같은 중간 상태가 원천 차단됩니다 — 풀-DB로 얻은 가장 큰 이점입니다.

06 — 상태 보호

상태 DB는 어떻게 보호되는가 — 인증 암호화 + 자동 복원

Archive.db는 유저 PC의 Saved/ 폴더에 있는 로컬 파일입니다 — DB Browser로 열어 Gold를 고칠 수 있죠. 서버 없는 싱글 게임에서 "완전 차단"은 불가능하지만, 인증 암호화로 "고쳐도 게임이 알아서 되돌린다"를 만듭니다. (구체 컬럼 타입·암호 코드는 05 스키마 명세에서 다룹니다.)

핵심 아이디어 — 암호화 사본이 "진짜 값"이다

player 행의 치트 유인이 큰 값(Gold·Shards·AccountXp)을 암호화한 사본(Secured 블롭)으로 같은 행에 함께 저장합니다. 이 블롭이 값의 정본(source of truth)이고, 평문 3칸은 게임이 빠르게 읽고 개발 중 눈으로 확인하는 편의 사본일 뿐입니다.

💾 저장할 때 (쓰기)

EconomySubsystem의 트랜잭션 안에서 평문 3칸 + Secured 블롭을 함께 갱신합니다. 둘은 항상 같은 COMMIT으로 묶여, 어긋난 채로 남지 않습니다.

UPDATE player평문 = 값Secured = 암호화(값)

📂 로드할 때 (읽기)

Secured 블롭을 복호화·인증해 진짜 값을 얻고, 평문과 비교합니다. 평문이 밖에서 편집됐으면 복호화한 값을 채택 = 자동 복원. 아래 흐름이 그 판정입니다.

복호화비교자동 복원
flowchart LR
    D1{"Secured 블롭\n복호화·인증 통과?"} -- 아니오 --> A["복원 불가 → 경고·새 게임/백업"]
    D1 -- 예 --> D2{"평문 == 복호화 값?"}
    D2 -- 예 --> B["정상 진행"]
    D2 -- 아니오 --> C["자동 복원 — 복호화 값 채택 + 알림"]

왜 인증 암호화 하나면 되나

인증 암호화(예: AES-GCM)는 블롭 안에 진짜 값(복원용)과 위조 감지(인증 태그)를 한 번에 담습니다. 그래서 별도의 체크섬 컬럼이 필요 없습니다 — 블롭이 변조되면 복호화 단계에서 인증이 실패해 바로 들통나고, 평문만 변조되면 복호화한 값으로 되돌립니다. 탐지와 복원을 한 메커니즘이 동시에 처리합니다.

⚠️
솔직한 한계 두 가지. ① 키는 클라이언트에 심깁니다 — 빌드에서 키를 추출한 아주 소수는 블롭까지 정상적으로 다시 만들 수 있습니다(완전 방어는 서버 권위뿐). 캐주얼~중급 변조는 막지만 절대적이지는 않습니다. ② 평문+블롭을 둘 다 파괴하면 복원이 불가능해 새 게임으로 갈 수밖에 없습니다. 방치형 싱글 게임에는 이 선(무결성 검사 + 자동 복원)이 비용 대비 적정합니다.
🎯

무엇을 보호하나(범위). 치트 유인이 큰 값 — player의 재화·레벨 — 만 Secured로 감쌉니다. owned_*·buildings까지 확장할지는 비용 대비 실익으로 판단합니다(05에서 표로 정리). archives(기록)는 위·변조 유인이 적어 대상에서 제외합니다. SQLite 트랜잭션이 이미 자연 손상(크래시)은 막으므로, 불일치는 대부분 고의 편집 신호입니다.

07 — 확장 규칙

새 데이터를 더할 때

배치는 두 질문으로 정해집니다 — 껐다 켜도 남아야 하나(→DB), 기획이 시트로 튜닝하나(→ROW). 그다음 축(계정·길드)과, 치트 유인이 있으면 보호(Secured)를 지정합니다.

✓ 이렇게

  • 새 능력치는 DT_Ability_Type에 종류 행 + DT_Ability_Step·DT_Ability_Enhance에 행 추가. 새 건물은 DT_Building + DT_Building_Upgrade
  • 밸런스 수치만 바꿀 땐 시트→CSV로 표의 값만 고친다(코드·DB 무변경)
  • 영속 상태는 DB 테이블 — 상태=행 UPDATE / 기록=행 INSERT. 저장 변수마다 계정/길드 축을 소유 결정표에 기록
  • 기록 삭제는 DeletedAt UPDATE(소프트 삭제) — 행 DELETE는 휴지통 비우기에만
  • 여러 테이블을 고치는 작업(추출·강화)은 BEGIN…COMMIT 트랜잭션으로 묶기
  • 치트 유인이 큰 값을 player에 추가하면 Secured도 같은 트랜잭션에서 갱신
  • 스키마 변경: 개발 중엔 DB 파일 삭제(09 절차), 출시 후엔 PRAGMA user_version + ALTER TABLE

✗ 이건 피하기

  • 표에서 조회 가능한 값(효과값·비용)을 DB 행에 복사하기 — 밸런스 패치가 안 먹는다
  • 파생 가능한 값(레벨 = XP 파생)을 저장하기 — 시간·XP와 어긋난다
  • 여러 테이블 변경을 트랜잭션 없이 쓰기 — 크래시 시 "조각만 사라진" 중간 상태
  • 같은 값을 두 서브시스템이 쓰기 — 주인은 하나(나머지는 읽기만)
  • 보호 값을 평문만 갱신하고 Secured를 빼먹기 — 로드 시 자동 복원이 옛 값으로 되돌려버린다
  • 영속 데이터를 다시 SaveGame(.sav)으로 저장하기 — 풀-DB로 통일했다
