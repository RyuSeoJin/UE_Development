데이터 설계
DataTable · SaveGame

V2의 데이터가 어디에 살고(정적 표 vs 런타임 세이브), 어떻게 서로 연결되는지의 지도입니다. 핵심 원칙은 "밸런스 값은 표에서 조회하고, 세이브에는 내가 뽑아 정해진 것만 담는다"입니다.

01 — 지도 읽는 법

수명 배지 먼저

이 문서의 모든 데이터에는 수명 배지가 붙습니다. "이게 어디 살고 언제 사라지는가"를 배지 색으로 먼저 구분하세요.

배지의미V2 예시
ROWDataTable 행 — 손으로 채우는 밸런스 데이터(에디터/CSV)DT_Ability_Step, DT_Gacha_Rate_ByLevel
SAVESaveGame으로 디스크에 저장 — 껐다 켜도 남음FOwnedCharacter, Gold, Shards
RUNTIME런타임 휘발 — 계산값, 세션 종료와 함께 소멸CurrentGuildId, 산출 계산 임시값
CONFIGDeveloperSettings 상수 — 기획자가 에디터에서 조정EstateFocusBaseNx, IdleOfflineCapHours
🧭

한 문장 원칙. ROW는 밸런스(손으로 채움), SAVE는 내 소유물(인스턴스가 정한 것). 세이브는 표에 있는 값을 복사하지 않고 RowName으로 참조합니다 — 그래서 수치 튜닝은 표만 고치면 됩니다.

02 — 테이블/파일 지도

무엇이 어디 사는가

🚀 이 문서로 작업 시작하기

🧭

전제 조건 — 01 문서 완료(세이브 구조·RecordTalesTypes.h 존재). 이 문서는 착수 순서 2번 — 02·04의 시스템이 조회할 테이블을 여기서 실물로 만듭니다. 공통 절차는 위키 『C++ 클래스 생성 · 빌드 워크플로우』.

작업방법비고
① 행 구조체 헤더 RecordTalesTableRows.h클래스 위저드 ▸ 부모 None(빈 헤더) → 이 문서의 행 구조체들(FTableRowBase 상속 USTRUCT) 작성테이블 1종 = 행 구조체 1개
② 재빌드에디터 종료 → IDE 빌드 → 재실행새 헤더이므로 Live Coding 불가
③ DT_* 애셋 4종 생성Content Browser에서 Content/Data/ 폴더 생성 → 우클릭 ▸ Miscellaneous ▸ Data Table ▸ 행 구조체 선택 → 이름 DT_Character · DT_Ability · DT_Gacha · DT_Building구조체를 먼저 컴파일해야 목록에 나타남 — ②를 건너뛰면 선택지가 비어 있음
④ 행 데이터 입력DT 애셋 더블클릭 → ＋ Add로 행 추가, Row Name은 영문 키(예: archer_01)대량 입력은 우클릭 ▸ Export/Import CSV 활용 가능
💡

코드 외 작업이 곧 본체인 문서입니다 — 이 문서 산출물의 절반은 코드(행 구조체)가 아니라 에디터에서 만드는 DT 애셋과 그 안의 데이터입니다. 완료 확인: 4개 DT 애셋이 Content/Data/에 있고, 각 애셋을 열면 이 문서의 열 구성이 그대로 보여야 합니다.

03 — 소유 결정

표에서 조회? 세이브에 저장?

새 데이터를 추가할 때 "표 vs 세이브"를 정하는 기준입니다. 파생 가능하면 저장하지 않고 계산·조회합니다.

데이터둘 곳근거
능력치 효과값 (예: 집중배율 3.5)ROW 조회DT_Ability_Step.Value에서 (AbilityId, 단계)로 조회. 세이브엔 CurrentStep만.
강화 비용ROW 조회DT_Ability_Step.ShardCostToNext. 튜닝은 표만 고침.
건물 레벨별 산출·슬롯ROW 조회DT_Building_Upgrade. 세이브엔 Level만.
캐릭터 등급 · 능력치 현재 단계SAVE뽑을 때 인스턴스에 롤·이후 강화로 변함 — 파생 불가, 저장 필수.
Gold · 조각 · 보유 목록 · 건물 상태SAVE내 소유물. 껐다 켜도 남아야 함.
길드 레벨RUNTIME 계산FocusSeconds / 3600로 매번 파생 — 저장하면 시간과 어긋날 여지가 생김.
04 — 연결 지도

세이브가 표를 가리키는 법

세이브의 인스턴스는 값을 복사하지 않고 RowName(키)으로 DataTable을 참조합니다. 아래는 그 참조 연결입니다.

FOwnedCharacter.CharacterId
SAVE (FName)
RowName 참조→
DT_Character
ROW (정체성)
FOwnedAbility.AbilityId + CurrentStep
SAVE
(AbilityId+단계)로 조회→
DT_Ability_Step
ROW (값·비용)
FBuildingState.BuildingId + Level
SAVE
(건물id+레벨)로 조회→
DT_Building_Upgrade
ROW (산출·슬롯)
💡

왜 참조인가. 값을 세이브에 복사해두면, 밸런스를 바꿀 때 이미 저장된 수천 개의 값이 옛 수치로 남습니다. RowName만 저장하면 표를 고치는 순간 모든 인스턴스가 새 밸런스를 즉시 반영합니다.

05 — 흐름 추적

시나리오: 뽑기 한 번

캐릭터 하나를 만드는 데 표들이 어떤 순서로 쓰이는지 따라가 봅니다(이 과정을 3번 반복해 3장을 보여주고 1택).

1
캐릭터 등급 롤
DT_Gacha_Rate_ByLevel[계정 레벨] → 캐릭터 등급(노말~레전더리). 계정 레벨은 AccountFocusSeconds에서 파생.
2
개화 슬롯 수
DT_..._Open_Chance[등급] → 4슬롯 중 몇 개가 개화되나.
3
슬롯마다 능력치 결정
개화된 슬롯마다 DT_..._Select_Chance[등급] → 능력치 등급 → DT_Ability_Type에서 능력치 선택 → DT_Ability_Step의 RollWeight로 시작 단계 롤.
4
FOwnedCharacter 생성(SAVE)
CharacterId · Rarity · Abilities[AbilityId + CurrentStep]만 저장. 효과값은 저장 안 함(표에서 조회).

시나리오: 능력치 강화

5
가중 랜덤 대상 + 비용 지불
개화된 비-최대 능력치 중 가중 랜덤(=남은 단계)으로 하나 선택 → 현재 단계의 DT_Ability_Step.ShardCostToNext만큼 조각 지불 → CurrentStep += 1. 세이브는 CurrentStep만 바뀜.
06 — 확장 규칙

새 데이터를 더할 때

✅

이렇게: 새 능력치는 DT_Ability_Type에 종류 행 + DT_Ability_Step에 단계 행들을 추가한다. 새 건물은 DT_Building + DT_Building_Upgrade. 밸런스 수치만 바꿀 땐 표의 값만 고친다(코드·세이브 무변경). 새 상수는 DeveloperSettings에.

⚠️

이건 피하기: 표에서 조회 가능한 값을 세이브에 복사하지 말 것 — 밸런스 변경이 안 먹는다. 파생 가능한 값(레벨=시간/3600)을 저장하지 말 것. SaveGame 구조를 바꾸면 반드시 SaveVersion += 1 + 로드 시 마이그레이션 분기를 둔다.
