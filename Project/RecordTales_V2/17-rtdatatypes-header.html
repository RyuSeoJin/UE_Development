RTDataTypes.h
전체판 · 복사 정본

DB 행을 담는 공용 그릇(enum 4종 · 캐시 struct 6종)의 단일 보관함을 한 상자에 담았습니다 — 복사해서 붙여 넣으면 파일이 완성됩니다. 저장 계층 확정분(길드 · 아카이브)은 마스터 페이지 ①과 동일하고, 가챠 · 영지 그릇은 Phase 2 구현 시 그대로 사용할 예정분입니다. 구조(컬럼 · 타입)의 정본은 05 데이터 스키마 명세입니다.

enum 4종 · struct 6종 복사 → 붙여넣기 → 빌드 02-B 기능 구현 명세
01 — 개요

이 파일이 맡는 것

RTDataTypes.h는 여러 시스템이 공유하는 타입(모양)만 모아두는 헤더입니다 — 로직 · 값 · 상태가 전혀 없습니다. DB(Archive.db)의 행을 메모리로 읽어올 때 담는 캐시 struct와, 등급 · 종류를 표현하는 enum이 삽니다. 어떤 헤더에도 의존하지 않으므로(엔진 기본 제외) 모든 서브시스템 · UI가 순환 참조 걱정 없이 include할 수 있습니다.

🧭

정본 관계. 컬럼 · 타입 구조의 정본 = 05 데이터 스키마 명세 / 저장 계층 확정분 코드 = 마스터 페이지 ①(이 문서와 문자 그대로 동일 유지) / 이 문서 = 파일 전체판(Phase 2 그릇 포함)의 복사 정본. 설계가 바뀌면 셋을 함께 개정합니다. 시트 행 구조체(F*Row)는 이 파일이 아니라 RTTables.h 소관입니다.

🚀 이 문서로 작업 시작하기

🧭

전제 조건 — C++ 프로젝트면 충분합니다. SQLite 배관 · 문지기와 무관하게 이 파일 하나만 먼저 만들어도 되고(타입만이라 의존 0), 마스터 페이지의 "일곱 파일" 순서에서는 1번(가장 먼저)입니다.

만들 파일부모 클래스 (위저드 검색어)비고

💡

02 — 구성

enum 4종 · struct 6종 — 무엇에 대응하나

각 타입이 어느 DB 테이블 · 표와 짝인지, 그리고 지금 가동 중인 확정분인지 Phase 2 예정분인지의 지도입니다. 예정분도 코드에 미리 넣어 두는 이유: 타입 선언은 비용이 0이고, 가챠 · 영지 구현 때 이 파일을 다시 고치면(새 헤더 수정) Live Coding 중단이 한 번 더 생기기 때문입니다.

타입대응상태
ERecordSourcearchives.Source 컬럼(INTEGER)확정 — 가동 중
FGuildDataguilds 1행 (+파생 함수 GetLevel())확정 — 가동 중
FArchiveCategoryarchive_categories 1행확정 — 가동 중

EEffectTypeDT_Ability_Type.EffectType 컬럼Phase 2 예정

여기 없는 것들이 없는 이유. player의 값(Gold · Shards · AccountFocusSeconds · LastRewardedLevel)은 항상 1행이라 struct 없이 EconomySubsystem 멤버로 직접 들고, FSecuredAccount는 봉인 전용이라 RTSecured.h에, 시트 행 구조체(F*Row 11종)는 RTTables.h에 삽니다.

03 — 전체 코드

복사해서 붙여 넣으면 됩니다

우측 상단 복사 버튼으로 전문을 복사해 RTDataTypes.h에 붙여 넣으세요. 구획 주석(/* ==== */)으로 확정분과 Phase 2 예정분을 나눠 두었습니다. 

// RTDataTypes.h — DB 행을 담는 공용 그릇(enum·struct)의 단일 보관함. 로직 없음
// 구조 정본: 05 데이터 스키마 명세 · 저장 계층 확정분은 마스터 페이지 ①과 동일
#pragma once
#include "CoreMinimal.h"
#include "RTDataTypes.generated.h"   // UENUM·USTRUCT 리플렉션 코드 — 항상 맨 마지막 include

/* ============================================================
   enum 4종 — 등급·종류의 공용 어휘
   ============================================================ */

// 캐릭터 등급 — 뽑을 때 롤되어 owned_characters.Rarity에 숫자로 저장된다.
// 표 조회 시 "Character_" 접두사 + enum 이름(Rarity_Normal…)을 조합해 RowName을 만든다
UENUM(BlueprintType)
enum class ECharacterRarity : uint8
{
    Rarity_Normal    UMETA(DisplayName = "노말"),      // 순서 = 등급 오름차순. DB에 0~3 숫자로 저장되므로
    Rarity_Rare      UMETA(DisplayName = "레어"),      // 순서를 바꾸면 기존 세이브와 어긋남 — 새 항목은 뒤에만 추가
    Rarity_Epic      UMETA(DisplayName = "에픽"),
    Rarity_Legendary UMETA(DisplayName = "레전더리")
};

// 능력치 등급 — 능력치 자체의 등급(효과 크기 tier). 캐릭터 등급과 별개 축이라 타입을 분리한다.
// 표 조회 시 "Ability_" 접두사 + enum 이름 조합
UENUM(BlueprintType)
enum class EAbilityRarity : uint8
{
    Rarity_Normal    UMETA(DisplayName = "노말"),
    Rarity_Rare      UMETA(DisplayName = "레어"),
    Rarity_Epic      UMETA(DisplayName = "에픽"),
    Rarity_Legendary UMETA(DisplayName = "레전더리")
};

// 능력치 효과 종류 — DT_Ability_Type.EffectType 컬럼에 대응. 전부 골드 건물 배치 시 적용
UENUM(BlueprintType)
enum class EEffectType : uint8
{
    FlatGold,    // 건물 산출식의 flat 골드 가산
    IdleMult,    // 유휴(오프라인) 배율 보너스
    FocusMult    // 집중 중 배율 보너스
};

// 기록 출처 — archives.Source 컬럼(INTEGER)에 대응.
// 순서를 바꾸면 기존 DB의 숫자와 어긋나므로 새 항목은 반드시 뒤에만 추가할 것
UENUM(BlueprintType)
enum class ERecordSource : uint8
{
    Timer,       // 집중 타이머
    Stopwatch,   // 스톱워치
    Quest        // 퀘스트 완료
};

/* ============================================================
   가챠·캐릭터 그릇 — Phase 2 예정분 (구현 시 그대로 사용)
   ============================================================ */

// 개화된 능력치 하나 — owned_abilities 1행에 대응. 효과값·강화 비용은 표에서 조회하므로 담지 않는다
USTRUCT(BlueprintType)
struct FOwnedAbility
{
    GENERATED_BODY()
    UPROPERTY(BlueprintReadOnly) int32 Slot = 0;   // 0~3 — 4개 능력치 슬롯 중 어디에 개화됐나
    UPROPERTY(BlueprintReadOnly) FName AbilityId;  // DT_Ability_Type / _Step의 RowName 참조 키
    UPROPERTY(BlueprintReadOnly) int32 Step = 0;   // 현재 강화 단계 — 효과값은 DT_Ability_Step에서 조회(복사 저장 금지)
};

// 보유 캐릭터 하나 — owned_characters 1행 + 딸린 owned_abilities N행을 한데 담는다
USTRUCT(BlueprintType)
struct FOwnedCharacter
{
    GENERATED_BODY()
    UPROPERTY(BlueprintReadOnly) int32 InstanceId = 0;   // PK — 건물 배치가 이 번호를 참조(재사용 없음)
    UPROPERTY(BlueprintReadOnly) FName CharacterId;      // DT_Character의 RowName — 이름·아이콘 등 정체성은 표에서 조회
    UPROPERTY(BlueprintReadOnly) ECharacterRarity Rarity = ECharacterRarity::Rarity_Normal; // 뽑을 때 롤된 결과 — 재계산 불가라 저장 필수
    UPROPERTY(BlueprintReadOnly) TArray<FOwnedAbility> Abilities;   // 개화된 능력치(최대 4)
};

/* ============================================================
   영지 그릇 — Phase 2 예정분 (구현 시 그대로 사용)
   ============================================================ */

// 경영 건물 하나 — buildings 1행에 대응
USTRUCT(BlueprintType)
struct FBuildingState
{
    GENERATED_BODY()
    UPROPERTY(BlueprintReadOnly) FName BuildingId;                  // DT_Building / DT_Building_Upgrade의 RowName 참조 키
    UPROPERTY(BlueprintReadOnly) int32 Level = 1;                   // 현재 레벨 — 산출·슬롯 수는 표에서 조회
    UPROPERTY(BlueprintReadOnly) TArray<int32> DeployedInstanceIds; // 배치된 캐릭터 InstanceId 목록 — FK 아님, 정합성은 코드가 관리
    UPROPERTY(BlueprintReadOnly) FDateTime LastCollectTime;         // 유휴 산출 누적의 기준 시각 — 오프라인 캡 계산에 사용
};

/* ============================================================
   길드·아카이브 그릇 — 저장 계층 확정분 (가동 중, 마스터 페이지 ①과 동일)
   ============================================================ */

// 길드 하나 — guilds 1행. 캐릭터·Gold는 여기 없음(계정 공유). 레벨은 파생(저장 안 함)
USTRUCT(BlueprintType)
struct FGuildData
{
    GENERATED_BODY()
    UPROPERTY(BlueprintReadOnly) int32 GuildId = 0;        // guilds.GuildId — AUTOINCREMENT 발급(재사용 없음)
    UPROPERTY(BlueprintReadOnly) FString GuildName;
    UPROPERTY(BlueprintReadOnly) FString Category;         // 대분류 1개(길드 간 중복 허용) — 2축 카테고리 모델
    UPROPERTY(BlueprintReadOnly) int32 FocusSeconds = 0;   // 길드 레벨의 원천 — 기록을 지워도 이 값은 유지
    UPROPERTY(BlueprintReadOnly) FDateTime CreatedAt;      // 생성 시각(UTC)
    UPROPERTY(BlueprintReadOnly) int32 SortOrder = 0;      // UI 목록 정렬 순서
    UPROPERTY(BlueprintReadOnly) FString Icon;             // UI 아이콘 식별자

    int32 GetLevel() const { return FocusSeconds / 3600; }   // 60분=1레벨 — 저장하지 않고 호출 때마다 계산(정직한 레벨)
};

// 소분류 카테고리 하나 — archive_categories 1행. 이름·색·아이콘이 UI에 항상 쌍으로 다닌다
USTRUCT(BlueprintType)
struct FArchiveCategory
{
    GENERATED_BODY()
    UPROPERTY(BlueprintReadOnly) FString Name;         // 플레이어가 정한 이름 — 테이블의 PK
    UPROPERTY(BlueprintReadOnly) FLinearColor Color = FLinearColor::White; // DB엔 "RRGGBBAA" 헥스 TEXT — 로드 시 FColor::FromHex로 복원
    UPROPERTY(BlueprintReadOnly) FString Icon;         // 플레이어가 고른 아이콘 식별자
};

// 아카이브 기록 하나 — archives 1행. 목록 조회는 살아있는 기록(DeletedAt IS NULL)만 담아 반환
USTRUCT(BlueprintType)
struct FArchiveRecord
{
    GENERATED_BODY()
    UPROPERTY(BlueprintReadOnly) FString Id;                    // "YYYYMMDDhhmmssSSS-{GUILDID}" — 발급 후 불변
    UPROPERTY(BlueprintReadOnly) int32 GuildId = 0;             // 길드 스코프 태그 — 길드별 필터 키
    UPROPERTY(BlueprintReadOnly) FString Category;              // 소분류 — archive_categories 풀의 이름 태그
    UPROPERTY(BlueprintReadOnly) ERecordSource Source = ERecordSource::Timer; // 타이머/스톱워치/퀘스트 중 출처
    UPROPERTY(BlueprintReadOnly) FDateTime CreatedAt;           // 정렬의 정본 — 문자열 Id로 정렬하지 않는다
    UPROPERTY(BlueprintReadOnly) int32 DurationSeconds = 0;     // 세션 길이(초)
    UPROPERTY(BlueprintReadOnly) FString Title;                 // 기록 제목(플레이어 입력)
};
04 — 등록 · 확인

붙여 넣은 뒤 3단계

1
파일 생성 — 빈 클래스로
에디터 도구 > 새로운 C++ 클래스 → None(빈 클래스) → 이름 RTDataTypes → 생성. 분리 레이아웃이면 .h는 Source/[프로젝트]/Public/에 생깁니다. 생성된 .h 내용을 03 코드로 전체 교체하고, .cpp는 #include "RTDataTypes.h" 한 줄만 남깁니다.
2
generated.h 규칙 확인
#include "RTDataTypes.generated.h"는 반드시 include 목록의 맨 마지막이어야 합니다 — 언리얼 헤더 툴(UHT)이 리플렉션 코드를 이 지점에 주입하기 때문이며, 순서가 어긋나면 generated.h should be the last #include 컴파일 에러가 납니다. 파일 이름을 바꾸면 이 include의 이름도 함께 바뀝니다(과거 RTTypes → RTDataTypes 개명 때와 동일).
3
빌드 → 블루프린트에서 확인
에디터 종료 → 솔루션 빌드 → 재실행. 확인: 블루프린트에서 Break FGuildData 노드가 검색되고, 변수 타입 목록에 ECharacterRarity가 보이면 리플렉션 등록까지 성공한 것입니다.
05 — 확장 규칙

이 파일에 타입을 더할 때

✓ 이렇게

  • enum 새 항목은 목록 끝에만 추가 — DB에 저장된 기존 숫자(0~N)를 보존
  • 새 DB 테이블이 생기면 대응하는 캐시 struct를 여기에 추가(테이블 1 : struct 1)
  • 파생 가능한 값은 GetLevel()처럼 함수로 — 필드로 저장하지 않기
  • 타입 추가 후에는 05 스키마 명세·마스터 페이지 ①도 함께 개정(정본 동기화)
  • struct 필드에는 항상 기본값을 지정 — 초기화 안 된 쓰레기 값 차단

✗ 이건 피하기

  • enum 항목 순서 변경·중간 삽입 (기존 세이브의 숫자가 다른 의미가 됨)
  • 시트 행 구조체(F*Row)를 여기에 넣기 — FTableRowBase 상속은 RTTables.h 소관
  • 표에서 조회 가능한 값(능력치 효과값 등)을 struct 필드로 복사
  • 이 파일에 함수 로직·상태 변수 넣기 — 타입 선언만 (파생 게터 한 줄은 예외)
  • 다른 프로젝트 헤더 include — 이 파일의 의존 0을 유지(순환 참조 방지)
