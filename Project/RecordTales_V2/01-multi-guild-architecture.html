플레이어가 활동 도메인마다 길드를 운영하되, 수집·경제는 계정 하나로 깊게 유지하는 하이브리드 C 구조입니다. 무엇을 공유하고 무엇을 나누는지, 어떤 서브시스템이 그것을 소유하는지, 데이터가 어떻게 흐르는지를 다룹니다.

하이브리드 C 계정 공유 + 길드별 02-A 아키텍처 패턴
01 — 개념

왜 멀티 길드인가

한 사람이 그림·운동·공부처럼 성격이 다른 활동을 하나의 기록으로 뒤섞으면 혼잡합니다. 멀티 길드는 활동 도메인마다 별도의 길드를 두어, 분야별로 독립된 아카이브·레벨·정체성을 갖게 합니다. "내 그림 길드 Lv40"처럼 분야별 투자가 한눈에 읽힙니다.

단, 도메인마다 가챠·캐릭터 수집을 따로 두면 각 도메인의 게임 경제가 얕아져 재미가 죽습니다. 그래서 하이브리드 C를 택합니다 — 수집·경제는 계정 하나로 공유하고, 기록·성장·정체성만 길드별로 나눕니다.

🧭

핵심 용어. 계정 공유 = 모든 길드가 함께 쓰는 것(캐릭터 로스터·Gold·조각·가챠·영지). 길드별 = 길드마다 따로 갖는 것(아카이브·레벨·카테고리). 현재 길드 = 지금 들어가 작업 중인 길드 — 이 세션의 기록이 이 길드로 태그됩니다.

02 — 관계

무엇이 어디에 사는가

03 — 책임 배치

계정에? 길드에?

새 데이터를 추가할 때 "계정 공유인가, 길드별인가"를 정하는 기준표입니다. 판단 기준은 "이게 도메인마다 달라야 하는가(길드별), 아니면 모든 도메인이 함께 쓰는가(계정)"입니다.

데이터 / 기능둘 곳근거
캐릭터 로스터 · Gold · 조각 · 가챠 확률 계정 공유 수집·경제를 길드마다 쪼개면 얕아짐. 한 번 뽑아 어느 길드 활동으로도 키움.
영지 · 경영 건물(배치·산출) 계정 공유 캐릭터가 계정 공유이므로 그 캐릭터가 일하는 건물도 계정 하나. 홈에서 접근.
아카이브(기록) 길드별(태그) 통합 원장에 저장하되 각 기록을 GuildId로 태그 → 길드별 필터 + 전체 조망 둘 다.
레벨(정직한 레벨) 길드별 도메인마다 투자 시간이 달라야 함. 60분=1레벨, 그 길드 집중 시간에서 파생.
카테고리 길드별 Guild→Category 2단. 그림 길드=캐릭터/배경/프롭처럼 도메인마다 다름.
가챠 확률의 입력(계정 XP) 계정 합산 전 길드 활동 시간 합. 어느 도메인을 하든 공용 수집이 전진.
04 — 흐름

세션 한 번이 흐르는 길

집중 세션을 완료하면 시간이 두 곳(길드 레벨 + 계정 XP)으로 라우팅되고, 기록이 현재 길드로 태그됩니다.

1
길드 진입 → 현재 길드 설정
홈에서 길드 건물을 선택하면 UGuildSubsystem.CurrentGuildId가 그 길드로 설정됩니다. 이후 기록·활동이 이 길드로 태그됩니다.
2
활동 → 시간 두 곳 적립
집중 N초 완료 시: 그 길드 FocusSeconds += N (길드 레벨) 그리고 계정 AccountFocusSeconds += N (가챠 확률). 정직한 레벨이라 캐릭터·Gold는 여기 관여하지 않습니다.
3
기록 생성 → 통합 원장에 태그 저장
UArchiveSubsystem이 기록을 만들어 (현재 GuildId → 카테고리 → 소스)로 태그해 통합 원장에 저장합니다.
4
Gold 지급 → 공유 지갑
활동 보상 Gold는 계정 공유 지갑(FEconomyState.Gold)으로. 이후 가챠·업그레이드·해금에 쓰입니다.
💡

느슨한 결합. "시간 적립 → 기록/경제"는 직접 호출보다 델리게이트(Delegate, 콜백)로 잇는 것을 권장합니다. UGuildSubsystem이 UEconomySubsystem·UArchiveSubsystem의 존재를 몰라도 이벤트를 흘려보내면, 나중에 시스템을 추가·교체할 때 수정 범위가 줄어듭니다.

05 — 스켈레톤

뼈대 코드

🚀 이 문서로 작업 시작하기

🧭

전제 조건 — 00 개요의 공통 전제(C++ 프로젝트·IDE) 충족. 이 문서가 착수 순서 1번입니다 — 다른 시스템(02·03·04)이 여기서 만드는 세이브 구조·서브시스템에 의존하므로 가장 먼저 만듭니다. 클래스 생성·컴파일의 공통 절차는 위키 『C++ 클래스 생성 · 빌드 워크플로우』(core-cpp)를 따릅니다.

만들 파일 (위저드 순서대로)부모 클래스 (위저드 검색어)비고
RecordTalesTypes.hNone (빈 헤더)FGuildData 등 공용 USTRUCT의 집 — 여러 클래스가 공유하므로 별도 헤더

💡

코드 외 작업 — ① Subsystem들은 엔진 등록 불필요(GameInstance 수명에 자동 생성·소멸) ② SaveGame도 애셋·설정 등록 없음(코드의 슬롯 이름으로 저장/로드) → 이 문서 범위에선 에디터 작업이 없습니다. 검증은 PIE에서 UE_LOG로 Initialize 호출 확인부터. ③ 세이브와 서브시스템을 잇는 로드·저장 타이밍(Initialize 로드 → 변경 즉시 저장 → Deinitialize 보험, InitializeDependency 순서 보장)은 위키 『Subsystem ↔ SaveGame 연동』(framework/) 문서의 패턴을 그대로 적용하세요.

⚠️
컴파일 주의 — 여기서 만드는 것은 전부 새 클래스·헤더이므로 Live Coding 대상이 아닙니다. 클래스를 다 만든 뒤 에디터를 끄고 IDE에서 빌드 → 재실행이 안전합니다(상세: 위키 C++ 워크플로우 04).

핵심 구조의 뼈대입니다(세부 구현 아님). 길드별 데이터는 세이브 안의 배열 원소로, 계정 공유는 세이브 최상위에 둡니다.

FGuildData — 길드 하나의 데이터 (per-guild)

// 길드 하나 = 계정 세이브 안의 배열 원소. 계정 공유 자산은 여기 넣지 않는다.
USTRUCT()
struct FGuildData
{
    GENERATED_BODY()
    UPROPERTY() int32   GuildId = 0;          // 생성순 번호(계정 내 고유, 삭제해도 재사용 없음)
    UPROPERTY() FString GuildName;             // "그림 길드" 등 사용자 지정 이름
    UPROPERTY() TArray<FString> Categories;    // 이 길드의 카테고리 세트(Guild→Category 2단)
    UPROPERTY() int32   FocusSeconds = 0;     // 이 길드 누적 집중 시간(초) — 레벨의 원천
    UPROPERTY() bool    bDeleted = false;      // 소프트 삭제 플래그(휴지통)
    UPROPERTY() FDateTime DeletedTime;          // 삭제 시각(무기한 보존)

    // 레벨은 저장하지 않고 시간에서 파생 — 60분=1레벨. 이게 '정직한 레벨'의 핵심.
    int32 GetLevel() const { return FocusSeconds / 3600; }
};

URecordTalesSaveGame — 계정 공유 + 길드 배열

UCLASS()
class URecordTalesSaveGame : public USaveGame
{
    GENERATED_BODY()
public:
    // ── 계정 공유 (게임 자산 — 모든 길드가 함께 씀) ──
    UPROPERTY() TArray<FOwnedCharacter> Roster;   // 뽑은 캐릭터(중복 허용, 보유 무제한)
    UPROPERTY() TArray<FBuildingState>  Estate;   // 영지(경영 건물 — 계정 하나)
    UPROPERTY() FEconomyState           Economy;  // Gold · Shards
    UPROPERTY() int32 AccountFocusSeconds = 0;    // 전 길드 집중 시간 합 → 가챠 확률 입력
    UPROPERTY() int32 NextInstanceId = 1;         // 캐릭터 인스턴스 번호 카운터(중복 구분)

    // ── 길드별 (per-guild) ──
    UPROPERTY() TArray<FGuildData>    Guilds;   // 멀티 길드 배열
    UPROPERTY() TArray<FArchiveRecord> Archive;  // 통합 원장(각 기록에 GuildId 태그)
    UPROPERTY() int32 NextGuildId = 1;             // 길드 생성순 번호 카운터(재사용 없음)
};

UGuildSubsystem — 길드 로직 + 현재 길드 컨텍스트

UCLASS()
class UGuildSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()
public:
    // 지금 들어가 있는 길드 — 세션·기록이 이 값으로 태그된다
    int32 CurrentGuildId = 0;

    // 활동 완료 시: 그 길드 FocusSeconds + 계정 AccountFocusSeconds 양쪽에 적립
    void AddFocusTime(int32 Seconds);

    // 새 길드 생성 — NextGuildId를 부여(재사용 없음)하고 그 번호를 반환
    int32 CreateGuild(const FString& Name);

    // 소프트 삭제 / 복원 / 완전 삭제 (휴지통)
    void SoftDeleteGuild(int32 GuildId);  // bDeleted=true, 그 길드 기록도 함께 숨김(cascade)
    void RestoreGuild(int32 GuildId);     // bDeleted=false로 복원
    void PurgeGuild(int32 GuildId);       // 실제 제거 + 그 길드 기록도 완전 삭제
};
🧭

레벨을 저장하지 않는 이유. GetLevel()이 FocusSeconds / 3600으로 매번 계산하므로, 레벨은 항상 시간과 일치합니다. 저장된 레벨 값이 시간과 어긋날 여지가 없어, "레벨 = 투자 시간"의 정직함이 코드 수준에서 보장됩니다.

06 — 확장

새 기능을 얹을 때

이 아키텍처 위에서 무엇을 어떻게 추가하는지, 그리고 피해야 할 것입니다.

✅

이렇게: 새 데이터를 추가할 때 03 결정표로 "계정 vs 길드"를 먼저 판단한다. 새 길드는 CreateGuild로 NextGuildId를 받아 만든다(번호 재사용 금지). 삭제는 항상 소프트 삭제부터 — bDeleted 플래그로 숨기고, 완전 삭제는 별도 명령으로만.

⚠️

이건 피하기: 캐릭터·Gold·영지를 길드 안(FGuildData)에 넣지 말 것 — 계정 공유가 깨져 수집이 얕아진다. 레벨을 세이브에 숫자로 저장하지 말 것 — 시간과 어긋날 수 있다. 삭제한 길드의 GuildId를 재사용하지 말 것 — 과거 기록의 태그가 엉킨다.
