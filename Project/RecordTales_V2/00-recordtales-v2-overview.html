RecordTales V2
프로젝트 개요

집중 관리 + 멀티 길드 + 영지 경영 게임화를 언리얼 엔진으로 구현하는 재설계(V2)의 전체 그림입니다. 이 문서는 각 시스템이 어떻게 맞물리는지, 핵심 어휘가 무엇인지를 먼저 잡아 이후 상세 문서의 토대를 만듭니다.

멀티 길드 하이브리드 C 정직한 레벨 (60분=1레벨) 영지 · 경영 · 가챠
01 — RecordTales V2란?

무엇을 만드나

RecordTales는 포모도로(Pomodoro, 25분 집중 + 5분 휴식) 기반 집중 관리에 게임화(Gamification)를 결합한 생산성 앱입니다. V2는 이 앱을 크게 재설계합니다. 사용자는 집중 타이머·스톱워치로 작업하고 퀘스트(할 일)를 수행하며, 그 활동으로 레벨을 올리고 Gold를 벌어 가챠(캐릭터 뽑기)와 영지 경영을 즐깁니다.

V2의 두 축은 이렇습니다. 첫째, 정직한 레벨 — 레벨은 오직 실제 집중 시간으로만 오릅니다(60분 = 1레벨). 캐릭터나 Gold로는 못 올려, 레벨이 곧 "내가 그 분야에 투자한 시간"의 거울이 됩니다. 둘째, 멀티 길드 — 활동 도메인마다(예: 그림·운동·공부) 별도의 길드를 두어 분야별로 깔끔하게 나눠 기록·성장합니다.

02 — 시스템 지도

큰 순환: 활동 → 성장 → 게임

아래는 각 시스템이 무엇을 주고받는지를 보여주는 관계도입니다. 화살표는 데이터가 흐르는 방향입니다. 핵심 순환은 "실제 활동으로 레벨·Gold를 얻고 → Gold로 캐릭터를 뽑아 → 영지 경영 건물에 배치해 다시 Gold·조각을 번다"입니다.

03 — 핵심 개념 · 어휘

먼저 알아둘 용어

V2에서 반복 등장하는 핵심 용어를 여기서 한 번씩 풀어둡니다. 이후 문서들은 이 정의를 재사용합니다.

용어정의
멀티 길드활동 도메인마다 두는 길드. 계정 공유(로스터·Gold·조각·가챠·영지) + 길드별(아카이브·레벨·카테고리)로 나뉘는 하이브리드 구조.
정직한 레벨실제 집중 시간으로만 오르는 길드 레벨(60분=1레벨). 캐릭터·Gold로 못 올림. 레벨 = 투자 시간의 거울.
계정 XP모든 길드의 활동 시간을 합산한 값. 가챠 등급 출현 확률을 결정.
영지(Estate)계정 단위의 홈 화면. 경영 건물을 배치·관리하는 공간(모든 길드 공유).
골드 건물캐릭터를 배치하면 그 능력치로 Gold를 산출하는 경영 건물.
기억 거래소캐릭터를 추출해 조각을 얻는 건물(배치·상시 산출 없음).
능력치 개화캐릭터의 능력치 슬롯(최대 4)이 열리는 것. 캐릭터 등급이 개화 슬롯 수와 뜨는 능력치 등급을 결정.
추출(Extract)보유한 캐릭터를 소멸시켜 조각(Shard)을 얻는 행동. 기억 거래소에서 실행하며, 캐릭터 등급이 높을수록 더 많은 조각을 지급.
조각(Shard)추출(Extract)로 얻는 강화 재료. 능력치 단계를 올리는 데 씀.
소프트 삭제 · 휴지통길드·기록을 삭제해도 데이터를 남겨 휴지통에서 복원 가능. 완전 삭제는 별도(로컬 저장).
🧭

등급이 둘 있습니다. "캐릭터 등급"(뽑을 때 롤됨 — 개화 수·능력치 등급을 결정)과 "능력치 등급"(각 능력치 자체의 등급 — 효과 크기를 결정)은 서로 다른 개념입니다. 상세는 가챠 설계 문서를 참조하세요.

04 — 계정 공유 vs 길드별

무엇이 어디에 속하나

멀티 길드의 핵심은 "무엇을 공유하고 무엇을 나누느냐"입니다. 수집·경제는 계정 하나로 깊게 유지하고, 정체성·기록은 길드마다 나눕니다.

계층속하는 것이유
계정 공유 캐릭터 로스터 · Gold · 조각 · 가챠 확률 · 영지(경영 건물) 수집·경제를 길드마다 쪼개면 얕아짐. 한 번 뽑아 여러 길드에서 활용.
길드별 아카이브(기록) · 레벨 · 카테고리 도메인마다 독립된 성장·정체성. "내 그림 길드 Lv40"처럼 분야별로 읽힘.
💡

기록 구조는 Guild → Category 2단입니다. 각 길드가 자기 카테고리 세트를 가집니다(예: 그림 길드 = 캐릭터/배경/프롭). 아카이브는 계정 단위 통합 원장에 저장되며, 각 기록이 (길드 → 카테고리 → 소스)로 태그됩니다.

05 — 데이터 큰 그림

밸런스는 표에, 소유물은 세이브에

V2의 데이터는 두 갈래입니다. 정적 DataTable(손으로 채우는 밸런스 데이터)과 런타임 SaveGame(플레이 중 생기는 내 소유물). 값은 표에서 조회하고, 세이브에는 "내가 뽑아 정해진 것"만 담아 저장을 가볍게 유지합니다. 상세 스키마는 데이터 설계 문서를 참조하세요.

// 정적 DataTable — 밸런스 (손으로 채움) DT_Character_Ability_OpenChance ← 등급 → 개화 슬롯 수 확률 DT_Character_Ability_GradeChance ← 등급 → 능력치 등급 분포 DT_Ability_Type ← 능력치 종류(효과×등급) DT_Ability_Step ← 단계별 값·시작 가중치 DT_Ability_Enhance ← 능력치 등급별 강화 비용(조각, 단계 무관) DT_Character ← 캐릭터 정체성 DT_GachaRateByLevel ← 계정 레벨 → 캐릭터 등급 DT_Building / DT_Building_Upgrade ← 건물 정체성 · 레벨별 산출 DT_Character_Extract ← 등급 → 추출 조각 수(고정) // 런타임 SaveGame — 인스턴스 (계정 단위) FOwnedCharacter { InstanceId, CharacterId, Rarity, Abilities[≤4] } FBuildingState { BuildingId, Level, DeployedInstanceIds, LastCollectTime } FEconomyState { Gold, Shards }
🧭

능력치는 "현재 단계"만 저장합니다. 효과값은 DT_Ability_Step, 강화 비용은 DT_Ability_Enhance에서 조회하므로, 수치 튜닝은 표만 고치면 되고 세이브는 건드리지 않습니다.

기록 먼저, 게임은 그 위에

🚀 작업 시작하기 (프로젝트 공통 전제)

🧭

전제 조건 — ① UE5 프로젝트 생성 완료(템플릿: Blank, Blueprint여도 무방 — 첫 클래스 생성 시 자동으로 C++ 프로젝트로 전환) ② IDE 준비(Windows: Visual Studio 2022 + "Game development with C++" 워크로드 / macOS: Xcode) ③ 클래스 생성·Build.cs·컴파일(Live Coding vs 재빌드)의 공통 절차는 위키의 『C++ 클래스 생성 · 빌드 워크플로우』(core-cpp) 문서가 담당합니다 — 각 문서의 "작업 시작하기" 블록은 그 문서 고유의 파일 목록·코드 외 작업만 담습니다.

착수 순서문서이 문서에서 만드는 것

💡

이 순서는 아래 Phase 구분과 별개로 "코드 의존성" 기준의 착수 순서입니다 — 뒤 문서가 앞 문서의 클래스·테이블을 참조하므로 거꾸로 시작하면 컴파일이 안 됩니다.

게임(가챠·영지)은 기록·집중 본체 위에 얹혀 돌아갑니다(가챠 확률은 활동 XP에서, 건물은 활동 결과로 산출). 그래서 제작도 두 단계로 나눕니다.

1
Phase 1 — 기록 · 집중 본체 (게임 없음)
집중/스톱워치/퀘스트 · 정직한 레벨(60분=1레벨) · 멀티 길드 생성 · 아카이브(기록·복원) · 홈 "오늘 할 일" · Gold 기본 획득. 이것만으로도 "정직하게 기록하는 생산성 앱"이 완성됩니다.
2
Phase 2 — 게임 레이어
가챠(캐릭터·능력치 개화·조각 강화) · 영지 경영 건물(배치·산출·업그레이드) · 자원 순환. 수집·경영의 재미를 Phase 1 위에 얹습니다.
