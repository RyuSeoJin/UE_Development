02 — 설계 결정

왜 이렇게 만드나

결정내용근거
문지기 분리DatabaseSubsystem은 게임 기능 0개 — 열닫·스키마·트랜잭션 창구만기능 시스템(예: Archive)이 DB 문지기를 겸직하면 모든 시스템이 그 클래스에 숨은 의존을 갖게 됨 — 테이블이 7종으로 늘기 전에 분리(11 결정)
핸들 대여기능 시스템은 GetDb()로 FSQLiteDatabase*를 빌려서만 쓴다 — 소유·Close() 금지열닫 주체가 둘이면 "닫힌 핸들에 쿼리" 사고가 생김 — 수명 관리 지점을 하나로
스키마는 한 곳에서CreateTables()가 8테이블을 FK 순서대로 전부 생성① 기능 시스템이 깨어날 땐 테이블 존재가 보장 ② owned_abilities→owned_characters 같은 FK 생성 순서를 한 파일에서 관리 ③ 이후 PRAGMA user_version 마이그레이션도 이곳 하나만 고침
트랜잭션 창구 = 문지기BeginTransaction / CommitTransaction / RollbackTransaction 3함수 제공 + 중첩 방지 플래그10·12 소유 결정표: "다중 테이블 트랜잭션의 단일 창구 = DatabaseSubsystem". 12에서는 기능 시스템이 BEGIN을 직접 실행했지만, 추출·강화처럼 두 시스템에 걸친 변경이 오기 전에 창구를 승격(이 문서의 결정)
player 1행 보장CHECK(Id=1) + 생성 직후 INSERT OR IGNORE계정은 하나니 행도 하나 — "행이 없어서 UPDATE가 조용히 실패"하는 사고를 스키마가 차단(11)
NextGuildId 유지(미사용)player에 컬럼은 있으나 코드가 안 씀길드 번호는 guilds의 AUTOINCREMENT가 발급(12) — 05 DDL 정본과의 일치를 위해 컬럼은 남겨 둠(제거는 05 개정과 함께)

핵심 용어 — 이 문서에서 계속 쓰는 세 가지
· 문지기(Gatekeeper) — DB의 열닫·스키마·트랜잭션 창구를 전담하는 인프라 서브시스템. 게임 기능이 없어서 어떤 기능 시스템과도 대등합니다.
· InitializeDependency — "저 서브시스템을 나보다 먼저 초기화해 달라"는 엔진 API. UGameInstanceSubsystem의 Initialize()는 앱 시작 순간 엔진이 자동 호출하는데, 호출 순서는 기본적으로 복불복입니다. 이 한 줄이 "문지기 먼저"를 보장하므로 기능 시스템은 깨어나는 순간 이미 DB가 준비돼 있습니다.
· 트랜잭션(Transaction) — 여러 SQL 변경을 "전부 성공 아니면 전부 취소"로 묶는 단위(BEGIN…COMMIT/ROLLBACK). 중간에 크래시해도 절반만 반영된 상태가 남지 않습니다.

1
클래스 위저드로 DatabaseSubsystem 생성
부모는 GameInstanceSubsystem — 수명 = 앱 세션

에디터에서 도구 > 새로운 C++ 클래스 → 모든 클래스 탭 → 검색창에 GameInstanceSubsystem → 부모로 선택 → 이름 DatabaseSubsystem → 생성.

UGameInstanceSubsystem이라서 게임 시작 시 Initialize가, 종료 시 Deinitialize가 자동으로 불립니다. DB를 언제 열고 닫을지 직접 관리할 필요가 없습니다 — 서브시스템의 수명이 곧 DB의 수명입니다.

🔎

확인 — DatabaseSubsystem.h와 DatabaseSubsystem.cpp가 둘 다 생겼다. 분리 레이아웃이면 각각 Public/·Private/에, 플랫이면 같은 폴더에. 헤더만 보이면 Private/ 폴더에서 .cpp를 찾으세요(사라진 게 아니라 형제 폴더에 있습니다).

2
헤더 작성
역할 3+1 — 열고, 스키마 만들고, 닫고, 트랜잭션 창구

아래를 DatabaseSubsystem.h에 붙여 넣으세요. 11의 정본에 트랜잭션 3함수와 중첩 방지 플래그가 더해진 최종형입니다.

// DatabaseSubsystem.h — DB 문지기: Archive.db 열닫·스키마·트랜잭션의 단일 창구
#pragma once
#include "Subsystems/GameInstanceSubsystem.h"   // UGameInstanceSubsystem — 수명 = 앱 세션
#include "SQLiteDatabase.h"                      // FSQLiteDatabase를 "값 멤버"로 품으므로 전방 선언이 아니라 실제 포함이 필요
#include "DatabaseSubsystem.generated.h"          // UHT 생성 헤더 — 항상 맨 마지막 include

// DB 문지기 — 게임 기능 0개. 열고, 스키마를 만들고, 닫고, 트랜잭션 창구를 빌려준다.
// Economy·Archive·Guild·Gacha·Estate 같은 기능 시스템들은 이 클래스"에만" 의존한다
UCLASS()
class RECORDTALES_API UDatabaseSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()

public:
    // 게임(앱) 시작 시 엔진이 자동 1회 호출 — 여기서 DB를 열고 8테이블을 보장한다
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;
    // 게임 종료 시 자동 호출 — 열었던 곳에서 닫는다(수명 대칭)
    virtual void Deinitialize() override;

    // 기능 시스템들에게 핸들을 "빌려준다" — 소유·수명 관리는 오직 여기.
    // 빌린 쪽은 FSQLiteDatabase* 멤버로 들고, 쓰기 전 IsValid()를 확인한다(6단계 관례)
    FSQLiteDatabase* GetDb() { return &Database; }

    // 다중 테이블 변경(추출·강화·카테고리 이름 변경)의 단일 창구 —
    // "값 쓰기"는 각 주인의 일이지만, 묶음의 시작/끝은 문지기의 일(10 소유 결정표)
    bool BeginTransaction();      // BEGIN — 이미 열려 있으면 false(중첩 금지)
    bool CommitTransaction();     // COMMIT — 묶은 변경을 한 번에 확정
    bool RollbackTransaction();   // ROLLBACK — 중간 실패 시 전부 원상 복귀

private:
    void CreateTables();          // CREATE TABLE IF NOT EXISTS 8종 — FK 순서 보장(4단계)
    FSQLiteDatabase Database;      // DB 핸들 실체 — 이 서브시스템만 소유(수명 = 앱 세션)
    bool bInTransaction = false;   // 중첩 BEGIN 방지 — SQLite는 트랜잭션 중첩을 허용하지 않는다
};

3
열고 닫기 구현 — Initialize / Deinitialize
경로는 반드시 Saved/ — 열기 실패는 크래시 대신 우회

아래를 DatabaseSubsystem.cpp에 붙여 넣으세요. 09에서 ArchiveSubsystem이 갖고 있던 "열기" 코드가 그대로 이 자리로 온 것입니다(11의 이관).

// DatabaseSubsystem.cpp — 열기·PRAGMA·스키마·닫기. "그릇 준비"가 사는 유일한 파일
#include "DatabaseSubsystem.h"
#include "Misc/Paths.h"   // FPaths::ProjectSavedDir()

void UDatabaseSubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
    Super::Initialize(Collection);   // 부모 초기화 먼저 — 관례상 항상 맨 위

    // DB 파일은 반드시 Saved/ 아래 — Content/는 패키징 후 쓰기 금지 구역(09의 disk I/O error 함정)
    const FString DbPath = FPaths::ProjectSavedDir() / TEXT("Archive.db");
    UE_LOG(LogTemp, Log, TEXT("[DB] 경로: %s"), *DbPath);   // 실제 경로를 로그로 남겨 확인

    // ReadWriteCreate = 없으면 새로 만들고, 있으면 읽기/쓰기로 연다
    if (!Database.Open(*DbPath, ESQLiteDatabaseOpenMode::ReadWriteCreate))
    {
        // 열기 실패 시 크래시 대신 로그 — 이후 기능 시스템들은 GetDb()->IsValid()==false로 안전하게 우회
        UE_LOG(LogTemp, Error, TEXT("[DB] 열기 실패: %s"), *DbPath);
        return;
    }

    Database.Execute(TEXT("PRAGMA foreign_keys = ON;"));   // FK 강제 — 연결(세션)마다 다시 켜야 함
    CreateTables();                                        // 8테이블 보장(4단계에서 작성)
    UE_LOG(LogTemp, Log, TEXT("[DB] 준비 완료"));           // 이 로그가 8단계 확인 기준
}

void UDatabaseSubsystem::Deinitialize()
{
    if (Database.IsValid()) Database.Close();   // 열었던 곳에서 닫는다 — 빌린 쪽은 절대 Close 금지
    Super::Deinitialize();
}

생성 순서가 중요합니다 — owned_abilities는 owned_characters를 FK로 참조하므로 참조되는 쪽이 먼저 있어야 합니다. 아래 표의 순서를 코드가 그대로 따릅니다.

void UDatabaseSubsystem::CreateTables()
{
    // IF NOT EXISTS = 이미 있으면 그대로 둔다 — 매 실행 안전. 단, 스키마를 "바꿀" 땐
    // 기존 테이블이 안 바뀌므로 개발 중엔 .db 파일 삭제로 리셋한다(검증 섹션 참고)

    // ① player — 계정 상태(항상 1행). CHECK(Id=1): Id≠1인 행은 DB가 거부
    Database.Execute(TEXT(
        "CREATE TABLE IF NOT EXISTS player ("
        "  Id                  INTEGER PRIMARY KEY CHECK (Id = 1),"
        "  Gold                INTEGER NOT NULL DEFAULT 0,"
        "  Shards              INTEGER NOT NULL DEFAULT 0,"
        "  AccountFocusSeconds INTEGER NOT NULL DEFAULT 0,"   // 계정 XP 원천(레벨 = ÷3600, 저장 안 함)
        "  NextInstanceId      INTEGER NOT NULL DEFAULT 1,"   // 캐릭터 인스턴스 카운터(재사용 없음)
        "  NextGuildId         INTEGER NOT NULL DEFAULT 1,"   // (미사용) 길드 번호는 guilds AUTOINCREMENT가 발급
        "  Secured             BLOB"                          // 보호 값 3종의 인증 암호화 사본(11의 7~8단계)
        ");"));
    // 기본 행 보장 — 이미 1행 있으면 OR IGNORE가 조용히 무시(첫 실행에만 삽입됨)
    Database.Execute(TEXT("INSERT OR IGNORE INTO player (Id) VALUES (1);"));

    // ② guilds — 길드(길드당 1행). AUTOINCREMENT: 삭제해도 GuildId 영구 미재사용(12의 결정)
    Database.Execute(TEXT(
        "CREATE TABLE IF NOT EXISTS guilds ("
        "  GuildId      INTEGER PRIMARY KEY AUTOINCREMENT,"  // 재사용 금지를 엔진이 보장
        "  GuildName    TEXT    NOT NULL,"
        "  Category     TEXT,"                                // 대분류 1개(길드 간 중복 허용 — 2축 모델)
        "  FocusSeconds INTEGER NOT NULL DEFAULT 0,"          // 길드 레벨 원천(레벨 = ÷3600)
        "  CreatedAt    TEXT    NOT NULL,"                    // 생성 시각(ISO8601 UTC)
        "  SortOrder    INTEGER NOT NULL DEFAULT 0,"          // UI 정렬 순서
        "  Icon         TEXT,"                                // UI 아이콘 식별자
        "  DeletedAt    TEXT,"                                // 소프트 삭제 시각(NULL = 살아있음)
        "  Secured      BLOB"                                 // GuildId+FocusSeconds 봉인(12의 6단계)
        ");"));

    // ③ archive_categories — 소분류 공유 풀(계정 축). 이름 자체가 키라 중복이 자동 차단되고,
    //    기록(archives.Category)이 이름 문자열로 태그하므로 그대로 조인·비교 가능
    Database.Execute(TEXT(
        "CREATE TABLE IF NOT EXISTS archive_categories ("
        "  Name      TEXT PRIMARY KEY,"
        "  Color     TEXT NOT NULL DEFAULT 'FFFFFFFF',"       // 플레이어 지정 색 — "RRGGBBAA" 헥스 문자열
        "  Icon      TEXT,"                                   // 플레이어 지정 아이콘 식별자
        "  CreatedAt TEXT NOT NULL,"
        "  SortOrder INTEGER NOT NULL DEFAULT 0"              // 목록 표시 순서(관리 UI 대비)
        ");"));

    // ④ owned_characters — 계정 로스터(캐릭터당 1행). ⑤가 참조하므로 반드시 먼저 생성
    Database.Execute(TEXT(
        "CREATE TABLE IF NOT EXISTS owned_characters ("
        "  InstanceId  INTEGER PRIMARY KEY,"                  // 고유 번호(건물 배치·능력치가 참조)
        "  CharacterId TEXT    NOT NULL,"                     // → DT_Character(RowName)
        "  Rarity      INTEGER NOT NULL"                      // 뽑을 때 롤된 등급 — 파생 불가라 저장 필수
        ");"));

    // ⑤ owned_abilities — 개화 능력치(캐릭터 1:N). ON DELETE CASCADE:
    //    캐릭터 행을 지우면(추출) 그 캐릭터의 능력치 행이 자동으로 함께 삭제된다
    Database.Execute(TEXT(
        "CREATE TABLE IF NOT EXISTS owned_abilities ("
        "  AbilityRowId INTEGER PRIMARY KEY AUTOINCREMENT,"
        "  InstanceId   INTEGER NOT NULL,"                    // FK → owned_characters
        "  Slot         INTEGER NOT NULL,"                    // 0~3(4슬롯 중 어디)
        "  AbilityId    TEXT    NOT NULL,"                    // → DT_Ability_Type(효과·등급은 표 조회)
        "  Step         INTEGER NOT NULL DEFAULT 0,"          // 현재 강화 단계(값 = DT_Ability_Step 조회)
        "  FOREIGN KEY (InstanceId) REFERENCES owned_characters(InstanceId) ON DELETE CASCADE"
        ");"));

    // ⑥ buildings — 경영 건물(건물당 1행). 영지는 계정 축
    Database.Execute(TEXT(
        "CREATE TABLE IF NOT EXISTS buildings ("
        "  BuildingId          TEXT    PRIMARY KEY,"          // → DT_Building
        "  Level               INTEGER NOT NULL DEFAULT 1,"   // → DT_Building_Upgrade
        "  DeployedInstanceIds TEXT,"                         // JSON 배열(배치된 캐릭터 InstanceId)
        "  LastCollectTime     TEXT"                          // 유휴 누적 기준(오프라인 캡)
        ");"));

    // ⑦ archives — 아카이브 기록(기록당 1행, 길드 축). 09 배관에서 가동 중인 테이블의 최종형
    Database.Execute(TEXT(
        "CREATE TABLE IF NOT EXISTS archives ("
        "  Id              TEXT    PRIMARY KEY,"              // "YYYYMMDDhhmmssSSS-{GUILDID}"(불변)
        "  GuildId         INTEGER NOT NULL,"                 // 길드 스코프 태그(참조·필터 키)
        "  Category        TEXT    NOT NULL,"                 // 소분류(archive_categories 풀의 이름 태그)
        "  Source          INTEGER NOT NULL,"                 // ERecordSource(타이머/스톱워치/퀘스트)
        "  CreatedAt       TEXT    NOT NULL,"                 // 정렬 정본(문자열 ID로 정렬 금지 — 07 규칙)
        "  DurationSeconds INTEGER NOT NULL DEFAULT 0,"
        "  Title           TEXT,"
        "  DeletedAt       TEXT"                              // 소프트 삭제 시각(NULL = 살아있음)
        ");"));

    // ⑧ settings — 유저 환경설정 키-값(05 v0.21). FK 없음 — 순서 무관이라 맨 끝.
    //   주인(USettingsSubsystem)은 아카이브 뷰 구현 시 신설 — 표는 문지기가 미리 보장한다
    Database.Execute(TEXT(
        "CREATE TABLE IF NOT EXISTS settings ("
        "  Name  TEXT PRIMARY KEY,"                           // 설정 이름(예: 'WeekStartsOn')
        "  Value TEXT NOT NULL"                               // 값은 전부 문자열 — 읽는 쪽이 파싱. 행 없음 = 코드 기본값
        ");"));
}

5
트랜잭션 창구 3함수
여러 테이블을 함께 고칠 때 "절반만 반영"을 원천 차단

구현은 세 함수 모두 Execute 한 줄 + 상태 플래그입니다. 핵심은 중첩 방지 — SQLite는 BEGIN이 열린 상태에서 또 BEGIN하면 에러를 냅니다.

bool UDatabaseSubsystem::BeginTransaction()
{
    // DB가 안 열렸거나 이미 트랜잭션 중이면 거절 — 중첩 BEGIN은 SQLite 에러
    if (!Database.IsValid() || bInTransaction) return false;
    bInTransaction = Database.Execute(TEXT("BEGIN TRANSACTION;"));
    return bInTransaction;
}

bool UDatabaseSubsystem::CommitTransaction()
{
    if (!Database.IsValid() || !bInTransaction) return false;   // 시작한 적 없으면 확정할 것도 없다
    bInTransaction = false;
    return Database.Execute(TEXT("COMMIT;"));   // 묶인 변경 전부를 이 순간 한 번에 확정
}

bool UDatabaseSubsystem::RollbackTransaction()
{
    if (!Database.IsValid() || !bInTransaction) return false;
    bInTransaction = false;
    return Database.Execute(TEXT("ROLLBACK;"));   // BEGIN 이후의 모든 변경을 되돌린다
}
// 사용 예 — 12의 소분류 이름 변경을 창구 경유로 다시 쓰면(GuildSubsystem 안):
// 두 테이블을 함께 고치므로 "풀만 새 이름, 기록은 옛 이름" 중간 상태가 원천 차단된다
if (!Gate->BeginTransaction()) return false;   // Gate = InitializeDependency로 받아 둔 문지기

const bool bOk =
    UpdateCategoryName(OldName, NewName)     // ① archive_categories.Name 갱신
 && RetagArchives(OldName, NewName);        // ② archives.Category 일괄 갱신

if (bOk) Gate->CommitTransaction();          // 둘 다 성공 → 한 번에 확정
else     Gate->RollbackTransaction();        // 하나라도 실패 → 전부 원상 복귀
return bOk;

04 — 검증 & 트러블슈팅

FK와 트랜잭션까지 눈으로 확인

1
FK CASCADE 실험 — "캐릭터를 지우면 능력치도 함께"
게임 종료 상태에서 DB Browser의 SQL 실행 탭에 차례로: ① PRAGMA foreign_keys = ON;(DB Browser는 자체 연결이라 다시 켜야 합니다) ② INSERT INTO owned_characters VALUES (1, 'Char_Test', 1); ③ INSERT INTO owned_abilities (InstanceId, Slot, AbilityId, Step) VALUES (1, 0, 'AB_FlatGold_N', 0); ④ DELETE FROM owned_characters WHERE InstanceId = 1; → owned_abilities를 조회하면 능력치 행이 자동으로 사라져 있어야 합니다. 실험 후 테스트 행이 남았다면 지워 두세요.
2
FK 거부 실험 — 존재하지 않는 캐릭터에 능력치 넣기
같은 탭에서 INSERT INTO owned_abilities (InstanceId, Slot, AbilityId, Step) VALUES (999, 0, 'AB_X', 0); → FOREIGN KEY constraint failed 에러가 나면 성공입니다(FK가 일하고 있다는 뜻 — 3단계에서 켠 PRAGMA의 효과).
3
트랜잭션 실험 — ROLLBACK이 정말 되돌리는지
같은 탭에서 ① BEGIN TRANSACTION; ② UPDATE player SET Gold = 12345 WHERE Id = 1; ③ SELECT Gold FROM player; → 12345 확인 ④ ROLLBACK; ⑤ 다시 SELECT → 원래 값으로 돌아와 있으면 성공. 게임 코드의 RollbackTransaction()이 하는 일이 정확히 이것입니다.

⚠️

CREATE TABLE IF NOT EXISTS는 테이블이 이미 있으면 아무것도 하지 않습니다. 컬럼을 추가·변경했다면: 게임/에디터 완전 종료 → [프로젝트]/Saved/의 Archive.db와 *.db-wal·*.db-shm·*.db-journal을 함께 삭제 → 재실행(새 스키마로 처음부터 생성). 테스트 데이터라 지워도 되는 것이고, 출시 후에는 파일 삭제 금지 — PRAGMA user_version으로 스키마 버전을 관리하고 ALTER TABLE … ADD COLUMN으로 데이터를 지키며 확장합니다. 그 마이그레이션 코드가 들어갈 자리도 CreateTables() 하나입니다(09의 같은 규칙).
