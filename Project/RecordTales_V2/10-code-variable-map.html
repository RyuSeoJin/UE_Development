V2의 축 — 계정 공유 vs 길드별

멀티 길드(하이브리드 C)라서 모든 저장 변수에는 소속 축이 있습니다. 계정 공유 = Gold · 조각(Shard) · 캐릭터 로스터 · 계정 XP/레벨(가챠 확률 결정). 길드별 = 아카이브 기록 · 길드 XP/레벨 · 카테고리. 소유 결정표(03섹션)의 "축" 열이 이걸 고정합니다.

🗄️

저장 전략 = 풀-DB (2026-07-20 확정) — 영속 데이터는 전부 SQLite Archive.db(09 배관 검증)에 저장합니다. 상태(재화·로스터·레벨·건물)는 테이블 행을 UPDATE, 기록(아카이브)은 INSERT로 쌓습니다. 채택 이유: ① 저장소 단일화(이중 저장 정합성 걱정 제거) ② 트랜잭션 — 추출처럼 여러 테이블을 고치는 작업을 BEGIN…COMMIT 하나로 묶어 "중간 상태"가 불가능 ③ DB Browser로 모든 데이터를 눈으로 확인. SaveGame(.sav)은 쓰지 않습니다 — 03·05·07 문서의 SaveGame 기반 설계는 이 지도가 대체하며, 해당 문서 개정은 별도 작업입니다.

💡

파일명 참고 — 실코드(09)의 DB 파일명은 Archive.db인데, 이제 범위가 "아카이브"가 아니라 계정 전체이므로 recordtales.db로 개명을 권장합니다(5단계 코드의 TEXT("...") 한 줄 + 기존 파일 삭제).

06 — 확장 규칙

새 변수가 생기면 이 순서로 배치한다

flowchart LR
    A{"껐다 켜도 남아야 하나?"} -- 아니오 --> R["RUNTIME 멤버"]
    A -- 예 --> B{"누적·검색형 기록인가?"}
    B -- 예 --> D["archives류 테이블 — 행 INSERT"]
    B -- 아니오 --> C{"기획이 시트로 튜닝하는 값?"}
    C -- 예 --> T["DataTable 행 (ROW)"]
    C -- 아니오 --> S["상태 테이블 — 행 UPDATE"]

배치가 정해지면 마지막으로 축을 지정합니다 — 이 값은 계정 공유인가, 길드별인가? (길드별 상태면 guilds처럼 길드당 1행으로, 기록이면 GuildId 컬럼으로.)

✓ 이렇게

  • 밸런스 수치의 원본은 구글 시트 — 문서·코드가 시트를 따라간다
  • 행 구조체 프로퍼티명 = 시트 헤더와 철자 동일(CSV 임포트 조건)
  • 저장 변수마다 계정/길드 축을 소유 결정표에 기록
  • 기록 삭제는 DeletedAt UPDATE(소프트 삭제) — 행 DELETE는 휴지통 비우기에만
  • 여러 테이블을 고치는 작업(추출·강화)은 BEGIN…COMMIT 트랜잭션으로 묶기
  • 스키마 변경: 개발 중엔 DB 파일 삭제(09 절차), 출시 후엔 user_version+ALTER TABLE
  • Lv21+ 가챠는 20행 plateau 폴백으로 처리

✗ 이건 피하기

  • 밸런스 값(확률·비용)을 런타임 DB 테이블에 넣기 — 원본은 시트→DataTable(ROW)
  • 여러 테이블을 고치는 작업을 트랜잭션 없이 쓰기 — 크래시 시 "조각만 사라진" 중간 상태
  • 같은 값을 두 서브시스템이 쓰기 — 주인은 하나
  • 구 명칭(DT_Ability_UpgradeCost 등)으로 새 코드 작성 — 01섹션 매핑 표 참조
  • DataTable 값을 코드에 하드코딩 — 시트 수정이 무효가 됨
