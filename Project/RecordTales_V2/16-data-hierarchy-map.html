위계 지도

V2의 모든 데이터를 계층 → 그룹 → 항목 위계로 한 장에 내려다봅니다 — ROW(밸런스 표 10종) · DB(Archive.db 7테이블) · RUNTIME(캐시 struct · 파생값). 이 문서는 조망 전용입니다: 배치 근거는 03, 변수 · 여정은 10, 컬럼 정본은 05, 구현 방식은 15가 담당합니다.

01 — 지도 읽는 법

배지 세 개와 위계 한 줄

이 지도의 모든 항목에는 수명 배지가 붙습니다 — 그 데이터가 어디에 살고 언제까지 남는지를 나타냅니다. V2는 풀-DB 전환으로 배지가 세 종류뿐입니다(USaveGame의 SAVE, UDeveloperSettings의 CONFIG는 폐기).

배지계층뜻
ROWROW — 밸런스 표DataTable 행. 기획(시트)이 소유, 코드는 읽기만. 껐다 켜도 에셋에 남음.
DBDB — 영속 상태SQLite Archive.db의 행. 플레이어 진행이 소유. 껐다 켜도 파일에 남음.
RUNRUNTIME — 메모리서브시스템 멤버 · 파생값. 앱을 끄면 사라짐 — 저장하지 않는 것이 규칙.
🧭

핵심 용어. 위계 = "누가 이 값을 소유하는가"의 사슬 — 위(원본)에서 아래(사용처)로만 흐릅니다. 축 = DB 안에서 데이터가 갈라지는 기준: 계정 축(모든 길드 공유)과 길드 축(길드별 분리). 참조(→DT_*) = DB 행이 값을 복사하지 않고 표의 RowName을 가리키는 것 — 값은 항상 표에서 조회. FK(외래키) = DB 행이 다른 테이블의 행을 가리키는 제약 — 부모가 지워지면 자식도 함께(CASCADE).

02 — 전체 위계

한 장 조망 — 개요도와 위계 트리

먼저 세 계층이 어떻게 이어지는지 개요도로 보고, 그다음 트리로 항목 이름까지 내려갑니다. 화살표 방향이 곧 위계입니다 — 시트가 ROW를 채우고, ROW와 DB가 RUNTIME으로 합류하며, RUNTIME의 변경은 트랜잭션으로 DB에 되돌아갑니다.

03 — 계층 카탈로그

항목별 한 줄 요지

트리의 항목 하나하나를 "행 단위 · 키 · 무엇을 담나" 한 줄로 요약합니다. 컬럼 전체와 타입 · DDL은 05 데이터 스키마 명세가 정본이므로 여기 요지와 다르면 05를 따릅니다.

Static ROW — DataTable 10종 (밸런스) 기획이 시트로 채우는 마스터 데이터 · 코드는 읽기만
DT_Ability_TypeTableROW 행=효과×등급 — 능력치의 정체성(등급 · 효과 종류 · 뽑힐 가중치)
DT_Ability_StepTableROW 행=능력치+단계 — 단계별 효과값 · 시작 가중치(등급 높을수록 행 많음)
DT_Ability_EnhanceTableROW 행=능력치 등급 — 한 단계 강화 조각 비용(단계 무관 고정)
DT_CharacterTableROW 행=캐릭터 id — 이름 · 종족 · 소속 · 직업 · 스토리 · 아이콘 · 매칭 테마(등급 없음 — 뽑을 때 롤)
DT_GachaRateByLevelTableROW 행=계정 레벨(1~20) — 캐릭터 등급 출현 확률 4열, 합=100, L20 plateau
DT_Character_Ability_OpenChanceTableROW 행=캐릭터 등급 — 4슬롯 각각의 개화 확률(계단형 → 왼쪽부터 연속 개화)
DT_Character_Ability_GradeChanceTableROW 행=캐릭터 등급 — 개화 슬롯에 뜨는 능력치 등급 분포 4열
DT_Character_ExtractTableROW 행=캐릭터 등급 — 추출 시 지급 조각 수(고정)
DT_BuildingTableROW 행=건물 id — 이름 · 타입 · 테마 · 해금 조건 · 최대 슬롯
DT_Building_UpgradeTableROW 행=건물+레벨 — 기본 산출 · 열린 슬롯 · 업그레이드 비용(건물별 개별 곡선)

04 — 참조선

위계를 실제로 잇는 선

트리는 소속을 보여주지만, 계층 사이는 참조로 이어집니다. DB 행은 값을 복사하지 않고 표를 가리키고(→DT_*), 테이블끼리는 FK로 묶이며, RUNTIME struct는 테이블과 1:1로 대응합니다.

출발 (자식)도착 (부모)의미
owned_characters.CharacterIdDT_Character RowName정체성(이름 · 아이콘 · 테마)은 표에서 조회 — DB엔 id만
owned_abilities.AbilityId · StepDT_Ability_Type · DT_Ability_Step효과값 · 비용을 DB에 복사하지 않음 — Step만 저장, 값은 표 조회
owned_abilities.InstanceIdowned_characters.InstanceIdFK · ON DELETE CASCADE — 캐릭터 추출 시 능력치 행 자동 동반 삭제
buildings.BuildingId · LevelDT_Building · DT_Building_Upgrade건물 정체성 · 레벨별 산출 곡선은 표가 정본
buildings.DeployedInstanceIdsowned_characters.InstanceId배치 캐릭터를 번호로 가리킴(JSON 배열 — FK 아님, 코드가 정합성 관리)

💡

여기까지가 이 문서의 몫. 이 참조선들이 실제 호출 · 델리게이트로 어떻게 흐르는지(연결 전도 · 변수 여정)는 10 코드/변수 지도가, 판정 다이어그램(Secured 복원)은 03 데이터 설계가 담당합니다.

05 — 확장 규칙

새 데이터는 위계 어디에 꽂히나

새 값이 생기면 아래 세 질문을 순서대로 통과시킵니다. 각 단계의 상세 근거는 03(배치 기준) · 10(소유 결정표)이 담당합니다.

1
누가 이 값을 소유하는가?
기획이 튜닝하는 수치 → ROW(표 컬럼으로 추가, 시트 → CSV). 플레이어 진행이 만드는 상태 → DB. 계산 가능하거나 세션에서만 쓰는 값 → RUNTIME(저장 금지).
2
DB라면 — 어느 축, 어느 테이블, 쓰기 주인은?
계정 축(모든 길드 공유)인지 길드 축(길드별)인지 정하고, 기존 7테이블에 컬럼으로 붙일지 새 테이블일지 판단합니다. 쓰기 주인은 테이블당 하나 — 소유 결정표(10)에 먼저 등록합니다.
3
스키마 · 지도 갱신
컬럼 · DDL은 05에 먼저 반영(출시 후엔 PRAGMA user_version 마이그레이션)하고, 이 위계 지도 · 카탈로그에도 항목을 추가해 "지도에 없는 데이터"를 만들지 않습니다.
