# 템플릿 카탈로그 (TEMPLATE-CATALOG)

> 문서 작업에 사용할 템플릿의 **단일 목록 + 판별 기준**입니다.
> 문서 생성 절차(STEP 3 작업 판별)에서 AI는 이 카탈로그를 기준으로 Case A/B/C를 판별합니다.
> 새 템플릿이 생기면(Case C) **반드시 이 파일에 등록**합니다. 절차 규약: `doc-process/DOC-GENERATION-PLAYBOOK.md`

---

## 판별 기준 (요청 성격 → 템플릿)

| 요청 성격 | 템플릿 |
|---|---|
| 이미 존재하는 **에셋**(위젯·머티리얼·BP 등)의 구조를 **분석·설명** | `01` 에셋 분석 |
| 프레임워크 **패턴/시스템을 설계·설명** (GameMode·SaveGame·PlayerController 구조 등) | `02-A` 아키텍처 패턴 |
| 구체적 **기능 구현 요청** (복붙 가능한 .h/.cpp 포함) | `02-B` 기능 구현 명세 |
| 프로젝트 전체 **개발 순서·Phase·의존성·마일스톤** 로드맵 | `03` 개발 플로우 로드맵 |
| 변수(데이터)가 **어느 .h/.cpp 파일에 살고, 시스템 사이를 어떻게 흘러 연결되는지** 배치·흐름을 시각화 | `04` 데이터 설계 지도 |

- 어느 행에도 부합하지 않으면 → **Case C(신규 템플릿 필요)**로 판별하고 사용자에게 확인을 받습니다.
- **결과물 저장 위치·허브·용어집은 이 카탈로그가 아니라 `doc-process/DOC-TRACKS.md`의 트랙 규칙을 따릅니다.** (카탈로그는 "어떤 구조로 쓰나"만 담당)

## 템플릿 목록

### 01 — 에셋 분석 (`01-ue-asset-analysis.html`)

- **용도**: 언리얼 MCP로 받은 에셋(위젯·머티리얼·블루프린트 등)의 구조 분석 문서.
- **섹션 구조 (독자 이해 순서)**: 7섹션 — 개념 → 구조 → 구성요소 → 데이터타입 → 사용 → 응용 → 팁. ("이게 뭔지" 파악 → 내부를 뜯어보기 → 직접 써보기 → 응용·주의점 순서)
- **전용 컴포넌트**: 에셋 트리(`.tree`), 에셋 카드, 의사 코드(`.snip`).
- **버전**: v1.2

### 02-A — 아키텍처 패턴 (`02-ue-architecture.html` 템플릿 A)

- **용도**: 프레임워크 패턴/시스템의 설계·설명 문서 (코드는 스켈레톤 수준).
- **섹션 구조 (독자 이해 순서)**: 개념 → 관계 → 책임 → 흐름 → 스켈레톤 → 확장. (등장 클래스를 알기 → 서로 어떻게 연결·분담하는지 → 실행 흐름 → 뼈대 코드 → 확장 방향 순서)
- **전용 컴포넌트**: 클래스 관계도(`.rel-map`), 클래스 계층 트리, 파이프라인(`.pipe-step`).
- **버전**: v1.2

### 02-B — 기능 구현 명세 (`02-ue-architecture.html` 템플릿 B)

- **용도**: 복붙 가능한 `.h`/`.cpp`를 포함한 구체적 기능 구현 문서.
- **섹션 구조 (독자 이해 순서)**: 요구사항 → 설계결정 → 클래스구성 → 파일별코드 → 등록 → 검증. (무엇을 만들지 합의 → 왜 이렇게 나누는지 → 코드 따라치기 → 엔진에 등록 → 동작 확인 순서)
- **전용 컴포넌트**: **책임 배치 결정표**(`.tbl .owner`) — 02-B의 핵심. 어떤 데이터/기능을 어느 클래스에, 왜 두는지 근거와 함께 기록. 코드 탭(`.codeblock`).
- **버전**: v1.2

### 03 — 개발 플로우 로드맵 (`03-ue-dev-flow.html`)

- **용도**: 프로젝트 전체의 개발 순서·시스템 의존성·페이즈별 구현·마일스톤을 담는 로드맵 문서.
- **섹션 구조 (독자 이해 순서)**: 4블록 17섹션 — 기반 지식(1~6) → 설계 이해(7~8) → 구현 Phase 1~6(9~14) → 마무리(15~17). (개발 전제·컨벤션을 먼저 확립 → 의존성 맵으로 전체 그림 → Phase별 "선행 조건 → 구현 → 중간 검증" 반복 → 리스크·마일스톤 총괄 순서)
- **전용 컴포넌트**: Phase 파이프라인(`.pipe-step` 6단계), Phase 블록 패턴(선행 조건 `.callout.warn` → 구현 `.pipe-step` → 중간 검증 `.callout.tip`), 의존성 맵(`.rel-map`), 마일스톤 체크리스트(`.tbl`), 리스크 매트릭스(`.tbl`), 트러블슈팅(`.pitfall`).
- **버전**: v1.0

### 04 — 데이터 설계 지도 (`04-ue-data-map.html`)

- **용도**: 프로젝트/시스템의 변수(데이터)가 어느 `.h`/`.cpp` 파일에 살고(정적 배치), 시스템 사이를 어떻게 흘러 하나로 연결되는지(동적 연결)를 시각화하는 문서.
- **섹션 구조 (독자 이해 순서)**: 6섹션 — 지도 읽는 법 → 파일 지도 → 소유 결정 → 연결 지도 → 흐름 추적 → 확장 규칙. (정적 배치를 먼저 눈에 익히고 → 배치의 근거 → 동적 연결 → 실제 시나리오로 검증 → 새 변수에 스스로 적용하는 순서)
- **전용 컴포넌트**: 파일-변수 카드(`.var-card` + 수명 배지 `.vbadge` 4종: SAVE/RUNTIME/CONFIG/ROW), 변수 흐름 연결 전도(`.flow-hub` + 범례 `.flow-legend`, 연결 유형 4색: 호출/델리게이트/직렬화/조회), 변수 여정 시퀀스(`.lane-seq`). 선택(v1.1): 옵시디언 그래프 뷰 스타일 관계망(`.graph-net`), Mermaid 스타일 구워넣은 SVG 플로차트(`.diagram`) — 항상 Mermaid 소스(`*.mmd` 단일 탭 `.codeblock`) 병기, 런타임 렌더링 금지. 재사용: 폴더 지도(`.tree`), 소유 결정표(`.tbl .owner`), Do/Don't(`.dd-grid`).
- **02-A와의 구분**: 클래스의 역할·책임 관계가 주제면 02-A, 변수(데이터)의 배치·흐름이 주제면 04.
- **버전**: v1.1

## 공통 준수 사항 (모든 템플릿)

- 스타일은 `design-guide/design-guide-ue-master.css`(마스터)에서만 가져옵니다. 시각 배치는 `design-guide/design-guide-ue-master-visual.html`을 본뜹니다. (결과물은 마스터 CSS를 `<style>`에 inline — 생성 시점 버전 스냅샷. 템플릿 파일 자체는 `<link>` 참조 유지. 단일 소스: `doc-process/DOC-TRACKS.md`)
- **섹션 구성의 기준은 각 템플릿의 "섹션 구조(독자 이해 순서)"입니다** — 위 템플릿 목록에 정의된 순서를 따르되, 요청 내용에 맞게 섹션을 추가·흡수·축약할 수 있습니다. 템플릿마다 구조가 다르므로 고정된 공통 목차는 없습니다.
- 각 템플릿에 정의된 **섹션 순서와 컴포넌트 사용법**을 그대로 따릅니다.
- 초보자 친화 원칙(용어 풀이·구체 절차·전제와 결과·근거)을 충족합니다. → `doc-process/DOC-GENERATION-PLAYBOOK.md` 작성 원칙 참조.

> 참고(설계 배경): 01·02 계열 템플릿의 섹션 구조는 초기의 "다섯 기둥(개념 · 구조 · 사용/구현 · 응용 · 팁)" 골격에서 파생되었습니다. 이는 배경 설명일 뿐이며, 신규 템플릿이 다섯 기둥을 따라야 하는 것은 아닙니다.

## 템플릿 관리 규칙 (STEP 4 갱신 원칙과 연동)

템플릿과 이 카탈로그는 **누적 자산**입니다. 문서를 만들 때마다 성장하되, 한 번 등록된 것은 함부로 줄이지 않습니다.

1. **미사용 ≠ 제거** — 어떤 문서가 템플릿의 특정 섹션·컴포넌트를 쓰지 않더라도(흡수·축약 허용), 그 이유로 **템플릿에서 해당 섹션·컴포넌트를 제거하지 않습니다.** 다른 문서가 이미 그 구조에 의존하고 있을 수 있습니다.
2. **없는 것만 추가** — 템플릿 갱신이 필요하면(Case B에서 마스터에 새 컴포넌트가 추가되어 템플릿 데모가 필요해진 경우 등) 없는 내용만 보태고, 기존 섹션 구조·컴포넌트 규칙은 재정의하지 않습니다.
3. **기존 템플릿 변형으로 Case C를 흡수하지 않기** — 부합 템플릿이 없으면 기존 템플릿에 섹션을 덧대지 말고 **새 번호(NN)의 신규 템플릿**을 만듭니다(아래 등록 절차).
4. **카탈로그 등록 = 템플릿 인정** — 이 카탈로그에 등록되지 않은 템플릿 파일은 판별 대상이 아닙니다. 템플릿의 판별 기준·버전·이력 관리는 카탈로그가 담당합니다.
5. **수정·제거 기록** — 템플릿을 수정하면 해당 템플릿 파일 CHANGELOG + 이 카탈로그의 버전 표기를 함께 갱신합니다. 템플릿(또는 그 일부)의 **제거는 사용자 별도 승인**을 받은 뒤에만 진행하고, 카탈로그 CHANGELOG에 `(제거)`로 명시합니다.

## 신규 템플릿 등록 절차 (Case C 승인 시)

1. **템플릿 파일 생성** — `design-template/NN-<이름>.html` (다음 번호 부여, 예: `03-…`).
   - 마스터 CSS를 `<link rel="stylesheet" href="../design-guide/design-guide-ue-master.css">` 로 참조합니다.
   - 기존 템플릿(01/02)과 같은 구성으로: 섹션 표준 구성 + 전용 컴포넌트의 Live 데모 + 사용법 설명.
   - 새 컴포넌트/토큰이 필요하면 결과물이 아니라 **마스터부터 갱신**합니다(갭 갱신 루프, STEP 4).
2. **이 카탈로그에 등록** — 판별 기준 표에 행 추가 + 템플릿 목록에 항목 추가(용도·**섹션 구조(독자 이해 순서 포함)**·전용 컴포넌트·버전). 섹션 구조에는 "독자가 어떤 순서로 이해에 도달하는지"의 근거를 한 줄 덧붙입니다.
3. **CHANGELOG 기록** — 새 템플릿 파일 상단 주석 + 이 파일 하단 변경 이력에 기록(STEP 6).
4. **결과물 저장 위치 확인** — 저장 위치는 `doc-process/DOC-TRACKS.md`의 트랙 규칙을 따릅니다. 기존 트랙에 부합하지 않는 새 저장 형태가 필요하면 DOC-TRACKS.md에 트랙/위치를 먼저 정의하고 CLAUDE.md·플레이북 폴더 구조에 반영합니다.
