설계 템플릿

"이 기능을 어느 클래스에, 어떻게 나눠 설계할지"와 "그대로 복사할 수 있는 .h/.cpp 코드"를 담는 문서용 컴포넌트·템플릿입니다. GameMode·PlayerController·GameInstance·SaveGame 같은 프레임워크 클래스의 책임 분배를 시각화하고, 하이라이팅이 구워진 코드 탭으로 오프라인에서도 완결됩니다. 토대는 디자인 마스터를 따릅니다.

04b — 코드 탭 (시그니처)

.h / .cpp 복사 가능한 코드 블록

탭으로 헤더와 소스를 전환하고, 복사 버튼으로 현재 탭 전체를 그대로 가져갑니다. 하이라이팅은 생성 시점에 <span>으로 구워넣어, 인터넷·스크립트 없이도 색이 유지됩니다. (탭·복사만 오프라인 inline JS 사용)

#pragma once

#include "CoreMinimal.h"
#include "Engine/GameInstance.h"
#include "MyGameInstance.generated.h"

UCLASS()
class MYGAME_API UMyGameInstance : public UGameInstance
{
    GENERATED_BODY()

public:
    // 점수를 저장 슬롯에 기록한다.
    UFUNCTION(BlueprintCallable, Category = "Save")
    void SaveProgress(int32 Score);

    // 저장된 점수를 읽어 반환한다. 없으면 0.
    UFUNCTION(BlueprintCallable, Category = "Save")
    int32 LoadProgress() const;

private:
    // 저장 슬롯 이름 — 저장/로드가 같은 문자열을 쓰도록 한곳에서 관리
    static const FString SlotName;
};
#include "MyGameInstance.h"
#include "MySaveGame.h"
#include "Kismet/GameplayStatics.h"

// 슬롯 이름의 실제 값 정의 — static 멤버는 .cpp에서 한 번만 정의한다
const FString UMyGameInstance::SlotName = TEXT("PlayerSave");

void UMyGameInstance::SaveProgress(int32 Score)
{
    // 1) 메모리에 빈 세이브 객체를 만든다 (아직 디스크에 안 씀)
    UMySaveGame* Save = Cast<UMySaveGame>(
        UGameplayStatics::CreateSaveGameObject(UMySaveGame::StaticClass()));

    // 2) 저장할 값을 채운 뒤 3) 슬롯 이름으로 디스크에 기록 (마지막 0은 유저 인덱스)
    Save->HighScore = Score;
    UGameplayStatics::SaveGameToSlot(Save, SlotName, 0);
}

int32 UMyGameInstance::LoadProgress() const
{
    // 슬롯에서 읽기 — 저장 파일이 없으면 nullptr가 돌아온다
    UMySaveGame* Save = Cast<UMySaveGame>(
        UGameplayStatics::LoadGameFromSlot(SlotName, 0));

    // nullptr 체크 필수 — 첫 실행처럼 저장이 없을 때 크래시 방지
    return Save ? Save->HighScore : 0;
}
⚠️

탭·복사 <script>는 문서 맨 아래에 inline으로 둡니다(오프라인 동작). 복사 버튼은 현재 활성 탭의 <pre> 텍스트를 그대로 클립보드에 넣으므로, 구워넣은 span은 복사 결과에 섞이지 않습니다.

토큰 색 매핑 (highlight.js 클래스)

클래스대상색
.hljs-keywordclass · public · return · const · void인디고 --accent
.hljs-type / .hljs-built_inFString · int32 · AActor · Cast · TEXT틸 --accent-3
.hljs-meta#include · UCLASS · UPROPERTY · UFUNCTION오렌지 --ue-macro
.hljs-string"..." 문자열·경로그린 --ue-demo
.hljs-number / .hljs-literal0 · true · nullptr퍼플 --accent-2
.hljs-title.function_함수 이름밝은 텍스트 --text
.hljs-comment// 주석흐림 --text-dim (이탤릭)
💡

실제 문서 생성 시: highlight.js로 코드를 한 번 하이라이팅한 결과 HTML을 <pre class="hljs"> 안에 붙여넣으면 됩니다. 위 클래스명이 그대로 이 팔레트에 매핑됩니다.

05 — 문서 템플릿 A

아키텍처 패턴 레퍼런스

"GameMode는 어떻게 설계하나", "SaveGame 저장 구조" 처럼 하나의 프레임워크 패턴/시스템을 설명하는 문서입니다. 요청 1·2(구조 설계 이해 + C++ 설계 요소)에 해당합니다.

01
개념 · 역할
이 클래스/시스템이 무엇이고 언제 쓰는지. 한 문장 정의 + 핵심 용어 callout + 구성 스탯.
02
클래스 관계도
클래스 계층 트리 + 관계도(소유·possess·참조) + 범례. 엔진 클래스와의 상속 관계를 명확히.
03
책임 분배
책임 배치 결정표로 "무엇을 어디에, 왜". 대표 클래스별 주요 멤버/함수 표.
04
데이터 흐름 · 생명주기
Arch 다이어그램으로 실행 순서(InitGame → Login → Possess 등). 서버/클라이언트 권한·복제 주의.
05
구현 스켈레톤
.h/.cpp 코드 탭으로 최소 동작 뼈대. 프로젝트/월드 세팅 등록 방법 파이프라인.
06
확장 & 주의
응용 아이디어 카드 그리드 + Pitfall 리스트(흔한 null 참조·권한 실수 등).
05 — 문서 템플릿 B

기능 구현 명세

"더블 점프를 넣어줘", "체크포인트 저장을 구현해줘" 처럼 구체 기능 요청을 받아 설계 + 복붙 코드로 답하는 문서입니다. 요청 3에 해당합니다.

01
요구사항
무엇을 만들지 한 문단 + 완료 기준 callout(체크리스트). 범위와 비범위를 분명히.
02
설계 결정
책임 배치 결정표로 "이 기능/데이터를 어느 클래스에, 왜". 후보 대안과 트레이드오프를 짧게.
03
클래스 구성
새로 만들/수정할 클래스 목록 카드 + 관계도. 변경 지점을 한눈에.
04
파일별 코드
파일마다 .h/.cpp 코드 탭(복사 가능, 하이라이팅 구워넣기). 그대로 붙여 컴파일되는 완결 코드.
05
등록 & 연결
프로젝트/월드 세팅 등록, 입력 매핑, 참조 연결을 번호 파이프라인으로.
06
검증 체크리스트
동작 확인 표(항목 → 기대 결과) + 컴파일/런타임 흔한 오류 Pitfall.
🧭

두 템플릿 모두 개념 → 관계 → 책임 → 코드 → 검증의 뼈대를 공유합니다. A는 "패턴을 이해"시키고, B는 "요청을 구현"합니다. 단순 기능이면 03을 02에 흡수해도 됩니다.
