지도 템플릿

프로젝트의 변수가 어느 .h/.cpp 파일에 살고, 시스템 사이를 어떻게 흘러 하나로 연결되는지를 시각화하는 문서용 템플릿입니다. 정적 배치(파일 지도)와 동적 연결(흐름 지도)을 겹쳐 읽으면 프로젝트 전체가 유기적으로 보입니다.

00 — 구성 개요

섹션 구조 (독자 이해 순서)

독자는 정적 배치를 먼저 눈에 익히고 → 배치의 근거를 이해하고 → 동적 연결을 읽고 → 실제 시나리오로 검증한 뒤 → 스스로 적용하는 순서로 이해에 도달합니다. 요청 내용에 맞게 섹션을 추가·흡수·축약할 수 있습니다.

#섹션답하는 질문핵심 컴포넌트
01지도를 읽는 법이 문서의 기호·색·용어는 무엇을 뜻하나.callout(핵심 용어) · .flow-legend · .vbadge
02파일 지도어떤 변수가 어느 파일에 사는가.tree(폴더 지도) · .var-card(파일-변수 카드)
03소유 결정왜 그 파일에 두는가.tbl .owner(소유 결정표)
04연결 지도변수가 시스템 사이를 어떻게 흘러가는가.flow-hub(연결 전도) + .flow-legend
05흐름 추적실제 시나리오에서 정말 그렇게 흐르는가.lane-seq(변수 여정) · .codeblock(근거 코드)
06확장 규칙새 변수가 생기면 어디에 두는가.pipe-step(결정 절차) · .dd-grid(Do/Don't)
🗺️

이 템플릿의 핵심 아이디어 — 같은 변수를 세 지도에서 세 번 만나게 합니다. 파일 지도에서 "여기 산다"를, 연결 지도에서 "이렇게 이어진다"를, 흐름 추적에서 "실제로 이렇게 움직인다"를 확인하면, 독자는 변수 하나를 축으로 프로젝트 전체 구조를 입체적으로 기억하게 됩니다.

01 — 전용 컴포넌트 · 파일-변수 카드

.var-card — 어떤 변수가 어느 파일에 사는가

헤더/소스 파일 쌍 하나당 카드 하나. 그 파일이 소유(선언·갱신 책임)하는 변수를 전수 나열합니다. 각 행은 변수명 · 타입 · 수명 배지 · 한 줄 설명으로 구성합니다.

02 — 전용 컴포넌트 · 변수 흐름 연결 전도

.flow-hub — 변수는 시스템 사이를 어떻게 흘러가는가

허브(중앙) 노드를 사이에 두고 좌 → 중 → 우로 데이터가 흐릅니다. 각 노드의 연결은 색 칩(.fh-link)으로 표기하고, 색의 의미는 반드시 범례(.flow-legend)로 문서 안에서 고정합니다. 화살표 방향(→/←)을 칩 텍스트에 포함해 흐름의 방향을 명시합니다.

03 — 전용 컴포넌트 · 변수 여정 시퀀스

.lane-seq — 변수 하나를 끝까지 따라가기

대표 시나리오에서 변수 하나가 어느 파일의 어느 함수를 거쳐 어떻게 변하는지를 단계로 추적합니다. 각 단계는 파일 칩(어디서) · 사건 제목(무엇이) · 함수(어떻게) · 변수 변화(결과)로 구성합니다.

04 — 다이어그램 확장 (선택)

.graph-net · .diagram — 옵시디언/Mermaid 시각 언어

연결을 한눈에 조감하고 싶을 때는 옵시디언 그래프 뷰 스타일 관계망(.graph-net)을, 분기·절차를 그리고 싶을 때는 Mermaid 스타일 플로차트(.diagram)를 씁니다. 데모와 내부 클래스 규칙은 마스터 비주얼 08 섹션을 따릅니다.

컴포넌트어울리는 자리필수 동반
.graph-net 관계망연결 지도 섹션 도입부 — 전체 조감 후 주제별 전도(.flow-hub)로 확대연결 색 범례(.flow-legend) + Mermaid 소스
.diagram 플로차트확장 규칙 섹션 — 결정 절차(.pipe-step)의 시각 요약Mermaid 소스
🧩

3원칙 — ① SVG는 생성 시점에 구워넣기(mermaid.js 등 런타임 렌더링 금지 — 오프라인 단일 파일 유지), ② Mermaid 소스를 .codeblock 단일 탭(*.mmd)으로 병기해 복사 → 옵시디언에 붙여넣기 가능하게, ③ 관계망·전도·전수 표는 역할이 다를 때만 병행(조감 / 주제별 상세 / 레퍼런스) — 같은 정보의 삼중 반복 금지.

05 — 사용 규칙

이 템플릿의 Do & Don't

✓ 이렇게

  • 변수 하나의 주인(소유 파일)은 반드시 하나로 지정
  • 파일 지도(02)와 소유 결정표(03)의 변수 목록을 1:1로 일치시키기
  • 연결 전도의 색은 문서 안 범례(.flow-legend)로 항상 고정
  • 흐름 추적의 각 단계에 실제 함수 이름·값 변화를 명시
  • 수명 배지(.vbadge)는 카드·표 어디서든 같은 의미로 재사용

✗ 이건 피하기

  • 같은 변수를 두 파일의 소유로 표기 (참조와 소유 혼동)
  • 범례 없이 색 칩만 나열하기
  • 연결 전도에 모든 시스템을 한 장으로 욱여넣기 — 유형별 분할
  • 파일 지도에 없는 변수를 흐름 추적에 등장시키기
  • rel-map(1:1 관계도)이 맞는 자리에 flow-hub를 쓰기
⚠️

02-A(아키텍처 패턴)와의 구분 — 클래스의 역할·책임 관계가 주제면 02-A, 변수(데이터)의 배치·흐름이 주제면 04입니다. "누가 무엇을 하나"는 02-A, "무엇이 어디 살고 어디로 가나"는 04.
