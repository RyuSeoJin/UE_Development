# UE 문서 자동 제작 규약 (DOC-GENERATION-PLAYBOOK)

> 사용자가 "~~에 대한 문서를 만들어줘"라고 요청하면, AI(Claude)는 **이 규약을 처음부터 끝까지 따릅니다.**
> 이 문서는 사람이 아니라 AI가 매 요청 시 실행하는 절차서입니다.

---

## 작성 원칙 — 초보자 친화 (모든 문서 공통, 필수)

생성하는 모든 문서는 **언리얼을 처음 접하는 사람도 혼자 읽고 따라 할 수 있을 만큼 자세한 절차**를 담아야 합니다. "안다고 가정하고 건너뛰기"를 하지 않습니다.

이를 위해 다음을 지킵니다.

- **용어는 처음 나올 때 풀어서 설명합니다.** 예: "Possess(폰을 컨트롤러가 소유·조종하는 것)". 이후에는 용어만 써서 반복을 피합니다. 단, 아래 보완책을 함께 적용합니다.
  - **문서 내 "핵심 용어" 블록**: 개념 섹션에 주요 용어를 모아 `callout`으로 정리해, 독자가 언제든 되짚게 합니다.
  - **멀리 재등장 시 짧은 재힌트**: 처음 설명에서 여러 섹션 떨어져 다시 나오면 한 번 더 짧게 힌트합니다. 예: "Possess(앞서 설명한 폰 소유)".
  - **한 개념엔 한 용어**: 같은 개념을 다른 말로 바꿔 부르지 않습니다(폰/캐릭터/조종 대상 혼용 금지).
  - **중앙 용어집 연동**: 어려운 용어는 `design-guide-ue-index.html`의 용어집에도 등록하고, 용어를 본격 설명한 위치에 **앵커 ID**(예: `id="term-possess"`)를 부여해 "자세히 →"가 정확히 점프하게 합니다. (STEP 6 참조)
- **절차는 '무엇을 · 어디서 · 어떻게'까지 구체적으로** 적습니다. 메뉴/버튼 위치, 노드 이름, 패널 이름, 파일 경로, 클래스 이름을 명시합니다. ("설정한다" X → "Project Settings ▸ Maps & Modes ▸ Default GameMode 에서 지정합니다" O)
- **전제 조건과 결과를 명시합니다.** 각 단계 시작 전 필요한 것, 끝난 뒤 확인되어야 할 상태를 적습니다.
- **왜 그렇게 하는지 근거를 함께** 적습니다. 방법뿐 아니라 이유를 알아야 응용할 수 있습니다.
- **중간 과정을 생략하지 않습니다.** 숙련자에게 당연한 단계도 빠뜨리지 않습니다.
- **막히기 쉬운 지점은 Pitfall(⚠️)로** 미리 경고합니다.

> 균형: 자세하되 장황하지 않게. 반복·군더더기 대신 "필요한 모든 단계를 빠짐없이"가 목표입니다.

## 0. 언제 발동하나

사용자가 언리얼 엔진과 관련된 **설명/분석/설계 문서(HTML)** 제작을 요청할 때입니다. 예:

- "인벤토리 시스템 구조를 설계하는 문서 만들어줘" → 아키텍처(02)
- "이 위젯 에셋 분석 문서 만들어줘" → 에셋 분석(01)
- "더블 점프 기능을 구현하는 문서 만들어줘" → 기능 구현 명세(02-B)

산출물은 원칙적으로 **단일 HTML 파일**(오프라인에서 열리는 완결형)입니다.

---

## 폴더 구조 & 출력 위치

```
UE_Development/
├── design-guide/        (템플릿·마스터·규약 — 편집 대상)
│   ├── design-guide-ue-master.css          스타일 마스터
│   ├── design-guide-ue-master-visual.html  시각 레퍼런스
│   ├── 01-ue-asset-analysis.html           에셋 분석 템플릿
│   ├── 02-ue-architecture.html             아키텍처/기능 템플릿
│   └── DOC-GENERATION-PLAYBOOK.md           이 규약
└── ue-wiki/             (생성 결과물 위키)
    ├── ue-asset/        ← 에셋 분석 결과 문서를 저장
    └── ue-architect/    ← 아키텍처/기능 구조 문서 + 허브
        └── design-guide-ue-index.html      문서 허브 & 용어집
```

- **에셋 분석 결과** → `ue-wiki/ue-asset/` 에 저장합니다.
- **아키텍처·기능 구현 결과** → `ue-wiki/ue-architect/` 에 저장합니다.
- 결과물은 CSS를 inline으로 구워넣으므로(STEP 5) 폴더 위치와 무관하게 단독으로 열립니다. 결과물에는 상대경로 링크를 쓰지 않습니다.
- 문서 허브(`ue-wiki/ue-architect/design-guide-ue-index.html`)만 예외로 마스터 CSS를 `../../design-guide/` 상대경로로 링크합니다. 허브가 다른 문서를 링크할 때도 폴더 기준 상대경로를 씁니다(같은 폴더면 `./`, 에셋 문서면 `../ue-asset/`).

## 1. 디자인 소스 (STEP 1)

두 파일이 **디자인의 유일한 기준(Single Source of Truth)**입니다.

| 파일 | 역할 | AI가 참조하는 방법 |
|---|---|---|
| `design-guide-ue-master.css` | **스타일 그 자체.** 모든 색·글꼴·간격·컴포넌트 규칙 | 결과물의 스타일은 반드시 이 파일에서만 가져옵니다 |
| `design-guide-ue-master-visual.html` | **시각적 정리 방법의 참고서.** 각 컴포넌트가 "어떻게 생겼고 어떻게 배치되는지" | 레이아웃·컴포넌트를 고를 때 이 문서의 예시를 그대로 본뜹니다 |

**규칙**

- 색은 의미가 있을 때만 씁니다. 타입 컬러 맵(에셋/프레임워크 클래스/파일)은 트리·관계도·태그·카드 보더에서만 사용합니다.
- 강조 요소는 화면당 하나로 제한합니다. 밝은 그라데이션·두꺼운 사이드바는 쓰지 않습니다.
- 에셋 이름·클래스·변수·경로·값은 항상 IBM Plex Mono로 표기합니다.

## 2. 포맷 판별 & 템플릿 준수 (STEP 2)

요청의 성격을 판별해 **01 또는 02의 문서 구조**를 따릅니다.

### 판별 기준

| 요청 성격 | 따라갈 템플릿 |
|---|---|
| 이미 존재하는 **에셋**(위젯·머티리얼·BP 등)의 구조를 **분석·설명** | `01-ue-asset-analysis.html` → 7섹션(개념·구조·구성요소·데이터타입·사용·응용·팁) |
| 프레임워크 **패턴/시스템을 설계·설명** (GameMode·SaveGame·PlayerController 구조 등) | `02-ue-architecture.html` → **템플릿 A**(개념·관계·책임·흐름·스켈레톤·확장) |
| 구체적 **기능 구현 요청** (복붙 가능한 .h/.cpp 포함) | `02-ue-architecture.html` → **템플릿 B**(요구사항·설계결정·클래스구성·파일별코드·등록·검증) |

### 준수 사항

- 해당 가이드 문서(01/02)에 정의된 **섹션 순서와 컴포넌트 사용법**을 그대로 따릅니다.
- 다섯 기둥(**개념 · 구조 · 사용/구현 · 응용 · 팁**)은 항상 포함합니다. 내용이 단순하면 섹션을 흡수·축약할 수 있습니다.
- 각 섹션은 위의 **[작성 원칙 — 초보자 친화]**를 충족해야 합니다. 단계·용어·근거를 빠짐없이 담습니다.
- 02-B(기능 구현)의 핵심은 **책임 배치 결정표**입니다. "이 데이터/기능을 어느 클래스(GameMode/GameState/PlayerController/Pawn/GameInstance/SaveGame/Subsystem)에, 왜 두는지"를 먼저 결정하고 근거와 함께 문서에 남깁니다.

## 3. 갭 발생 시 마스터 갱신 (STEP 3) — 핵심 루프

문서를 만들다가 **기존 `master.css` / `master-visual.html`로 표현할 수 없는 디자인**이 필요하면, 임의로 새 스타일을 결과물에만 넣지 말고 **먼저 마스터를 갱신**합니다.

```
갭 감지
  │
  ├─(1) design-guide-ue-master.css 에 필요한 토큰/컴포넌트를 추가합니다.
  │      · 새 색이 필요하면 기존 색을 재활용하지 말고, 팔레트 톤에 맞는 1색만 추가합니다.
  │      · 클래스명·변수명은 기존 네이밍 관습을 따릅니다.
  │
  ├─(2) design-guide-ue-master-visual.html 에 그 사용법을 등록합니다.
  │      · 색이면 색상표 스와치에, 컴포넌트면 "Live 데모 + 설명"으로 추가합니다.
  │
  ├─(3) STEP 4 에 따라 두 파일에 변경 이력(CHANGELOG)을 기록하고 버전을 올립니다.
  │
  └─(4) 갱신된 마스터를 기준으로 결과물 문서 작업을 이어갑니다.
         (결과물은 갱신된 CSS를 inline 복사)
```

> 즉 마스터가 항상 "이미 그 디자인을 담고 있는" 상태가 되도록 유지합니다. 결과물에만 존재하는 일회성 스타일을 남기지 않습니다.

## 4. 변경 이력 기록 (STEP 4)

초기 버전과 달라진 점이 생기면 **영향받은 각 HTML(및 css)의 상단 주석 블록**에 빠짐없이 기록합니다. (형식: 소스 주석 블록)

### 형식

- 위치: 각 HTML은 `<head>` 안, `<link>` 바로 다음에 둡니다. CSS는 파일 상단 헤더 주석 안에 둡니다.
- 최신 버전이 위로 오도록 내림차순으로 적습니다. 날짜는 절대일자(YYYY-MM-DD)로 표기합니다.

```html
<!-- ============================================================
     CHANGELOG — <파일명>
     <역할 한 줄>. 작업 규약: design-guide/DOC-GENERATION-PLAYBOOK.md (4단계)
     v1.2 · 2026-MM-DD
       - (추가) …
       - (변경) …
       - (제거) …
     v1.1 · 2026-07-01
       - …
     ============================================================ -->
```

### 버전 규칙

- **마스터(css/visual)를 STEP 3으로 건드리면 → 마이너 버전 +1** 합니다(v1.1 → v1.2). 세 가이드 파일과 css의 버전 표기(헤더·footer의 `vX.Y`)를 함께 올립니다.
- 결과물 문서는 자체 CHANGELOG를 갖되, 재생성·수정 시 그 문서에 기록합니다.
- "빠짐없이": 추가/변경/제거를 카테고리로 나눠 하나도 빠뜨리지 않습니다.

## 5. 결과물 출력 규칙 (사용자와 합의된 산출 방식)

- **CSS**: `design-guide-ue-master.css` 내용을 결과물 HTML의 `<style>`에 **통째로 복사(inline)**합니다. → 단일 파일·오프라인 무결성. (링크 참조 X)
- **C++ 코드 하이라이팅**: 생성 시점에 highlight.js `hljs-*` `<span>`을 **미리 구워넣습니다.** 런타임 스크립트 X.
- **탭 전환·복사 버튼**: `.codeblock`용 inline `<script>`를 문서 하단에 포함합니다(오프라인 동작). 복사는 `<pre>.innerText`를 넣어 span 잔재 없이 순수 코드만 복사되게 합니다.
- **품질 기준선**: 모바일 반응형, 키보드 포커스 표시, `prefers-reduced-motion`을 존중합니다.

## 6. 문서 허브 & 용어집 유지 (STEP 6)

`ue-wiki/ue-architect/design-guide-ue-index.html`는 (1) 생성 문서 목록과 (2) 중앙 용어집을 담는 허브입니다. **문서를 하나 만들 때마다** 다음을 갱신합니다.

1. **문서 목록에 등록** — 허브의 `#docs` 표에 한 행 추가: 문서명 · 유형(에셋/아키텍처/기능) · 날짜(YYYY-MM-DD) · `열기 →` 링크. 링크 경로는 허브 기준 상대경로로: 아키텍처 문서는 같은 폴더(`./파일.html`), 에셋 문서는 `../ue-asset/파일.html`.
2. **용어집에 어려운 용어 추가** — 새 문서에 초보자가 막힐 만한 용어가 있으면 `#glossary` 표에 `용어 · 한 줄 뜻 · 자세히 →` 행을 추가하고, "자세히"는 그 용어를 설명한 문서(+섹션)로 링크합니다.
3. **앵커 ID 부여** — 문서 안에서 용어를 본격 설명하는 위치(핵심 용어 callout 또는 해당 섹션)에 `id="term-<용어>"`를 달아, 용어집 링크가 정확히 그 지점으로 점프하게 합니다. 세밀한 앵커가 없으면 섹션 단위 앵커(`#components` 등)로 연결합니다.

> 링크는 파일들이 같은 폴더에 함께 있을 때만 작동합니다. 각 문서는 용어를 자체적으로 한 번 풀어 설명하므로(작성 원칙 ①), 링크가 없어도 단독으로 이해됩니다. 용어집 링크는 보조 수단입니다.

## 7. 실행 체크리스트 (매 문서 제작 시)

- [ ] 요청 성격을 판별합니다 → 01 / 02-A / 02-B 중 선택
- [ ] master.css · master-visual.html 를 기준으로 컴포넌트를 선택합니다
- [ ] **초보자 친화 원칙 충족**: 용어 풀이 · 구체 절차(위치/이름/경로) · 전제·결과 · 근거를 담았는지 확인
- [ ] **용어 규칙**: 처음 1회 설명 · 핵심 용어 블록 · 재힌트 · 한 개념 한 용어 적용
- [ ] 다섯 기둥(개념·구조·사용/구현·응용·팁)을 포함했는지 확인
- [ ] (02-B) 책임 배치 결정표로 설계 근거를 명시했는지 확인
- [ ] 표현 갭이 있으면 → master.css/visual 을 먼저 갱신(STEP 3) → 버전+1 → CHANGELOG
- [ ] 결과물: CSS inline + hljs span 구워넣기 + 탭/복사 inline JS
- [ ] 반응형·포커스·reduced-motion 확인
- [ ] **허브 갱신(STEP 6)**: 문서 목록 등록 + 어려운 용어 용어집 추가 + 앵커 ID 부여
- [ ] 변경사항이 있으면 각 파일 CHANGELOG 주석에 빠짐없이 기록
- [ ] 브라우저로 렌더링·탭/복사·링크를 검증
