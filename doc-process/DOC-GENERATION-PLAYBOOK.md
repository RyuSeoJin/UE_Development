## 작성 원칙 — 초보자 친화 (모든 문서 공통, 필수)

생성하는 모든 문서는 **언리얼을 처음 접하는 사람도 혼자 읽고 따라 할 수 있을 만큼 자세한 절차**를 담아야 합니다. "안다고 가정하고 건너뛰기"를 하지 않습니다.

이를 위해 다음을 지킵니다.

- **용어는 처음 나올 때 풀어서 설명합니다.** 예: "Possess(폰을 컨트롤러가 소유·조종하는 것)". 이후에는 용어만 써서 반복을 피합니다. 단, 아래 보완책을 함께 적용합니다.
  - **문서 내 "핵심 용어" 블록**: 개념 섹션에 주요 용어를 모아 `callout`으로 정리해, 독자가 언제든 되짚게 합니다.
  - **멀리 재등장 시 짧은 재힌트**: 처음 설명에서 여러 섹션 떨어져 다시 나오면 한 번 더 짧게 힌트합니다. 예: "Possess(앞서 설명한 폰 소유)".
  - **한 개념엔 한 용어**: 같은 개념을 다른 말로 바꿔 부르지 않습니다(폰/캐릭터/조종 대상 혼용 금지).
  - **중앙 용어집 연동**: 어려운 용어는 `design-guide-ue-index.html`의 용어집에도 등록하고, 용어를 본격 설명한 위치에 **앵커 ID**(예: `id="term-possess"`)를 부여해 "자세히 →"가 정확히 점프하게 합니다. (STEP 8 참조)
- **절차는 '무엇을 · 어디서 · 어떻게'까지 구체적으로** 적습니다. 메뉴/버튼 위치, 노드 이름, 패널 이름, 파일 경로, 클래스 이름을 명시합니다. ("설정한다" X → "Project Settings ▸ Maps & Modes ▸ Default GameMode 에서 지정합니다" O)
- **전제 조건과 결과를 명시합니다.** 각 단계 시작 전 필요한 것, 끝난 뒤 확인되어야 할 상태를 적습니다.
- **왜 그렇게 하는지 근거를 함께** 적습니다. 방법뿐 아니라 이유를 알아야 응용할 수 있습니다.
- **중간 과정을 생략하지 않습니다.** 숙련자에게 당연한 단계도 빠뜨리지 않습니다.
- **막히기 쉬운 지점은 Pitfall(⚠️)로** 미리 경고합니다.
- **예제 코드(.h/.cpp)에는 설명 주석을 꼼꼼하게 답니다.** 코드도 본문과 같은 초보자 친화 원칙을 따릅니다 — 독자가 주석만 따라 읽어도 코드의 흐름과 의도를 이해할 수 있어야 합니다.
  - **의미 있는 모든 줄/블록에** 주석을 답니다: 매크로·지정자의 의미(예: `UPROPERTY`가 GC 추적에 미치는 효과, `BlueprintType`의 노출 범위), 함수가 **엔진에 의해 언제 호출되는지**(예: `SetupPlayerInputComponent`는 폰 빙의 시 자동 호출), 값의 **범위·단위**(예: 축 입력 Value는 -1.0 ~ 1.0), 순서가 있는 로직은 **단계 번호**(1) 2) 3)).
  - **'무엇'보다 '왜'를** 적습니다: 초기값이 왜 필수인지, `nullptr` 체크가 없으면 언제 크래시하는지, 전방 선언을 왜 쓰는지 등 이유와 근거 중심. 코드를 그대로 읽어주는 주석(예: "변수에 값을 대입한다")은 금지.
  - **형식**: 한국어, 짧은 블록은 코드 오른쪽 한 줄(`// …`), 설명이 길거나 블록 단위면 코드 위 줄에 답니다. 결과물 HTML에서는 hljs 규칙에 따라 `hljs-comment` `<span>`으로 구워넣습니다.
  - `.h`와 `.cpp` 모두 대상이며, 탭으로 나뉜 코드블록의 **모든 탭**에 동일 기준을 적용합니다.
- **구현성 문서에는 "🚀 이 문서로 작업 시작하기" 블록을 포함합니다.** (v3.9 신설) 코드/스켈레톤 섹션 첫머리에 다음 4요소를 문서 맞춤으로 담습니다 — 독자가 "월요일에 뭐부터"를 문서 안에서 찾을 수 있게:
  1. **전제 조건** (`callout info`) — 선행 문서·프로젝트 상태·이 문서의 착수 순서상 위치.
  2. **만들 파일 표** (`.tbl`) — 파일명 · 부모 클래스(**클래스 위저드 검색어** 그대로) · 비고. 기존 파일에 내용을 채우는 경우 "새 파일 아님 — ○○에 추가"로 구분합니다.
  3. **코드 외 작업** (`callout tip`) — 에디터에서 할 일(애셋 생성·데이터 입력·Project Settings·콜리전 등). **작업이 없으면 "없음"을 명시**합니다(없다는 것도 정보).
  4. **컴파일 주의** (`pitfall`) — 이 문서 작업 중 Live Coding으로 되는 범위와 재빌드가 필요한 지점.

  클래스 위저드·Build.cs·컴파일 규칙 등 **모든 문서가 공유하는 반복 절차는 블록에서 반복하지 않고** 중앙 문서 `ue-wiki/ue-architect/core-cpp/ue-cpp-class-workflow.html`가 담당합니다 — 블록에서는 『C++ 클래스 생성 · 빌드 워크플로우』라고 **이름으로만 언급**합니다(문서 간 상대 링크 금지 규약 유지, 연결은 허브·용어집 담당). 코드가 없는 순수 개념/설계 문서는 이 블록을 생략할 수 있습니다.

> 균형: 자세하되 장황하지 않게. 반복·군더더기 대신 "필요한 모든 단계를 빠짐없이"가 목표입니다.

## 0. 언제 발동하나

사용자가 언리얼 엔진과 관련된 **설명/분석/설계 문서(HTML)** 제작을 요청할 때입니다.

요청은 세 가지 유형으로 나뉘며, 유형에 따라 **트랙(저장 위치·허브·용어집)**이 달라집니다. 트랙 규칙의 단일 소스는 `doc-process/DOC-TRACKS.md`입니다.

| 유형 | 예 | 트랙 → 저장 위치 |
|---|---|---|
| **① 엔진 자체 기능/프레임워크** | "GameMode 구조를 설명하는 문서 만들어줘" | Track A → `ue-wiki/ue-architect/<카테고리>/` |
| **② 판매 에셋 MCP 분석** | "이 마켓 위젯 에셋 분석 문서 만들어줘" | Track A → `ue-wiki/ue-asset/<판매에셋이름>/` |
| **③ 프로젝트 기능 개발** | "RecordTales에 더블 점프 기능 구현 문서 만들어줘" | Track B → `Project/<프로젝트이름>/` |

산출물은 원칙적으로 **단일 HTML 파일**(오프라인에서 열리는 완결형)입니다.

### 전체 흐름 한눈에

```
문서 요청
  → STEP 2  초안 아웃라인 작성 (+ 템플릿·트랙 가판별) → 사용자 확인 ① (방향)
  → STEP 3  작업 판별 (Case A/B/C) → 사용자 확인 ② (템플릿/마스터/신규 템플릿 + 저장 위치)
  → STEP 4  (Case B/C일 때) 마스터 또는 템플릿 갱신
  → STEP 5  상세 설계
  → 본문 작성 (STEP 7 출력 규칙)
  → STEP 8  트랙별 허브·용어집 갱신 (DOC-TRACKS.md) / STEP 6  CHANGELOG 기록
```

**사용자 확인 ①·②를 받기 전에는 본문 작성(파일 생성)을 시작하지 않습니다.**

- **결과물 저장 위치·허브·용어집·출력 방식(CSS) 규칙은 `doc-process/DOC-TRACKS.md`가 단일 소스**입니다. (유형 ① → `ue-wiki/ue-architect/<카테고리>/`(카테고리 판별: DOC-TRACKS.md §2-1), ② → `ue-wiki/ue-asset/<판매에셋이름>/`, ③ → `Project/<프로젝트이름>/`)
- **CSS 참조**: 모든 결과물 문서(Track A·B 공통)는 마스터 CSS를 `<style>`에 **inline**합니다(생성 시점 마스터 버전 스냅샷 — 이후 마스터 개정에 영향받지 않음, 단독 열람 보장). `<link>` 참조는 디자인 규칙서 3건(master-visual·템플릿 01·02)에만 허용. 상세는 STEP 7·DOC-TRACKS.md.
- 결과물 본문에서 다른 문서로의 상대경로 링크는 쓰지 않습니다(문서 연결은 허브·용어집 담당).
- 중앙 허브·프로젝트 허브도 결과물과 동일하게 CSS를 inline하며, 다른 문서를 폴더 기준 상대경로로 링크합니다(중앙 허브 기준: 아키텍처 문서 `./ue-architect/<카테고리>/`, 에셋 문서 `./ue-asset/<판매에셋이름>/`, 프로젝트 허브 `../Project/<프로젝트이름>/`).
- 템플릿(`design-template/*.html`)은 마스터 CSS를 `../design-guide/design-guide-ue-master.css` 로 링크합니다.

## 2. 초안 아웃라인 & 사용자 확인 ① (STEP 2)

작업 판별을 하기 **전에**, 먼저 문서의 **큰 뼈대(초안 아웃라인)**를 잡습니다. 이 단계의 목적은 "어떤 섹션이 필요하고, 각 섹션에 어떤 컴포넌트(트리·관계도·코드 탭·결정표 등)를 쓸지"를 대략적으로 파악하는 것입니다. 대략적으로 파악함으로써, **사용자에게 어떻게 설명하면 혼자 읽고 따라 할 수 있을지 고민해야 합니다.** 즉 아웃라인은 단순한 목차 나열이 아니라, [작성 원칙 — 초보자 친화]를 어떤 구성으로 실현할지(어느 섹션에서 어떤 용어를 풀고, 어떤 컴포넌트로 절차를 보여줄지)를 미리 설계하는 단계입니다.

### 수행 내용

1. **섹션 목록 나열** — 요청 내용을 기준으로, 독자가 이해에 도달하는 순서에 맞춰 이 문서에 포함될 섹션 제목을 나열합니다. **이해 순서는 템플릿마다 다르므로** `design-template/TEMPLATE-CATALOG.md`에 정의된 템플릿별 섹션 구조를 기준으로 삼습니다(예: 01 에셋 분석은 개념 → 구조 → 구성요소 → … → 팁, 02-B 기능 구현은 요구사항 → 설계결정 → … → 검증). 요청 내용에 맞게 섹션을 추가·흡수·축약할 수 있습니다.
2. **각 섹션의 핵심 컴포넌트 명시** — 각 섹션에서 사용할 컴포넌트를 짧게 적습니다. 이때 "이 컴포넌트가 독자의 이해를 어떻게 돕는지"를 선택 기준으로 삼습니다. (예: 순서 있는 절차 → 파이프라인 `.pipe-step`, 클래스 구조 → 관계도 `.rel-map`, 복붙할 코드 → 코드 탭 `.codeblock`)
3. **섹션별 설명 전략 메모** — 각 섹션에서 독자가 막힐 지점(처음 등장하는 용어·전제 조건·건너뛰기 쉬운 중간 단계)을 표시하고, 혼자 따라 할 수 있게 만들 설명 방식을 한 줄씩 메모합니다. 상세 용어 계획·앵커 ID는 STEP 5에서 확정하므로 여기서는 대략만 잡습니다.
4. **템플릿·트랙 가판별** — 아웃라인이 `design-template/TEMPLATE-CATALOG.md`의 어느 템플릿(01/02-A/02-B/…)에 얹히는지, 그리고 요청 유형(①②③)에 따라 어느 트랙·저장 위치(`doc-process/DOC-TRACKS.md`)로 가는지 내부적으로 판단합니다. 유형 ①이면 **카테고리 폴더까지 판별**합니다(DOC-TRACKS.md §2-1 — 기존 카테고리에 없으면 신규 폴더 후보도 정함). (사용자 확정은 STEP 3에서 받습니다.)
5. **사용자에게 공유 (확인 ①)** — 초안 아웃라인과 섹션별 설명 전략 요지를 함께 사용자에게 보여주고, **방향이 맞는지 확인을 받습니다.** 유형 ①이면 **저장 카테고리 폴더(신규 생성 필요 여부 포함)도 함께 공유**합니다. 방향이 어긋나면 아웃라인을 수정해 다시 확인받습니다.

> **왜 이 단계가 먼저인가**: 문서의 뼈대를 잡아야 "기존 마스터에 없는 컴포넌트가 필요한지", "기존 템플릿으로 표현 가능한지"를 정확히 판단할 수 있습니다. 뼈대 없이 판별하면 작업 중에 뒤늦게 갭이 발견되어 마스터를 다시 건드리게 됩니다.

## 3. 작업 판별 & 사용자 확인 ② (STEP 3)

확인 ①로 방향이 확정된 아웃라인을 근거로, 이 작업이 아래 **세 가지 케이스 중 어디에 해당하는지 판별**하고 사용자에게 **재확인 질문**을 합니다. 판별 기준은 `design-template/TEMPLATE-CATALOG.md`가 단일 소스입니다.

| 케이스 | 조건 | 사용자에게 할 질문 (문구 준수) |
|---|---|---|
| **Case A** | 기존 템플릿으로 작업 가능하고, 아웃라인의 모든 컴포넌트가 마스터에 이미 존재 | "**~~ 요청에 대한 문서 작업은 [템플릿명] 문서 템플릿을 기반으로 작성되어 [저장 위치]에 저장될 예정인데 괜찮을까요?**" |
| **Case B** | 기존 템플릿으로 작업 가능하지만, 아웃라인의 컴포넌트 중 기존 `master.css`/`master-visual.html`로 표현할 수 없는 디자인이 있음 | "**~~ 요청에 대한 문서 작업은 [템플릿명] 문서 템플릿을 기반으로 작성되어 [저장 위치]에 저장될 예정인데, {특정 기능}에 대해 마스터를 갱신하고 진행해도 괜찮을까요?**" — {특정 기능}에는 어떤 신규 디자인인지 간략 설명을 반드시 포함합니다(예: "좌우 3열 비교 카드", "단계별 타임라인 컴포넌트") |
| **Case C** | 카탈로그의 어떤 템플릿에도 부합하지 않음 | "**~~ 요청에 대한 문서 작업은 현재 존재하는 템플릿에 부합하는 케이스가 없는데, 새로운 템플릿 규칙서를 작성 후 그에 따라 작업을 진행해도 괜찮을까요? (저장 위치는 [저장 위치] 예정)**" |

- **[저장 위치]**는 트랙 판별 결과입니다: 유형 ① → `ue-wiki/ue-architect/<카테고리>/`(카테고리까지 포함해 명시, 판별 기준: DOC-TRACKS.md §2-1), ② → `ue-wiki/ue-asset/<판매에셋이름>/`, ③ → `Project/<프로젝트이름>/`. (기준: `doc-process/DOC-TRACKS.md`)

### 규칙

- 사용자가 **승인하면** 다음 단계로 진행합니다. Case A → STEP 4 건너뛰고 STEP 5로. Case B/C → STEP 4 수행 후 STEP 5로.
- 사용자가 **거절하거나 다른 방향을 제시하면** 그 피드백을 반영해 STEP 2(아웃라인) 또는 판별부터 다시 수행합니다.
- Case B에서 갭 판단 기준: 아웃라인에 나열한 컴포넌트를 하나씩 `master-visual.html`의 데모와 대조합니다. 색·변형(variant)만 다른 것도 갭입니다(임의로 결과물에만 새 스타일을 넣는 것 금지).

## 4. 마스터/템플릿 갱신 (STEP 4) — Case B·C 전용

사용자 확인 ②에서 승인받은 내용대로 마스터 또는 템플릿을 갱신합니다. **Case A는 이 단계를 건너뜁니다.**

### 갱신 공통 원칙 — "추가는 하되, 제거하지 않는다"

마스터와 템플릿은 **누적 자산**입니다. 문서 하나를 만들 때마다 필요한 것을 보태며 성장하고, 한 번 들어간 것은 다른 문서들이 이미 의존하고 있을 수 있습니다.

1. **승인 범위만 갱신** — 확인 ②에서 사용자에게 설명하고 승인받은 신규 디자인/템플릿 범위만 갱신합니다. 승인받지 않은 항목을 끼워 넣지 않습니다.
2. **없는 것만 추가** — 마스터/템플릿에 **없는 내용만** 새로 추가합니다. 이미 존재하는 토큰·컴포넌트·섹션 규칙은 재정의하지 않고 그대로 재사용합니다. 비슷한 컴포넌트가 이미 있으면 새로 만들지 말고 변형(variant)으로 추가합니다.
3. **미사용 ≠ 제거** — 기존에 있던 토큰·컴포넌트·섹션이 **이번 문서 작성 구조에서 쓰이지 않더라도 제거하지 않습니다.** "이 문서에 안 쓰임"은 제거 사유가 아닙니다(다른 문서·템플릿이 사용 중일 수 있음).
4. **제거·변경은 별도 승인 사항** — 기존 항목의 제거나 동작이 달라지는 변경이 정말 필요하면, 이 STEP에서 임의로 하지 말고 **사용자에게 별도로 승인**을 받은 뒤 진행하고, CHANGELOG에 `(제거)`/`(변경)`으로 명시합니다(STEP 6).

### Case B — 마스터 갱신 (갭 갱신 루프)

확인 ②에서 승인받은 **신규 디자인 갭만** 마스터에 추가합니다. 기존 템플릿 자체는 수정하지 않으며(템플릿으로 표현 가능한 경우가 Case B의 전제), 위 공통 원칙에 따라 기존 마스터 내용도 건드리지 않습니다.

```
갭 감지 (STEP 3에서 승인됨)
  │
  ├─(1) design-guide/design-guide-ue-master.css 에 필요한 토큰/컴포넌트를 추가합니다.
  │      · 새 색이 필요하면 기존 색을 재활용하지 말고, 팔레트 톤에 맞는 1색만 추가합니다.
  │      · 클래스명·변수명은 기존 네이밍 관습을 따릅니다.
  │
  ├─(2) design-guide/design-guide-ue-master-visual.html 에 그 사용법을 등록합니다.
  │      · 색이면 색상표 스와치에, 컴포넌트면 "Live 데모 + 설명"으로 추가합니다.
  │
  ├─(3) STEP 6 에 따라 두 파일에 변경 이력(CHANGELOG)을 기록하고 버전을 올립니다.
  │
  └─(4) 갱신된 마스터를 기준으로 다음 단계(STEP 5)를 이어갑니다.
         (결과물은 갱신된 CSS를 inline 복사)
```

> 즉 마스터가 항상 "이미 그 디자인을 담고 있는" 상태가 되도록 유지합니다. 결과물에만 존재하는 일회성 스타일을 남기지 않습니다.

### Case C — 신규 템플릿 생성 & 카탈로그 관리

기존 템플릿을 고쳐서 끼워 맞추지 않고, **새 템플릿을 만들어 카탈로그에서 관리**합니다. `design-template/TEMPLATE-CATALOG.md`의 **"신규 템플릿 등록 절차"**를 따릅니다. 요약:

1. **기존 템플릿은 수정하지 않습니다** — 부합하는 템플릿이 없다는 판별(Case C)이 이미 내려졌으므로, 기존 템플릿에 섹션을 덧대 변형하지 말고 `design-template/NN-<이름>.html`을 새 번호로 생성합니다(마스터 CSS 기반 `../design-guide/` 링크, 섹션 표준 구성 + 전용 컴포넌트 Live 데모 포함).
2. 새 템플릿에 마스터에 없는 컴포넌트가 필요하면 **위 Case B 루프를 먼저 수행**합니다(마스터 → 템플릿 순).
3. **카탈로그에 등록해야 템플릿으로 인정됩니다** — `TEMPLATE-CATALOG.md`에 판별 기준 행 + 템플릿 항목(용도·섹션 구조(독자 이해 순서)·전용 컴포넌트·버전)을 등록합니다. 이후 이 템플릿의 판별·버전·이력 관리는 카탈로그가 담당합니다.
4. 결과물 저장 위치가 기존 트랙(`doc-process/DOC-TRACKS.md`)에 부합하지 않으면 DOC-TRACKS.md에 트랙/위치를 먼저 정의하고 이 규약과 CLAUDE.md의 폴더 구조에 반영합니다.
5. STEP 6에 따라 CHANGELOG를 기록합니다(새 템플릿은 v1.0부터).

## 5. 문서 체계화 설계/구조화 (STEP 5) — 상세 설계

확인 ①·②가 끝나 **템플릿과 마스터가 확정된 상태**에서, STEP 2의 아웃라인을 바탕으로 **문서의 상세 구조를 설계**합니다.

### 수행 내용

1. **섹션별 상세 구성 확정** — 각 섹션의 소제목(h3), 사용할 컴포넌트, 담을 내용 요지를 구체적으로 결정합니다.
2. **컴포넌트 배치 순서 확정** — 트리·관계도·결정표·코드 탭 등을 어느 섹션의 어느 위치에 넣을지 결정합니다.
3. **용어 계획** — 핵심 용어 블록에 넣을 용어, 앵커 ID(`id="term-..."`)를 사전에 정리합니다.
4. **코드 예제 범위 확정** — 코드 탭으로 넣을 파일 목록과 각 파일의 핵심 포인트를 정합니다.

> 이 단계에서 확정된 설계를 그대로 따라 문서 본문을 작성(STEP 7)하므로, 작성 중 구조 변경이 최소화됩니다.

### 템플릿 준수 사항

- 확정된 템플릿에 정의된 **섹션 순서와 컴포넌트 사용법**을 그대로 따릅니다. 섹션 구성의 기준은 `design-template/TEMPLATE-CATALOG.md`의 **해당 템플릿 섹션 구조(독자 이해 순서)**이며, 요청 내용에 맞게 섹션을 추가·흡수·축약할 수 있습니다.
- 각 섹션은 위의 **[작성 원칙 — 초보자 친화]**를 충족해야 합니다. 단계·용어·근거를 빠짐없이 담습니다.
- 02-B(기능 구현)의 핵심은 **책임 배치 결정표**입니다. "이 데이터/기능을 어느 클래스(GameMode/GameState/PlayerController/Pawn/GameInstance/SaveGame/Subsystem)에, 왜 두는지"를 먼저 결정하고 근거와 함께 문서에 남깁니다.

## 6. 변경 이력 기록 (STEP 6)

초기 버전과 달라진 점이 생기면 **영향받은 각 HTML(및 css/md)의 상단 주석 블록**에 빠짐없이 기록합니다. (형식: 소스 주석 블록)

### 형식

- 위치: 각 HTML은 `<head>` 안, `<link>` 바로 다음에 둡니다. CSS는 파일 상단 헤더 주석 안에, md는 상단 개정 이력 또는 하단 CHANGELOG 섹션에 둡니다.
- 최신 버전이 위로 오도록 내림차순으로 적습니다. 날짜는 절대일자(YYYY-MM-DD)로 표기합니다.

```html
<!-- ============================================================
     CHANGELOG — <파일명>
     <역할 한 줄>. 작업 규약: doc-process/DOC-GENERATION-PLAYBOOK.md (STEP 6)
     v1.2 · 2026-MM-DD
       - (추가) …
       - (변경) …
       - (제거) …
     v1.1 · 2026-07-01
       - …
     ============================================================ -->
```

### 버전 규칙

- **마스터(css/visual)를 STEP 4으로 건드리면 → 마이너 버전 +1** 합니다(v1.1 → v1.2). 마스터 두 파일과 템플릿들의 버전 표기(헤더·footer의 `vX.Y`)를 함께 올립니다.
- 신규 템플릿(Case C)은 v1.0부터 시작하고, `TEMPLATE-CATALOG.md`에도 버전을 기록합니다.
- 결과물 문서는 자체 CHANGELOG를 갖되, 재생성·수정 시 그 문서에 기록합니다.
- "빠짐없이": 추가/변경/제거를 카테고리로 나눠 하나도 빠뜨리지 않습니다.

## 7. 결과물 출력 규칙 (STEP 7)

- **모든 결과물은 마스터 CSS를 `<style>`에 통째로 inline** (트랙 공통) — 단일 소스: `doc-process/DOC-TRACKS.md` 각 트랙의 "출력 방식(CSS)".
  - 생성 시점의 마스터 버전이 문서에 스냅샷으로 고정됩니다. 기준 버전을 문서 CHANGELOG와 footer(`Master vX.Y`)에 기록합니다.
  - 마스터 개정을 기존 문서에 반영하려면 해당 문서를 개별 재생성하는 명시적 작업으로만 수행합니다.
  - `<link>` 참조는 디자인 규칙서 3건(master-visual·템플릿 01·02)에만 허용됩니다.
- **문서 간 상대 링크 금지(트랙 공통)**: 결과물 본문에서 다른 문서로의 상대경로 링크는 쓰지 않습니다. 문서 연결은 허브·용어집이 담당합니다.
- **C++ 코드 하이라이팅**: 생성 시점에 highlight.js `hljs-*` `<span>`을 **미리 구워넣습니다.** 런타임 스크립트 X. (트랙 공통)
- **다이어그램(Mermaid/옵시디언 그래프 뷰 스타일)**: 생성 시점에 **SVG로 미리 렌더링해 구워넣습니다**(`.diagram`·`.graph-net`, master v1.5). mermaid.js 등 런타임 렌더링 라이브러리는 넣지 않습니다. 대신 **Mermaid 소스를 `.codeblock` 단일 탭(`*.mmd`)으로 병기**해 복사 → 옵시디언·GitHub 재사용을 보장합니다. (트랙 공통, v3.7)
- **탭 전환·복사 버튼**: `.codeblock`용 inline `<script>`를 문서 하단에 포함합니다(오프라인 동작). 복사는 `<pre>.innerText`를 넣어 span 잔재 없이 순수 코드만 복사되게 합니다. (트랙 공통)
- **품질 기준선**: 모바일 반응형, 키보드 포커스 표시, `prefers-reduced-motion`을 존중합니다. (트랙 공통)

## 8. 트랙별 허브 & 용어집 유지 (STEP 8)

**문서를 하나 만들 때마다** 트랙에 맞는 허브·용어집을 갱신합니다. 상세 규칙(어느 허브·어느 용어집·부트스트랩 절차)은 `doc-process/DOC-TRACKS.md`가 단일 소스입니다.

| 트랙 | 갱신 대상 | 용어집 규칙 |
|---|---|---|
| **Track A** (유형 ①·②) | 중앙 허브 `ue-wiki/design-guide-ue-index.html` — 문서 목록 + 중앙 용어집 | **범용 UE API/구조 용어만.** 특정 에셋·프로젝트 고유명사 금지(에셋 고유명사는 문서 내 자체 설명으로만) |
| **Track B** (유형 ③) | 프로젝트 허브 `Project/<프로젝트이름>/<이름>-index.html` — 문서 목록 + 프로젝트 용어집. (중앙 허브에는 프로젝트 허브 행 1개만) | **프로젝트 고유명사 허용.** 정의에 프로젝트 수치·규칙이 들어가는 용어의 홈 |

트랙과 무관한 **공통 원칙**:

1. **문서 목록 등록** — 해당 허브의 `#docs` 표에 한 행 추가: 문서명 · 유형 · 날짜(YYYY-MM-DD) · `열기 →` 링크(허브 기준 상대경로).
2. **용어집에 어려운 용어 추가** — 배치할 용어집은 DOC-TRACKS.md §4 용어 배치 결정 기준("정의 문장에서 프로젝트/에셋 이름을 지워도 성립하는가?")으로 판단합니다. `용어 · 한 줄 뜻 · 자세히 →` 행을 추가하고, "자세히"는 그 용어를 설명한 문서(+섹션)로 링크합니다.
3. **앵커 ID 부여** — 문서 안에서 용어를 본격 설명하는 위치(핵심 용어 callout 또는 해당 섹션)에 `id="term-<용어>"`를 달아, 용어집 링크가 정확히 그 지점으로 점프하게 합니다. 세밀한 앵커가 없으면 섹션 단위 앵커(`#components` 등)로 연결합니다.

> 링크는 파일들이 저장소 구조 그대로 함께 있을 때만 작동합니다. 각 문서는 용어를 자체적으로 한 번 풀어 설명하므로(작성 원칙 ①), 링크가 없어도 단독으로 이해됩니다. 용어집 링크는 보조 수단입니다.

- [ ] **상세 설계(STEP 5)**: 확정된 마스터·템플릿 기준으로 섹션별 상세 구성·컴포넌트 배치·용어 계획·코드 범위를 확정

- [ ] **용어 규칙**: 처음 1회 설명 · 핵심 용어 블록 · 재힌트 · 한 개념 한 용어 적용
- [ ] 카탈로그에 정의된 해당 템플릿의 섹션 구조(독자 이해 순서)를 따랐는지 확인
- [ ] (02-B) 책임 배치 결정표로 설계 근거를 명시했는지 확인
- [ ] 결과물: 마스터 CSS inline(트랙 공통, 생성 시점 버전 스냅샷 — CHANGELOG·footer에 기록) + hljs span 구워넣기 + 탭/복사 inline JS (STEP 7)

- [ ] **트랙별 허브 갱신(STEP 8)**: 트랙에 맞는 허브(중앙/프로젝트)에 문서 목록 등록 + 용어 배치 기준(DOC-TRACKS.md §4)에 따라 용어집 추가 + 앵커 ID 부여
