# 문서 트랙 규약 (DOC-TRACKS)

> 모든 문서 요청은 `doc-process/DOC-GENERATION-PLAYBOOK.md`(공통 절차)를 처음부터 끝까지 따릅니다.
> 이 문서는 그 위에서 **요청 유형에 따라 달라지는 것** — 결과물 저장 위치 · 허브 · 용어집 — 을 정의합니다.
> **저장 위치·허브·용어집의 단일 소스는 이 파일입니다.** (템플릿 구조의 단일 소스는 `design-template/TEMPLATE-CATALOG.md`)

## 1. 요청 유형 판별 (트랙 판별)

| 유형 | 요청 성격 | 트랙 | 결과물 저장 위치 |
|---|---|---|---|
| **①** | 언리얼 엔진이 **자체 제공하는 기능·구조(프레임워크)** 문서 (GameMode 구조, Subsystem, DataTable 등) | **Track A** | `ue-wiki/ue-architect/<카테고리>/` (§2-1) |
| **②** | 마켓플레이스 등에서 **판매되는 에셋**을 언리얼 MCP로 **분석** | **Track A** | `ue-wiki/ue-asset/<판매에셋이름>/` |
| **③** | 사용자가 **특정 프로젝트의 특정 기능**을 개발하기 위해 요청 | **Track B** | `Project/<프로젝트이름>/` |

- ②는 ①의 기능을 직간접적으로 이용하므로 둘 다 Track A(UE 지식 위키)로 묶입니다.
- ②의 에셋별 폴더는 **같은 에셋의 후속 분석 문서도 같은 폴더에 축적**해 에셋 단위로 관리합니다. (폴더명은 판매 에셋 이름, 공백 대신 하이픈)

## 2. Track A — UE 지식 위키 (유형 ①·②)

**성격**: 프로젝트와 무관하게 재사용되는 언리얼 지식. 누가 언제 봐도 유효한 범용 문서.

1. **저장**: ① → `ue-wiki/ue-architect/<카테고리>/` (카테고리 판별은 아래 **§2-1**), ② → `ue-wiki/ue-asset/<판매에셋이름>/`.
2. **출력 방식(CSS)**: Track B와 동일하게 마스터 CSS를 **`<style>`에 통째로 inline**합니다(중앙 허브 포함) — 생성 시점의 마스터 버전을 스냅샷으로 고정해, 이후 마스터가 개정돼도 완성된 문서의 모습이 사후에 변하지 않고, 파일을 단독으로 열거나 저장소 밖으로 옮겨도 깨지지 않습니다.
   - 기준 마스터 버전을 문서 CHANGELOG와 footer(`Master vX.Y`)에 기록합니다.
   - 마스터 개정을 기존 문서에 반영하려면 그 문서를 **개별 재생성**하는 명시적 작업으로만 수행합니다(해당 문서 CHANGELOG에 기록).
   - `<link>` 참조는 **디자인 규칙서 3건**(design-guide-ue-master-visual.html, design-template/01·02)에만 허용됩니다 — 규칙서는 항상 최신 마스터를 보여줘야 하기 때문입니다.
3. **중앙 허브 등록**: 문서를 만들 때마다 `ue-wiki/design-guide-ue-index.html`(위키 루트)의 문서 목록(`#docs`)에 등록합니다. 유형 ① 문서는 **해당 카테고리 소구획의 표**에 등록합니다(카테고리가 새로 생겼으면 허브에 소구획도 추가). 링크는 허브 기준: 아키텍처 문서 `./ue-architect/<카테고리>/파일.html`, 에셋 문서 `./ue-asset/<판매에셋이름>/파일.html`.
4. **중앙 용어집 갱신**: 어려운 용어는 허브 용어집(`#glossary`)에 등록하되, 아래 **용어 배치 기준(§4)**을 따릅니다.
   - **중앙 용어집에는 범용 UE API/구조 용어만** 넣습니다.
   - **특정 에셋의 고유 명사는 넣지 않습니다.** 에셋 고유 명사는 해당 분석 문서 안에서 자체적으로 풀어 설명합니다(작성 원칙 ①로 충분).

### 2-1. 유형 ① 카테고리 규칙 (ue-architect 하위 폴더)

유형 ① 문서는 `ue-wiki/ue-architect/` 바로 아래가 아니라 **반드시 카테고리 하위 폴더 안에** 저장합니다. **이 표가 카테고리의 단일 소스**입니다 — 새 카테고리 폴더를 만들면 반드시 이 표에 행을 추가합니다.

| 폴더명 | 범위 (판별 기준) | 예 |
|---|---|---|
| `core-cpp/` | C++ 언어 구조·리플렉션·UHT·모듈·빌드 등 **"코드 자체"의 구조** | 클래스 계층, UCLASS/UPROPERTY 매크로 |
| `framework/` | 게임플레이 프레임워크·Subsystem·상태 머신 등 **엔진이 제공하는 게임 구조/패턴** | GameMode, GameInstanceSubsystem |
| `ui/` | UMG·Slate·Common UI 등 **UI 계층 전반** | 위젯 블루프린트, SWidget, 레이어 스택 |
| `data/` | DataTable·DataAsset·Config 등 **데이터 관리** | 마스터 데이터, 행 구조체 |

**폴더 이름 규칙**: 영문 소문자 kebab-case, 단수형 우선, `ue-` 접두어 금지(상위 폴더가 이미 ue-architect). 파일명 규칙(`ue-….html`)은 기존 그대로.

**배치 판별 절차** (문서 1건당 카테고리는 1개):

1. 문서의 **핵심 주제 시스템**이 위 표의 판별 기준에 맞으면 그 폴더에 저장합니다. 두 영역에 걸치면 **"독자가 이 문서를 찾을 때 먼저 떠올릴 영역"**을 따릅니다(예: UMG+C++ 연동 → `ui/`).
2. 맞는 카테고리가 없으면 아래 **표준 후보 목록**에서 골라 새 폴더를 생성합니다:
   `rendering/`, `animation/`, `networking/`, `ai/`, `physics/`, `audio/`, `editor-tools/`, `build-pipeline/`
3. 후보 목록에도 없으면 kebab-case 이름을 새로 제안하되, 신규 폴더 생성은 **플레이북 STEP 2 사용자 확인 ① 시점에 저장 폴더(기존/신규 여부 포함)를 함께 공유**해 확인받습니다.
4. 새 폴더를 만들면 **① 위 카테고리 표에 행 추가, ② 중앙 허브 문서 목록에 해당 카테고리 소구획 추가** — 두 가지를 반드시 함께 수행합니다.

## 3. Track B — 프로젝트 개발 문서 (유형 ③)

**성격**: 특정 프로젝트의 설계 결정·수치·규칙이 담긴 문서. 프로젝트 맥락 안에서만 유효.

1. **저장**: `Project/<프로젝트이름>/`.
2. **출력 방식(CSS)**: 마스터 CSS를 **`<style>`에 통째로 inline**합니다(프로젝트 허브 포함) — 프로젝트 폴더는 통째로 이동·외부 공유될 수 있으므로 문서가 자립해야 합니다. 상대경로 링크는 쓰지 않습니다.
3. **프로젝트 허브 갱신**: 각 프로젝트는 자체 허브 `<프로젝트이름 소문자>-index.html`을 가집니다(예: `recordtales-index.html`). 문서를 만들 때마다 프로젝트 허브의 **문서 목록**과 **프로젝트 용어집**을 갱신합니다(허브 내 문서 링크는 같은 폴더 `./`만 사용 — 폴더째 이동에 안전).
4. **프로젝트 용어집**: 해당 **프로젝트의 고유명사를 넣어도 됩니다**(권장). 가챠 규칙·수치, 시스템 이름, 구조체 이름 등 프로젝트 정의 용어의 홈입니다.
5. **중앙 허브에는 프로젝트 허브 행 1개만**: 중앙 허브 문서 목록에는 프로젝트당 "『프로젝트명』 프로젝트 허브" 행 하나만 등록합니다. 개별 문서는 프로젝트 허브가 관리합니다.

### 신규 프로젝트 부트스트랩 (프로젝트 첫 문서 요청 시)

1. `Project/<프로젝트이름>/` 폴더 생성.
2. 프로젝트 허브 `<프로젝트이름 소문자>-index.html` 생성 — 구성: 문서 목록 · 아키텍처 개요 · 프로젝트 용어집. (마스터 CSS inline, 기존 `recordtales-index.html` 구성 참고)
3. 중앙 허브 문서 목록에 프로젝트 허브 행 1개 등록 (허브 기준 `../Project/<프로젝트이름>/…-index.html`).

## 4. 용어 배치 결정 기준

용어집에 등록하기 전에 아래 기준으로 **어느 용어집에 둘지** 판단합니다.

| 용어 성격 | 배치 | 예 |
|---|---|---|
| 범용 UE API·클래스·패턴 (프로젝트가 없어도 정의가 성립) | **중앙 용어집** | `GameInstanceSubsystem`, `DataTable`, `FTimerHandle`, 상태 머신 |
| 정의에 프로젝트의 수치·규칙·이름이 들어감 | **프로젝트 용어집** | 가챠(Gold 1,000 규칙), 스쿼드, 마스터 레코드, FFocusRecord |
| 일반 개발/업무 기법이지만 이 저장소에선 프로젝트 맥락으로만 정의됨 | **프로젝트 용어집** | 포모도로(RecordTales 포커스 시스템의 25/5 사이클) |
| 특정 판매 에셋의 고유 명사 | **용어집 미등록** — 분석 문서 안에서 자체 설명 | 에셋 내부 컴포넌트/노드 이름 |

- 판단 요령: **"이 용어의 정의 문장에서 프로젝트/에셋 이름을 지워도 성립하는가?"** 성립하면 중앙, 아니면 프로젝트(또는 미등록).
- 같은 용어가 양쪽 성격을 다 가지면: 범용 정의는 중앙에, 프로젝트 특화 정의는 프로젝트 용어집에 각각 등록할 수 있습니다(정의 문장은 서로 달라야 함).

## 5. 범용 개념 승격 절차 (Track B → Track A)

프로젝트 문서 작성 중 나온 개념이 **범용 지식으로 중앙에 필요**해지면:

1. 프로젝트 내용(수치·규칙·이름)을 **제거하고 일반화**한 **유형 ① 문서**를 별도로 작성합니다 — 플레이북 절차(아웃라인 확인 ①·판별 확인 ②)를 새로 거칩니다.
2. `ue-wiki/ue-architect/<카테고리>/`에 저장하고(카테고리 판별은 §2-1) 중앙 허브 문서 목록에 등록합니다.
3. 중앙 용어집의 해당 용어 "자세히 →" 링크를 새 범용 문서로 연결합니다.
4. 원본 프로젝트 문서는 수정하지 않습니다(프로젝트 맥락 설명은 그대로 유효).

> 전용 범용 문서가 아직 없는 범용 용어는 중앙 용어집에 두되, 링크가 임시로 프로젝트 문서를 가리킬 수 있습니다. 이 경우 소스 주석으로 `TODO` 표기하고, 범용 문서가 생기면 교체합니다.
