Subsystem ↔ SaveGame 연동
로드 · 저장 타이밍 패턴

SaveGame 문서와 Subsystem 문서를 각각 읽어도 남는 질문 — "그래서 게임이 켜질 때 누가 로드하고, 값이 바뀔 때 누가 저장을 부르지?" 이 문서는 그 배선도입니다. EconomySubsystem(골드)을 예제로, 시작 → 변경 → 종료의 세 시점에 무슨 코드가 어디서 불리는지를 완전한 예제로 잇습니다.

시스템 연동 패턴 UE5 · 싱글플레이/로컬 SaveGame + GameInstanceSubsystem
01 — 개념

상자와 일꾼 — 역할부터 가르기

가장 흔한 오해는 "Subsystem을 SaveGame에 저장한다"입니다. 아닙니다 — 둘은 저장하고/되는 관계가 아니라 일꾼과 상자의 관계입니다.

📦 SaveGame = 상자

UPROPERTY 필드만 있는 데이터 덩어리입니다. 로직이 없고, 스스로 아무것도 하지 않습니다. 디스크에 통째로 직렬화되는 것은 이 상자뿐입니다.

Gold보유 목록SaveVersion

🔧 Subsystem = 일꾼

로직과 런타임 상태를 가진 살아있는 서비스입니다. 디스크에 저장되지 않으며, 시작할 때 상자에서 값을 꺼내 오고 값이 바뀌면 상자에 다시 넣습니다.

AddGold()검증 로직델리게이트 방송

따라서 "연동"이란 딱 세 가지 시점의 규칙입니다: ① 시작할 때 상자를 열어 읽기 ② 값이 바뀔 때 상자에 쓰고 디스크에 반영 ③ 끝날 때 보험 저장. 이 셋만 정확하면 나머지는 각 문서(SaveGame·Subsystem)가 이미 다룬 내용입니다.

📌

핵심 용어 — 변경 즉시 저장(Save-on-Change): 값을 바꾸는 함수가 그 자리에서 저장까지 부르는 전략. 크래시·강제 종료에도 데이터가 살아남는다. / 저장 주인(Save Owner): SaveGame 객체를 보유하고 로드·저장을 전담하는 단 하나의 Subsystem — 이 문서에선 USaveSubsystem. / InitializeDependency: Subsystem끼리는 초기화 순서가 보장되지 않으므로, "나보다 먼저 켜져야 하는 Subsystem"을 Collection.InitializeDependency<T>()로 명시해 순서를 강제하는 장치.

1
엔진이 GameInstance를 만든다
게임 실행(또는 PIE 시작) 시 자동. 이때 모든 GameInstanceSubsystem의 Initialize()가 순서 보장 없이 불립니다 — 그래서 2·3단계의 순서 장치가 필요합니다.
2
저장 주인이 상자를 연다
USaveSubsystem::Initialize: DoesSaveGameExist? → 있으면 LoadGameFromSlot + 캐스트, 없으면(첫 실행) CreateSaveGameObject로 새 상자. 결과: 이 시점 이후 GetSave()는 항상 유효한 상자를 돌려줌.
3
도메인 일꾼이 주인에게 연결한다
UEconomySubsystem::Initialize 첫 줄에서 Collection.InitializeDependency<USaveSubsystem>() — "저장 주인 먼저 켜라"를 엔진에 강제한 뒤 참조를 캐시. 이 한 줄이 없으면 실행 순서 복불복으로 nullptr 크래시가 간헐 발생합니다.
4
값이 바뀔 때마다: 쓰고 → 저장
AddGold(100) = ① 검증 ② 상자 필드 갱신(Save->Gold += …) ③ SaveSub->RequestSave() ④ 델리게이트 방송(UI 갱신용). 변경과 저장이 한 함수 안 — 다른 어디서도 저장을 신경 쓸 필요가 없어집니다.
5
종료: 보험 저장
GameInstance가 닫힐 때 USaveSubsystem::Deinitialize가 자동 호출 → 마지막 RequestSave(). 4단계가 본선이므로 이건 안전망입니다(크래시 시엔 안 불리지만, 4단계 덕분에 잃는 게 없음).

#include "CoreMinimal.h"
#include "GameFramework/SaveGame.h"

class UAccountSaveGame : public USaveGame
{
	GENERATED_BODY()

public:
	// --- Economy 구역 ---
	UPROPERTY()
	int64 Gold = 0;                // 초기값 = 첫 실행(새 상자)의 기본값이 된다

	// --- 다른 도메인 구역이 여기 늘어난다 (수집·길드 등) ---

	// 상자 구조를 바꿀 때 구버전 파일을 구분하기 위한 버전 표식 (SaveGame 문서의 마이그레이션 참조)
	UPROPERTY()
	int32 SaveVersion = 1;

#pragma once

#include "CoreMinimal.h"
#include "Subsystems/GameInstanceSubsystem.h"
#include "SaveSubsystem.generated.h"

class UAccountSaveGame;   // 전방 선언 — 헤더 의존 최소화(빌드 시간)

// 저장 주인 — SaveGame 객체를 보유하고 디스크를 만지는 "유일한" 클래스.
// 다른 Subsystem은 GetSave()로 상자를 빌려 읽고 쓸 뿐, 직접 Load/Save를 부르지 않는다.
UCLASS()
class USaveSubsystem : public UGameInstanceSubsystem
{
	GENERATED_BODY()

public:
	// 엔진 호출 시점: GameInstance 생성 직후(게임 시작·PIE 시작) — 여기서 상자를 연다
	virtual void Initialize(FSubsystemCollectionBase& Collection) override;
	// 엔진 호출 시점: GameInstance 종료(정상 종료) — 보험 저장
	virtual void Deinitialize() override;

	// 도메인 Subsystem들이 상자를 빌리는 창구 — Initialize 이후 항상 유효
	UAccountSaveGame* GetSave() const { return Current; }

	// 디스크 반영 요청 — 값을 바꾼 쪽이 마지막에 부른다 (변경 즉시 저장)
	void RequestSave();

private:
	// UPROPERTY 필수! — 없으면 GC가 상자를 수거해 "갑자기 저장이 안 되는" 유령 버그가 된다
	UPROPERTY()
	TObjectPtr<UAccountSaveGame> Current = nullptr;

	// 슬롯 이름은 상수 한 곳에 — 오타로 인한 "다른 슬롯에 저장" 사고 방지
	static const FString SlotName;
};
#include "SaveSubsystem.h"
#include "AccountSaveGame.h"
#include "Kismet/GameplayStatics.h"   // LoadGameFromSlot / SaveGameToSlot 제공

const FString USaveSubsystem::SlotName = TEXT("Account");   // 결과 파일: Saved/SaveGames/Account.sav

void USaveSubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
	Super::Initialize(Collection);

	// 1) 기존 세이브가 있으면 로드 — UserIndex 0 = 로컬 단일 사용자
	if (UGameplayStatics::DoesSaveGameExist(SlotName, 0))
	{
		// LoadGameFromSlot은 부모 타입(USaveGame*)을 돌려주므로 우리 상자 타입으로 캐스트
		Current = Cast<UAccountSaveGame>(UGameplayStatics::LoadGameFromSlot(SlotName, 0));
	}

	// 2) 없거나(첫 실행) 캐스트 실패(파일 손상·타입 변경) → 새 상자 생성
	//    이 분기 덕분에 GetSave()는 "항상 유효"가 보장된다
	if (!Current)
	{
		Current = Cast<UAccountSaveGame>(
			UGameplayStatics::CreateSaveGameObject(UAccountSaveGame::StaticClass()));
		UE_LOG(LogTemp, Log, TEXT("세이브 없음 — 새 상자 생성 (첫 실행)"));
	}
}

void USaveSubsystem::RequestSave()
{
	// 상자가 없으면 저장할 것도 없다 — Initialize 전에 불리는 비정상 경로 방어
	if (!Current) { return; }

	// 동기 저장 — 상자가 작을 땐 충분. 커지면 SaveGameToSlot 대신 Async 버전으로 교체 지점
	UGameplayStatics::SaveGameToSlot(Current, SlotName, 0);
}

void USaveSubsystem::Deinitialize()
{
	// 보험 저장 — 본선은 각 도메인의 변경 즉시 저장(4단계). 크래시 시엔 안 불린다는 걸 전제
	RequestSave();
	Super::Deinitialize();
}
#pragma once

#include "CoreMinimal.h"
#include "Subsystems/GameInstanceSubsystem.h"
#include "EconomySubsystem.generated.h"

class USaveSubsystem;

// 골드 변경을 UI 등에 통지하는 방송 (새 골드 값 1개를 실어 보냄)
DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(FOnGoldChanged, int64, NewGold);

// 도메인 일꾼 — 골드의 규칙(검증·증감)을 알고, 진실은 상자(SaveGame)에만 둔다.
UCLASS()
class UEconomySubsystem : public UGameInstanceSubsystem
{
	GENERATED_BODY()

public:
	virtual void Initialize(FSubsystemCollectionBase& Collection) override;

	// BlueprintCallable: UI(BP)에서 직접 부를 수 있게 — 획득/소비의 유일한 통로
	UFUNCTION(BlueprintCallable, Category = "Economy")
	bool AddGold(int64 Delta);      // 반환: 성공 여부 (잔액 부족 시 false)

	UFUNCTION(BlueprintPure, Category = "Economy")
	int64 GetGold() const;          // 읽기는 항상 상자에서 — 사본 없음

	// UI가 구독하는 방송 — BlueprintAssignable로 BP 바인딩 허용
	UPROPERTY(BlueprintAssignable, Category = "Economy")
	FOnGoldChanged OnGoldChanged;

private:
	// 저장 주인 참조 — Initialize에서 채워지며 이후 항상 유효 (순서는 InitializeDependency가 보장)
	UPROPERTY()
	TObjectPtr<USaveSubsystem> SaveSub = nullptr;
};
#include "EconomySubsystem.h"
#include "SaveSubsystem.h"
#include "AccountSaveGame.h"

void UEconomySubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
	// ★ 이 문서의 가장 중요한 한 줄 ★
	// Subsystem 초기화 순서는 기본적으로 복불복 — 이 호출이 "SaveSubsystem을 먼저
	// 켠 뒤 나를 켜라"를 엔진에 강제한다. 없으면 아래 GetSubsystem이 간헐적 nullptr.
	Collection.InitializeDependency<USaveSubsystem>();
	Super::Initialize(Collection);

	// 저장 주인 참조 캐시 — 위 줄 덕분에 이 시점에 반드시 존재한다
	SaveSub = GetGameInstance()->GetSubsystem<USaveSubsystem>();
}

bool UEconomySubsystem::AddGold(int64 Delta)
{
	UAccountSaveGame* Save = SaveSub->GetSave();

	// 1) 검증 — 도메인 규칙은 일꾼의 몫 (소비 결과가 음수면 거절)
	if (Save->Gold + Delta < 0)
	{
		return false;
	}

	// 2) 상자 필드 갱신 — 진실은 상자에 하나, 사본을 만들지 않는다
	Save->Gold += Delta;

	// 3) 변경 즉시 저장 — 이 함수가 끝나는 순간 디스크에도 반영 완료
	SaveSub->RequestSave();

	// 4) 방송 — UI는 이 방송을 구독해 표시만 갱신 (델리게이트 문서의 원칙)
	OnGoldChanged.Broadcast(Save->Gold);
	return true;
}

int64 UEconomySubsystem::GetGold() const
{
	// 읽기도 상자에서 직접 — Economy에는 골드 사본이 존재하지 않는다
	return SaveSub->GetSave()->Gold;
}

새 도메인을 잇는 법 — 이 패턴이 자리 잡으면 확장은 기계적입니다: ① 상자에 구역 필드 추가 ② 도메인 Subsystem 생성 + InitializeDependency 한 줄 ③ 변경 함수 = 검증 → 상자 갱신 → RequestSave → 방송. Gacha·Guild 등 어떤 도메인이든 같은 4박자입니다.
