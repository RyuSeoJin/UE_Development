블루프린트 인터페이스
Blueprint Interface

"이 함수를 갖고 있겠다"는 약속(시그니처)만 정의하고 구현은 각자 하게 만드는 장치입니다. 문·NPC·상자처럼 서로 전혀 다른 클래스를, 상대가 정확히 어떤 클래스인지 몰라도 같은 한 마디("상호작용해!")로 부르는 방법을 다룹니다 — 캐스트 없이, 결합 없이.

오브젝트 간 통신 엔진 버전 무관 Blueprint · C++ 양쪽 구현
01 — 개념

인터페이스는 "구현 없는 함수 약속"이다

인터페이스(Interface)는 함수의 이름·입력·출력(시그니처)만 정의하고 본문은 비워 둔 약속 목록입니다. 어떤 클래스든 이 약속에 서명(구현)하면 "나 그 함수 있어"라고 선언한 셈이 되고, 호출하는 쪽은 상대의 실제 클래스가 무엇이든 약속만 믿고 함수를 부를 수 있습니다.

비유하면 220V 콘센트 규격입니다. 벽 콘센트(호출자)는 꽂히는 게 냉장고인지 드라이어인지 모릅니다. "220V 플러그 규격(인터페이스)을 지킨 기기"라는 것만 알면 전기를 보낼 수 있습니다. 기기마다 전기를 받아 무엇을 할지(구현)는 각자의 몫입니다.

왜 쓰나 — 캐스트 지옥에서 벗어나기

인터페이스가 없다면 "E키로 상호작용" 하나를 만들 때 호출자는 이렇게 해야 합니다: 대상을 BP_Door로 캐스트해 보고, 실패하면 BP_NPC로, 또 실패하면 BP_Chest로… 상호작용 대상이 늘 때마다 플레이어 블루프린트를 고쳐야 하고, 플레이어가 모든 대상 클래스를 직접 참조하므로 결합도(서로 묶인 정도)가 계속 올라갑니다. 인터페이스를 쓰면 플레이어는 BPI_Interact 하나만 알면 되고, 새 대상은 플레이어를 건드리지 않고 약속에 서명만 하면 됩니다.

📌

핵심 용어 — 블루프린트 인터페이스(BPI): 함수 시그니처만 담는 전용 애셋(그래프·변수 없음). / 구현(Implement): 클래스가 인터페이스를 등록하고 함수 본문을 채우는 것. / 인터페이스 메시지(Message): 대상이 인터페이스를 구현했는지 엔진이 확인하고 함수를 호출해 주는 노드(✉ 아이콘) — 구현하지 않은 대상이면 에러 없이 조용히 무시. / Does Implement Interface: 대상의 구현 여부를 bool로 돌려주는 확인용 순수 함수. / 시그니처(Signature): 함수의 이름·입력·출력 규격.

02 — 통신 방식 비교

언제 인터페이스를 쓰나

블루프린트에서 오브젝트끼리 통신하는 대표 수단은 세 가지입니다. 각각 역할이 다르므로 "무엇을 대체하는 기술"이 아니라 "언제 무엇을 꺼내 쓰는가"로 이해해야 합니다.

04 — 호출 흐름

메시지 한 통이 전달되는 5단계

호출자가 인터페이스 메시지(앞서 설명한 ✉ 노드)를 부른 순간부터 대상의 구현이 실행되기까지, 엔진 내부에서 일어나는 일입니다.

1
대상 확보
호출자가 말을 걸 대상(Actor 참조)을 얻습니다 — 라인 트레이스의 Hit Actor, 오버랩된 액터 등. 전제: 대상 참조가 유효해야 합니다.
2
(선택) 구현 여부 확인 — Does Implement Interface
대상이 약속에 서명했는지 bool로 확인합니다. 필수는 아니지만, "상호작용 가능" UI 표시처럼 호출 전에 분기가 필요할 때 씁니다.
3
Message 노드 호출
호출자 그래프에서 On Interact (Message) 노드를 실행하고 Target 핀에 대상을 연결합니다.
4
엔진이 검사 · 라우팅
엔진이 Target의 클래스를 조사해 구현했으면 그 클래스의 함수로 연결하고, 구현하지 않았으면 아무 일 없이 넘어갑니다(에러·크래시 없음).
5
대상의 구현 실행
대상 클래스에 작성해 둔 Event On Interact 본문이 실행됩니다. 결과: 문이 열리거나, NPC가 말을 걸거나 — 대상마다 다른 반응.
⚠️
"조용한 무시"는 양날의 검 — 4단계 덕분에 안전하지만, 구현을 깜빡한 대상에게 메시지를 보내면 아무 로그도 없이 그냥 아무 일도 안 일어납니다. "분명 호출했는데 반응이 없다"면 십중팔구 ① 대상이 인터페이스를 Implement 안 했거나 ② 컴파일을 안 했거나입니다. 디버깅할 땐 2단계 Does Implement Interface + Print String으로 먼저 확인하세요.
05 — 구현 ① 블루프린트

블루프린트로 만들기 (노드 그래프 포함)

예제 시나리오를 블루프린트만으로 완성합니다. 전제: 프로젝트에 플레이어 캐릭터 블루프린트(BP_PlayerCharacter)와 상호작용 대상 블루프린트(BP_Door)가 이미 있다고 가정합니다. 아래 그래프의 노드 색·핀 모양은 실제 에디터와 동일한 규칙입니다.

이벤트 함수 호출·메시지 순수 함수 흐름 제어 실행 핀 ▷ Object 핀 Bool 핀 String 핀

단계 A — 인터페이스 애셋 만들기

1
애셋 생성
Content Browser 빈 곳에서 우클릭 → Blueprints → Blueprint Interface를 선택하고 이름을 BPI_Interact로 짓습니다. (접두어 BPI_는 인터페이스 애셋임을 이름만 보고 알게 하는 관례입니다.)
2
함수 시그니처 정의
애셋을 더블클릭해 열면 함수 목록만 보입니다. My Blueprint 패널 Functions 옆 ＋를 눌러 함수 이름을 OnInteract로 바꾸고, 우측 Details 패널 Inputs ▸ ＋로 입력 Interactor(타입: Actor ▸ Object Reference)를 추가합니다. "누가 나에게 상호작용을 걸었나"를 구현자에게 알려주는 통로입니다.
3
컴파일 · 저장
툴바 Compile → Save. 결과 확인: Content Browser에 BPI_Interact 애셋이 생기고, 다른 블루프린트에서 이 이름이 검색됩니다.
🧭

출력(Output)이 있으면 '함수', 없으면 '이벤트'로 구현됩니다. OnInteract처럼 출력이 없는 시그니처는 구현 클래스의 이벤트 그래프에 빨간 이벤트 노드로 나타나고, 출력을 하나라도 추가하면(예: 성공 여부 bool 반환) 이벤트 그래프가 아니라 My Blueprint ▸ Interfaces 아래 더블클릭으로 여는 함수로 구현됩니다. "이벤트가 안 보여요"의 원인 1순위입니다.

단계 B — 대상 클래스에서 구현하기 (BP_Door)

1
인터페이스 등록
BP_Door를 열고 툴바 Class Settings를 클릭 → Details 패널 Interfaces ▸ Implemented Interfaces ▸ Add 드롭다운에서 BPI_Interact를 선택합니다. 바로 Compile — 컴파일해야 함수 목록이 나타납니다.
2
이벤트 구현
My Blueprint 패널에 새로 생긴 Interfaces ▸ On Interact를 우클릭 → Implement event. 이벤트 그래프에 빨간 Event On Interact 노드가 놓입니다. 여기서부터가 "문의 반응"입니다.
3
반응 로직 연결
아래 그래프처럼 우선 Print String으로 동작만 확인합니다(실전에서는 문 회전 Timeline·사운드로 교체). 결과: 메시지를 받으면 화면 좌상단에 문자열이 출력됩니다.

단계 C — 호출자에서 메시지 보내기 (BP_PlayerCharacter)

1
대상 확보
E키 입력 이벤트(InputAction Interact)에서 시작해, 카메라 방향 LineTraceByChannel의 Hit Actor를 변수 Hit Actor에 저장해 둡니다(트레이스 부분은 이 문서 범위 밖이라 변수로 대신합니다).
2
구현 여부 확인 후 메시지 호출
Does Implement Interface(Interface 드롭다운: BPI_Interact) → Branch → True에서 대상 핀을 드래그해 검색창에 "On Interact"를 치고 봉투(✉) 아이콘이 붙은 "On Interact (Message)"를 선택합니다. 같은 이름의 다른 노드(자기 자신의 이벤트 등)와 헷갈리지 않도록 반드시 ✉를 확인하세요.
3
동작 확인
플레이(PIE) → 문 앞에서 E키. 결과: 화면에 "문이 열렸습니다"가 출력됩니다. NPC·상자를 추가하고 싶으면 그 클래스에 단계 B만 반복하면 됩니다 — 플레이어는 그대로.
💡
Does Implement Interface + Branch는 생략해도 됩니다. Message 노드 자체가 미구현 대상을 조용히 무시하므로(04-4단계), 단순히 "되면 하고 안 되면 말고"라면 InputAction → On Interact (Message)만으로 충분합니다. 확인 노드는 호출 전에 UI 표시 등 분기가 필요할 때만 쓰세요.

(U 접두어, UINTERFACE)		리플렉션 시스템용 "등록 카드" — 엔진·에디터가 인터페이스의 존재를 아는 통로항상 비워 둠 (GENERATED_BODY만)
IInteractInterface
(I 접두어, 일반 클래스)		실제 약속 — 함수 시그니처가 사는 곳. 구현 클래스가 다중 상속하는 대상UFUNCTION 함수 선언들

함수 지정자 선택 — 누가 구현하게 할 것인가

지정자구현 주체언제 쓰나
BlueprintNativeEventC++ 기본 구현(…_Implementation) + BP가 오버라이드 가능기본 동작은 C++로 깔고, 특수 케이스만 BP에서 바꾸고 싶을 때 (권장 기본값)
BlueprintImplementableEventBP 전용 — C++ 본문을 쓰면 링크 에러반응이 전부 연출(사운드·이펙트)이라 디자이너가 소유해야 할 때
지정자 없음 (순수 C++ 가상 함수)C++만BP 노출이 전혀 필요 없을 때 — 이 경우 굳이 UINTERFACE 없이 일반 C++ 인터페이스도 가능
#pragma once

#include "CoreMinimal.h"
#include "UObject/Interface.h"          // UInterface 부모 클래스 제공
#include "InteractInterface.generated.h" // UHT 생성 코드 — 항상 마지막 include

// (1) U 클래스 — 리플렉션용 "등록 카드". 본문은 항상 비워 둔다.
//     Blueprintable: 블루프린트 클래스도 Class Settings에서 이 인터페이스를
//                    Implement할 수 있게 허용 (빼면 C++ 클래스만 구현 가능)
//     MinimalAPI: 타입 정보만 다른 모듈에 노출해 불필요한 export를 줄임
UINTERFACE(MinimalAPI, Blueprintable)
class UInteractInterface : public UInterface
{
	GENERATED_BODY()
};

// (2) I 클래스 — 실제 약속(시그니처)이 사는 곳.
//     구현 클래스는 이 클래스를 다중 상속하고 본문을 채운다.
class IInteractInterface
{
	GENERATED_BODY()

public:
	// BlueprintNativeEvent: C++ 기본 구현(OnInteract_Implementation)을 두되
	//                       블루프린트가 오버라이드할 수 있는 하이브리드 방식
	// BlueprintCallable:    블루프린트 그래프에서 이 함수를 호출(✉ Message)할 수 있게 함
	// 반환값이 없으므로 BP에서는 '이벤트'로 구현된다 (05 섹션의 규칙과 동일)
	UFUNCTION(BlueprintNativeEvent, BlueprintCallable, Category = "Interact")
	void OnInteract(AActor* Interactor);   // Interactor = 상호작용을 건 쪽(플레이어 등)
};
#pragma once

#include "CoreMinimal.h"
#include "GameFramework/Actor.h"
#include "InteractInterface.h"      // (1) 약속에 서명하려면 인터페이스 헤더가 필요
#include "InteractDoor.generated.h"

// 문 액터 — AActor를 상속하면서 IInteractInterface를 "다중 상속"해
// "나는 상호작용 가능한 물건이다"를 선언한다. (I 클래스를 상속, U 클래스가 아님!)
UCLASS()
class AInteractDoor : public AActor, public IInteractInterface
{
	GENERATED_BODY()

public:
	// (2) BlueprintNativeEvent의 실제 본문은 "함수명_Implementation"에 작성한다.
	//     엔진이 BP 오버라이드가 없을 때 이 함수를 대신 호출해 준다.
	//     virtual + override: 인터페이스가 만든 가상 함수를 재정의한다는 명시
	virtual void OnInteract_Implementation(AActor* Interactor) override;

private:
	// 문이 이미 열렸는지 — 중복 상호작용을 막는 상태 플래그.
	// 초기값 필수: C++ 기본 타입은 초기화하지 않으면 쓰레기값이 들어간다.
	bool bIsOpen = false;
};
#include "InteractDoor.h"

// OnInteract의 C++ 기본 구현.
// 호출 시점: 호출자가 Execute_OnInteract(문, 플레이어)를 부르면 엔진이
//            이 액터(또는 자식 BP)의 오버라이드 유무를 확인한 뒤 여기로 도착한다.
void AInteractDoor::OnInteract_Implementation(AActor* Interactor)
{
	// 1) 이미 열린 문이면 즉시 종료 — 같은 문을 연타해도 한 번만 반응(재진입 방지)
	if (bIsOpen)
	{
		return;
	}

	// 2) 상태를 '먼저' 바꾼다 — 아래 연출이 길어져도 그 사이 중복 실행이 막힌다
	bIsOpen = true;

	// 3) 문 열림 처리 — 실전에서는 여기에 회전 애니메이션·사운드·이펙트를 채운다.
	//    GetNameSafe: Interactor가 nullptr여도 크래시 없이 "None"을 돌려주는 안전한 이름 조회
	UE_LOG(LogTemp, Log, TEXT("%s 가 문을 열었습니다"), *GetNameSafe(Interactor));
}
// BP_PlayerCharacter의 C++ 부모 등, 호출자 쪽 코드 발췌.
// 호출 시점: Enhanced Input의 상호작용 액션 바인딩에서 호출된다고 가정.
void AMyPlayerCharacter::TryInteract()
{
	// 1) 시선 방향 라인 트레이스 등으로 대상 액터를 얻는다 (프로젝트마다 구현이 다른 부분)
	AActor* Target = GetActorInSight();

	// 2) 대상이 없거나 파괴 중이면 즉시 종료 — nullptr에 Execute_를 부르면 크래시한다
	if (!IsValid(Target))
	{
		return;
	}

	// 3) 구현 여부 확인 — Implements<U클래스>()는 C++ 구현체와 "BP에서만
	//    Implement한 클래스"를 모두 잡아낸다. (BP의 Does Implement Interface와 동일)
	//    ⚠ Cast<IInteractInterface>(Target)은 BP 구현체에서 실패하므로 쓰지 말 것
	if (Target->Implements<UInteractInterface>())
	{
		// 4) 인터페이스 함수는 반드시 I클래스::Execute_함수명(대상, 인자…)로 호출한다.
		//    직접 OnInteract()를 부르면 BP 오버라이드를 건너뛰어 버린다(아래 Pitfall)
		IInteractInterface::Execute_OnInteract(Target, this);
	}
}
⚠️
직접 호출 금지 — 반드시 Execute_ · BlueprintNativeEvent/BlueprintImplementableEvent 함수는 Target->OnInteract(...)처럼 직접 부르면 BP 오버라이드가 무시되거나 링크 에러가 납니다. UHT가 만들어 주는 정적 함수 IInteractInterface::Execute_OnInteract(대상, 인자)가 BP/C++ 구현을 올바르게 라우팅하는 유일한 공식 통로입니다.
⚠️
Cast<IInteractInterface>는 반쪽짜리 · C++ 클래스가 상속으로 구현한 경우에만 성공하고, 블루프린트의 Class Settings로 Implement한 대상에게는 실패합니다. 구현 여부 확인은 항상 Implements<UInteractInterface>()(U 클래스!)를 쓰세요.

정의 통로 비교 — BPI 애셋 vs C++ 정의

여기까지 인터페이스를 만드는 두 통로(05의 에디터 애셋 / 이 섹션의 C++ 타입)를 모두 봤습니다. 중요한 것은 두 통로의 결과물이 대칭이 아니라는 점입니다 — C++로 정의한 인터페이스는 양쪽 세계 모두에서 구현·호출할 수 있지만, BPI 애셋은 블루프린트 세계 안에서만 쓸 수 있습니다.

정의 방식BP에서 구현BP에서 호출(✉)C++에서 구현C++에서 호출
C++ 정의
UINTERFACE(Blueprintable)		✅ 가능✅ 가능✅ 가능✅ 가능
BPI 애셋 정의
Content Browser에서 생성		✅ 가능✅ 가능❌ 불가❌ 사실상 불가

BPI 애셋을 C++에서 못 쓰는 이유는 단순합니다: 애셋은 에디터에만 존재하므로 C++ 컴파일 시점에 참조할 헤더 자체가 없습니다. 반대 방향이 가능한 것은 리플렉션(코드 구조를 런타임에 조회하는 시스템) 덕분입니다.

🧭

"C++로 정의하면 블루프린트가 자동 생성되나요?" — 아니요. Content Browser에 애셋이 생기지 않습니다. UHT가 인터페이스를 리플렉션에 등록하면 에디터가 그 정보를 읽어 Class Settings 드롭다운과 노드 검색 목록에 비춰줄 뿐입니다 — C++ 클래스가 "Pick Parent Class" 다이얼로그에 나타나는 것과 같은 원리입니다. 블루프린트 쪽 사용법(Implement·✉ Message)은 두 통로가 완전히 동일해서, 작업자는 그 인터페이스가 애셋인지 C++ 타입인지 구분할 필요가 없습니다.

BPI 애셋의 강점 — 에디터 가시성

Content Browser에서 눈에 보이고, 더블클릭으로 시그니처를 확인하고, 우클릭 → Reference Viewer로 "누가 구현하나"를 그래프로 추적할 수 있습니다. 발견 경로가 에디터 안에서 완결됩니다.

C++ 정의의 강점 — 양쪽 세계 참여

C++ 클래스(터렛·시스템 코드)도 같은 약속에 참여합니다. 로직이 텍스트 코드라 버전 관리·검색·이식에도 유리합니다. 대신 이름을 모르면 에디터에서 발견하기 어렵습니다.

C++ 구현 가능diff·검색 용이C++ 혼용 팀
💡
C++ 정의의 발견성 보완 3가지 — ① UINTERFACE(Blueprintable, meta = (DisplayName = "상호작용 인터페이스", ToolTip = "E키 상호작용 대상이 구현하는 약속"))처럼 메타데이터를 달면 Class Settings 검색에서 한글 이름·설명이 보입니다. ② 대표 구현체 BP 하나(예: BP_Door)를 "살아있는 견본"으로 유지하세요. ③ 프로젝트에 어떤 인터페이스가 있는지는 팀 위키·문서가 알려주는 것이 C++ 중심 팀의 방식입니다.
📐

선택 기준은 "약속을 누가 소유·관리하는가" — 디자이너가 인터페이스를 직접 만들고 자주 고치는 팀이면 BPI 애셋(가시성·추적이 우선), 프로그래머가 약속을 관리하고 작업자는 구현·호출만 하면 C++ 정의(양쪽 참여가 우선)를 고르세요. C++가 조금이라도 있는 프로젝트라면 C++ 정의가 안전한 기본값입니다.

다중 인터페이스
Implemented Interfaces에 여러 개
한 클래스가 BPI_Interact + BPI_Damageable처럼 여러 약속에 동시에 서명할 수 있습니다. 상속과 달리 부모를 하나로 정할 필요가 없는 것이 인터페이스의 힘.
🌉
C++ 정의 + BP 구현
BlueprintNativeEvent
약속은 C++로 한 번 정의하고, 문은 BP로·터렛은 C++로 구현하는 혼용이 실무 표준입니다. C++ 인터페이스도 BP의 Class Settings에 똑같이 나타납니다.
📡
역할 분담
인터페이스 ↔ 디스패처
"시키기"는 인터페이스, "알리기"는 디스패처(델리게이트). 문이 열린 뒤 퀘스트 시스템에 알리는 건 문의 OnDoorOpened 디스패처가 맡는 식으로 조합합니다.

흔한 실수

✓ 이렇게

  • 인터페이스 이름은 BPI_ 접두어로 통일 (검색·구분)
  • 함수는 "대상이 하는 일" 기준으로 최소한만 (OnInteract 하나면 충분한 경우가 많음)
  • 메시지 호출 전 대상 유효성 확인 (IsValid)
  • 반응이 없으면 Does Implement Interface + Print String으로 구현 여부부터 확인
  • C++ 호출은 항상 Execute_ + Implements<U…>()

✗ 이건 피하기

  • 모든 통신을 인터페이스로 — 1:N 통지는 디스패처가 맞는 도구
  • 인터페이스에 "기본 동작"을 기대 — 로직이 필요하면 부모 클래스(상속)나 컴포넌트로
  • 시그니처 잦은 변경 — 구현 클래스 전부에 컴파일 경고·재작업이 번짐
  • 같은 개념에 인터페이스 여러 개 난립 (BPI_Use vs BPI_Interact 혼재)
  • ✉ 없는 동명 노드 호출 — 자기 이벤트를 부르는 실수
⚠️
함수 이름을 바꾸면 구현이 끊길 수 있습니다 — BPI의 함수 이름을 바꾸면 이미 구현해 둔 클래스들의 이벤트가 "고아"가 되어 조용히 호출되지 않게 될 수 있습니다. 이름 변경은 에디터의 리네임 기능으로 하고, 변경 후 구현 클래스를 모두 열어 컴파일 경고를 확인하세요.
💡
Message 노드가 검색에 안 나오면 — ① BPI 애셋을 컴파일·저장했는지, ② 검색창의 Context Sensitive를 잠시 꺼 보기, ③ 함수에 BlueprintCallable(C++ 정의 시)이 있는지 순서로 확인하세요.
