UE Architecture · Type ① · 02-A

GameInstance
Subsystem

게임이 켜질 때 자동으로 생성되고 꺼질 때 자동으로 정리되는 전역 서비스 객체를 만드는 방법입니다. 점수 관리·사운드 매니저처럼 "레벨이 바뀌어도 살아 있어야 하는 기능"을 싱글턴 없이 안전하게 만드는 언리얼의 표준 패턴을 다룹니다.

엔진 자체 기능 UE 4.22+ / UE5 C++ · Blueprint
01 — 개념 · 역할

왜 싱글턴을 직접 만들지 않아도 되는가

게임을 만들다 보면 "어디서든 접근할 수 있고, 게임이 실행되는 동안 계속 살아 있는 객체"가 필요해집니다. 점수 집계, 배경음 관리, 저장 요청 큐 같은 것들이죠. C++에서 흔히 쓰는 해법은 싱글턴(Singleton — 프로그램 전체에서 인스턴스를 하나만 두는 패턴)이지만, 직접 만들면 언제 생성하고 언제 파괴할지를 전부 수동으로 관리해야 하고, 에디터의 PIE(Play In Editor — 에디터 안에서 게임을 실행하는 기능)처럼 게임이 여러 번 시작/종료되는 환경에서 잔여 상태가 남는 버그가 생기기 쉽습니다.

GameInstanceSubsystem은 이 문제를 엔진이 대신 풀어줍니다. UGameInstanceSubsystem을 상속한 클래스를 만들어 두기만 하면, 엔진이 GameInstance가 만들어질 때 함께 생성하고, GameInstance가 사라질 때 함께 정리합니다. 등록 코드도, 파괴 코드도 쓸 필요가 없습니다.

🔑

핵심 용어 — Subsystem: 엔진이 생성·파괴 시점을 대신 관리해 주는 서비스 객체의 총칭. · GameInstance: 게임 프로세스가 시작될 때 1개 만들어져 레벨(맵)이 바뀌어도 유지되고, 게임이 끝날 때 사라지는 최상위 객체. · 수명(Lifetime): 객체가 생성되어 파괴될 때까지의 구간. Subsystem의 수명은 자신을 소유한 객체(여기선 GameInstance)의 수명과 같습니다. · 싱글턴(Singleton): 인스턴스를 하나만 두는 전통적 패턴 — Subsystem이 대체합니다.

Subsystem은 4가지 수명 스코프가 있습니다

부모 클래스를 무엇으로 하느냐에 따라 "언제 만들어지고 언제 사라지는지"가 달라집니다. 필요한 수명에 맞는 부모를 고르는 것이 Subsystem 설계의 전부라고 해도 과언이 아닙니다.

부모 클래스수명 (생성 ~ 파괴)이런 기능에 적합
UEngineSubsystem엔진 시작 ~ 엔진 종료에디터 포함 전체 프로세스 서비스
UGameInstanceSubsystem게임 시작 ~ 게임 종료 (레벨 전환 무관)점수·사운드·세이브 큐 등 게임 전역 서비스 — 이 문서의 주제
UWorldSubsystem월드(레벨) 로드 ~ 언로드레벨에 종속된 스폰 관리·연출
ULocalPlayerSubsystem로컬 플레이어 생성 ~ 제거분할 화면 등 플레이어별 상태
03 — 책임 분배

무엇을 여기에 두고, 무엇을 두지 말아야 하나

GameInstanceSubsystem이 편하다고 모든 로직을 몰아넣으면 "신(God) 클래스"가 됩니다. 아래 결정표를 기준으로 판단하세요 — 판단 기준은 그 데이터/기능이 필요한 수명입니다.

데이터 / 기능배치왜
세션 전역 상태 (누적 점수, 재화, 설정 캐시)UGameInstanceSubsystem레벨이 바뀌어도 유지되어야 하므로 GameInstance 수명이 필요
전역 서비스 (사운드 매니저, 알림 큐, 분석 이벤트 수집)UGameInstanceSubsystem어디서든 접근해야 하고 상태를 계속 들고 있어야 함
레벨(맵)에 종속된 로직 — 스폰 지점 관리, 레벨 연출UWorldSubsystem레벨이 내려가면 함께 정리되어야 함. GameInstance 수명이면 이전 레벨의 상태가 남는 버그 발생
디스크에 남아야 하는 데이터 (재실행 후에도 유지)USaveGameSubsystem은 메모리 객체일 뿐 — 게임을 끄면 사라짐. Subsystem이 SaveGame을 읽고 쓰는 중개자 역할을 맡기는 것이 정석
매치 규칙·승패 판정AGameMode매치(레벨) 단위 규칙은 프레임워크 클래스의 몫
💡

망설여질 때의 질문: "레벨을 전환해도 이 값이 살아 있어야 하는가?" — 예라면 GameInstanceSubsystem, 아니오라면 WorldSubsystem이나 액터 쪽이 맞습니다.

04 — 생명주기 흐름

엔진이 대신 해주는 일의 순서

우리가 클래스를 정의해 두면, 게임 실행 시 엔진이 아래 순서를 자동으로 진행합니다. 각 단계에서 무엇이 보장되고 무엇이 아직 없는지가 중요합니다.

1 · 게임 시작
GameInstance 생성
프로세스당 1회. 이 시점에 모든 GameInstanceSubsystem 클래스가 수집됩니다.
→
2 · 자동 생성
Initialize() 호출
Subsystem 인스턴스가 만들어지고 초기화 훅이 불립니다. 아직 월드(레벨)는 없을 수 있습니다.
→
3 · 플레이 중
레벨 전환에도 유지
OpenLevel로 맵을 몇 번을 바꿔도 같은 인스턴스가 계속 살아 있습니다.
→
4 · 게임 종료
Deinitialize() 호출
타이머·델리게이트 해제 등 정리 코드를 두는 곳. 이후 GC가 회수합니다.
⚠️
Initialize()에서 월드를 만지지 마세요 — Initialize 시점에는 아직 레벨이 로드되기 전일 수 있어 GetWorld()가 nullptr일 수 있습니다. 월드가 필요한 초기화는 첫 접근 시점으로 미루거나 월드 델리게이트(예: FWorldDelegates::OnPostWorldInitialization)에 걸어 두세요.
💡
다른 Subsystem에 의존한다면 — Subsystem끼리 초기화 순서는 보장되지 않습니다. Initialize(FSubsystemCollectionBase& Collection) 안에서 Collection.InitializeDependency<UOtherSubsystem>()을 먼저 호출하면 의존 대상이 먼저 초기화됨을 보장할 수 있습니다.
05 — 구현 스켈레톤

최소 동작 코드 — 점수 서비스 예시

🚀 이 문서로 작업 시작하기

🧭

전제 조건 — 단독 실습 가능합니다. 위저드·Build.cs·컴파일의 공통 절차는 위키 『C++ 클래스 생성 · 빌드 워크플로우』(core-cpp)를 따릅니다.

만들 파일 / 작업 대상부모 클래스 (위저드 검색어)비고
ScoreSubsystem.h/.cppGameInstanceSubsystem수명 자동 관리 전역 서비스
💡

코드 외 작업 — 없음 — Subsystem은 자동 등록됩니다(이 구조의 장점). 확인: BP 아무 그래프에서 Get ScoreSubsystem 노드가 검색되면 성공.

⚠️
컴파일 주의 — 새 클래스 생성은 재빌드, AddScore 등 본문 다듬기는 Live Coding.

전제: C++ 프로젝트, 모듈 이름이 MyGame이라고 가정합니다(MYGAME_API 부분을 자기 모듈 이름으로 바꾸세요). 아래 두 파일을 Source/MyGame/ 아래에 만들고 컴파일하면 그 외 등록 절차는 없습니다 — 엔진이 클래스를 자동 발견해 인스턴스화합니다.

#pragma once

#include "CoreMinimal.h"
#include "Subsystems/GameInstanceSubsystem.h"
#include "ScoreSubsystem.generated.h"  // 반드시 마지막 #include

// 게임 전체에서 점수를 관리하는 전역 서비스.
// UGameInstanceSubsystem 상속 = 게임 시작~종료 동안 자동으로 1개 유지.
// UCLASS(): 리플렉션 시스템 등록 — 엔진이 이 클래스를 자동 발견해 인스턴스화하는 전제 조건
UCLASS()
class MYGAME_API UScoreSubsystem : public UGameInstanceSubsystem   // MYGAME_API: 모듈(DLL) 외부에 클래스를 노출하는 매크로 — 자기 모듈 이름으로 교체
{
    GENERATED_BODY()   // 리플렉션용 보일러플레이트가 삽입되는 자리 — UCLASS()와 한 쌍

public:
    // 생성 직후 1회. 월드는 아직 없을 수 있음(04 섹션 참고).
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;

    // 게임 종료 직전 1회. 타이머·델리게이트 정리는 여기서.
    virtual void Deinitialize() override;

    // BP에서도 호출 가능하게 노출.
    UFUNCTION(BlueprintCallable, Category = "Score")
    void AddScore(int32 Amount);

    // BlueprintPure: 실행 핀 없는 '읽기 전용' 노드로 노출 — 상태를 바꾸지 않으므로 const와 짝을 이룸
    UFUNCTION(BlueprintPure, Category = "Score")
    int32 GetScore() const { return CurrentScore; }

private:   // 점수는 AddScore/GetScore를 통해서만 접근 — 외부의 임의 수정 차단
    // 지정자 없는 UPROPERTY()라도 리플렉션에 등록되어 값이 안전하게 관리됨 (UObject 포인터라면 GC 추적까지)
    UPROPERTY()
    int32 CurrentScore = 0;   // 레벨이 바뀌어도 유지되는 값
};
#include "ScoreSubsystem.h"

void UScoreSubsystem::Initialize(FSubsystemCollectionBase& Collection)
{
    Super::Initialize(Collection);   // 부모 호출을 잊지 마세요
    CurrentScore = 0;   // 헤더의 기본값과 중복이지만, 초기화 시점을 명시해 의도를 드러냄
    UE_LOG(LogTemp, Log, TEXT("ScoreSubsystem ready"));   // 에디터 Output Log에서 생성 시점을 눈으로 확인하는 용도
}

void UScoreSubsystem::Deinitialize()
{
    // 타이머·델리게이트를 여기서 해제 (예: GetGameInstance()->GetTimerManager().ClearAllTimersForObject(this))
    Super::Deinitialize();
}

void UScoreSubsystem::AddScore(int32 Amount)
{
    // Subsystem이 GameInstance 수명을 따르므로 레벨이 바뀌어도 누적값이 유지됨
    CurrentScore += Amount;   // 음수 방지·상한선 등 검증 로직을 넣는다면 이 지점이 확장 위치
}

어디서든 접근하는 방법

C++
GetSubsystem<T>()
UGameInstance* GI = GetGameInstance(); (액터라면 GetWorld()->GetGameInstance()) → UScoreSubsystem* Score = GI->GetSubsystem<UScoreSubsystem>(); — 반환값 null 체크 후 사용.
BP
자동 노출 노드
블루프린트 그래프에서 우클릭 → "Score Subsystem" 검색 → Get ScoreSubsystem 노드가 자동으로 생겨 있습니다. 별도 노출 작업이 필요 없습니다 — 결과: 노드 출력 핀에서 AddScore/GetScore 호출 가능.

⚠️
영속 저장과 혼동 — Subsystem의 값은 게임을 끄면 사라집니다. "재실행 후에도 남아야 한다"면 반드시 USaveGame과 연동하세요(03 섹션 결정표).
⚠️
WorldSubsystem과 혼동 — 레벨 종속 상태를 GameInstanceSubsystem에 두면 이전 레벨의 값이 다음 레벨로 새어 들어갑니다. 수명 기준(02 섹션 표)으로 부모를 고르세요.
💡
추상 부모로 생성 제외 — 공통 로직만 담은 중간 부모 클래스는 UCLASS(Abstract)로 표시하면 엔진이 인스턴스를 만들지 않습니다. 조건부 생성이 필요하면 ShouldCreateSubsystem()을 오버라이드하세요.
