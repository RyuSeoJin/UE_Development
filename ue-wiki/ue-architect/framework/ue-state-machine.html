상태 머신
State Machine

"지금 어떤 상태인가"와 "어떤 조건에서 다른 상태로 넘어가는가"를 명시적으로 정의해 동작을 제어하는 패턴입니다. if문 덩어리로 뒤엉키기 쉬운 게임 로직(타이머, 캐릭터 행동, UI 모드)을 상태·전이·이벤트 세 가지 개념으로 정리하는 방법을 다룹니다.

설계 패턴 엔진 버전 무관 C++ · Blueprint · AnimBP
01 — 개념 · 역할

신호등은 왜 헷갈리지 않는가

신호등은 빨강·노랑·초록 중 항상 정확히 하나의 상태에만 있고, "빨강 → 초록"처럼 정해진 경로로만 바뀝니다. 그래서 복잡한 교차로에서도 동작을 예측할 수 있습니다. 상태 머신은 이 아이디어를 코드에 옮긴 것입니다.

상태 머신 없이 기능을 키우다 보면 이런 코드가 됩니다: if (bIsRunning && !bIsPaused && RemainingTime > 0 && !bFinished) { ... }. bool 변수가 4개면 조합은 16가지인데, 그중 실제로 의미 있는 조합은 몇 개뿐이고 나머지는 "있어서는 안 되는 상태"입니다. 버그는 대부분 그 틈에서 나옵니다. 상태 머신(State Machine)은 의미 있는 상태만 열거형으로 선언하고, 상태 사이의 이동(전이)을 함수 한 곳으로 모아 불가능한 상태 조합 자체를 없애 버리는 설계입니다.

🔑

핵심 용어 — 상태(State): 객체가 놓일 수 있는 명확한 국면 하나 (예: Idle, Running). 항상 정확히 하나만 활성. · 전이(Transition): 한 상태에서 다른 상태로의 이동. 허용된 경로만 존재. · 이벤트(Event): 전이를 일으키는 계기 (버튼 입력, 시간 만료). · 가드 조건(Guard): 이벤트가 와도 전이를 허용할지 검사하는 조건문. · Enter/Exit 처리: 상태에 들어갈 때/나갈 때 1회 실행하는 부수 작업.

03 — 구현 방식 선택

세 가지 구현 방식과 선택 기준

enum 기반 (가벼움)

상태 4~7개추천 기본값

UENUM으로 상태를 선언하고 switch와 전이 함수 하나로 관리. 코드가 한눈에 들어오고 디버깅이 쉽습니다. 대부분의 게임플레이 기능(타이머·문 열림·간단한 AI)은 이걸로 충분합니다.

클래스 기반 (State 패턴)

상태 8개+상태별 로직 큼

상태 하나를 클래스 하나로 만들어 Enter/Tick/Exit을 가상 함수로 분리. 상태별 코드가 길어질 때 switch 비대화를 막지만, 파일 수와 간접 참조가 늘어나는 비용이 있습니다.

상황방식왜
타이머·UI 모드·간단한 기믹 (상태 4~7개)enum 기반가장 적은 코드로 불가능한 상태를 제거. 이 문서의 스켈레톤(05)
보스 AI처럼 상태별 로직이 수십 줄 이상클래스 기반 (State 패턴)상태별 파일 분리로 유지보수 확보. enum으로 시작했다가 승격해도 늦지 않음
캐릭터 애니메이션 (걷기·점프·착지 블렌딩)AnimBP 스테이트 머신애니메이션 블루프린트에 시각적 상태 머신 에디터가 내장되어 있음 — 애니메이션 전환은 코드로 재발명하지 말 것
행동 "선택"이 핵심인 AI (우선순위·조건 판단)Behavior Tree상태 머신은 "지금 무엇인가", BT는 "다음에 무엇을 할까"에 특화 — 둘은 경쟁이 아니라 용도가 다름
04 — 전이 처리 흐름

이벤트 하나가 처리되는 5단계

어떤 구현 방식이든 전이 처리의 뼈대는 같습니다. 이 순서를 SetState 함수 한 곳에 고정해 두면, 상태 변경 버그를 추적할 때 볼 곳이 한 군데로 줄어듭니다.

01
이벤트 수신
입력·타이머 만료·외부 호출이 전이 요청 함수를 부릅니다. (예: RequestStart())
02
가드 조건 검사
현재 상태에서 허용된 전이인지 확인. 전이도(02 섹션)에 없는 경로면 아무 것도 하지 않고 반환 — 조용한 무시가 정답입니다.
03
Exit 처리
이전 상태에서 나가며 정리할 것 (예: Running에서 나갈 때 타이머 해제).
04
상태 갱신 + Enter 처리
상태 변수를 새 값으로 바꾸고, 새 상태에 들어가며 할 일 실행 (예: Running 진입 시 타이머 시작).
05
변경 알림
델리게이트로 "상태가 바뀌었다"를 방송 — UI 등 관심 있는 쪽이 구독. 상태 머신이 UI를 직접 알 필요가 없어집니다.
⚠️
전이 중 재진입(re-entrancy) — Enter 처리나 변경 알림 안에서 또 SetState를 부르면 반쯤 바뀐 상태 위에 전이가 겹칩니다. Enter/알림에서는 전이를 직접 부르지 말고, 필요하면 "다음 프레임에 전이 예약" 방식으로 미루세요.
05 — 구현 스켈레톤

enum 기반 최소 구현

🚀 이 문서로 작업 시작하기

🧭

전제 조건 — 상태 머신을 담을 소유자(Subsystem·액터)가 있는 프로젝트에 적용합니다. 위저드·Build.cs·컴파일의 공통 절차는 위키 『C++ 클래스 생성 · 빌드 워크플로우』(core-cpp)를 따릅니다.

만들 파일 / 작업 대상부모 클래스 (위저드 검색어)비고
TimerMachine.h/.cppObject (All Classes ▸ Object — Subsystem 아님 주의)상태 UENUM도 같은 헤더에
💡

코드 외 작업 — 없음 — 자동 등록되지 않는 일반 UObject이므로 소유자가 NewObject로 생성해 UPROPERTY 멤버로 보관합니다(본문 참조 — GC 추적을 위해 UPROPERTY 필수).

⚠️
컴파일 주의 — UENUM·헤더 작업은 재빌드, 전이 로직 다듬기는 Live Coding.

전제: C++ 클래스(액터·컴포넌트·Subsystem 어디든) 안에 넣을 수 있는 형태입니다. 모듈 이름 MYGAME_API는 자기 프로젝트로 바꾸세요. 핵심은 상태 변수를 private으로 숨기고, 변경은 SetState 한 곳으로만 하는 것입니다.

#pragma once

#include "CoreMinimal.h"
#include "TimerMachine.generated.h"  // 반드시 마지막 #include

// 상태: 의미 있는 국면만 열거. bool 조합 대신 이것 하나가 진실.
// UENUM(BlueprintType): 블루프린트 변수·Switch 노드에서 사용 가능하게 노출 — uint8 기반 지정이 필수 조건
UENUM(BlueprintType)
enum class ETimerState : uint8
{
    Idle,       // 대기
    Running,    // 진행 중
    Paused,     // 일시정지
    Finished    // 종료
};

// 상태 변경 방송: UI 등이 구독 (State, OldState)
// TwoParams = 인자 2개짜리 동적 멀티캐스트 델리게이트 — (타입, 이름) 쌍을 인자 수만큼 나열하는 형식
DECLARE_DYNAMIC_MULTICAST_DELEGATE_TwoParams(FOnTimerStateChanged, ETimerState, NewState, ETimerState, OldState);

// UCLASS(): 리플렉션 등록 — GC 추적·블루프린트 노출의 전제 조건
UCLASS()
class MYGAME_API UTimerMachine : public UObject   // UObject 상속 — 월드 배치·Tick이 필요 없는 순수 로직이라 액터보다 가벼움
{
    GENERATED_BODY()   // 리플렉션 보일러플레이트 삽입 자리 — UCLASS()와 한 쌍

public:
    // 이벤트 = 전이 요청. 허용 여부는 내부 가드가 판단.
    UFUNCTION(BlueprintCallable) void RequestStart();
    UFUNCTION(BlueprintCallable) void RequestPause();
    UFUNCTION(BlueprintCallable) void RequestFinish();

    // BlueprintPure: 실행 핀 없는 읽기 전용 노드 — 상태 조회는 언제든 안전
    UFUNCTION(BlueprintPure)
    ETimerState GetState() const { return State; }

    // BlueprintAssignable: BP에서 이 이벤트에 바인딩 가능 — 상태 변화를 구독하는 공식 통로
    UPROPERTY(BlueprintAssignable)
    FOnTimerStateChanged OnStateChanged;

private:
    // 상태 변경의 유일한 통로. 외부에서 State를 직접 못 만짐.
    void SetState(ETimerState NewState);

    UPROPERTY()   // 지정자 없는 UPROPERTY라도 리플렉션에 등록 — 에디터 디버깅·직렬화 대상이 됨
    ETimerState State = ETimerState::Idle;   // 초기 상태를 Idle로 명시 — 초기화 누락 시 쓰레기 값 상태로 시작하는 사고 방지
};
#include "TimerMachine.h"

void UTimerMachine::RequestStart()
{
    // 가드: Idle에서만 시작 허용 (전이도에 있는 경로만)
    if (State == ETimerState::Idle)
    {
        SetState(ETimerState::Running);
    }
}

void UTimerMachine::RequestPause()
{
    // Running ⇄ Paused 토글
    if (State == ETimerState::Running)      { SetState(ETimerState::Paused); }
    else if (State == ETimerState::Paused)  { SetState(ETimerState::Running); }
}

void UTimerMachine::RequestFinish()
{
    // 가드: Running·Paused 어느 쪽에서든 종료 가능 — Idle/Finished에서 온 요청은 조용히 무시
    if (State == ETimerState::Running || State == ETimerState::Paused)
    {
        SetState(ETimerState::Finished);
    }
}

void UTimerMachine::SetState(ETimerState NewState)
{
    if (State == NewState) { return; }   // 같은 상태로의 전이는 무시 — 불필요한 Exit/Enter·중복 방송 방지

    const ETimerState OldState = State;   // 방송에 실어 보낼 이전 상태를 갱신 전에 백업

    // (03) Exit 처리: 이전 상태 정리 — 예: Running에서 나가면 틱 타이머 해제

    State = NewState;                     // (04) 상태 갱신

    // (04) Enter 처리: 새 상태 진입 작업 — 예: Running 진입 시 틱 타이머 시작

    OnStateChanged.Broadcast(State, OldState);   // (05) 변경 알림
}
🧭

주석의 (03)(04)(05)는 04 섹션 전이 처리 5단계와 같은 번호입니다. Exit/Enter에 실제 작업(타이머 시작·해제 등)을 채우는 것이 이 뼈대를 자기 기능으로 완성하는 방법입니다.

06 — 확장 & 주의

더 큰 상태 머신으로 갈 때

🎬
AnimBP 전환
Animation State Machine
애니메이션 전환은 AnimBP 내장 상태 머신 에디터로. 게임플레이 상태 머신이 "의미"를, AnimBP가 "겉모습"을 담당.
🌲
계층 상태 머신
HFSM
상태 안에 하위 상태 머신을 중첩 (전투 상태 안에 근접/원거리). 상태 폭발을 구조로 눌러 담는 기법.
🧠
행동 선택은 BT로
Behavior Tree
"다음 행동 고르기"가 복잡해지면 상태 머신을 늘리기보다 Behavior Tree와 역할 분담.
⚠️
상태 폭발 — "PausedButAlmostFinished" 같은 조합 상태가 생기기 시작하면 신호입니다. 상태는 국면만 담고, 세부 수치(남은 시간 등)는 일반 변수로 두세요. 상태 수가 8개를 넘으면 계층화(HFSM)나 클래스 기반 전환을 검토하세요.
⚠️
Tick에서 직접 전이 판정 남발 — 매 프레임 조건을 검사해 SetState를 부르면 한 프레임에 여러 번 전이될 수 있습니다. 전이는 이벤트(입력·타이머 만료·델리게이트)에서 요청하고, Tick은 현재 상태의 "진행"만 담당하게 하세요.
💡
디버깅 팁 — SetState 안에 UE_LOG로 OldState → NewState를 남기면 상태 흐름 전체가 로그 한 줄씩으로 보입니다. 변경 통로가 한 곳이라 가능한 이점입니다.
