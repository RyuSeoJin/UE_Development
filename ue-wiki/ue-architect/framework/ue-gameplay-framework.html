01 — 개념 이해

게임플레이 프레임워크 — 왜 이렇게 많은 클래스로 나뉘었는가

게임플레이 프레임워크(Gameplay Framework)는 언리얼 엔진이 게임의 핵심 시스템(규칙·상태·입력·물리적 표현)을 관리하기 위해 제공하는 모듈식 클래스 모음입니다. 하나의 거대한 "Game" 클래스를 만드는 대신, 역할별로 클래스를 쪼개서 각각 독립적으로 교체·확장할 수 있게 설계되어 있습니다.

이 설계를 이해하려면 먼저 3가지 핵심 설계 철학을 잡아야 합니다. 이 철학이 "왜 이 클래스가 존재하는가"에 대한 답입니다.

철학 1 — 의도와 몸의 분리 (Controller vs Pawn)

"무엇을 할지 결정하는 것"과 "물리적으로 실행하는 것"을 별도 클래스로 나눕니다. Controller(컨트롤러)는 의도(입력·AI 판단)를 담당하고, Pawn(폰)은 월드에서의 물리적 존재(이동·충돌·메시)를 담당합니다. 컨트롤러가 폰을 Possess(빙의)하면 조종이 시작됩니다.

이 분리 덕분에: 캐릭터가 죽어도 Controller는 살아있어 리스폰이 자연스럽고, 같은 Controller가 다른 Pawn을 Possess하면 차량 탑승·빙의 전환이 가능합니다.

철학 2 — 규칙과 상태의 분리 (GameMode vs GameState)

"게임이 어떻게 진행되는가"(규칙)와 "지금 어떤 상태인가"(점수·진행도)를 나눕니다. GameMode는 규칙을 정의하고 서버에만 존재합니다(클라이언트가 규칙을 조작할 수 없게). GameState는 현재 상태를 담고 모든 클라이언트에 복제됩니다(모두가 점수를 알아야 하므로).

철학 3 — 수명 범위별 데이터 배치

데이터가 얼마나 오래 살아야 하는가에 따라 둘 클래스가 달라집니다. 앱 전체 수명이 필요한 데이터(설정, 세이브)는 GameInstance에, 레벨 단위는 GameMode/GameState에, 플레이어 단위는 PlayerState/Controller에 둡니다.

📖

핵심 용어 블록
Possess(빙의) — 컨트롤러가 폰을 소유·조종하는 관계. Controller->Possess(Pawn)으로 시작
Replication(복제) — 서버의 데이터 변경을 모든 클라이언트에 자동 전송하는 네트워크 동기화

BeginPlay — 액터가 월드에 완전히 준비된 후 최초 1회 호출되는 함수

03 — 책임

클래스별 깊은 설계 분석

각 클래스의 역할, 주요 멤버·함수, "여기에 두어야 할 것"과 "두면 안 되는 것"을 분석합니다.

UGameInstance — 세션 전역 영속 객체

UGameInstance는 엔진이 시작될 때 생성되어 종료될 때까지 유지되는 유일한 클래스입니다. 레벨이 바뀌어도 파괴되지 않습니다. AActor가 아닙니다(UObject를 상속) — 월드에 배치되지 않고, Tick도 없습니다.

구분내용
주요 함수Init() — 최초 초기화, OnStart() — 게임 시작 시, Shutdown() — 종료 시
여기에 두어야 할 것세이브/로드 관리자, 유저 설정(볼륨·해상도), 레벨 간 전달할 데이터, Subsystem 등록
두면 안 되는 것레벨별 점수, 현재 매치 규칙, 입력 처리 — 이것들은 수명이 더 짧은 클래스의 몫
💡
GameInstanceSubsystem을 활용하면 GameInstance 수명에 묶인 서비스를 모듈식으로 분리할 수 있습니다. GameInstance 자체를 비대하게 만드는 대신, 기능별 Subsystem을 만드는 것이 권장됩니다.

AGameModeBase — 게임 규칙과 권한 (서버 전용)

AGameModeBase는 게임의 규칙·구조·진행 방식을 정의합니다. "이 매치에서 몇 명이 플레이하는가", "어떤 Pawn을 스폰하는가", "승리 조건은 무엇인가"를 결정합니다. 서버에만 존재하며, 클라이언트에서 GetWorld()->GetAuthGameMode()를 호출하면 nullptr이 반환됩니다.

구분내용
주요 프로퍼티DefaultPawnClass — 스폰할 폰 클래스, PlayerControllerClass — 사용할 컨트롤러 클래스, GameStateClass — 사용할 게임스테이트 클래스
주요 함수InitGame() — 게임 초기화, PreLogin() — 접속 검증, PostLogin() — 접속 후 처리, HandleStartingNewPlayer() — 신규 플레이어 셋업, SpawnDefaultPawnFor() — 폰 스폰, RestartPlayer() — 리스폰
여기에 두어야 할 것매치 규칙(시간 제한·인원), 플레이어 접속/퇴장 처리, 스폰 위치 결정, 승패 판정 로직, 게임 단계 전환(대기→시작→종료)
두면 안 되는 것점수 데이터(→ GameState), 입력 처리(→ Controller), 시각 효과(→ Pawn/HUD)
⚠️
GameMode는 클라이언트에서 접근할 수 없습니다. 클라이언트가 알아야 하는 정보(남은 시간, 점수)는 반드시 GameState에 복제된 변수로 넣어야 합니다. GameMode에 넣으면 클라이언트 UI에서 표시할 수 없습니다.
🧭

AGameModeBase vs AGameMode: AGameModeBase는 경량 베이스 클래스이고, AGameMode는 그 위에 매치 상태 머신(Waiting → Playing → Ended)을 추가한 확장 클래스입니다. 멀티플레이어 대전 게임이 아니라면 AGameModeBase로 충분합니다.

AGameStateBase — 모든 클라이언트가 공유하는 전체 상태

AGameStateBase는 모든 플레이어가 알아야 하는 게임 전체 정보를 담습니다. 팀 점수, 남은 시간, 현재 라운드, 접속 중인 플레이어 목록 등입니다. GameMode가 자동으로 생성하고, 모든 클라이언트에 복제됩니다.

구분내용
주요 프로퍼티PlayerArray — 현재 접속된 모든 PlayerState 배열, AuthorityGameMode — 서버에서만 유효한 GameMode 포인터
주요 함수GetServerWorldTimeSeconds() — 서버 기준 게임 시간
여기에 두어야 할 것팀 점수, 매치 타이머, 현재 라운드/웨이브 번호, 목표 달성 상태 — "팀 단위"이거나 "게임 전체"인 데이터
두면 안 되는 것개별 플레이어 체력·탄약(→ PlayerState), 규칙 로직(→ GameMode), 세이브 데이터(→ GameInstance)
💡
GameState는 논리적 액터입니다. 월드에 스폰되지만 눈에 보이는 메시나 충돌이 없습니다. 순수하게 데이터를 담고 복제하는 역할만 합니다.

APlayerState — 개별 플레이어의 공유 데이터

APlayerState는 특정 플레이어 한 명의 데이터 중 다른 플레이어도 알아야 하는 정보를 담습니다. 이름, 점수, 팀 번호 등입니다. 플레이어가 접속하면 GameMode가 자동 생성하고, 모든 클라이언트에 복제됩니다.

구분내용
주요 프로퍼티PlayerName — 플레이어 이름 (FString), PlayerId — 고유 ID, Score — 점수 (float, 기본 제공), bIsABot — AI 여부
여기에 두어야 할 것킬/데스 수, 팀 번호, 플레이어 레벨, 선택한 캐릭터 종류 — "이 플레이어에 대해 다른 플레이어도 볼 수 있어야 하는" 데이터
두면 안 되는 것입력 매핑(→ Controller), 나만 아는 인벤토리 상세(→ Controller 또는 Pawn), 월드 전체 상태(→ GameState)
🧭

PlayerState vs PlayerController 구분 기준: "다른 플레이어도 이 정보를 알아야 하는가?" 예(이름·점수·팀) → PlayerState(전원 복제). 아니오(내 입력·UI 상태) → PlayerController(소유자만 복제).

APlayerController — 인간 플레이어의 의도 처리

APlayerController는 인간 플레이어의 입력을 받아 의도를 Pawn에 전달하는 클래스입니다. 키보드·마우스·게임패드 입력을 읽고, HUD·카메라·UI를 관리합니다. 물리적 표현(메시·충돌)이 없고, 소유 클라이언트에만 복제됩니다.

구분내용
주요 함수SetupInputComponent() — 입력 바인딩, Possess(APawn*) — 폰 빙의, UnPossess() — 빙의 해제, GetPawn() / GetCharacter() — 현재 조종 중인 폰 가져오기, SetViewTarget() — 카메라 전환
소유하는 것PlayerInput — 입력 시스템, PlayerCameraManager — 카메라 블렌딩, MyHUD — HUD 인스턴스, PlayerState — 플레이어 상태
여기에 두어야 할 것입력 처리, UI/메뉴 전환, 카메라 설정, 채팅 입력, 서버 RPC 전송 — "이 플레이어의 의도와 표시"
두면 안 되는 것이동 로직(→ Pawn/MovementComponent), 게임 규칙(→ GameMode), 다른 플레이어가 볼 데이터(→ PlayerState)
⚠️
Pawn이 죽어도 PlayerController는 살아있습니다. 리스폰 대기 화면, 관전 모드 전환 등은 Controller에서 처리합니다. Pawn의 Destroyed()에 게임 종료 로직을 넣으면, Controller가 새 Pawn을 받을 기회를 잃습니다.

AAIController — AI의 행동 결정

AAIController는 PlayerController의 AI 대응물입니다. 인간 입력 대신 비헤이비어 트리(Behavior Tree), 스테이트 트리(State Tree), EQS(Environment Query System) 등을 사용해 판단합니다. 네비게이션 시스템과 직접 연동되어 AI 폰의 이동 경로를 계산합니다.

구분내용
주요 함수RunBehaviorTree() — 비헤이비어 트리 실행, MoveToLocation() / MoveToActor() — 네비게이션 이동, GetPerceptionComponent() — AI 감각(시각·청각)
여기에 두어야 할 것AI 판단 로직, 네비게이션 요청, 감각 시스템 설정, 비헤이비어 트리 참조
두면 안 되는 것이동 물리(→ MovementComponent), 애니메이션(→ Pawn/Mesh), 체력(→ Pawn 또는 PlayerState)

APawn — 월드의 물리적 표현

APawn은 월드에서 물리적으로 존재하는 몸입니다. 컨트롤러가 Possess하면 조종이 시작됩니다. AActor를 상속하므로 Transform(위치·회전·스케일), Tick, 컴포넌트를 가집니다. 가장 기본적인 "조종 가능한 것"이며, 이동 방식은 자유입니다.

구분내용
주요 함수SetupPlayerInputComponent() — 입력 바인딩 (Controller가 호출), GetController() — 현재 조종 중인 컨트롤러, AddMovementInput() — 이동 입력 전달
기본 컴포넌트충돌(RootComponent), 이동(MovementComponent — 선택적)
파생 클래스ADefaultPawn — 에디터 기본 폰 (구체 메시 + 이동), ASpectatorPawn — 관전용 (충돌 없이 자유 비행), ACharacter — 보행 전용 (아래 참조)
여기에 두어야 할 것체력(싱글플레이), 충돌 설정, 물리 반응, 시각 효과(메시·파티클), 이동 능력
두면 안 되는 것입력 매핑(→ Controller), 게임 규칙(→ GameMode), 세이브 데이터(→ GameInstance)

ACharacter — 보행·점프·수영을 내장한 폰

ACharacter는 Pawn의 특화 서브클래스로, 인간형 캐릭터에 필요한 이족 보행 이동 시스템을 내장합니다. 대부분의 게임에서 플레이어 캐릭터는 이 클래스를 상속합니다.

내장 컴포넌트역할
UCharacterMovementComponent핵심. 걷기·달리기·점프·낙하·수영·비행·사용자 정의 이동 모드. 네트워크 이동 예측(클라이언트 측 예측 + 서버 보정)까지 내장
USkeletalMeshComponent애니메이션이 적용되는 메시. 캐릭터의 시각적 몸
UCapsuleComponent충돌 판정용 캡슐. 이동과 물리의 기준점
⚠️
Character를 쓸 필요가 없는 경우도 있습니다. 탑다운 전략 게임(클릭으로 유닛 이동), 차량 게임, 우주 비행 게임처럼 이족 보행이 필요 없으면 APawn을 직접 상속하고 자체 MovementComponent를 만드는 것이 더 가볍고 유연합니다.

AHUD & Camera — 화면 표시와 시점

AHUD는 화면에 오버레이되는 정보(체력바·미니맵·크로스헤어)를 그리는 클래스입니다. 인간 플레이어당 1개씩 PlayerController가 소유합니다. 현재는 UMG(Unreal Motion Graphics) 위젯이 HUD 역할을 대체하는 추세이지만, DrawHUD()를 통한 캔버스 직접 그리기는 디버그 표시에 여전히 유용합니다.

APlayerCameraManager는 카메라 전환·블렌딩을 관리합니다. SetViewTarget()으로 시점을 바꾸면 이 매니저가 부드러운 전환을 처리합니다.

UActorComponent — 컴포지션 패턴의 핵심

UActorComponent는 액터에 기능을 부여하는 모듈입니다. 상속 대신 컴포지션(조합)으로 기능을 조립합니다. "이 액터는 이동도 하고, 소리도 내고, 파티클도 뿌린다" → 각각 MovementComponent, AudioComponent, ParticleComponent를 붙이면 됩니다.

종류예역할
UActorComponent게임 로직 컴포넌트Transform 없음. 순수 로직(인벤토리·상태 머신)
USceneComponentSpringArm, ArrowTransform 있음. 부모-자식 부착 가능
UPrimitiveComponentStaticMesh, SkeletalMesh렌더링 + 충돌 + 물리
💡
설계 원칙: 상속보다 컴포지션을 선호하세요. "비행하는 캐릭터"를 만들 때 AFlyingCharacter를 새로 만드는 대신, 기존 Character에 UFlyingMovementComponent를 붙이는 것이 재사용성과 유연성이 높습니다.
04 — 흐름

게임 시작부터 플레이까지 — 초기화 파이프라인

"플레이어가 게임에 접속하면 무슨 일이 벌어지는가"를 시간순으로 추적합니다. 각 단계에서 어떤 클래스가 생성되고, 누가 누구를 호출하는지 확인하세요.

1
UGameInstance::Init()
엔진 시작 시 최초 1회. GameInstance가 생성되고 초기화됩니다. 세이브 로드, 전역 서비스 초기화 등을 여기서 수행합니다. GameInstanceSubsystem들도 이 시점에 자동 생성됩니다.
2
AGameModeBase::InitGame()
레벨 로드 직후. 엔진이 월드에 GameMode를 스폰합니다. GameMode는 자신의 GameStateClass로 GameState를 자동 생성합니다. 이 시점에서 아직 플레이어는 없습니다.
3
AGameModeBase::PreLogin()
플레이어 접속 요청 시. 서버가 이 플레이어를 받아들일지 검증합니다(서버 풀? 밴? 버전 불일치?). 거부하면 에러 문자열을 채워서 반환합니다. 싱글플레이에서는 호출되지 않습니다.
4
AGameModeBase::Login() → PlayerController 생성
접속이 승인되면 PlayerControllerClass에 지정된 클래스로 PlayerController를 스폰합니다. PlayerController는 자동으로 PlayerState를 생성합니다.
5
AGameModeBase::PostLogin()
접속 완료 후 처리. 여기서 HandleStartingNewPlayer()가 호출되어 PlayerStart 위치를 찾고, SpawnDefaultPawnFor()로 Pawn을 스폰합니다. 환영 메시지, 팀 배정 등 초기 셋업을 오버라이드할 수 있는 핵심 시점입니다.
6
Controller::Possess(Pawn)
PlayerController가 스폰된 Pawn을 Possess(빙의)합니다. 이 순간부터 입력이 Pawn에 전달되고, 카메라가 Pawn 시점으로 전환됩니다. Pawn의 PossessedBy()와 Controller의 OnPossess()가 호출됩니다.
7
AActor::BeginPlay()
모든 액터가 준비된 후. 월드의 모든 액터에 대해 BeginPlay()(앞서 핵심 용어 블록에서 설명)가 호출됩니다. 여기서부터 게임 로직이 정상 실행됩니다. Tick()도 이 이후에 시작됩니다.

시각적 흐름 요약

Phase 1
엔진 시작
GameInstance 생성
→
Phase 2
레벨 로드
GameMode + GameState 생성
→
Phase 3
플레이어 접속
Controller + PlayerState + Pawn + Possess
⚠️
생성자에서 게임 로직을 실행하지 마세요. 생성자는 CDO 생성 시에도 호출되므로, 월드에 대한 가정(다른 액터 존재, 타이머 설정 등)이 통하지 않습니다. 게임 로직은 반드시 BeginPlay() 이후에 시작하세요.
05 — 스켈레톤

커스텀 프레임워크 최소 구성 — 4파일 세트

자기만의 게임 뼈대를 만들려면 최소 4개 클래스를 커스텀해야 합니다: GameMode, GameState, PlayerController, Character. 아래는 복사해서 바로 사용할 수 있는 최소 코드입니다.

1. 커스텀 GameMode

#pragma once
#include "CoreMinimal.h"
#include "GameFramework/GameModeBase.h"
#include "MyGameMode.generated.h"   // UHT(언리얼 헤더 툴)가 자동 생성하는 리플렉션 코드 — 반드시 include 목록의 맨 마지막이어야 컴파일됨

// UCLASS(): 이 클래스를 언리얼 리플렉션 시스템에 등록 — 에디터·블루프린트·GC(가비지 컬렉터)가 인식하는 전제 조건
UCLASS()
class AMyGameMode : public AGameModeBase   // AGameModeBase 상속 → 서버에만 존재하는 규칙 클래스가 됨
{
    GENERATED_BODY()   // 리플렉션용 보일러플레이트가 이 자리에 삽입됨 — UCLASS()와 항상 한 쌍

public:
    // 생성자 — CDO(클래스 기본값 템플릿) 생성 시에도 호출되므로 '기본값 지정'만 하고 게임 로직은 금지
    AMyGameMode();

    // 플레이어 접속 완료 시 호출 — 팀 배정, 환영 처리 등
    virtual void PostLogin(APlayerController* NewPlayer) override;
};
#include "MyGameMode.h"
#include "MyGameState.h"
#include "MyPlayerController.h"
#include "MyCharacter.h"

AMyGameMode::AMyGameMode()
{
    // 이 게임에서 사용할 클래스를 지정
    // StaticClass(): 해당 클래스의 UClass(타입 정보)를 반환 — 엔진은 이 정보로 실제 인스턴스를 스폰함
    GameStateClass       = AMyGameState::StaticClass();   // 레벨 로드 시 GameMode가 이 클래스로 GameState를 자동 생성
    PlayerControllerClass = AMyPlayerController::StaticClass();   // 플레이어 접속(Login) 시 이 클래스로 컨트롤러 스폰
    DefaultPawnClass     = AMyCharacter::StaticClass();   // RestartPlayer 시 이 클래스로 폰(캐릭터) 스폰
}

// PostLogin: 접속이 완전히 승인된 직후 서버에서 호출 — 이 시점부터 PlayerState 접근이 안전함
void AMyGameMode::PostLogin(APlayerController* NewPlayer)
{
    Super::PostLogin(NewPlayer);   // 부모 처리 필수 — 기본 폰 스폰·시작 처리가 여기서 진행됨. 생략하면 폰이 스폰되지 않음

    // 여기에 팀 배정, 환영 메시지 등 초기 셋업
    // GetPlayerName()은 FString 반환 — 서식 문자열 %s에 넘기려면 앞에 *(TCHAR* 변환)가 필수
    UE_LOG(LogTemp, Log, TEXT("Player joined: %s"),
        *NewPlayer->GetPlayerState<APlayerState>()->GetPlayerName());
}

2. 커스텀 GameState

#pragma once
#include "CoreMinimal.h"
#include "GameFramework/GameStateBase.h"
#include "MyGameState.generated.h"

UCLASS()
class AMyGameState : public AGameStateBase
{
    GENERATED_BODY()

public:
    // 모든 클라이언트에 복제되는 팀 점수
    // Replicated: 서버에서 값이 바뀌면 엔진이 자동으로 전 클라이언트에 전송
    // BlueprintReadOnly: 블루프린트에서는 읽기만 허용 — 클라이언트 측 임의 수정(치팅) 차단
    // UPROPERTY 자체는 이 변수를 리플렉션·GC 추적 대상으로 만듦
    UPROPERTY(Replicated, BlueprintReadOnly, Category = "Score")
    int32 TeamScore;

    // 복제할 프로퍼티 목록을 엔진에 알려주는 함수
    // Replicated 지정자를 하나라도 쓰면 반드시 override — 구현이 없으면 링크 에러 발생
    virtual void GetLifetimeReplicatedProps(
        TArray<FLifetimeProperty>& OutLifetimeProps) const override;

    // 서버에서 호출: 점수 추가
    // UFUNCTION(BlueprintCallable): 블루프린트 그래프에서 노드로 호출할 수 있게 노출
    UFUNCTION(BlueprintCallable, Category = "Score")
    void AddTeamScore(int32 Amount);
};
#include "MyGameState.h"
#include "Net/UnrealNetwork.h"   // DOREPLIFETIME 매크로가 정의된 헤더 — 복제 구현부(.cpp)에 필수

// 엔진이 복제 시스템 초기화 시 1회 호출 — 여기 등록된 프로퍼티만 네트워크로 전송됨
void AMyGameState::GetLifetimeReplicatedProps(
    TArray<FLifetimeProperty>& OutLifetimeProps) const
{
    Super::GetLifetimeReplicatedProps(OutLifetimeProps);   // 부모(PlayerArray 등)가 등록한 복제 목록 유지 — 생략하면 부모 프로퍼티 복제가 끊김
    // TeamScore를 모든 클라이언트에 복제하도록 등록
    DOREPLIFETIME(AMyGameState, TeamScore);
}

void AMyGameState::AddTeamScore(int32 Amount)
{
    // Authority(서버)에서만 수정 → Replicated로 자동 전파
    // 이 체크가 없으면 클라이언트가 로컬에서만 값을 바꿔 서버와 어긋나고, 다음 복제 때 덮어써짐
    if (HasAuthority())
    {
        TeamScore += Amount;   // 서버 값 변경 → 엔진이 자동으로 전 클라이언트에 새 값 전송
    }
}

4. 커스텀 Character

#pragma once
#include "CoreMinimal.h"
#include "GameFramework/Character.h"
#include "MyCharacter.generated.h"

UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    AMyCharacter();

    // EditAnywhere: 에디터에서 클래스 기본값·레벨에 배치된 인스턴스 양쪽 모두 수정 가능 (기획자가 밸런스 조정)
    // BlueprintReadWrite: 블루프린트에서 읽기·쓰기 모두 허용
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Category = "Stats")
    float MaxHealth = 100.0f;   // 기본값 필수 — 초기화 없이 두면 쓰레기 값으로 시작해 체력 계산이 전부 깨질 수 있음

    // VisibleAnywhere: 에디터에서 보이지만 수정 불가 — 런타임 계산값이라 손으로 고칠 이유가 없음
    UPROPERTY(VisibleAnywhere, BlueprintReadOnly, Category = "Stats")
    float CurrentHealth;   // 초기화는 BeginPlay에서 — 에디터에서 조정된 MaxHealth를 반영하기 위해 (아래 .cpp 참조)

    // BlueprintCallable: 블루프린트 그래프에서 노드로 호출 가능
    UFUNCTION(BlueprintCallable, Category = "Combat")
    void ApplyDamage(float Amount);

protected:
    virtual void BeginPlay() override;   // 스폰 완료 후 첫 틱 전에 엔진이 1회 호출
};
#include "MyCharacter.h"

AMyCharacter::AMyCharacter()
{
    // CharacterMovementComponent는 ACharacter가 자동 생성
    // 여기서 이동 속도 등을 조정
    GetCharacterMovement()->MaxWalkSpeed = 600.0f;   // 단위: cm/s (언리얼 기본 단위는 센티미터) → 600 = 초당 6m
}

void AMyCharacter::BeginPlay()
{
    Super::BeginPlay();
    // 생성자가 아닌 BeginPlay에서 초기화 — 에디터/블루프린트에서 바꾼 MaxHealth 값이 적용된 뒤에 복사되므로
    CurrentHealth = MaxHealth;
}

void AMyCharacter::ApplyDamage(float Amount)
{
    CurrentHealth -= Amount;   // 1) 피해 적용
    if (CurrentHealth <= 0.0f)   // 2) 사망 판정
    {
        CurrentHealth = 0.0f;   // 3) 음수 방지 클램프 — UI 체력바가 음수로 그려지는 것을 막음
        // 사망 처리: Controller에 알림 → 리스폰 요청 등
    }
}

Project Settings에 등록하기

위 4개 클래스를 만든 뒤, Project Settings > Maps & Modes > Default GameMode에서 AMyGameMode를 지정합니다. 나머지(GameState·Controller·Pawn)는 GameMode 생성자에서 지정했으므로 자동으로 적용됩니다.

💡
월드별로 다른 GameMode를 쓸 수도 있습니다. 레벨 에디터에서 World Settings > GameMode Override를 지정하면, 해당 레벨에서만 다른 GameMode가 적용됩니다. 메인 메뉴용 GameMode와 인게임 GameMode를 분리할 때 유용합니다.
06 — 확장

수명 범위별 데이터 배치 전략

"이 데이터를 어느 클래스에 두어야 하나?"는 게임플레이 프레임워크에서 가장 자주 마주치는 설계 질문입니다. 아래 판단 가이드를 따라가면 답이 나옵니다.

데이터 배치 결정표

데이터 성격배치 클래스이유
앱 전체 수명 (설정, 세이브 슬롯, 유저 프로필)GameInstance레벨 전환에도 유지되는 유일한 클래스
현재 매치의 규칙 (시간 제한, 인원, 스폰 규칙)GameMode서버 전용 → 치팅 불가, 규칙은 서버만 정의
전체 게임 상태 (팀 점수, 라운드, 목표)GameState모든 클라이언트에 복제 → UI에서 표시 가능
플레이어별 공개 데이터 (이름, 킬수, 팀)PlayerState다른 플레이어도 볼 수 있어야 → 전원 복제
플레이어별 비공개 데이터 (입력, UI 상태)PlayerController나만 알면 됨 → 소유자만 복제
물리적 상태 (위치, 체력, 무기, 애니메이션)Pawn / Character월드에서 보이는 것 → 전원 복제, 파괴/리스폰됨

빠른 판단 플로차트

데이터를 어디에 둘지 모를 때 아래 질문을 순서대로 따라가세요.

Q1
레벨이 바뀌어도 유지되어야 하는가?
예 → GameInstance (또는 SaveGame으로 디스크 저장)
Q2
서버만 알면 되는 규칙인가?
예 → GameMode (클라이언트에 복제되지 않음)
Q3
모든 플레이어가 알아야 하는 전체 상태인가?
예 → GameState (전원 복제)
Q4
특정 플레이어의 데이터인가?
다른 플레이어도 봐야 하면 → PlayerState, 나만 알면 → PlayerController
Q5
월드에서 물리적으로 존재하는 것인가?
예 → Pawn / Character (죽으면 파괴되고 리스폰됨)

싱글플레이 vs 멀티플레이 — 무엇이 달라지나

싱글플레이

모든 클래스가 같은 프로세스에 있으므로 복제 걱정이 없습니다. GameMode에 점수를 넣어도 동작은 합니다.

하지만 나중에 멀티를 추가할 가능성이 조금이라도 있다면, 처음부터 프레임워크 규칙대로 분리하는 것이 좋습니다.

복제 불필요 구조 단순

멀티플레이

"누가 이 데이터를 볼 수 있는가"가 핵심 설계 기준이 됩니다. GameMode는 서버 전용, GameState와 PlayerState는 복제 대상입니다.

복제 범위를 잘못 잡으면 클라이언트에서 null 참조, 점수 미표시, 치팅 등의 문제가 발생합니다.

복제 범위 설계 Authority 관리
💡

가장 흔한 실수: "체력을 어디에 둘까?"에 대해 — 싱글플레이라면 Pawn에 두면 간단합니다. 멀티플레이라면 Pawn에 Replicated UPROPERTY로 두되, 서버에서만 수정(HasAuthority() 체크)하고 클라이언트는 복제된 값을 읽기만 합니다.
