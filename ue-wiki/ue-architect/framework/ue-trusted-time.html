신뢰 시간
시계 치트 방지와 Steam 서버 시간

일일 보상·스트릭·출석 체크는 전부 사용자가 마음대로 돌릴 수 있는 OS 시계 위에 서 있습니다. 이 문서는 스팀 출시작이 별도 서버 없이 확보할 수 있는 신뢰 시계 — GetServerRealTime() — 를 축으로, 시계 조작 3종을 각각 어떤 규칙으로 잡는지, 그리고 잡았을 때 처벌 대신 조용히 보류해야 하는 이유를 다룹니다. 목표는 "조작 불가능"이 아니라 "조작해도 수지가 안 맞게"입니다.

01 — 개념 · 위협 모델

적을 먼저 정의한다 — 시계 조작 3종

방어를 설계하려면 공격부터 명확히 해야 합니다. 로컬 시계로 시간을 판정하는 앱에서 가능한 조작은 세 가지뿐입니다.

조작수법노리는 것잡는 규칙 (§03)
A. 크게 앞으로OS 날짜를 +1일 이상 변경캐치업 루프를 속여 며칠치 일일 보상 선수령오프셋 범위 검증
B. 자정 미세 조작23:59 → 00:01로 몇 분만 이동날짜 키만 바꿔 하루 보상을 하루 두 번신뢰 경과시간 AND 조건
C. 뒤로 돌리기날짜를 과거로 변경(또는 시간대 서쪽 이동)치트라기보다 데이터 꼬임 유발클램프 가드 (FDateTime 문서)

핵심 용어

📖

신뢰 시간(Trusted Time) — 사용자가 조작할 수 없는 출처에서 온 현재 시각. 이 문서에서는 밸브 서버가 알려주는 UTC(Steamworks ISteamUtils::GetServerRealTime(), 유닉스 타임스탬프)입니다.

시간 권위자(Time Authority) — "지금 몇 시인가"에 대한 답을 한 곳에서만 내리는 서브시스템. 모든 시간 판정 코드가 로컬 시계를 직접 읽지 않고 이 창구를 거치게 합니다 — 값의 창구를 하나로 좁히는 소유(Ownership) 원칙의 시간판입니다.

유닉스 타임스탬프(Unix Timestamp) — 1970-01-01 00:00 UTC부터의 경과 초(정수). FDateTime::FromUnixTimestamp()로 변환합니다.

오프셋(Offset) — 로컬 Now() − 신뢰 UTC. 정상값은 그 사용자의 시간대 차이뿐이므로 지구상 한계(약 −12h ~ +14h)를 벗어날 수 없습니다 — 이 물리적 상한이 조작 탐지의 근거가 됩니다.

💡
왜 UtcNow()로는 안 되는가 — FDateTime::UtcNow()도 결국 OS 시계를 읽습니다. 사용자가 OS 시계를 돌리면 로컬과 UTC가 같이 돌아가므로 방어가 되지 않습니다. 신뢰 시간은 반드시 기기 밖에서 와야 합니다.

역할 분리가 핵심입니다 — 로컬 시계는 "사용자의 자정이 언제인가"(시간대 반영)를 계산하는 데는 계속 쓰입니다. 신뢰 시계는 "실제로 시간이 흘렀는가"를 승인하는 데만 씁니다. 둘은 경쟁 관계가 아니라 질문이 다른 두 심판입니다.

03 — 판정 규칙

규칙 4개 — 조작 3종과의 1:1 대응

규칙내용잡는 조작근거
① 오프셋 범위 검증|로컬 − 신뢰UTC|가 시간대 물리 한계(15시간 = 최대 +14h + DST 여유)를 넘으면 조작 판정A 크게 앞으로+24h 조작은 어떤 시간대로도 설명 불가 — 오탐 없는 확정 판정
② 신뢰 경과시간 AND 조건날짜 키가 바뀌어도 마지막 승인 롤오버로부터 신뢰 시계 기준 20시간 이상 지나야 승인B 자정 미세 조작달력을 어떻게 주물러도 실제 시간이 안 흘렀으면 보상 없음. 20h(24h가 아니라)인 이유: DST·시간대 이동의 정상 하루 단축 흡수
③ 클램프 가드마지막 처리일이 미래면 오늘로 당기고 종료 (FDateTime 문서의 패턴 그대로)C 뒤로 돌리기데이터 무결성 담당 — ①②와 역할 분리
④ 오프라인 폴백 + 보류신뢰 시계가 없으면(스팀 오프라인) 롤오버는 실행하되 보상만 '보류' 표시 → 다음 온라인 검증 때 확정 지급A·B의 오프라인 우회오프라인 앞돌리기는 원리상 탐지 불가 — 지급 시점을 검증 가능한 순간으로 미루는 것이 유일한 수
⚠️
정책 원칙 — 처벌하지 말고 보류하라. 시간대 변경·듀얼부팅·CMOS 배터리 방전 같은 정상 유저의 오탐이 반드시 발생합니다. 감지 시 계정 제재·경고창은 금물이고, 해당 롤오버 보상만 조용히 보류했다가 신뢰 시간이 따라잡으면 자동 지급하세요. 정상 유저는 존재조차 모르게 하는 것이 목표입니다.
04 — 승인 흐름

롤오버 승인 플로차트

일일 리셋 시스템이 "오늘 롤오버를 실행해도 되는가?"를 물었을 때 권위자가 내리는 판정 전체입니다.

flowchart TD
    Q1{"로컬 날짜 키가 바뀌었는가?"} -- 아니오 --> N["할 일 없음"]
    Q1 -- 예 --> Q2{"Steam 신뢰 시계 사용 가능?"}
    Q2 -- 아니오 --> HOLD1["롤오버 실행 + 보상 보류 (다음 온라인 때 확정)"]
    Q2 -- 예 --> Q3{"|로컬 − 신뢰UTC| ≤ 15시간?"}
    Q3 -- 아니오 --> HOLD2["조작 확정 — 롤오버는 실행, 지급만 보류"]
    Q3 -- 예 --> Q4{"마지막 승인 후 신뢰 기준 20h 경과?"}
    Q4 -- 아니오 --> WAIT["아직 이르다 — 대기"]
    Q4 -- 예 --> OK["승인 — 롤오버 실행 + 신뢰 시각 기록"]
💡

"롤오버는 실행, 지급만 보류"인 이유 — 퀘스트 목록 갱신 같은 상태 전환은 막으면 앱이 이상해 보입니다(어제 퀘스트가 계속 떠 있음). 막는 것은 가치가 있는 것(보상·스트릭 증가)만으로 좁혀야 오탐의 피해도 최소화됩니다.

#pragma once
#include "Subsystems/GameInstanceSubsystem.h"
#include "TrustedTimeSubsystem.generated.h"   // UHT 생성 리플렉션 코드 — 반드시 마지막 #include

// 롤오버 승인 요청의 판정 결과 — 플로차트(§04)의 출구 4개와 1:1
// UENUM(BlueprintType): 이 열거형을 블루프린트 변수·비교 노드에서 쓸 수 있게 노출 — uint8 기반 지정이 필수 조건
UENUM(BlueprintType)
enum class ERolloverVerdict : uint8
{
    NotYet,        // 날짜 안 바뀜 / 경과시간 미달 — 대기
    Approved,      // 승인 — 보상 포함 전부 실행
    HoldOffline,   // 오프라인 — 실행하되 보상 보류
    HoldSuspicious // 오프셋 이탈 — 실행하되 보상 보류
};

// UCLASS(): 리플렉션 등록 — 엔진이 이 Subsystem을 자동 발견·생성 / GameInstanceSubsystem 상속 = 앱 수명 내내 1개 유지
UCLASS()
class MYGAME_API UTrustedTimeSubsystem : public UGameInstanceSubsystem   // MYGAME_API: 모듈 외부 노출 매크로 — 자기 모듈 이름으로 교체
{
    GENERATED_BODY()   // 리플렉션 보일러플레이트 삽입 자리 — UCLASS()와 한 쌍
public:
    // 신뢰 UTC. 성공 시 true — 오프라인이면 false (그때 로컬을 쓸지는 호출자가 아님,
    // 이 클래스의 판정 함수가 결정한다. 시간 판정 창구는 여기 하나뿐.
    bool TryGetTrustedUtcNow(FDateTime& OutUtc) const;

    // 일일 리셋 시스템이 부르는 유일한 질문
    UFUNCTION(BlueprintCallable, Category = "TrustedTime")
    ERolloverVerdict ApproveRollover();

private:
    // ↓ 두 값은 SaveGame에 영속 (로드/저장은 소유자 경유 — SaveGame 문서)
    FDateTime LastApprovedTrustedUtc;   // 마지막 승인 시점의 "신뢰" 시각
    FString   LastLocalDateKey;         // 마지막 승인의 로컬 날짜 키 "YYYYMMDD"

    static constexpr double MaxOffsetHours  = 15.0;  // 규칙 ① — 시간대 한계 + 여유
    static constexpr double MinElapsedHours = 20.0;  // 규칙 ② — DST·이동 흡수
};
#include "TrustedTimeSubsystem.h"
// Steamworks 직접 호출 — Build.cs의 Steamworks 서드파티 의존이 전제
// THIRD_PARTY_INCLUDES_START/END: 외부 SDK 헤더의 컴파일 경고를 엔진이 억제하도록 감싸는 매크로 쌍
THIRD_PARTY_INCLUDES_START
#include <steam/steam_api.h>
THIRD_PARTY_INCLUDES_END

bool UTrustedTimeSubsystem::TryGetTrustedUtcNow(FDateTime& OutUtc) const
{
    // OnlineSubsystemSteam이 SteamAPI를 초기화한 뒤에만 SteamUtils()가 유효.
    // 오프라인 모드·스팀 미실행이면 null — 그것이 곧 "신뢰 시계 없음" 신호다.
    if (SteamUtils() == nullptr) return false;

    const uint32 UnixUtc = SteamUtils()->GetServerRealTime();   // Valve 서버의 현재 UTC를 유닉스 초로 반환 — 로컬 시계 설정과 무관한 값
    OutUtc = FDateTime::FromUnixTimestamp((int64)UnixUtc);   // 유닉스 초 → FDateTime 변환. 인자가 int64라서 명시 캐스팅 필요
    return true;   // true = OutUtc가 신뢰할 수 있는 시각으로 채워졌음을 보장
}

ERolloverVerdict UTrustedTimeSubsystem::ApproveRollover()
{
    // Q1 — 사용자의 "오늘"은 로컬로 계산한다 (시간대 반영, FDateTime 문서)
    const FString TodayKey = FDateTime::Now().ToString(TEXT("%Y%m%d"));   // 예: "20260708" — 날짜만 비교하기 위한 키 문자열
    if (TodayKey == LastLocalDateKey) return ERolloverVerdict::NotYet;   // 마지막 승인과 같은 날짜 = 아직 넘어가지 않음 — 이후 검사 불필요

    // Q2 — 신뢰 시계 확보 시도
    FDateTime TrustedUtc;
    if (!TryGetTrustedUtcNow(TrustedUtc))
    {
        LastLocalDateKey = TodayKey;   // 상태 전환은 진행, 지급만 보류 (규칙 ④)
        return ERolloverVerdict::HoldOffline;
    }

    // Q3 — 규칙 ①: 오프셋이 시간대 물리 한계를 벗어나면 조작 확정
    const FTimespan Offset = FDateTime::Now() - TrustedUtc;   // 로컬 현재시각 − 서버 UTC = 시간대 차이 + (있다면) 조작량
    if (FMath::Abs(Offset.GetTotalHours()) > MaxOffsetHours)   // Abs: 동·서 어느 방향의 이탈이든 잡음 — 지구 시간대는 ±14h가 물리 한계
    {
        LastLocalDateKey = TodayKey;
        return ERolloverVerdict::HoldSuspicious;
    }

    // Q4 — 규칙 ②: 달력이 아니라 "실제 흐른 시간"이 승인한다
    const double ElapsedH = (TrustedUtc - LastApprovedTrustedUtc).GetTotalHours();
    if (ElapsedH < MinElapsedHours) return ERolloverVerdict::NotYet;

    // 승인 — 신뢰 시각을 기록해 다음 판정의 기준점으로
    LastApprovedTrustedUtc = TrustedUtc;
    LastLocalDateKey = TodayKey;
    // SaveNow() — 소유자 경유 저장
    return ERolloverVerdict::Approved;
}
💡

보류(Hold)의 구현 — 보류 판정을 받은 보상은 "보류 큐"(예: TArray<FPendingReward>, SaveGame 영속)에 넣어 두고, 다음에 Approved가 나오는 순간 함께 지급하면 됩니다. 첫 구현에서는 큐 없이 "보류 = 그날 보상 스킵"으로 시작해도 충분합니다 — 정책 강도는 나중에 올릴 수 있습니다.

06 — 확장 & 주의

연동에서 미끄러지는 지점들

⚠️
① SteamAPI 초기화 전 호출 — SteamUtils()는 OnlineSubsystemSteam이 초기화를 마친 뒤에만 유효합니다. GameInstance Init() 극초반이나 에디터 PIE(스팀 미연결 상태)에서는 null이 정상 — null = 오프라인 폴백 경로로 설계돼 있으므로 크래시 없이 흘러가는지 확인하세요.
⚠️
② 에디터에서는 대부분 오프라인 판정 — PIE는 SteamDevAppId=480 설정이 있어도 스팀 연결이 불안정합니다. 개발 중 판정 로직을 테스트하려면 TryGetTrustedUtcNow를 콘솔 변수로 강제 성공/실패시키는 테스트 훅을 붙이는 것이 편합니다.
⚠️
③ 임계값을 24h로 잡는 실수 — 경과시간 조건을 24h로 하면 서머타임 시작일(23h짜리 하루)과 동쪽으로 여행한 유저의 정상 롤오버가 거부됩니다. 20h가 "정상 하루의 최소 길이"를 흡수하는 안전값입니다. 오프셋 한계도 ±14h가 아니라 15h로 여유를 두는 같은 이유입니다.
