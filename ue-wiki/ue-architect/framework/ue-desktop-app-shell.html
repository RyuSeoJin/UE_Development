UE Architecture · Type ① · 02-A

데스크톱 앱 셸
창 모드 · 크기 · 백그라운드 동작

UE 프로젝트를 패키징해 실행하면 모니터를 가득 채운 풀스크린 3D 게임처럼 뜹니다 — 타이머·할 일 앱을 만들었어도요. 이 문서는 "창 기반 데스크톱 앱"이라는 셸을 만드는 방법을 다룹니다: 창 모드와 시작 크기, 마우스 커서, 발열을 잡는 FPS 상한, 그리고 창을 최소화했을 때 타이머는 계속 가는가라는, 생산성 앱에는 치명적인 질문까지.

엔진 자체 기능 UE4 / UE5 공통 C++ · ini
01 — 개념 · 역할

창의 주인은 GameUserSettings다

해상도·창 모드·수직동기 같은 "실행 환경 설정"은 흩어져 있지 않고 UGameUserSettings 한 클래스가 소유합니다. 값을 바꾸고 ApplySettings()를 부르면 창이 실제로 변하고, 결과가 ini에 저장돼 다음 실행에도 유지됩니다.

핵심 용어

📖

UGameUserSettings — 해상도·창 모드·그래픽 품질·FPS 상한을 담는 엔진 표준 설정 객체. GEngine->GetGameUserSettings()로 얻습니다. 유저별 값이므로 SaveGame이 아니라 전용 ini(Saved/Config/…/GameUserSettings.ini)에 저장됩니다.

창 모드(Window Mode) 3종 — Fullscreen(독점 전체화면, 게임용) / WindowedFullscreen(테두리 없는 전체창, Alt-Tab 빠름) / Windowed(일반 창 — 데스크톱 앱의 선택지).

DefaultGameUserSettings.ini — 유저 ini가 아직 없는 첫 실행에 쓰일 기본값을 정하는 프로젝트 파일(Config/, 버전 관리 대상). "처음 켰을 때 어떤 창으로 뜨는가"는 이 파일이 결정합니다.

백그라운드 틱 — 창이 포커스를 잃거나 최소화된 동안에도 게임 스레드(틱·타이머)가 도는 것. 패키징된 UE 앱은 기본적으로 계속 돕니다 — 렌더링만 절약되고 로직은 살아 있습니다(§06에서 예외와 함께).

💡
DeveloperSettings와 헷갈리지 않기 — 밸런스 상수(DeveloperSettings)는 게임 전체 공통이고 기획자가 만지는 값, GameUserSettings는 유저마다 다른 실행 환경입니다. 보관처 경계 감각은 DeveloperSettings 문서 §03과 같은 문제의식입니다.
03 — 셸 설계 결정

데스크톱 앱이 정해야 할 다섯 가지

결정데스크톱 앱 권장근거
창 모드Windowed다른 작업과 나란히 쓰는 앱 — 전체화면은 생산성 앱의 목적과 충돌
시작 크기예: 1280×800 (UI 설계 기준 크기)UI 프레임워크가 검증한 기준 해상도로 시작 — 유저가 조절하면 ini에 자동 기억
마우스 커서항상 표시 + UI 입력 모드게임 기본은 커서 숨김·게임 입력 — 앱에서는 PlayerController에서 뒤집어야 함
FPS 상한60 (t.MaxFPS 또는 SetFrameRateLimit)정적 UI를 수백 FPS로 그리면 발열·팬소음 — 노트북에서 앱 평판을 깎는 1순위
백그라운드 정책틱 유지(기본값) + 절전은 캐치업타이머 앱은 최소화 중에도 시간이 가야 함 — §06에서 검증 방법과 예외
04 — 첫 실행 절차

ini 기본값 → 코드 보정 → 커서 → FPS

1
DefaultGameUserSettings.ini 작성
Config/DefaultGameUserSettings.ini를 만들고(없는 파일) 아래 블록을 넣습니다. 결과: 첫 실행부터 1280×800 일반 창으로 뜹니다.
[/Script/Engine.GameUserSettings]
ResolutionSizeX=1280 · ResolutionSizeY=800
FullscreenMode=2 (0=Fullscreen, 1=WindowedFullscreen, 2=Windowed)
bUseDesiredScreenResolution=False · FrameRateLimit=60.000000
2
코드 보정 — 신뢰할 수 있는 이중화
ini는 유저가 지우거나 깨뜨릴 수 있으므로, GameInstance 시작 시점에 같은 정책을 코드로도 강제합니다(아래 스켈레톤). 전제: GEngine 유효 시점 — Init()에서 안전합니다.
3
마우스 커서 + UI 입력 모드
PlayerController에서 bShowMouseCursor = true, SetInputMode(FInputModeGameAndUI())(또는 UIOnly). 결과: 커서가 보이고 위젯 클릭이 자연스럽게 동작합니다. 게임패드형 입력 처리(Possess된 폰 이동)는 이 앱엔 불필요합니다.
4
검증
패키징 후: ① 일반 창으로 뜨는가 ② 창 크기 조절 후 재실행하면 기억되는가(Saved/…/GameUserSettings.ini 갱신 확인) ③ 작업 관리자에서 GPU 사용률이 상한에 맞게 낮은가.
05 — 구현 스켈레톤

셸 세팅 코드 — GameInstance와 PlayerController

// MyGameInstance.cpp — 앱 셸 정책은 시작 시점에 한 번
#include "GameFramework/GameUserSettings.h"   // UGameUserSettings 정의 헤더 — 창 모드·해상도·FPS 상한의 단일 창구

// Init() 등 시작 시점에서 1회 호출하는 자체 함수 — 셸 정책을 한 곳에 모음
void UMyGameInstance::ApplyDesktopShell()
{
    UGameUserSettings* S = GEngine->GetGameUserSettings();   // GEngine: 엔진 전역 싱글턴 — 설정 객체는 엔진이 1개만 관리
    if (!S) return;   // null 방어 — 엔진 초기화 극초반이면 아직 없을 수 있음. 체크 없이 역참조하면 크래시

    S->LoadSettings();   // 유저 ini가 있으면 그 값을 먼저 존중

    // 정책 강제: 데스크톱 앱은 어떤 경우에도 일반 창
    S->SetFullscreenMode(EWindowMode::Windowed);   // LoadSettings 뒤에 강제 — 유저 ini에 풀스크린이 남아 있어도 덮어씀

    // 첫 실행(저장된 해상도 없음)일 때만 기준 크기 지정
    if (S->GetScreenResolution() == FIntPoint::ZeroValue)   // ZeroValue(0,0) = 저장된 해상도가 없다는 신호
    {
        S->SetScreenResolution(FIntPoint(1280, 800));   // 단위: 픽셀(가로×세로). 이후 실행은 유저가 조절한 크기를 존중
    }

    // 발열 억제 — 정적 UI 앱에 수백 FPS는 낭비
    S->SetFrameRateLimit(60.f);   // 단위: FPS. 0은 무제한 — GPU가 쓸데없이 최대치로 돌게 됨

    S->ApplySettings(false);   // false = 해상도 확인 대화 없이 즉시 적용+저장
}
// MyPlayerController.cpp — 커서·입력 모드는 컨트롤러 소관
#include "MyPlayerController.h"

// BeginPlay: 액터가 월드에 완전히 준비된 후 엔진이 1회 호출 — 커서 설정은 이 시점이 안전
void AMyPlayerController::BeginPlay()
{
    Super::BeginPlay();   // 부모의 기본 초기화 선행 — override한 엔진 콜백의 관례

    bShowMouseCursor = true;   // 기본값 false(게임용 숨김) — 데스크톱 앱은 커서가 항상 보여야 함
    bEnableClickEvents = true;   // 액터 대상 마우스 클릭 이벤트(OnClicked) 활성화
    bEnableMouseOverEvents = true;   // 마우스 오버/아웃 이벤트 활성화 — 호버 반응 UI의 전제

    // UI 중심 앱: 게임 입력과 UI 입력을 함께 허용 (단축키 + 위젯 클릭)
    FInputModeGameAndUI Mode;   // 대안: FInputModeUIOnly(게임 입력 차단) / FInputModeGameOnly(UI 클릭 불가)
    Mode.SetHideCursorDuringCapture(false);   // 드래그(마우스 캡처) 중에도 커서 유지 — 숨기면 앱이 아니라 게임처럼 느껴짐
    SetInputMode(Mode);   // 구성한 입력 모드를 실제로 적용 — 이 호출을 빠뜨리면 위 설정은 무효
}
06 — 확장 & 주의

최소화·절전 — 타이머 앱의 진짜 시험대

💡
최소화 = 틱 유지 (기본) — 패키징된 UE 앱은 포커스를 잃거나 최소화돼도 게임 스레드가 계속 돕니다. World 타이머(25분 카운트다운)도 계속 갑니다. 단, 에디터 PIE는 다르게 동작할 수 있으므로 이 검증은 반드시 패키징 빌드로 하세요: 타이머 시작 → 최소화 → 5분 뒤 복귀 → 남은 시간이 정확히 5분 줄었는지.
⚠️
① OS 절전은 별개 — 노트북 덮개를 닫으면 프로세스 전체가 얼어붙고 World 타이머도 함께 얼어붙습니다. 깨어난 순간 벽시계와의 차이로 재구성해야 합니다: 세션 시작 시각(FDateTime)을 기록해 두고 복귀 시 경과를 재계산 — FDateTime 문서의 "저장된 시각 + 현재 시각 비교" 원리이며, FCoreDelegates::ApplicationHasEnteredForegroundDelegate류 복귀 신호에서 체크 함수를 부릅니다.
⚠️
② 백그라운드 CPU 절약과의 균형 — 틱이 계속 돈다는 건 최소화 중에도 CPU를 쓴다는 뜻입니다. FPS 상한(60)이 1차 방어이고, 더 줄이고 싶으면 포커스 상실 시 상한을 10~15로 낮췄다가 복귀 시 되돌리는 정책을 쓸 수 있습니다 — 단, 오디오 재생 중에는 낮추지 않는 조건을 함께.
⚠️
③ ApplySettings를 빠뜨림 — Set 함수들은 값만 바꿉니다. ApplySettings()(또는 ApplyResolutionSettings())를 불러야 창이 실제로 변하고 ini에 저장됩니다. "코드는 도는데 창이 안 바뀌는" 원인 1순위.
💡
창 최소 크기·고정 비율 — 엔진 표준 설정으로는 창 최소 크기 강제가 없습니다. 1차 해법은 UI를 반응형(앵커)으로 만들어 작은 창도 견디게 하는 것이고, 꼭 필요하면 네이티브 윈도(GEngine->GameViewport->GetWindow())의 SWindow에 접근해 제한을 거는 고급 경로가 있습니다 — 첫 출시엔 반응형으로 충분합니다.
