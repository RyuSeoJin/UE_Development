2D 사이드뷰 카메라
드래그 · 키보드 좌우 이동

2D 사이드뷰 월드를 바라보는 카메라를 화면 드래그 또는 A/D 키로 좌우로만 움직이게 만듭니다. 카메라 전용 Pawn · 직교 투영 · Enhanced Input · 픽셀→월드 환산까지, 한 번 실패한 뒤 재시도하는 독자를 기준으로 각 단계의 "끝나면 확인되어야 할 상태"와 실패 시 체크리스트를 함께 담았습니다.

카메라 · 입력 UE5 · Enhanced Input Blueprint 전용 (C++ 불필요)
01 — 요구사항 & 시작 전 정리

무엇을 만들고, 무엇을 알고 시작하나

항목요구
시점2D 사이드뷰 — 카메라가 옆에서 월드를 바라봄
입력 ①A/D 키로 좌우 이동
입력 ②마우스 왼쪽 버튼 드래그로 좌우 이동 (화면을 "잡아 끄는" 느낌)
제약이동은 좌우 1개 축만. 맵 경계 밖으로 나가지 않게 범위 제한

함정 ① — 축 관례: "Z축이 없다"가 아니라 "Y가 깊이"다

UE에서 Z는 위쪽입니다. 2D 사이드뷰의 표준 관례(Paper2D 포함)는 화면을 X-Z 평면에 놓고 카메라가 Y축에서 바라보는 것입니다. "X,Y만 쓰고 Z를 없앤" 배치는 UE 관례와 어긋나 카메라 방향·중력·스프라이트 정렬이 전부 꼬입니다 — 원인 불명으로 실패했다면 이것부터 의심하세요.

축2D 사이드뷰에서의 역할이 문서에서
X좌우 (화면 가로)카메라가 움직이는 유일한 축
Z상하 (화면 세로)고정
Y깊이 (화면 안쪽)카메라가 Y=-1000쯤에서 +Y 방향을 바라봄 (회전 Yaw 90°)

함정 ② — 카메라는 직교(Orthographic)로

직교 카메라는 원근(멀수록 작게)이 없는 평행 투영 카메라입니다. Ortho Width 하나가 "화면에 보이는 월드의 가로 폭"을 결정하므로, 뒤에서 만들 드래그의 픽셀→월드 환산이 상수 곱셈 한 번으로 끝납니다. 원근(Perspective) 카메라는 거리마다 배율이 달라져 드래그 이동량 계산이 복잡해지고 2D 스프라이트에 원근 왜곡도 생깁니다 — "드래그 속도가 이상했다"면 십중팔구 원근 카메라 때문입니다.

📌

핵심 용어 — 직교 카메라(Orthographic): 원근 없는 평행 투영. Ortho Width = 화면에 담기는 월드 가로 폭. / Pawn: 컨트롤러가 빙의(Possess)해 조종하는 대상 — 이 문서에선 카메라 자체가 Pawn. / 픽셀→월드 환산: 마우스가 움직인 화면 픽셀을 월드 거리로 바꾸는 계산(ΔX × OrthoWidth ÷ 뷰포트 가로).

02 — 설계 결정

책임을 어디에 둘 것인가

가장 중요한 결정: 캐릭터에 카메라를 붙이는 게 아니라, 카메라 자체를 조종 대상(Pawn)으로 만듭니다. 이 카메라는 따라다닐 주인공이 없고 입력으로 직접 움직이는 독립 개체이기 때문입니다.

데이터 / 기능배치왜 여기인가
카메라·이동 로직·bDragging·환산 계산BP_CameraPawn카메라가 곧 조종 대상. 입력을 받아 움직이는 주체가 상태와 로직을 소유
마우스 커서 표시·클릭 이벤트 허용PlayerController입력 장치 정책(커서를 보일지)은 "조종하는 쪽"의 소관 — Pawn이 바뀌어도 유지
시작 시 어떤 Pawn을 줄지 (Default Pawn)GameMode"이 레벨의 규칙"이므로 GameMode가 지정 (테스트용으론 Pawn의 Auto Possess도 가능)
이동 속도·경계값 MinX/MaxXBP_CameraPawn 변수 (Instance Editable)레벨마다 다른 값 — 레벨에 배치된 인스턴스에서 조정. 프로젝트가 커지면 DeveloperSettings로 승격 여지

채택 — 직교 + 전용 Pawn

환산이 상수 1개(OrthoWidth ÷ 뷰포트X), 왜곡 없음, 캐릭터와 독립적으로 카메라 연출 가능.

계산 단순2D 왜곡 없음독립 제어

기각 — 원근 + 캐릭터 부착

거리별 배율 계산 필요, 스프라이트 원근 왜곡, 캐릭터 없는 화면(메뉴·맵 뷰)에서 카메라를 못 씀.

환산 복잡왜곡종속
03 — 준비 (애셋 · 컴포넌트 · 변수)

구현 전에 만들어 둘 것들

단계 A — BP_CameraPawn 만들기

1
Pawn 블루프린트 생성
Content Browser 우클릭 → Blueprint Class → Pawn 선택, 이름 BP_CameraPawn. (Character가 아닙니다 — 중력·캡슐이 필요 없습니다.)
2
Camera Component 추가 · 직교 설정
Components 패널 + Add → Camera. Details에서 Projection Mode = Orthographic, Ortho Width = 2048(보이길 원하는 월드 가로 폭 — 나중에 조정). 결과 확인: 뷰포트 미리보기가 평면적으로 보입니다.
3
Floating Pawn Movement 추가
Components 패널 + Add → Floating Pawn Movement. 이 컴포넌트가 없으면 Add Movement Input이 조용히 아무 일도 하지 않습니다 — "키는 눌리는데 안 움직임"의 최다 원인. Max Speed는 기본값으로 시작.
4
변수 4개 추가
My Blueprint ▸ Variables ＋: bDragging(Boolean, 기본 false) · MinX/MaxX(Float, Instance Editable 체크 — 레벨에서 눈 아이콘으로 조정) · 필요 시 MoveSpeed(Float). 컴파일·저장.
5
레벨에 배치 · 방향 맞추기
Pawn을 레벨에 놓고 위치 Y = -1000, 회전 Yaw = 90°(+Y 방향을 바라보게). 스프라이트/배경이 X-Z 평면에 있다는 전제입니다(01 축 표). 결과 확인: 선택 후 우클릭 Pilot(조종)으로 보면 화면이 정면으로 보입니다.

단계 B — Enhanced Input 애셋 만들기

Enhanced Input은 UE5의 입력 시스템입니다: 키를 코드에 직접 묶지 않고 Input Action(IA — "무엇을 한다")과 Input Mapping Context(IMC — "어떤 키가 그 행동을 발동하나") 애셋으로 분리합니다. IMC는 런타임에 등록해야만 작동합니다(구현 그래프 ①).

1
Input Action 생성
Content Browser 우클릭 → Input → Input Action, 이름 IA_MoveCamera. 열어서 Value Type = Axis1D (float) — A/D가 -1.0 ~ +1.0 한 축을 공유하게 합니다.
2
Input Mapping Context 생성 · 키 매핑
우클릭 → Input → Input Mapping Context, 이름 IMC_Camera. 열어서 Mappings ＋ → IA_MoveCamera 선택 후 키 2개 등록: D는 그대로, A는 키 항목 아래 Modifiers ＋ → Negate 추가(값을 -1로 뒤집음). 결과 확인: IA 하나에 D(+1)·A(-1) 두 매핑이 보입니다.
04 — 구현 (블루프린트 노드 그래프)

BP_CameraPawn 이벤트 그래프 — 4개 묶음

아래 그래프 4건을 모두 BP_CameraPawn의 이벤트 그래프에 만듭니다. 노드 색·핀 모양은 실제 에디터와 동일한 규칙입니다.

이벤트 함수 호출 순수 함수 흐름 제어 실행 핀 ▷ Bool Float Vector Object

그래프 ① — Mapping Context 등록 (BeginPlay)

이 등록을 빼먹으면 Enhanced Input 이벤트가 전부 무시됩니다 — 재시도 시 반드시 먼저 확인할 지점. Get Player Controller → Get Enhanced Input Local Player Subsystem → Add Mapping Context 순서로 연결하고 Mapping Context 핀에 IMC_Camera를 지정합니다.

그래프 ② — A/D 키 이동

그래프 빈 곳 우클릭 → IA_MoveCamera 검색 → EnhancedInputAction IA_MoveCamera 이벤트 선택. Triggered 실행 핀을 Add Movement Input에 잇고, World Direction = (1, 0, 0)(X축), Scale Value ← Action Value를 연결합니다. D는 +1, A는 Negate 덕분에 -1이 들어와 한 그래프로 양방향이 처리됩니다. 실제 이동은 Floating Pawn Movement가 수행합니다.

그래프 ③ — 드래그 시작 / 종료

우클릭 → Left Mouse Button 키 이벤트 추가. Pressed에서 bDragging = true, Released에서 false. 이 이벤트가 아예 안 들어오면 05 등록 섹션의 PlayerController 설정(커서·클릭 이벤트)이 빠진 것입니다.

그래프 ④ — Tick에서 픽셀→월드 환산 이동

드래그 중일 때만(Branch) 마우스 이동량을 읽어 월드 이동으로 바꿉니다. 공식은 하나입니다:

🧮

이동량 = 마우스 ΔX × (OrthoWidth ÷ 뷰포트 가로 픽셀) × −1 — 직교라서 배율이 상수입니다. × −1이 "화면을 잡아 끄는" 방향을 만듭니다: 오른쪽으로 끌면 세상이 오른쪽으로 따라오도록 카메라는 왼쪽으로 이동. 이걸 빼먹으면 드래그가 반대로 움직입니다.

🧭

Ortho Width 변수 채우기 — BeginPlay에서 Camera(컴포넌트) → Get Ortho Width를 읽어 변수 Ortho Width에 한 번 저장해 두면 그래프 ④가 위처럼 단순해집니다. Get Input Mouse Delta의 Target에는 Get Player Controller를 연결하세요(그래프 ①의 순수 노드 재사용).

마무리 — 경계 제한 (클램프)

1
이동 직후 X를 자르기
그래프 ④의 AddActorWorldOffset 실행 핀 뒤에: GetActorLocation → Break Vector로 X를 꺼내 Clamp (Float)(Min = MinX, Max = MaxX 변수) → Make Vector(Y·Z는 Break에서 그대로) → SetActorLocation. 그래프 ②의 키 이동은 Floating Pawn Movement가 옮기므로, 같은 클램프를 Tick 마지막에 한 번 더 두면 두 입력 모두 잡힙니다.
05 — 등록 (엔진에 연결)

여기까지 하고도 안 되면, 대부분 이 섹션이 빠진 것

1
Pawn을 플레이어에게 주기
방법 A(권장): GameMode 블루프린트의 Class Defaults ▸ Default Pawn Class = BP_CameraPawn, World Settings ▸ GameMode Override에 그 GameMode 지정. 방법 B(빠른 테스트): 레벨에 배치한 BP_CameraPawn 선택 → Details ▸ Pawn ▸ Auto Possess Player = Player 0. 전제: 방법 B를 쓰면 GameMode가 스폰하는 기본 폰과 겹치지 않는지 확인(Default Pawn Class = None 권장).
2
PlayerController에서 마우스 허용
BP_PlayerController를 만들어(또는 기존 것) Class Defaults에서 Show Mouse Cursor ✓ · Enable Click Events ✓ 체크 → GameMode의 Player Controller Class로 지정. 이 체크가 없으면 그래프 ③의 마우스 이벤트가 아예 발생하지 않습니다.
3
Input Mode 확인
UI를 띄우며 Set Input Mode UI Only를 호출한 적이 있다면 게임 입력이 전부 막힙니다. 기본은 Game Only, UI와 병행하려면 Set Input Mode Game and UI를 쓰세요.
06 — 검증 & 트러블슈팅

순서대로 확인하고, 막히면 표에서 찾기

1
빙의 확인
PIE(플레이) 시작 → 화면이 배치해 둔 카메라 시점으로 보이면 통과. 엉뚱한 곳(원점 하늘 등)이면 1단계 등록(Default Pawn/Auto Possess) 실패.
2
키 이동 확인
A/D를 눌러 좌우로 움직이면 통과. 안 되면 아래 표의 ①②③ 순서로.
3
드래그 확인
커서가 보이고, 왼쪽 버튼을 누른 채 끌면 화면이 끄는 방향으로 따라오면 통과. 반대로 가면 −1 곱을 빼먹은 것.
4
경계 확인
맵 끝까지 이동해 MinX/MaxX 밖으로 나가지 않으면 완료.
