차이 1 — 리플렉션 시스템

표준 C++에서는 "이 클래스에 어떤 변수가 있는지"를 런타임에 알 수 없습니다. 언리얼은 리플렉션(Reflection)이라는 시스템으로 이 문제를 해결합니다. 리플렉션이란 코드의 구조 정보(클래스 이름, 변수 목록, 함수 목록 등)를 프로그램이 실행 중에 스스로 조회할 수 있는 능력입니다.

이를 위해 UCLASS(), UPROPERTY(), UFUNCTION() 같은 매크로를 코드에 붙입니다. 이 매크로를 붙이면 UHT(Unreal Header Tool)라는 도구가 빌드 전에 코드를 스캔해서 .generated.h 파일을 자동 생성하고, 엔진이 런타임에 해당 정보를 조회할 수 있게 됩니다.

차이 2 — 가비지 컬렉션(GC)

표준 C++에서는 new로 만든 객체를 반드시 delete로 직접 해제해야 합니다. 언리얼의 UObject 계열 클래스(뒤에서 설명)는 가비지 컬렉터(Garbage Collector, GC)가 자동으로 메모리를 관리합니다. GC는 주기적으로 "아무도 참조하지 않는 UObject"를 찾아 메모리에서 해제합니다.

단, GC가 올바르게 참조를 추적하려면 포인터 변수에 반드시 UPROPERTY() 매크로를 붙여야 합니다. 매크로 없이 일반 포인터(UObject* Ptr)로만 들고 있으면 GC가 그 참조를 모르기 때문에 오브젝트가 갑자기 사라질 수 있습니다.

차이 3 — 모듈 기반 빌드 시스템

CMake나 Visual Studio 프로젝트 파일 대신, 언리얼은 UBT(Unreal Build Tool)라는 자체 빌드 도구를 씁니다. 코드는 모듈(Module) 단위로 나뉘고, 각 모듈은 .Build.cs 파일에서 의존성을 선언합니다. 자세한 구조는 섹션 07에서 다룹니다.

차이 4 — 블루프린트 양방향 연동

언리얼의 비주얼 스크립팅 시스템인 블루프린트(Blueprint)는 C++과 분리된 것이 아니라, C++ 위에서 동작합니다. C++ 클래스를 블루프린트로 상속할 수 있고, C++ 함수를 블루프린트에서 호출하거나 그 반대도 가능합니다. 이 연동은 앞서 설명한 리플렉션 매크로(BlueprintCallable, BlueprintImplementableEvent 등)를 통해 이루어집니다.

02 — 네이밍 컨벤션

접두사 규칙과 명명 표준

언리얼 엔진은 엄격한 접두사 규칙을 따릅니다. 접두사만 보면 해당 타입의 역할과 상속 관계를 즉시 파악할 수 있습니다. 이 규칙을 어기면 UHT(Unreal Header Tool, 앞서 01에서 설명)가 빌드 에러를 냅니다.

접두사의미대표 예시
AActor 파생 클래스 — 게임 월드에 배치/스폰 가능한 오브젝트AActor, ACharacter, APlayerController, AGameModeBase
UUObject 파생 클래스 (Actor 제외) — GC 관리, 리플렉션 지원UActorComponent, USceneComponent, UUserWidget
F구조체 / 일반 C++ 클래스 — GC 대상이 아닌 값 타입FVector, FString, FTransform, FName, FText
E열거형 (Enum) — 상태, 모드, 채널 등 고정 선택지ECollisionChannel, EMovementMode, ENetRole
T템플릿 클래스 — 컨테이너, 스마트 포인터 등 범용 타입TArray, TMap, TSet, TSharedPtr, TWeakPtr
I인터페이스 — 다중 상속을 위한 순수 인터페이스 클래스IInteractable, IDamageable
bBoolean 변수 접두사 — 멤버 변수에만 적용 (클래스명 아님)bIsActive, bCanJump, bReplicates
⚠️

필수 규칙: .generated.h 파일은 반드시 마지막 #include여야 합니다. GENERATED_BODY()는 클래스 본문의 첫 번째 줄에 위치해야 합니다. 이 규칙을 어기면 UHT가 컴파일 에러를 발생시킵니다. 순서를 지키지 않아 발생하는 빌드 에러는 UE C++ 초보자가 가장 자주 겪는 문제 중 하나입니다.

올바른 헤더 구조 예시

MyCharacter.h — 올바른 헤더 구조
#pragma once   // 같은 헤더가 중복 포함되는 것을 방지 — 모든 헤더 첫 줄 관례

// 일반 헤더를 먼저 포함
#include "CoreMinimal.h"
#include "GameFramework/Character.h"
// .generated.h는 반드시 마지막!
#include "MyCharacter.generated.h"

// MYGAME_API: 다른 모듈에서도 이 클래스를 쓸 수 있게 내보내는(export) 매크로 — 모듈명_API 형식
UCLASS()
class MYGAME_API AMyCharacter : public ACharacter
{
    GENERATED_BODY()  // 반드시 첫 줄!

public:
    AMyCharacter();   // 생성자 — 기본값 설정 전용 (CDO에 기록됨)

03 — 클래스 계층 구조

UObject에서 시작하는 전체 클래스 트리

언리얼의 모든 클래스는 UObject를 최상위 부모로 삼는 하나의 거대한 트리를 이룹니다. UObject가 리플렉션/GC/직렬화를 제공하고, 그 아래로 역할이 세분화됩니다. 어떤 부모를 상속하느냐에 따라 사용할 수 있는 기능이 달라지므로, 이 계층을 파악하는 것이 언리얼 C++ 프로그래밍의 첫 단추입니다.

UObject — 모든 것의 뿌리

UObject는 언리얼 오브젝트 시스템의 최상위 기반 클래스입니다. 이 클래스를 상속하면 자동으로 리플렉션, 가비지 컬렉션, 직렬화, 에디터 노출 기능을 얻습니다. 게임 월드에 배치할 필요 없는 순수 데이터 객체(예: SaveGame, DataAsset)는 UObject를 직접 상속합니다.

AActor — 월드에 존재하는 것

AActor는 UObject의 자식으로, 게임 월드에 배치(Spawn)할 수 있는 오브젝트입니다. 위치(Transform)를 가질 수 있고, 컴포넌트를 붙일 수 있으며, 틱(매 프레임 업데이트)과 네트워크 복제를 지원합니다. 캐릭터, 무기, 문, 라이트 — 월드에 보이거나 존재하는 거의 모든 것이 AActor의 자손입니다.

UActorComponent — 기능 모듈

UActorComponent는 액터에 붙여서 기능을 부여하는 모듈입니다. 메시 렌더링, 물리, 오디오, 이동 등을 각각 컴포넌트로 분리해 조합합니다. 이 설계를 컴포지션(Composition)이라 부르며, 깊은 상속 대신 필요한 기능을 "꽂아 넣는" 방식입니다.

💡

상속보다 조합(Composition over Inheritance) — 액터는 "무엇인가"(캐릭터, 자동차, 문)를 정의하고, 컴포넌트는 "무엇을 할 수 있는가"(메시 표시, 물리 충돌, 이동)를 부여합니다. 새 기능이 필요하면 새 컴포넌트를 만들어 붙이세요.

04 — 리플렉션 매크로

UCLASS / UPROPERTY / UFUNCTION — 엔진과 대화하는 언어

리플렉션 매크로(앞서 01 섹션에서 설명한 리플렉션을 실현하는 코드 표식)는 언리얼 C++의 가장 독특한 부분입니다. 이 매크로를 올바르게 붙이지 않으면 변수가 에디터에 안 보이고, 함수를 블루프린트에서 호출할 수 없으며, GC가 참조를 추적하지 못합니다.

UCLASS() — 클래스 등록

UCLASS()는 클래스 전체를 언리얼 리플렉션에 등록합니다. UObject를 상속하는 모든 클래스 선언부 바로 위에 반드시 적어야 합니다.

주요 지정자(Specifier)의미사용 예
Blueprintable이 클래스를 블루프린트에서 상속 가능하게 허용커스텀 GameMode를 BP로 확장할 때
BlueprintType이 클래스 타입의 변수를 블루프린트에서 선언 가능BP 그래프에서 변수 핀으로 사용
Abstract직접 인스턴스화 금지, 반드시 자식 클래스로 사용기반 클래스를 순수 인터페이스로 강제
Config=Game이 클래스의 UPROPERTY(Config) 변수를 .ini에서 읽기게임 설정 값을 외부 파일에서 관리

UPROPERTY() — 변수 등록

UPROPERTY()는 멤버 변수를 리플렉션에 등록합니다. 등록된 변수만 에디터에 노출되고, GC가 추적하고, 네트워크로 복제되고, 직렬화(저장/로드)됩니다.

BP 그룹 (블루프린트 접근)

네트워크 그룹 (리플리케이션)

UFUNCTION()은 멤버 함수를 리플렉션에 등록합니다.

USTRUCT() & UENUM() — 구조체와 열거형

USTRUCT()는 구조체를, UENUM()은 열거형을 리플렉션에 등록합니다. GC 대상은 아니지만 블루프린트 노출/직렬화가 가능해집니다.

USTRUCT 예시
// BlueprintType: BP 변수 타입으로 사용 가능. 구조체는 F 접두사
USTRUCT(BlueprintType)
struct FItemData
{
    GENERATED_BODY()   // 구조체에도 필수

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    FString ItemName;   // FString은 생성자가 빈 문자열로 초기화해 주므로 초기값 생략 가능

    // 값 타입은 초기값 필수 — 없으면 쓰레기 값이 들어간다
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    int32 ItemCount = 0;

    // UObject 포인터는 UPROPERTY 덕분에 GC가 추적한다
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    UTexture2D* Icon = nullptr;
};
UENUM 예시
// BP 노출 enum은 반드시 uint8 기반 + E 접두사
UENUM(BlueprintType)
enum class EWeaponType : uint8
{
    // UMETA(DisplayName): 에디터/BP 드롭다운에 보일 이름
    Sword   UMETA(DisplayName="검"),
    Bow     UMETA(DisplayName="활"),
    Staff   UMETA(DisplayName="지팡이"),
    Shield  UMETA(DisplayName="방패"),
};

매크로가 붙은 클래스 예제

#pragma once

#include "CoreMinimal.h"
#include "GameFramework/Character.h"
#include "MyCharacter.generated.h"    // UHT가 생성한 파일 — 반드시 마지막 #include

// UCLASS: 이 클래스를 블루프린트에서 상속 가능하게 등록
UCLASS(Blueprintable)
class MYGAME_API AMyCharacter : public ACharacter
{
    GENERATED_BODY()          // 리플렉션 코드 자동 삽입 — 클래스 본문 첫 줄

public:
    AMyCharacter();   // 생성자 — CDO 기본값 설정 전용 (월드 접근 금지)

    // UPROPERTY: 에디터+BP에서 편집 가능한 변수
    // (매크로가 없으면 에디터에 안 보이고, UObject 포인터라면 GC 추적도 안 됨)
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Category = "Stats")
    float MaxHealth = 100.0f;   // 값 타입은 선언과 동시에 초기화 — 쓰레기 값 방지

    // UPROPERTY: BP에서 읽기만 가능한 변수
    UPROPERTY(VisibleAnywhere, BlueprintReadOnly, Category = "Stats")
    float CurrentHealth;   // 초기값은 생성자에서 MaxHealth로 설정 (.cpp 참조)

    // UFUNCTION: BP에서 호출 가능한 함수
    UFUNCTION(BlueprintCallable, Category = "Combat")
    void TakeDamage(float DamageAmount);

protected:
    // 액터가 월드에 스폰되어 게임에 참여한 직후, 첫 틱 전에 엔진이 1회 호출
    virtual void BeginPlay() override;
};
#include "MyCharacter.h"

AMyCharacter::AMyCharacter()
{
    // 생성자: CDO(Class Default Object) 값 설정
    CurrentHealth = MaxHealth;
}

void AMyCharacter::BeginPlay()
{
    Super::BeginPlay();        // 부모 클래스의 BeginPlay를 먼저 호출
    CurrentHealth = MaxHealth;   // 게임 시작 시 체력 초기화
}

void AMyCharacter::TakeDamage(float DamageAmount)
{
    CurrentHealth -= DamageAmount;   // 1) 받은 데미지만큼 체력 차감
    if (CurrentHealth <= 0.0f)       // 2) 0 이하로 떨어졌는지 확인
    {
        CurrentHealth = 0.0f;        // 3) 음수 체력 방지 — UI/BP가 음수를 받으면 오작동 여지
        // 사망 처리 로직을 여기에 추가
    }
}

GENERATED_BODY()와 CDO

GENERATED_BODY() 매크로는 클래스 본문 가장 첫 줄에 반드시 적어야 합니다. 이 매크로가 UHT가 생성한 리플렉션 코드를 클래스 안에 삽입하는 자리입니다.

CDO(Class Default Object)는 클래스마다 엔진이 자동으로 만드는 "기본값 템플릿 오브젝트"입니다. 생성자(Constructor)에서 설정한 값은 CDO에 기록되고, 이후 이 클래스의 인스턴스가 만들어질 때 CDO의 값이 복사됩니다. 따라서 생성자는 오직 기본값 설정에만 사용하고, 게임 로직(월드 접근, 다른 액터 찾기 등)은 BeginPlay()에서 해야 합니다.

05 — 게임플레이 프레임워크

7대 프레임워크 클래스 — 누가 무엇을 책임지나

언리얼은 게임의 구조를 7개의 핵심 클래스로 분리합니다. 각 클래스는 명확한 책임 경계를 갖고 있어서, "이 기능을 어디에 넣을지"를 결정하는 기준이 됩니다. 이 구조를 이해하면 코드가 자연스럽게 정리됩니다.

GameMode GameState PlayerController Pawn / Character PlayerState GameInstance SaveGame

클래스별 역할 상세

클래스수명존재 위치핵심 책임
GameMode현재 레벨서버에만 존재게임 규칙(승패 판정, 스폰 위치, 매치 상태 전환). 클라이언트는 접근 불가.
GameState현재 레벨모든 머신에 복제모든 플레이어가 공유하는 상태(점수판, 남은 시간, 팀 정보).
PlayerController현재 레벨소유 클라이언트 + 서버플레이어의 입력 처리, 카메라 관리, UI 소유. 폰이 죽어도 유지됨.
Pawn / Character스폰~소멸모든 머신에 복제물리적 몸. 이동, 충돌, 체력 등 물리 상태를 담당.
PlayerState현재 레벨모든 머신에 복제개별 플레이어의 공개 정보(이름, 점수). 다른 플레이어도 조회 가능.
GameInstance프로그램 전체로컬에만레벨 전환에도 유지되는 세션 전역 데이터(로그인 정보, 전환 중 임시 저장).
SaveGame디스크 영속로컬에만게임을 껐다 켜도 남아야 할 데이터(세이브 슬롯, 업적, 설정).

책임 배치 결정표 — "이 데이터를 어디에 둘까?"

Possess(빙의)란 컨트롤러가 폰을 "소유하고 조종하는 관계"를 말합니다. 플레이어가 게임에 접속하면 GameMode가 PlayerController와 Pawn을 각각 생성한 뒤, 컨트롤러가 폰을 Possess합니다. 캐릭터가 죽으면 UnPossess 후 새 폰을 Possess하는 식입니다.

06 — 액터 라이프사이클

생성부터 소멸까지 — 언제 무엇이 호출되나

언리얼 오브젝트(특히 액터)는 정해진 순서로 초기화되고 소멸됩니다. 이 순서를 모르면 "생성자에서 다른 액터를 찾으려고 했는데 null" 같은 문제에 반복적으로 부딪힙니다. 각 단계에서 안전하게 할 수 있는 일을 정확히 알아야 합니다.

Spawning 흐름 vs Loading 흐름

액터가 만들어지는 경로는 두 가지입니다. 코드로 런타임 생성(Spawning)하느냐, 레벨에 미리 배치된 것이 로드(Loading)되느냐에 따라 호출 순서가 다릅니다.

게임 시작 시 프레임워크 클래스 초기화 순서

1
GameInstance가 이미 존재
프로그램이 실행될 때 한 번 생성되어, 종료할 때까지 유지됩니다.
2
GameMode 생성 (서버)
레벨이 로드될 때 엔진이 GameMode를 생성합니다. 서버에만 존재합니다.
3
GameState 생성
GameMode가 GameState를 생성합니다. 모든 머신에 자동 복제됩니다.
4
PlayerController 생성
각 플레이어가 접속하면 GameMode가 PlayerController를 생성합니다.
5
PlayerState 생성
PlayerController와 함께 PlayerState가 생성됩니다.
6
Pawn/Character 스폰 + Possess
GameMode가 DefaultPawnClass에 지정된 폰을 스폰하고, PlayerController가 그 폰을 Possess(빙의)합니다.
7
모든 액터 BeginPlay
레벨의 모든 액터에 BeginPlay가 호출되며, 게임이 시작됩니다.
⚠️
Constructor ≠ BeginPlay — 가장 흔한 실수입니다. 생성자는 "CDO(클래스 기본 오브젝트, 앞서 04 섹션에서 설명)"를 만들 때도 호출됩니다. 이 시점에는 월드가 없으므로 GetWorld()는 nullptr을 반환합니다. 게임 로직은 반드시 BeginPlay() 이후에 작성하세요.
💡
Super:: 호출을 잊지 마세요 — BeginPlay(), Tick(), EndPlay()를 오버라이드할 때는 함수 본문 첫 줄에서 Super::BeginPlay()처럼 부모 함수를 먼저 호출해야 합니다. 이걸 빼먹으면 엔진의 내부 초기화가 누락되어 예측 불가능한 버그가 발생합니다.
07 — 빌드 시스템 & 모듈 구조

UBT / 모듈 / .Build.cs — 코드가 실행되기까지

언리얼은 CMake나 MSBuild를 직접 쓰지 않습니다. 대신 UBT(Unreal Build Tool)가 .Build.cs와 .Target.cs 파일을 읽어 빌드를 수행합니다. 코드는 모듈(Module) 단위로 조직됩니다.

모듈이란?

모듈(Module)은 언리얼에서 코드를 묶는 최소 단위입니다. 하나의 모듈은 하나의 .Build.cs 파일을 갖고, 자체 Public/Private 폴더를 갖습니다. 프로젝트를 만들면 프로젝트 이름과 같은 모듈이 하나 자동 생성됩니다.

프로젝트 폴더 구조

MyGame/ ← 프로젝트 루트 ├── Source/ │ ├── MyGame.Target.cs ← 게임 빌드 타깃 설정 │ ├── MyGameEditor.Target.cs ← 에디터 빌드 타깃 설정 │ └── MyGame/ ← 메인 모듈 폴더 │ ├── MyGame.Build.cs ← 모듈 의존성 선언 │ ├── MyGame.h ← 모듈 헤더 (보통 최소) │ ├── MyGame.cpp ← IMPLEMENT_PRIMARY_GAME_MODULE │ ├── Public/ ← 다른 모듈에 공개할 .h │ │ ├── MyCharacter.h │ │ └── MyGameMode.h │ └── Private/ ← 이 모듈 내부 전용 .cpp │ ├── MyCharacter.cpp │ └── MyGameMode.cpp ├── Content/ ← 에셋 (블루프린트, 텍스처, 메시 등) ├── Config/ ← .ini 설정 파일 └── MyGame.uproject ← 프로젝트 디스크립터

09 — 메모리 관리

GC / 스마트 포인터 — 메모리를 안전하게

언리얼에서 메모리 관리는 두 트랙으로 나뉩니다: UObject 계열은 가비지 컬렉터(GC, 앞서 01에서 설명)가, 비-UObject 계열은 스마트 포인터가 관리합니다.

트랙 1: UObject → GC가 관리

UObject를 상속한 모든 클래스는 GC가 메모리를 자동 해제합니다. 단, GC가 참조를 인식하려면 포인터 변수에 UPROPERTY()를 반드시 붙여야 합니다.

안전한 방법

  • UPROPERTY()가 붙은 포인터 — GC가 참조를 추적
  • TWeakObjectPtr<T> — 참조하되 GC 수거를 막지 않음
  • NewObject<T>(Outer)로 생성 — GC가 관리하는 객체
  • 액터는 GetWorld()->SpawnActor<T>()로 스폰

위험한 방법

  • UPROPERTY() 없는 일반 포인터 — GC가 모르고 수거할 수 있음
  • UObject에 new/delete 직접 사용 — GC와 충돌
  • std::shared_ptr로 UObject 관리 — GC와 이중 관리 충돌

트랙 2: 비-UObject → 스마트 포인터

UObject를 상속하지 않는 일반 C++ 클래스는 GC가 관리하지 않습니다. 이 경우 언리얼 전용 스마트 포인터를 씁니다.

스마트 포인터역할사용 시점
TSharedPtr<T>참조 카운트 기반 공유 소유권여러 곳에서 같은 객체를 공유할 때
TWeakPtr<T>소유하지 않는 약한 참조순환 참조를 끊을 때
TUniquePtr<T>단독 소유권 (복사 불가, 이동만 가능)소유자가 하나뿐일 때
TSharedRef<T>null이 될 수 없는 공유 참조절대 null이 아닌 것이 보장될 때
10 — 델리게이트 & 이벤트

델리게이트 — 오브젝트 간 안전한 통신

델리게이트(Delegate)는 함수 포인터를 안전하게 감싼 콜백 시스템입니다. "체력이 0이 되면 사망 처리 함수를 호출해줘"처럼 이벤트가 발생했을 때 등록된 함수를 자동 호출하는 패턴입니다. 표준 C++ 함수 포인터와 달리, 대상 오브젝트가 파괴되면 자동으로 바인딩이 해제되어 댕글링(dangling, 이미 해제된 메모리를 가리키는 포인터) 문제를 방지합니다.

델리게이트 4종류

종류매크로바인딩 수블루프린트사용 시점
단일 델리게이트DECLARE_DELEGATE1개X콜백이 딱 하나만 필요할 때
멀티캐스트DECLARE_MULTICAST_DELEGATE여러 개X여러 리스너가 같은 이벤트를 수신
다이나믹 단일DECLARE_DYNAMIC_DELEGATE1개OBP에서 바인딩할 단일 콜백
다이나믹 멀티캐스트DECLARE_DYNAMIC_MULTICAST_DELEGATE여러 개OBP 이벤트 디스패처 (가장 일반적)

델리게이트 사용 예제

#pragma once

#include "CoreMinimal.h"
#include "Components/ActorComponent.h"
#include "HealthComponent.generated.h"

// 다이나믹 멀티캐스트 델리게이트 선언
// — 파라미터: float NewHealth, float DamageAmount
DECLARE_DYNAMIC_MULTICAST_DELEGATE_TwoParams(
    FOnHealthChanged,         // 델리게이트 타입 이름
    float, NewHealth,         // 첫 번째 파라미터
    float, DamageAmount       // 두 번째 파라미터
);

// BlueprintSpawnableComponent: BP 에디터의 "컴포넌트 추가" 목록에 노출 — 없으면 BP에서 부착 불가
UCLASS(ClassGroup=(Custom), meta=(BlueprintSpawnableComponent))
class MYGAME_API UHealthComponent : public UActorComponent
{
    GENERATED_BODY()   // 리플렉션 코드 삽입 지점 — 클래스 본문 첫 줄 필수

public:
    UHealthComponent();

    // 블루프린트에서 바인딩 가능한 이벤트
    // (BlueprintAssignable은 멀티캐스트 델리게이트 전용 지정자)
    UPROPERTY(BlueprintAssignable, Category = "Health")
    FOnHealthChanged OnHealthChanged;

    // 데미지 입구를 함수 하나로 통일 — 외부에서 CurrentHealth를 직접 만지지 못하게 함
    UFUNCTION(BlueprintCallable, Category = "Health")
    void ApplyDamage(float DamageAmount);

private:
    // EditDefaultsOnly: BP 기본값만 편집 가능 — 레벨의 개별 인스턴스에서 실수로 바꾸는 것 방지
    UPROPERTY(EditDefaultsOnly, Category = "Health")
    float MaxHealth = 100.0f;

    // 런타임 전용 값 — UObject 포인터가 아닌 float이라 GC와 무관, 에디터 노출도 불필요해 UPROPERTY 생략
    float CurrentHealth;
};
#include "HealthComponent.h"

UHealthComponent::UHealthComponent()
{
    CurrentHealth = MaxHealth;   // 시작 체력은 최대치로
}

void UHealthComponent::ApplyDamage(float DamageAmount)
{
    // Clamp로 0 ~ MaxHealth 범위를 벗어나지 않게 (음수 체력 방지)
    CurrentHealth = FMath::Clamp(CurrentHealth - DamageAmount,
                                   0.0f, MaxHealth);

    // 등록된 모든 리스너에게 이벤트를 발송 (Broadcast)
    OnHealthChanged.Broadcast(CurrentHealth, DamageAmount);
}

언제 어떤 델리게이트를 쓸까?
C++ 내부에서만 → DECLARE_MULTICAST_DELEGATE (오버헤드 최소).
블루프린트에서도 바인딩 → DECLARE_DYNAMIC_MULTICAST_DELEGATE (리플렉션 비용 있지만 유연).
콜백이 딱 하나 → 단일 버전 사용.

11 — 블루프린트 연동 패턴

C++와 블루프린트를 연결하는 5가지 패턴

언리얼의 핵심 철학은 "C++로 기반을 만들고, 블루프린트로 확장한다"입니다. 다음 5가지 패턴을 익히면 프로그래머와 디자이너 간 효율적인 협업이 가능합니다. 각 패턴은 "C++과 블루프린트의 역할 분담"을 어떻게 할지 결정합니다.

패턴 1: BlueprintCallable — C++ 함수를 BP에서 호출

가장 기본적인 연동. C++에서 구현한 함수를 블루프린트 그래프에서 노드로 호출합니다. 실행 핀이 있고, 함수 내부에서 상태를 변경할 수 있습니다.

BlueprintCallable — 기본 호출
// 상태 변경 O, 실행 핀 O
UFUNCTION(BlueprintCallable, Category = "Inventory")
bool AddItem(FName ItemID, int32 Count);
// BP에서 "Add Item" 노드로 사용 가능
// 반환값이 있으면 BP에서 출력 핀으로 표시됨

패턴 2: BlueprintPure — 사이드 이펙트 없는 계산

실행 핀 없이 데이터만 반환하는 순수 함수. 상태를 변경하지 않는 getter/계산 함수에 사용합니다. const 한정자를 붙이는 것이 관례입니다.

BlueprintPure — 순수 함수
// 상태 변경 X, 실행 핀 X
UFUNCTION(BlueprintPure, Category = "Stats")
float GetHealthPercent() const
{
    return CurrentHealth / MaxHealth;
}
// BP에서 실행 핀 없이 값만 가져옴
// 여러 노드에 연결해도 매번 재계산됨 (캐싱 X)

패턴 3: BlueprintImplementableEvent — BP에서만 구현

C++에서 선언만 하고 구현은 블루프린트에서 합니다. C++ 본문을 작성하지 않습니다. BP에서 구현하지 않으면 호출해도 아무 일도 일어나지 않습니다.

BlueprintImplementableEvent
// 헤더에 선언만 (cpp에 구현 작성 금지!)
UFUNCTION(BlueprintImplementableEvent, Category = "Events")
void OnItemCollected(FName ItemID);

// C++에서 호출하면 BP 구현이 실행됨
OnItemCollected(FName("Potion_HP"));

패턴 4: BlueprintNativeEvent — C++ 기본 + BP 오버라이드

C++에서 기본 구현을 제공하고, 블루프린트에서 선택적으로 오버라이드합니다. _Implementation 접미사로 기본 구현을 작성합니다. BP에서 오버라이드하지 않으면 C++ 구현이 실행됩니다.

BlueprintNativeEvent
// 헤더 — 선언
UFUNCTION(BlueprintNativeEvent, Category = "Damage")
float CalculateDamage(float BaseDamage);

// cpp — _Implementation 접미사 필수!
float AMyCharacter::CalculateDamage_Implementation(float BaseDamage)
{
    return BaseDamage * DamageMultiplier;
}
// BP에서 오버라이드하면 BP 구현이 우선 실행
// BP에서 "Parent: CalculateDamage"를 호출하면 C++ 구현도 실행

패턴 5: BlueprintAssignable — 이벤트 디스패처 BP 노출

C++의 멀티캐스트 델리게이트를 블루프린트에서 바인딩할 수 있게 노출합니다. BP에서 "Bind Event" 또는 "Assign" 노드로 이벤트를 수신합니다.

BlueprintAssignable
// 델리게이트 타입 선언
DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(
    FOnHealthChanged, float, NewHealth);

// UPROPERTY로 노출
UPROPERTY(BlueprintAssignable, Category = "Events")
FOnHealthChanged OnHealthChanged;

// C++에서 브로드캐스트
OnHealthChanged.Broadcast(CurrentHealth);
// BP에서 "Bind Event" 또는 "Assign" 노드로 수신
🧭

어떤 상황에 어떤 패턴을 쓸까?
C++ 로직을 BP에서 사용 → BlueprintCallable (기본).
읽기 전용 값 반환 → BlueprintPure (실행 핀 없음).
C++ 선언, BP 구현 → BlueprintImplementableEvent (디자이너 자유도 최대).
C++ 기본값 + BP 오버라이드 → BlueprintNativeEvent (가장 유연).
이벤트를 BP에서 수신 → BlueprintAssignable (옵저버 패턴).

컴파일 주의 — 전부 새 클래스이므로 에디터 종료 → IDE 빌드 코스입니다.

지금까지 배운 내용을 실전으로 연결합니다. 커스텀 GameMode와 커스텀 Character를 만들고, GameMode에 등록하는 최소 뼈대 코드입니다.

프로젝트 설정 흐름

1
C++ 클래스 생성
언리얼 에디터 > Tools > New C++ Class를 클릭합니다. 부모 클래스로 Game Mode Base를 선택하고 이름을 MyGameMode로 지정합니다. 같은 방법으로 Character를 부모로 MyCharacter를 생성합니다.
2
GameMode에 기본 폰 등록
GameMode의 생성자에서 DefaultPawnClass를 커스텀 캐릭터로 지정합니다. 아래 코드 참조.
3
프로젝트 설정에 GameMode 등록
에디터 > Edit > Project Settings > Maps & Modes > Default GameMode에서 MyGameMode를 선택합니다.
4
컴파일 & 플레이
에디터 하단 또는 IDE에서 컴파일(Ctrl+Shift+B 또는 에디터의 Compile 버튼)한 뒤, Play를 누르면 커스텀 캐릭터가 스폰됩니다.

커스텀 Character (이동 + 카메라)

#pragma once

#include "CoreMinimal.h"
#include "GameFramework/Character.h"
#include "MyCharacter.generated.h"

// 전방 선언 — 헤더에서 포인터만 쓸 땐 #include 대신 이걸로 컴파일 의존성 최소화
class USpringArmComponent;
class UCameraComponent;

UCLASS()
class MYGAME_API AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    AMyCharacter();

protected:
    virtual void BeginPlay() override;
    // 입력 바인딩 전용 함수 — 엔진이 폰 빙의(Possess) 시 자동 호출
    virtual void SetupPlayerInputComponent(
        class UInputComponent* PlayerInputComponent) override;

private:
    // 스프링 암: 카메라와 캐릭터 사이 거리를 유지하고 벽 충돌 시 줄여주는 붐대
    UPROPERTY(VisibleAnywhere, Category = "Camera")
    USpringArmComponent* SpringArm;

    // VisibleAnywhere: 디테일 패널에 보이되 포인터 교체는 불가 — 컴포넌트 포인터의 표준 지정자
    UPROPERTY(VisibleAnywhere, Category = "Camera")
    UCameraComponent* Camera;

    // 축 입력 콜백 — Value는 -1.0 ~ 1.0 (W/S, A/D 또는 스틱 기울기)
    void MoveForward(float Value);
    void MoveRight(float Value);
};
#include "MyCharacter.h"
#include "GameFramework/SpringArmComponent.h"
#include "Camera/CameraComponent.h"

AMyCharacter::AMyCharacter()
{
    // ---- 컴포넌트 생성 (생성자에서만 가능) ----

    // 스프링 암을 루트(RootComponent) 아래에 붙입니다.
    SpringArm = CreateDefaultSubobject<USpringArmComponent>(
        TEXT("SpringArm"));
    SpringArm->SetupAttachment(RootComponent);
    SpringArm->TargetArmLength = 300.0f;  // 카메라 거리
    SpringArm->bUsePawnControlRotation = true;  // 마우스 회전을 스프링 암이 따라가게

    // 카메라를 스프링 암 끝에 붙입니다.
    Camera = CreateDefaultSubobject<UCameraComponent>(
        TEXT("Camera"));
    Camera->SetupAttachment(SpringArm);
}

void AMyCharacter::BeginPlay()
{
    Super::BeginPlay();
    // 게임 시작 시 추가 초기화가 필요하면 여기에 작성
}

void AMyCharacter::SetupPlayerInputComponent(
    UInputComponent* PlayerInputComponent)
{
    Super::SetupPlayerInputComponent(PlayerInputComponent);   // 부모 클래스의 바인딩 처리를 먼저 실행 — 누락 시 기본 입력이 깨질 수 있음

    // "MoveForward" 축 입력을 MoveForward 함수에 바인딩
    // (Project Settings > Input > Axis Mappings 에서 축을 등록해야 동작)
    PlayerInputComponent->BindAxis(
        TEXT("MoveForward"), this, &AMyCharacter::MoveForward);
    PlayerInputComponent->BindAxis(
        TEXT("MoveRight"), this, &AMyCharacter::MoveRight);

    // 마우스 회전
    PlayerInputComponent->BindAxis(
        TEXT("Turn"), this, &APawn::AddControllerYawInput);
    PlayerInputComponent->BindAxis(
        TEXT("LookUp"), this, &APawn::AddControllerPitchInput);
}

void AMyCharacter::MoveForward(float Value)
{
    if (Value != 0.0f)   // 입력이 없으면(0) 아무것도 안 함
    {
        // 카메라(컨트롤러)가 보는 방향 기준의 "앞" 방향을 구한다 (X축 = 전방)
        const FRotator Rot = Controller->GetControlRotation();
        const FVector Dir = FRotationMatrix(Rot).GetUnitAxis(EAxis::X);
        // 실제 이동은 CharacterMovementComponent가 처리 — 여기선 방향과 세기만 전달
        AddMovementInput(Dir, Value);
    }
}

void AMyCharacter::MoveRight(float Value)
{
    if (Value != 0.0f)
    {
        // 같은 원리로 "오른쪽" 방향 (Y축 = 우측)
        const FRotator Rot = Controller->GetControlRotation();
        const FVector Dir = FRotationMatrix(Rot).GetUnitAxis(EAxis::Y);
        AddMovementInput(Dir, Value);
    }
}
⚠️

입력 매핑 등록 필수 — 위 코드의 "MoveForward", "MoveRight", "Turn", "LookUp"은 에디터의 Project Settings > Engine > Input > Axis Mappings에서 미리 등록해야 합니다. 등록하지 않으면 아무 입력도 동작하지 않습니다.

자주 빠지는 함정 (Pitfall)

⚠️
생성자에서 월드/다른 액터에 접근 — 생성자는 CDO 생성 시에도 호출됩니다. 이때 월드가 없으므로 GetWorld()는 nullptr을 반환합니다. 해결: 게임 로직은 BeginPlay()에서 시작하세요.
⚠️
GameMode를 클라이언트에서 접근 — GameMode는 서버에만 존재합니다. 클라이언트에서 GetWorld()->GetAuthGameMode()를 호출하면 항상 nullptr입니다. 해결: 클라이언트가 필요한 정보는 GameState에 두어 복제하세요.
⚠️
UPROPERTY 없이 UObject 포인터 보관 — UPROPERTY() 매크로 없이 일반 포인터로 UObject를 들고 있으면 GC가 해당 참조를 모릅니다. 해결: UObject 포인터에는 반드시 UPROPERTY()를 붙이세요.
⚠️
.Build.cs에 모듈 추가 누락 — 새 엔진 기능을 쓸 때 unresolved external symbol 링크 에러가 나면, 해당 기능이 속한 모듈을 .Build.cs의 PublicDependencyModuleNames에 추가했는지 확인하세요.
⚠️
Super:: 호출 누락 — BeginPlay(), Tick(), EndPlay()를 오버라이드할 때 Super::를 호출하지 않으면 엔진의 내부 초기화 코드가 실행되지 않아 예측 불가능한 버그가 발생합니다.

Do & Don't 요약

이렇게 하세요

  • UObject 포인터에는 항상 UPROPERTY() 붙이기
  • 게임 로직은 BeginPlay() 이후에 작성
  • 오버라이드 함수 첫 줄에서 Super:: 호출
  • 클래스 접두사 규칙(A/U/F/T/E) 준수
  • .generated.h를 #include 맨 마지막에 배치
  • 새 엔진 모듈 사용 시 .Build.cs에 추가
  • 컴포지션(컴포넌트 조합) 우선, 깊은 상속 피하기

이건 피하세요

  • UObject에 new/delete 직접 사용
  • 생성자에서 GetWorld()/FindActor 등 호출
  • 클라이언트에서 GameMode에 접근
  • std::vector/std::map 사용 (TArray/TMap 사용)
  • Tick에서 매 프레임 무거운 검색/할당 반복
  • UPROPERTY() 없이 UObject 포인터 보관

자주 묻는 질문 (FAQ)

질문답변
BlueprintCallable과 BlueprintPure의 차이는?BlueprintCallable은 실행 핀이 있고 사이드 이펙트를 가질 수 있습니다. BlueprintPure는 실행 핀 없이 값만 반환하며 상태를 변경하지 않습니다. getter/계산 함수에는 Pure, 상태 변경 함수에는 Callable을 씁니다.
BlueprintImplementableEvent vs BlueprintNativeEvent?ImplementableEvent는 C++에 구현이 없고 BP에서만 구현합니다. NativeEvent는 C++에 _Implementation 기본 구현을 제공하고 BP에서 선택적으로 오버라이드합니다. 기본 동작이 필요하면 NativeEvent, BP 전용이면 ImplementableEvent.
EditAnywhere vs EditDefaultsOnly?EditAnywhere는 BP 기본값과 레벨 인스턴스 모두에서 편집 가능. EditDefaultsOnly는 BP 에디터의 기본값만 편집 가능하고, 레벨에 배치된 개별 인스턴스에서는 편집 불가. 인스턴스별 변경이 위험한 값(무기 데미지 공식 등)에는 EditDefaultsOnly.
TSharedPtr를 UObject에 쓰면 안 되는 이유?UObject는 가비지 컬렉터(GC)가 관리합니다. TSharedPtr는 참조 카운팅 기반이므로 GC와 충돌하여 이중 해제(double free) 또는 댕글링 포인터가 발생합니다. UObject에는 UPROPERTY() 또는 TWeakObjectPtr를 사용하세요.
FName vs FString vs FText는 언제 사용?FName: 식별자, 비교 연산이 잦은 경우 (에셋 경로, 소켓 이름). FString: 문자열 조작이 필요한 경우 (로깅, 파싱). FText: UI에 표시하거나 현지화(다국어 번역)가 필요한 경우.
Server RPC에 WithValidation이 왜 필요한가?클라이언트가 보내는 RPC는 치트/해킹 가능성이 있습니다. WithValidation을 추가하면 _Validate 함수에서 입력값을 검증하여, 비정상 요청을 서버에서 거부할 수 있습니다. 보안이 중요한 게임 로직(공격, 아이템 사용 등)에 필수.
