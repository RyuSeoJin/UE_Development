C++ 클래스 생성 · 빌드
코드를 프로젝트에 넣는 법

문서의 .h/.cpp 코드를 복사하기 전에 해야 하는 일 — 클래스 파일을 어디에 어떻게 만들고, 어떤 모듈을 추가하고, 언제 Live Coding으로 되고 언제 에디터를 꺼야 하는지. 이 위키의 모든 C++ 문서가 공통으로 전제하는 반복 절차를 한 곳에 모았습니다. 각 문서의 "작업 시작하기" 블록에서 이 문서를 가리킵니다.

공통 워크플로우 UE5 · Windows/macOS 모든 C++ 문서의 선행 지식
01 — 전제 조건

시작 전에 갖춰야 할 것

항목내용확인 방법
IDE (코드 편집·빌드 도구)Windows: Visual Studio 2022 + 설치 옵션에서 "Game development with C++" 워크로드 체크. macOS: Xcode (App Store). JetBrains Rider는 양쪽 모두 대안에디터 Edit ▸ Editor Preferences ▸ Source Code에서 사용할 IDE 선택 가능
C++ 프로젝트 여부프로젝트에 Source/ 폴더가 있으면 C++ 프로젝트. 블루프린트 전용 프로젝트여도 걱정 없음 — 아래 클래스 위저드로 클래스를 1개 만드는 순간 자동으로 C++ 프로젝트로 전환됩니다프로젝트 폴더에 Source/ 존재 여부
디스크 여유첫 C++ 빌드는 중간 산출물(Intermediate/, Binaries/)로 수 GB를 씁니다—
📌

핵심 용어 — 클래스 위저드(New C++ Class): 에디터가 부모 클래스 선택 → 파일 생성 → 프로젝트 파일 갱신까지 한 번에 해 주는 마법사. C++ 파일은 가급적 항상 이 통로로 만듭니다(손으로 만들면 프로젝트 파일 재생성이 별도로 필요). / Live Coding: 에디터를 켠 채 Ctrl+Alt+F11로 변경된 C++를 패치해 넣는 기능 — 함수 본문(.cpp) 수정에만 안전. / 모듈(Module): 코드의 빌드 묶음. 남의 모듈 기능을 쓰려면 내 .Build.cs에 이름을 적어야 링크됩니다.

02 — 클래스 생성 (위저드)

문서의 클래스를 실제 파일로 만들기

위키 문서가 "UGuildSubsystem (UGameInstanceSubsystem 상속)을 만든다"라고 하면, 실제 절차는 아래 5단계입니다.

1
위저드 열기
에디터 상단 메뉴 Tools ▸ New C++ Class… (또는 Content Browser의 C++ Classes 폴더에서 우클릭 ▸ New C++ Class).
2
부모 클래스 선택
첫 화면(Common Classes)에 없으면 상단 All Classes 탭으로 전환 → 검색창에 부모 이름 입력(예: GameInstanceSubsystem, SaveGame, DeveloperSettings). 문서가 지정한 부모를 그대로 검색하면 됩니다. 구조체·열거형만 담을 헤더가 필요하면 부모를 None으로 선택합니다.
3
이름 · 위치 지정
이름은 접두어 없이 입력합니다(GuildSubsystem이라고 치면 파일은 GuildSubsystem.h/.cpp, 클래스는 UGuildSubsystem으로 자동 생성). Public/Private 선택은 단일 모듈 프로젝트면 기본값 그대로 둡니다.
4
Create Class → IDE 확인
생성하면 파일이 Source/<프로젝트이름>/ 아래에 생기고 IDE가 자동으로 열립니다. 결과 확인: .h에 UCLASS()·GENERATED_BODY()가 이미 채워진 뼈대가 보입니다.
5
문서 코드 붙여넣기
위키 문서의 코드 탭에서 복사 버튼으로 .h/.cpp 내용을 가져와 뼈대에 맞춰 붙입니다. 이때 맨 위 #include "…generated.h"는 위저드가 만든 줄을 유지하고(항상 마지막 include), 클래스 선언부와 함수들을 문서 내용으로 채웁니다.
⚠️
파일을 IDE에서 직접 만들었다면 — 언리얼은 그 파일의 존재를 모릅니다. 에디터를 끄고 프로젝트 파일 재생성(Windows: .uproject 우클릭 ▸ Generate Visual Studio project files / macOS: 에디터 Tools ▸ Refresh Xcode Project) 후 다시 빌드하세요. 위저드를 쓰면 이 단계가 필요 없습니다.
03 — 모듈 · Build.cs

"없는 헤더" 에러의 90%는 여기서 해결

프로젝트 코드는 Source/<프로젝트이름>/<프로젝트이름>.Build.cs라는 빌드 설정 파일을 가집니다. 여기의 PublicDependencyModuleNames 목록에 적힌 모듈만 include·링크할 수 있습니다. 문서 코드가 UMG 위젯·Enhanced Input 등을 쓰는데 해당 모듈이 목록에 없으면 "cannot open include file" 또는 링크 에러가 납니다.

문서 코드가 이걸 쓰면추가할 모듈
UMG 위젯 C++ 연동 (UUserWidget, BindWidget)"UMG" (+ 대부분 "Slate", "SlateCore" 함께)
Enhanced Input (UInputAction, UInputMappingContext)"EnhancedInput"
DeveloperSettings (UDeveloperSettings 상속)"DeveloperSettings"
Paper2D 스프라이트"Paper2D"
JSON 읽기/쓰기"Json", "JsonUtilities"
기본 게임플레이 (Actor·GameMode·Subsystem·SaveGame·DataTable)추가 불필요 — 기본 4종(Core, CoreUObject, Engine, InputCore)에 포함
using UnrealBuildTool;   // UBT(언리얼 빌드 도구)의 타입을 사용

public class MyProject : ModuleRules
{
	public MyProject(ReadOnlyTargetRules Target) : base(Target)
	{
		PCHUsage = PCHUsageMode.UseExplicitOrSharedPCHs;   // 기본값 유지 (컴파일 가속 설정)

		// 이 목록에 적힌 모듈만 #include · 링크 가능.
		// 앞 4개는 위저드가 만들어 주는 기본값 — 지우지 말 것.
		// 그 뒤가 문서 코드에 맞춰 "추가"하는 부분 (위 표 참조).
		PublicDependencyModuleNames.AddRange(new string[] {
			"Core", "CoreUObject", "Engine", "InputCore",
			"UMG", "Slate", "SlateCore",   // 예: UMG C++ 연동 문서용
			"EnhancedInput"                    // 예: 2D 카메라 문서의 입력용
		});
	}
}
💡

Build.cs를 고쳤다면 Live Coding으로는 반영되지 않습니다 — 아래 04의 "에디터 종료 → 재빌드" 경로를 타야 합니다.

04 — 컴파일 워크플로우

무엇을 바꿨는지가 방법을 결정한다

가장 많이 헤매는 지점입니다. 판단 기준은 단 하나 — 리플렉션(UHT가 스캔하는 것)이 변했는가.

바꾼 것방법이유
.cpp 함수 본문만 수정Live Coding — 에디터 켠 채 Ctrl+Alt+F11 (또는 에디터 우하단 ⚡ 버튼)로직 패치는 실행 중 교체 가능
헤더(.h) 수정 — UPROPERTY/UFUNCTION 추가·변경, 멤버 추가, 시그니처 변경에디터 종료 → IDE에서 빌드 → 에디터 재실행
(VS: 솔루션에서 프로젝트 우클릭 ▸ Build / Rider·Xcode: Run 대상 빌드)		리플렉션 정보(.generated.h)가 다시 생성되어야 하는데, 켜진 에디터는 옛 정보를 물고 있음 — Live Coding으로 하면 겉으론 되지만 크래시·유령 값의 원인
새 클래스 추가 (위저드 사용 포함)
Build.cs·모듈 변경
⚠️
"어제는 됐는데 오늘 이상해요"의 단골 원인 — 헤더를 바꾸고 Live Coding으로 넘어간 세션입니다. 증상이 조금이라도 이상하면(값이 안 보임, 에디터 크래시, BP에 새 변수가 안 뜸) 에디터를 완전히 끄고 IDE에서 다시 빌드하세요. 구버전 Hot Reload는 같은 이유로 아예 쓰지 않는 것이 현대 UE의 통념입니다.
💡

빌드가 성공했는지 확인하는 습관: IDE 출력 창에 Build succeeded(VS) 또는 BUILD SUCCEEDED(Xcode)가 떠야 합니다. 에러 목록의 첫 번째 에러부터 읽으세요 — 뒤따르는 수십 개는 대부분 첫 에러의 연쇄입니다.

05 — 에디터 연결 (코드 밖 작업)

빌드 성공 후, 에디터에서 마저 할 일

C++ 클래스는 컴파일만으로는 게임에 등장하지 않는 경우가 많습니다. 문서별 "작업 시작하기" 블록의 코드 외 작업이 가리키는 대표 패턴들입니다.

1
클래스가 보이는지 확인
Content Browser 우상단 Settings ▸ Show C++ Classes 체크 → 좌측 트리에 C++ Classes 폴더가 생기고 방금 만든 클래스가 보입니다. 안 보이면 빌드 실패이거나 에디터 재시작 전입니다.
2
(대부분의 문서) BP 자식 클래스 만들기
C++ Classes 폴더에서 해당 클래스 우클릭 ▸ Create Blueprint class based on … → BP_ 접두어로 저장. 디자이너가 값을 만지는 지점은 C++가 아니라 이 BP 자식입니다(문서의 UPROPERTY EditAnywhere 값들이 여기 Details에 나타남).
3
(유형별) 엔진에 등록
클래스 유형에 따라 등록 위치가 다릅니다: GameMode·PlayerController·Pawn → Project Settings ▸ Maps & Modes 또는 World Settings / Subsystem → 등록 불필요(자동) / DeveloperSettings → Project Settings에 자동 노출 / SaveGame → 코드에서 슬롯 이름으로 사용(등록 없음) / DataTable 행 구조체 → Content Browser 우클릭 ▸ Miscellaneous ▸ Data Table ▸ 행 구조체 선택으로 DT_ 애셋 생성.
4
동작 확인 루틴
PIE 실행 → 해당 문서의 "검증" 섹션 절차 수행. 아무 반응이 없으면 UE_LOG/Print String을 진입점(BeginPlay·Initialize)에 꽂아 "코드가 불리기는 하는가"부터 가릅니다.
06 — 트러블슈팅

대표 에러 → 원인 → 처방

증상 / 에러 메시지원인처방
cannot open include file · unresolved external symbol (UMG·Input 등)Build.cs에 모듈 미추가03의 표대로 모듈 추가 → 에디터 종료 후 재빌드
…generated.h: No such file 또는 generated 관련 다발 에러#include "….generated.h"가 마지막 include가 아님, 또는 파일명과 불일치generated include를 include 목록 맨 아래로
GENERATED_BODY() 관련 매크로 에러UCLASS/USTRUCT 아래 GENERATED_BODY() 누락선언 첫 줄에 추가 (위저드 뼈대를 지우지 않았는지 확인)
빌드는 성공했는데 에디터에 클래스가 안 보임① Show C++ Classes 꺼짐 ② Live Coding 세션 잔재05-1 체크 → 에디터 재시작
BP 자식/Details에 새 변수가 안 뜸헤더 변경을 Live Coding으로 넘김에디터 종료 → 재빌드 (04)
IDE에서 만든 파일을 UE가 모름프로젝트 파일 미갱신02의 Pitfall — 프로젝트 파일 재생성
에디터가 켜지지 않음 (모듈 로드 실패 팝업)바이너리와 코드 불일치팝업의 "다시 빌드" 수락 또는 IDE에서 빌드 후 실행. 최후엔 Binaries/·Intermediate/ 삭제 후 프로젝트 파일 재생성 → 풀 빌드
💡

이 문서와 각 문서의 관계 — 위키·프로젝트 문서의 "🚀 이 문서로 작업 시작하기" 블록은 그 문서 고유의 것(만들 파일 목록·코드 외 작업)만 담고, 여기 있는 공통 절차(위저드·Build.cs·컴파일 규칙)는 반복하지 않습니다. 막히면 이 문서로 돌아오세요.
