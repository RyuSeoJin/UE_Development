델리게이트
서로 모르는 시스템을 잇는 방송

"점수가 바뀌면 UI도 바꿔야 하는데, 점수 시스템이 UI를 알게 하고 싶지는 않다" — 이 모순을 푸는 언리얼의 표준 장치가 델리게이트(Delegate)입니다. 값이 바뀐 쪽이 방송하고, 필요한 쪽만 구독해서 반응합니다. 이 문서는 델리게이트 3종의 판별 기준과, 컴파일은 되는데 콜백만 조용히 안 불리는 함정을 다룹니다.

엔진 자체 기능 UE4 / UE5 공통 C++ · Blueprint
01 — 개념 · 역할

직접 호출의 문제, 방송의 해법

점수 시스템이 UI 위젯의 함수를 직접 부르면(Widget->UpdateScore(500)) 두 가지가 무너집니다. 첫째, 점수 시스템이 UI 헤더를 include해야 하므로 UI 없이는 컴파일조차 안 됩니다. 둘째, 점수를 듣고 싶은 시스템이 늘어날 때마다(사운드, 업적, 저장…) 점수 시스템의 코드를 고쳐야 합니다.

델리게이트는 방향을 뒤집습니다. 점수 시스템은 "점수가 바뀌었다"고 방송만 하고, 누가 듣는지 모릅니다. 듣고 싶은 쪽이 스스로 구독(바인딩)을 겁니다. 새 구독자가 늘어도 방송자의 코드는 한 글자도 바뀌지 않습니다.

핵심 용어 — 이 다섯 개로 문서 전체가 읽힙니다

📖

델리게이트(Delegate) — "이 시그니처의 함수를 나중에 부르겠다"는 함수 예약 슬롯. C++ 함수 포인터의 안전한 언리얼 버전입니다.

바인딩(Binding) / 구독 — 델리게이트 슬롯에 "내 함수를 불러줘"라고 등록하는 것. AddDynamic, BindUObject 등이 이 동작입니다.

브로드캐스트(Broadcast) / 방송 — 델리게이트에 등록된 모든 함수를 호출하는 것. 방송자는 등록된 함수가 0개여도, 100개여도 같은 코드를 씁니다.

시그니처(Signature) — 델리게이트가 예약하는 함수의 파라미터 형태. 선언 매크로에서 정하며, 구독 함수는 이 형태와 정확히 일치해야 합니다.

페이로드(Payload) — 방송에 실어 보내는 값(바뀐 점수, 완료된 시간 등). 구독자는 페이로드만 받고, 방송자의 내부 상태를 뒤지지 않습니다.

02 — 종류 관계 · 판별

델리게이트 3종 — 무엇을 언제 쓰나

선언 매크로 이름이 곧 종류입니다. 판별 질문은 두 개뿐입니다: "듣는 쪽이 몇 명인가?"와 "블루프린트/직렬화가 필요한가?"

종류선언 매크로듣는 쪽BP 노출구독 함수 조건대표 용도
단일DECLARE_DELEGATE[_XParams]1명✗제한 없음 (람다·raw 함수 가능)비동기 작업 1건의 완료 콜백
멀티캐스트DECLARE_MULTICAST_DELEGATE[_XParams]N명✗제한 없음C++ 내부 이벤트 (성능 우선)
동적 멀티캐스트DECLARE_DYNAMIC_MULTICAST_DELEGATE[_XParams]N명✓ (BlueprintAssignable)반드시 UFUNCTION()시스템 → UI/BP 통지 (가장 흔함)

왜 "동적"만 UFUNCTION이 필수인가 — 리플렉션과의 관계

동적(Dynamic) 델리게이트는 함수를 포인터가 아니라 이름(문자열)으로 기억합니다. 방송 시점에 리플렉션 시스템에서 그 이름으로 함수를 찾아 호출하죠. 그래서 ① 블루프린트에서 바인딩할 수 있고 ② 직렬화가 가능하지만, ③ 리플렉션에 등록되지 않은 함수(UFUNCTION() 없는 함수)는 이름으로 찾을 수 없어 호출이 조용히 실패합니다. 리플렉션 등록 원리는 위키의 리플렉션 매크로 문서에서 다룹니다.

⚠️
이 문서에서 단 하나만 기억한다면 이것 — AddDynamic으로 구독하는 콜백 함수 선언 위에는 반드시 UFUNCTION()을 붙입니다. 빠뜨려도 컴파일 에러도, 런타임 에러도 없습니다. 그냥 콜백만 영원히 안 불립니다.
03 — 책임 · 배치 규칙

누가 선언하고, 누가 구독하는가

역할담당규칙과 근거
델리게이트 선언·소유방송자값이 바뀌는 곳이 방송 출구를 가집니다. 방송자의 .h 상단에 매크로 선언, 클래스 안에 멤버 변수로 배치
Broadcast() 호출방송자값이 바뀌는 바로 그 함수 안에서 호출 — 값 변경과 방송이 떨어져 있으면 누락됩니다
구독·해제구독자구독자가 자기 초기화 시점(Initialize/BeginPlay/NativeConstruct)에 걸고, 자기 정리 시점에 풉니다. 방송자는 구독 목록을 관리하지 않습니다
BP 노출방송자멤버에 UPROPERTY(BlueprintAssignable) — 블루프린트의 "Bind Event to ..." 노드가 이걸 찾습니다
💡
화살표 방향 = 아는 방향. 구독자는 방송자를 알지만(구독하러 찾아가야 하므로), 방송자는 구독자를 모릅니다. 코드에서 이 방향이 거꾸로 보이면(방송자가 구독자 헤더를 include) 설계가 뒤집힌 것입니다.
04 — 수명 흐름

선언 → 구독 → 방송 → 해제, 4박자

1
선언 — 방송자의 .h 상단
DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(FOnScoreChanged, int32, NewScore); — 매크로가 FOnScoreChanged라는 타입을 만듭니다. 파라미터는 (타입, 이름) 쌍으로 적습니다.
2
멤버 배치 — 방송자 클래스 안
UPROPERTY(BlueprintAssignable) FOnScoreChanged OnScoreChanged; — 이 멤버가 구독자 명단을 담는 슬롯입니다.
3
구독 — 구독자의 초기화 시점
Score->OnScoreChanged.AddDynamic(this, &UMyWidget::HandleScoreChanged); — 전제: HandleScoreChanged는 시그니처가 일치하고 UFUNCTION()이 붙어 있어야 합니다. 결과: 이후의 모든 방송을 받습니다.
4
방송 — 값이 바뀌는 함수 안
Score += Amount; OnScoreChanged.Broadcast(Score); — 값 변경과 방송을 한 함수에 붙여 두면 "바꿨는데 방송을 깜빡"하는 사고가 없습니다.
5
해제 — 구독자의 정리 시점
Score->OnScoreChanged.RemoveDynamic(this, &UMyWidget::HandleScoreChanged); 또는 RemoveAll(this). 구독자가 방송자보다 먼저 죽는 구조라면 필수입니다(§06).
flowchart LR
    OWNER[방송자 — 값의 주인] -. Broadcast .-> A[구독자 A — UI 위젯]
    OWNER -. Broadcast .-> B[구독자 B — 다른 시스템]
    A -- AddDynamic 구독 --> OWNER
    B -- AddDynamic 구독 --> OWNER
05 — 구현 스켈레톤

복사해서 시작하는 최소 코드

점수 서브시스템(방송자)과 위젯(구독자)의 최소 짝입니다. 클래스 이름만 바꿔 쓰면 됩니다.

// ScoreSubsystem.h — 방송자: 선언(①) + 멤버(②) + 방송(④)
#pragma once
#include "Subsystems/GameInstanceSubsystem.h"
#include "ScoreSubsystem.generated.h"

// ① 타입 선언 — 파라미터는 (타입, 이름) 쌍. 클래스 밖, .h 상단에 둔다.
DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(FOnScoreChanged, int32, NewScore);

UCLASS()
class MYGAME_API UScoreSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()
public:
    // ② 구독자 명단 슬롯 — BlueprintAssignable로 BP의 Bind Event 노드에도 노출
    UPROPERTY(BlueprintAssignable, Category = "Score")
    FOnScoreChanged OnScoreChanged;

    UFUNCTION(BlueprintCallable, Category = "Score")
    void AddScore(int32 Amount)
    {
        Score += Amount;
        OnScoreChanged.Broadcast(Score);   // ④ 값 변경과 같은 함수에서 방송
    }

private:
    int32 Score = 0;
};
// ScoreWidget.h — 구독자: 콜백은 반드시 UFUNCTION()
private:
    UFUNCTION()   // ← 이 한 줄이 없으면 AddDynamic이 조용히 실패한다
    void HandleScoreChanged(int32 NewScore);

// ScoreWidget.cpp — ③ 구독은 구독자의 초기화 시점에
void UScoreWidget::NativeConstruct()
{
    Super::NativeConstruct();
    UScoreSubsystem* Score =
        GetGameInstance()->GetSubsystem<UScoreSubsystem>();
    Score->OnScoreChanged.AddDynamic(this, &UScoreWidget::HandleScoreChanged);
}

void UScoreWidget::NativeDestruct()
{
    // ⑤ 위젯이 먼저 죽는 구조 — 반드시 해제
    if (UGameInstance* GI = GetGameInstance())
        if (UScoreSubsystem* Score = GI->GetSubsystem<UScoreSubsystem>())
            Score->OnScoreChanged.RemoveAll(this);
    Super::NativeDestruct();
}

void UScoreWidget::HandleScoreChanged(int32 NewScore)
{
    TXT_Score->SetText(FText::AsNumber(NewScore));   // 그리기만 한다
}
06 — 확장 & 주의

조용한 실패 4형제와 방어법

⚠️
① UFUNCTION 누락 — 증상: 방송은 도는데 내 콜백만 안 불림. 에러 없음. 확인법: 콜백 선언 위에 UFUNCTION()이 있는지부터 봅니다. AddDynamic 관련 문제의 8할입니다.
⚠️
② 시그니처 불일치 — 선언이 OneParam(F..., int32, V)인데 콜백이 (int64 V)면 동적 델리게이트는 바인딩에 실패합니다(컴파일은 통과할 수 있음). 파라미터 개수·타입·순서를 선언 매크로와 글자 단위로 맞추세요.
⚠️
③ 해제 누락 — 위젯처럼 자주 생성/파괴되는 구독자가 해제를 안 하면, 죽은 객체로의 호출 시도(동적은 자동 무시되지만 명단이 쌓임)나 중복 구독이 생깁니다. 파괴 시점에 RemoveAll(this)를 습관화하세요.
⚠️
④ 중복 바인딩 — 같은 함수를 두 번 AddDynamic하면 방송 한 번에 콜백이 두 번 불립니다(재구독되는 화면에서 흔함). 반복 진입 가능성이 있는 곳에서는 AddUniqueDynamic을 쓰세요.
💡
C++ 전용 구독이면 AddUObject — 구독자가 C++뿐이고 BP 노출이 필요 없다면 비동적 멀티캐스트(DECLARE_MULTICAST_DELEGATE + AddUObject)가 더 빠르고 UFUNCTION도 필요 없습니다. 판별 기준은 §02의 표입니다.
