Table

아이템 스펙·몬스터 능력치·상점 가격처럼 자주 바뀌는 수치 데이터를 코드 밖의 표로 빼서 관리하는 언리얼의 표준 방법입니다. 구조체 한 개를 정의하면 에디터에서 엑셀처럼 편집하고, CSV로 임포트하고, 코드 한 줄로 조회할 수 있습니다.

엔진 자체 기능 UE4 · UE5 C++ · Blueprint · CSV/JSON
01 — 개념 · 역할

왜 데이터를 코드 밖으로 빼는가

검의 공격력을 Damage = 25;처럼 코드에 직접 쓰면(하드코딩), 밸런스를 조금 고칠 때마다 프로그래머가 코드를 수정하고 다시 컴파일해야 합니다. 아이템이 수백 개가 되면 이 방식은 유지가 불가능하고, 기획자는 수치 하나 바꾸는 데도 프로그래머를 거쳐야 합니다.

DataTable은 이런 반복 데이터를 행(Row) 단위의 표로 만든 에셋입니다. 표의 열(컬럼) 구성은 C++ 구조체 하나로 정의하고, 내용은 에디터에서 직접 편집하거나 CSV/JSON 파일로 임포트합니다. 결과: 수치 변경에 컴파일이 필요 없고, 기획자가 스프레드시트로 밸런싱한 것을 그대로 게임에 반영할 수 있습니다.

🔑

핵심 용어 — DataTable: 같은 구조의 행이 모인 표 형태의 에셋. · FTableRowBase: DataTable의 행이 되려면 반드시 상속해야 하는 기반 구조체. · RowName: 각 행을 식별하는 고유 이름(FName) — 표의 "기본 키". · CSV 임포트: 쉼표로 구분된 텍스트 파일(엑셀/구글시트에서 내보내기 가능)을 DataTable로 읽어 들이는 것.

구조체 · 에셋 · 코드가 연결되는 방식

DataTable은 세 조각의 협업입니다: 구조체(형태 정의) → DataTable 에셋(내용 보관) → 조회 코드(사용).

⚠️

DataTable은 런타임 읽기 전용으로 쓰는 것이 원칙입니다. 플레이 중 값이 변해야 하는 데이터(현재 체력 등)는 DataTable이 아니라 액터·Subsystem의 변수이며, DataTable에는 그 초기값·정의만 둡니다.

03 — 데이터 보관 방식 선택

하드코딩 · DataTable · DataAsset, 언제 무엇을

DataTable

같은 구조 × 많은 행CSV 밸런싱

아이템 300개, 몬스터 80종처럼 동일한 형태가 반복되는 데이터. 스프레드시트에서 일괄 편집·임포트하는 워크플로우가 강점.

DataAsset

구조가 다른 낱개에셋 참조 중심

캐릭터 클래스 정의, 무기 설정처럼 하나하나가 개별 에셋인 데이터. 행 단위 표가 아니라 상속·에셋 참조가 필요할 때.

데이터 성격보관 방식왜
수백 행의 동일 구조 스펙 (아이템·몬스터·드롭률)UDataTable행 반복 구조 + CSV 일괄 편집. 이 문서의 주제
레벨·시간에 따라 연속으로 변하는 수치 곡선 (성장 곡선)UCurveTable / UCurveFloat이산 행이 아닌 보간 곡선이 필요 — 레벨 17.5의 값도 계산 가능
구조가 제각각인 설정 덩어리 (게임 모드 구성, 무기 정의)UPrimaryDataAsset표에 안 맞는 개별 에셋. 에셋 매니저로 비동기 로드 가능
거의 안 바뀌는 상수 한두 개코드 상수 / Config표를 만들 관리 비용이 더 큼 — 다만 밸런싱 대상이 되는 순간 DataTable로 승격
04 — 제작 · 임포트 절차

구조체 정의부터 조회까지 4단계

전제: C++ 프로젝트(구조체 정의용), 에디터 사용 가능. 각 단계의 끝에 확인해야 할 결과를 함께 적었습니다.

01
행 구조체 정의 (.h)
FTableRowBase를 상속한 USTRUCT(BlueprintType) 구조체를 만들고 컴파일합니다(05 섹션 코드). 결과 확인: 에디터 재시작 후 Row Type 목록에 구조체 이름이 나타남.
02
DataTable 에셋 생성
Content Browser에서 우클릭 ▸ Miscellaneous ▸ Data Table → "Pick Row Structure" 창에서 ItemRow 선택 → 이름을 DT_Items처럼 지정. 결과 확인: 더블클릭하면 표 편집기가 열림.
03
내용 채우기 — 에디터 또는 CSV
(방법 A) 표 편집기 상단 + Add 버튼으로 행 추가, RowName과 값을 직접 입력. (방법 B) 첫 열이 RowName인 CSV를 만들어 Content Browser로 드래그&드롭 → 임포트 창에서 Row Type 선택. 결과 확인: 행들이 표에 표시됨. CSV 재임포트는 에셋 우클릭 ▸ Reimport.
04
코드/BP에서 조회
C++은 FindRow<FItemRow>()(05 섹션), BP는 Get Data Table Row 노드 사용. 결과 확인: 존재하는 RowName으로 조회 시 값이, 없는 이름으로는 null/실패 핀이 반환됨.
⚠️
에디터 수정과 CSV 재임포트는 충돌합니다 — 표 편집기에서 고친 내용은 CSV 원본에 반영되지 않으므로, Reimport하면 에디터 수정분이 덮어써져 사라집니다. 팀에서 "진실은 CSV" 또는 "진실은 에셋" 중 하나를 정하고 한쪽으로만 편집하세요.

구글 스프레드시트 연동 — 시트를 데이터의 원본으로

언리얼에는 구글 스프레드시트에 직접 접속하는 내장 기능이 없습니다. 표준 워크플로우는 시트를 데이터의 원본(진실)으로 삼고, CSV로 내려받아 임포트/Reimport하는 것입니다 — 위 3단계의 "방법 B"를 시트 기준으로 확장한 흐름입니다. 기획자는 시트에서만 편집하고, 엔진 쪽에서는 편집하지 않는 것이 원칙입니다(위 충돌 함정 참조).

01
시트 작성 — 아래 "작성 규칙"대로
시트 탭 1개가 DataTable 1개에 대응하도록 만듭니다(탭 이름 = 에셋 이름, 예: DT_Items). 1행은 헤더(구조체 필드 이름), A열은 RowName. 결과 확인: 헤더가 05 섹션 구조체의 변수 이름과 글자 단위로 일치.
02
CSV 다운로드
구글 시트 메뉴에서 파일 ▸ 다운로드 ▸ 쉼표로 구분된 값(.csv). 현재 열려 있는 탭 1개만 내려받아지므로, 테이블이 여러 개면 탭마다 반복합니다. 파일 이름은 에셋과 같게(DT_Items.csv) 정리해 두면 추적이 쉽습니다. 구글 시트의 CSV는 항상 UTF-8이라 한글이 깨지지 않습니다.
03
최초 임포트 → 이후엔 Reimport
처음 1회는 CSV를 Content Browser로 드래그&드롭하고 Row Type(ItemRow)을 지정합니다. 이후 갱신은 에셋 우클릭 ▸ Reimport — 처음 임포트했던 경로의 CSV를 다시 읽으므로, 내려받은 새 CSV를 같은 경로·같은 이름으로 덮어쓴 뒤 Reimport합니다. 결과 확인: Output Log에 열 이름 불일치 등 임포트 경고가 없어야 합니다.
04
검증
표 편집기를 열어 행 수가 시트와 같은지, enum·에셋 경로 열이 기본값으로 채워지지 않았는지 확인합니다(형식이 틀린 셀은 에러 대신 조용히 기본값이 되는 경우가 많습니다). 결과 확인: 플레이 없이 에셋만 열어도 확인 가능.
💡

다운로드까지 자동화하고 싶다면 — 구글 시트의 파일 ▸ 공유 ▸ 웹에 게시에서 특정 탭을 CSV 형식으로 게시하면 내용이 갱신되는 고정 URL이 생깁니다. 에디터 파이썬 스크립트나 Editor Utility Widget으로 이 URL을 내려받아 Reimport까지 묶으면 버튼 한 번으로 동기화됩니다(고급 주제라 이 문서에서는 개요만). 주의: 게시된 URL은 링크를 아는 누구나 볼 수 있으므로, 공개 전 프로젝트의 밸런스 데이터라면 게시 범위를 신중히 정하세요.

CSV 작성 규칙 — 임포터가 요구하는 형식 (필수)

DataTable 임포터는 기대하는 형식이 정해져 있어서, 아래 규칙은 취향이 아니라 지키지 않으면 임포트가 깨지는 필수 사항입니다. 05 섹션의 FItemRow(DisplayName·Damage·Icon)에 맞춘 예시입니다.

---,DisplayName,Damage,Icon
sword_iron,철검,25,/Game/UI/Icons/T_SwordIron.T_SwordIron
bow_short,단궁,14,/Game/UI/Icons/T_BowShort.T_BowShort
potion_hp,체력 물약,0,
규칙내용
1행 = 헤더첫 칸의 내용은 임포터가 무시합니다(관례상 --- 또는 Name). 나머지 칸은 행 구조체 필드의 C++ 변수 이름과 정확히 일치해야 합니다(대소문자 구분, 에디터 표시 이름 아님). 일치하지 않는 열은 경고와 함께 무시됩니다.
첫 열 = RowName행마다 고유한 이름이어야 합니다. 중복되면 하나만 남고 나머지는 덮어써집니다.
빈 칸구조체에 선언해 둔 기본값으로 채워집니다(예: int32 Damage = 0이면 0). 에러가 나지 않으므로 실수를 눈치채기 어렵습니다.
쉼표·줄바꿈 포함 값값 전체를 큰따옴표로 감쌉니다 — 구글 시트가 다운로드 시 자동으로 처리해 주므로 시트에서는 그냥 입력하면 됩니다.
인코딩UTF-8이어야 한글이 깨지지 않습니다. 구글 시트 다운로드는 항상 UTF-8이라 안전합니다.

셀에 값을 적는 방법은 필드 타입마다 다릅니다. 특히 에셋 참조·구조체·배열은 형식이 틀리면 에러 없이 기본값이 되니 아래 표기를 그대로 따르세요.

필드 타입셀 표기주의
int32 / float25 · 3.5천 단위 쉼표 금지 — 1,500이라 쓰면 열이 둘로 갈라집니다. 시트 서식도 "자동/일반"으로.
boolTrue / False구글 시트 체크박스 열도 CSV로는 TRUE/FALSE로 내려받아져 그대로 동작합니다.
FString / FName / FText철검그대로 입력. 쉼표가 들어가면 따옴표(시트가 자동 처리).
UENUM 열거형Sword열거자 이름 그대로(EWeaponType::Sword처럼 타입 접두사 붙이지 않음). 오타는 기본값(첫 항목)이 됩니다.
에셋 참조 (TSoftObjectPtr 등)/Game/UI/Icons/T_Sword.T_Sword오브젝트 경로 — Content Browser에서 에셋 우클릭 ▸ Copy Reference로 복사한 값을 붙여넣으면 됩니다.
구조체 필드 (FVector 등)"(X=0,Y=0,Z=100)"괄호+이름=값 형식. 쉼표가 들어가므로 따옴표 필수(시트에서는 자동).
배열 (TArray)"(1,2,3)" · "((X=1,Y=2),(X=3,Y=4))"괄호로 감싼 목록. 구조체 배열은 괄호가 이중.
⚠️
엑셀로 CSV를 저장한다면 "CSV UTF-8" 형식을 선택하세요 — 엑셀의 기본 "CSV(쉼표로 분리)"는 한국어 Windows에서 CP949 인코딩으로 저장되어, 임포트하면 한글이 전부 깨집니다. 구글 시트에서 내려받으면 이 문제가 없습니다.

시트 작성 규칙 — 팀 컨벤션 제안

위 형식 규칙만 지키면 임포트는 됩니다. 하지만 여러 명이 시트를 몇 달씩 운영하다 보면 "임포트는 되는데 데이터가 조용히 잘못된" 사고가 생깁니다. 아래는 그 사고를 막기 위한 운영 컨벤션 제안입니다 — 팀 상황에 맞게 골라 쓰되, 정했으면 시트 첫 탭에 적어두고 전원이 따르는 것이 핵심입니다.

규칙이유
탭 1개 = DataTable 1개, 탭 이름 = 에셋 이름(DT_Items)시트 탭 ↔ CSV 파일 ↔ 에셋이 1:1로 대응해 "이 데이터가 어디로 가는지" 추적이 끊기지 않습니다.
1행(헤더)과 A열(RowName)은 틀 고정(보기 ▸ 고정)스크롤해도 기준이 보이고, 헤더 행을 실수로 밀어내는 사고를 막습니다.
RowName은 소문자_밑줄 + 분류 접두사(sword_iron, potion_hp)정렬하면 분류별로 묶이고, 띄어쓰기·대소문자 변형으로 인한 오타 여지가 줄어듭니다.
한 번 배포된 RowName은 변경·재사용 금지세이브 파일·코드·BP가 이름으로 행을 참조합니다. 이름을 바꾸면 컴파일 에러 없이 FindRow가 nullptr를 돌려주며 조용히 깨집니다. 폐기할 행은 삭제 대신 "사용 안 함" 표시 열로 관리하는 방법도 있습니다.
메모·계산 보조는 별도 탭에 — 내보내는 탭은 구조체와 1:1구조체에 없는 여분 열은 임포트 경고를 만들고, 병합 셀은 CSV로 내려받으면 형태가 깨집니다. 수식 자체는 사용해도 됩니다(CSV에는 계산 결과 값이 저장됨).
enum 열은 데이터 확인(드롭다운)으로 입력 제한삽입 ▸ 드롭다운으로 열거자 이름만 고르게 하면, "오타 → 조용히 기본값" 사고를 입력 단계에서 차단합니다.
변경 기록 탭 운영 (날짜 · 수정자 · 내용 · 반영 여부)"시트는 고쳤는데 Reimport를 안 한" 누락을 잡는 가장 값싼 장치입니다.

✓ 안전한 습관

  • 편집은 시트에서만 — 엔진 표 편집기는 열람용
  • Reimport 직후 행 수·enum 열을 눈으로 확인
  • RowName·enum은 드롭다운/규칙으로 입력 제한
  • 내려받은 CSV는 항상 같은 경로에 덮어쓰기

✗ 임포트를 조용히 깨뜨리는 실수

  • 숫자에 천 단위 쉼표(1,500) — 열이 갈라짐
  • 병합 셀 · 구조체에 없는 메모 열
  • 배포된 RowName 변경 — FindRow가 nullptr 반환
  • 시트와 에디터 양쪽에서 동시 수정
05 — 구현 스켈레톤

행 구조체 정의와 조회 코드

MYGAME_API는 자기 모듈 이름으로 바꾸세요 — 무엇이고 이름이 어떻게 정해지는지는 코드 아래 "스켈레톤 해부"에서 설명합니다. 조회 예시는 임의 액터 클래스 안의 함수로 작성했습니다.

#pragma once

#include "CoreMinimal.h"
#include "Engine/DataTable.h"       // FTableRowBase가 여기 있음
#include "ItemRow.generated.h"       // 반드시 마지막 #include

// DataTable의 행 = 이 구조체 하나. 필드가 곧 표의 열이 됩니다.
USTRUCT(BlueprintType)
struct MYGAME_API FItemRow : public FTableRowBase
{
    GENERATED_BODY()

    // 화면 표시용 이름 (FText = 현지화 대상 문자열)
    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Item")
    FText DisplayName;

    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Item")
    int32 Damage = 0;

    // 아이콘은 소프트 참조: 표를 로드해도 텍스처까지 전부 로드되지 않게
    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Item")
    TSoftObjectPtr<UTexture2D> Icon;
};
#include "ItemRow.h"
#include "Engine/DataTable.h"

// 헤더에 선언해 둔 DataTable 참조.
// UPROPERTY(EditDefaultsOnly, Category="Item") UDataTable* ItemTable;
// → 에디터에서 BP 디폴트로 DT_Items 에셋을 지정해 둡니다.

int32 AMyShop::GetItemDamage(FName RowName) const
{
    if (!ItemTable) { return 0; }   // 테이블 미지정 방어

    // 두 번째 인자(ContextString)는 실패 로그에 찍힐 위치 표시용
    const FItemRow* Row =
        ItemTable->FindRow<FItemRow>(RowName, TEXT("GetItemDamage"));

    // RowName 오타·행 삭제 시 nullptr — 반드시 체크
    if (!Row)
    {
        UE_LOG(LogTemp, Warning, TEXT("Item row not found: %s"), *RowName.ToString());
        return 0;
    }
    return Row->Damage;
}

// 전체 행 순회가 필요할 때 (도감·상점 목록 등)
void AMyShop::ListAllItems() const
{
    if (!ItemTable) { return; }

    TArray<FItemRow*> Rows;
    ItemTable->GetAllRows<FItemRow>(TEXT("ListAllItems"), Rows);

    for (const FItemRow* Row : Rows)
    {
        UE_LOG(LogTemp, Log, TEXT("%s (Damage %d)"), *Row->DisplayName.ToString(), Row->Damage);
    }
}

스켈레톤 해부 ① — 헤더 첫 줄들: #include는 에셋이 아니라 코드를 가져온다

위 ItemRow.h의 첫 세 줄은 어떤 DataTable 행 구조체를 만들든 똑같이 반복됩니다. 한 번만 정확히 이해해 두면 됩니다.

줄정체왜 필요한가
#include "CoreMinimal.h"언리얼 코드의 표준 시작 헤더 — FString·FName·TArray·FVector·UE_LOG 등 어디서나 쓰는 코어 타입 모음사실상 모든 언리얼 헤더의 첫 줄 관례. 이름 그대로 "최소한의 코어"만 들어 있어 AActor·UDataTable 같은 엔진 클래스는 여기 없음 — 그래서 다음 줄이 따로 필요
#include "Engine/DataTable.h"UDataTable·FTableRowBase 클래스가 선언된 엔진 소스 헤더FTableRowBase를 상속하려면 그 선언이 보여야 함. 경로는 엔진 모듈 소스 기준이라 어느 프로젝트에서든 항상 이 문자열 그대로 — 내 에셋의 위치와 무관
#include "ItemRow.generated.h"UHT가 자동 생성하는 리플렉션 코드 헤더USTRUCT가 있는 헤더의 필수 요소이며 반드시 마지막 #include여야 함 (상세: 리플렉션 매크로 문서)
🔑

#include는 "코드(타입 선언)"를 가져오는 것이지, 내 에셋을 가져오는 게 아닙니다. DT_Items 에셋을 Content 폴더 어디에 두든, 심지어 프로젝트가 달라져도 #include "Engine/DataTable.h"는 변하지 않습니다. 내 에셋과 코드가 실제로 연결되는 지점은 include가 아니라 UPROPERTY로 선언한 UDataTable* 변수에 에디터에서 에셋을 지정하는 순간입니다(위 ItemQuery.cpp 상단 주석의 ItemTable이 그 변수).

스켈레톤 해부 ② — 선언부 한 줄의 네 조각

struct MYGAME_API FItemRow : public FTableRowBase는 네 조각으로 나뉩니다. 실제로 여러분이 바꿔 쓸 부분은 ②(자기 모듈의 매크로)와 ③(구조체 이름) 두 곳뿐이고, 나머지 골격은 어떤 행 구조체든 동일합니다.

조각뜻
① structC++ 구조체 선언. 언리얼 관례로 동작(함수) 중심이면 class, 데이터 묶음이면 struct — DataTable 행은 순수 데이터라 struct
② MYGAME_API모듈 API 매크로 — 이 타입을 내 모듈(DLL) 바깥에서도 쓸 수 있게 공개한다는 표시. 바로 아래에서 상세히
③ FItemRow우리가 짓는 이름. 언리얼 구조체는 F 접두사 필수. 에디터의 Row Type 목록에는 F를 뗀 ItemRow로 표시됨
④ : public FTableRowBase"FTableRowBase의 자식"이라는 상속 선언 — DataTable 행이 되기 위한 필수 자격. 빼먹으면 컴파일은 되지만 에셋 생성 시 Row Type 목록(제작 절차 02단계)에 아예 나타나지 않음

모듈 API 매크로 — 여기서 "API"는 데이터가 아니라 C++ 심볼 공개

이름 때문에 오해하기 쉽지만, 이 매크로의 API는 웹 API나 데이터 접근과 무관합니다. "이 모듈이 바깥 코드에 공개하는 프로그래밍 접점"이라는 원래 의미의 API입니다. 언리얼 프로젝트는 모듈 단위로 컴파일되고 에디터 빌드에서 각 모듈은 별도 DLL이 되는데, DLL 안의 타입은 기본적으로 바깥에서 링크할 수 없습니다. 이 매크로가 컴파일 시 __declspec(dllexport/dllimport) 지시어로 바뀌면서 "이 타입은 DLL 경계 밖에서도 링크 가능"하다는 도장을 찍어 줍니다. 행 데이터가 .uasset으로 저장되는 것(런타임)과 타입 정의 코드가 모듈 밖에 공개되는 것(컴파일/링크 타임)은 전혀 다른 층의 이야기입니다.

질문답
이름은 어떻게 정해지나?<모듈 이름 대문자>_API — 정확히는 프로젝트가 아니라 그 헤더 파일이 속한 모듈 기준. 기본 프로젝트는 프로젝트와 같은 이름의 모듈 1개로 시작하므로 사실상 프로젝트 이름과 같음 (예: 프로젝트 RecordTales → RECORDTALES_API). 모듈을 추가하면 그 모듈의 코드는 그 모듈 이름을 따름
내가 정의해야 하나?아니요 — UBT(언리얼 빌드 툴)가 모듈 컴파일 시 자동 정의합니다. 마음대로 지을 수 없고, 철자가 다르면 정의된 적 없는 식별자가 되어 그 자리에서 즉시 컴파일 에러
안 붙이면 어떻게 되나?모듈이 하나뿐인 동안은 정상 동작. 하지만 나중에 다른 모듈(예: 별도 UI 모듈)이 이 타입을 쓰는 순간 링크 에러(unresolved external symbol). 그래서 언리얼 코드베이스 전체가 "일단 붙여두는" 관례를 따름
💡
직접 타이핑하지 마세요 — 프로젝트 생성 시 자동으로 만들어진 기존 클래스 헤더를 열어 거기 쓰인 매크로를 그대로 복사하거나, 에디터의 Tools ▸ New C++ Class로 클래스를 만들면 올바른 매크로가 자동으로 들어갑니다. 오타 여지 자체를 없애는 것이 가장 안전합니다.
