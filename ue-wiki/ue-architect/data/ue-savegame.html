SaveGame
디스크에 살아남는 데이터

앱을 껐다 켜도 남아야 하는 값(재화·설정·진행도)을 파일로 보존하는 언리얼의 표준 장치가 USaveGame입니다. 이 문서는 "어떤 멤버가 저장되고 어떤 멤버는 조용히 사라지는가"의 규칙, 저장/로드 왕복 절차, 그리고 세이브 구조가 바뀌었을 때 기존 유저의 데이터를 살리는 SaveVersion 마이그레이션을 다룹니다.

엔진 자체 기능 UE4 / UE5 공통 C++ · Blueprint
01 — 개념 · 역할

메모리는 휘발한다 — 살릴 값을 고르는 일

게임의 모든 변수는 메모리에 살고, 프로세스가 끝나면 전부 사라집니다. USaveGame은 그중 살아남아야 할 값만 골라 담는 전용 클래스입니다. 이 클래스의 객체를 통째로 파일로 바꿔 쓰고(저장), 파일에서 객체로 되살리는(로드) 과정이 전부입니다.

핵심 용어

📖

직렬화(Serialization) — 메모리 속 객체를 바이트 줄(파일)로 바꿔 쓰는 것. 역방향(파일 → 객체)은 역직렬화입니다. SaveGame에서는 엔진이 자동으로 해 주지만, 무엇을 직렬화할지는 UPROPERTY()로 표시한 멤버만입니다.

슬롯(Slot) — 세이브 파일 하나의 주소. 슬롯 이름(문자열) + UserIndex(정수) 조합이며, 프로젝트의 Saved/SaveGames/<슬롯이름>.sav로 저장됩니다(패키징 후에는 플랫폼별 사용자 데이터 폴더).

마이그레이션(Migration) — 세이브 구조가 바뀐 뒤에도 구버전 파일을 읽어 새 구조로 옮기는 작업. 버전 필드(SaveVersion)로 분기합니다.

붙은 멤버만 저장

💡

왜 소유자가 GameInstance인가 — 세이브 객체는 레벨이 바뀌어도 유지되어야 하고, 어느 시스템보다 먼저 로드되어야 합니다. 앱과 수명이 같은 유일한 표준 객체가 GameInstance입니다(상세: 위키 GameInstanceSubsystem 문서). 각 시스템은 소유자를 통해 자기 몫의 필드만 읽고 씁니다.

03 — 무엇을 저장하나

저장되는 것, 안 되는 것, 하면 안 되는 것

판별 기준은 세 줄입니다: ① UPROPERTY()가 붙어 있는가 ② 값 타입인가 ③ 다시 계산할 수 없는가. 셋 다 "예"일 때만 SaveGame 필드가 됩니다.

04 — 저장/로드 왕복

왕복 한 바퀴와 버전 분기

저장 — 값이 바뀔 때마다

1
밀어넣기
런타임 값의 주인(각 시스템)이 자기 몫의 SaveGame 필드를 갱신합니다. 전제: 세이브 객체가 이미 로드되어 있음.
2
디스크 쓰기
UGameplayStatics::SaveGameToSlot(SaveObj, TEXT("MySlot"), 0). 결과: .sav 파일 갱신. 반환값(bool)으로 성공을 확인합니다.

로드 — 앱 시작 시 1회

1
존재 확인
DoesSaveGameExist("MySlot", 0) — 없으면 CreateSaveGameObject로 새 객체를 만들고 기본값을 채웁니다(첫 실행).
2
읽기 + 캐스팅
Cast<UMySaveGame>(LoadGameFromSlot("MySlot", 0)) — 반환 타입이 USaveGame*라 반드시 내 타입으로 캐스팅합니다.
3
버전 분기 (마이그레이션)
if (Save->SaveVersion < 최신) Migrate(Save) — 구버전 필드를 새 필드로 옮기고 버전을 올립니다. 결과: 이후 코드는 항상 최신 구조만 상대합니다.
4
복원
각 시스템이 자기 몫의 필드를 런타임 멤버로 복사해 갑니다. 결과: 게임이 어제 상태로 이어집니다.
flowchart LR
    SYS[각 시스템 — 런타임 값의 주인] -- 값 밀어넣기 --> OBJ[UMySaveGame 객체]
    OBJ -- SaveGameToSlot --> FILE[(슬롯.sav 파일)]
    FILE -- LoadGameFromSlot --> OBJ2[UMySaveGame 객체]
    OBJ2 -- SaveVersion 분기 --> MIG{구버전인가?}
    MIG -- 예 --> M[Migrate: 구 필드 → 새 필드]
    MIG -- 아니오 --> RES[각 시스템으로 값 복원]
    M --> RES
05 — 구현 스켈레톤

세이브 클래스 + 소유자 왕복 코드

🚀 이 문서로 작업 시작하기

🧭

전제 조건 — 단독 실습 가능합니다. 위저드·Build.cs·컴파일의 공통 절차는 위키 『C++ 클래스 생성 · 빌드 워크플로우』(core-cpp)를 따릅니다.

만들 파일 / 작업 대상부모 클래스 (위저드 검색어)비고
MySaveGame.hSaveGame디스크에 남길 필드만 UPROPERTY로
저장·로드 함수새 파일 없음소유자(Subsystem 등) .cpp에 추가
💡

코드 외 작업 — 없음 — 애셋·설정 등록이 필요 없습니다(코드의 슬롯 이름으로 저장/로드). 완료 확인: 저장 후 프로젝트 폴더 Saved/SaveGames/에 .sav 파일 생성.

⚠️
컴파일 주의 — UPROPERTY(직렬화 대상) 변경은 재빌드. 필드 구조를 바꾸면 구버전 .sav 호환(마이그레이션, 본문 참조)을 함께 검토하세요.
// MySaveGame.h — 데이터 그릇. 로직을 넣지 않는다.
#pragma once   // 같은 헤더의 중복 include 방지 — 언리얼 헤더 공통 첫 줄
#include "GameFramework/SaveGame.h"   // 부모 클래스 USaveGame의 선언이 여기 있음
#include "MySaveGame.generated.h"   // UHT 자동 생성 헤더 — 반드시 마지막 #include

// UCLASS: 리플렉션 등록 — 이 등록이 있어야 아래 UPROPERTY 필드가 직렬화(저장) 대상이 됨
UCLASS()
class MYGAME_API UMySaveGame : public USaveGame   // USaveGame 상속 = 슬롯 저장 대상이 되기 위한 자격
{
    GENERATED_BODY()   // UHT가 만든 리플렉션 코드 삽입 지점 — UCLASS 본문 필수
public:
    // 구조를 바꾸면 이 값을 +1 하고 로드에서 분기한다
    // (= 1 초기값이 없으면 구버전 판별 기준 자체가 무너짐 — 첫 배포 버전부터 반드시 포함)
    UPROPERTY() int32 SaveVersion = 1;

    // 저장할 값들 — UPROPERTY()만 붙이면 에디터 노출 없이 직렬화에만 포함됨
    UPROPERTY() int64 Gold = 0;   // int64: int32 상한(약 21억) 초과 대비 — 저장 필드는 나중에 좁히기 어려움
    UPROPERTY() TArray<FName> UnlockedIds;   // 해금 항목 ID 목록 — 컨테이너(TArray)도 통째로 저장됨
    UPROPERTY() TMap<FString, int32> DailyCounters;   // 키-값 카운터 — TMap도 지원 타입

    // UPROPERTY가 없으면 이 멤버는 절대 저장되지 않는다 (조용한 무시)
    // int32 ThisWillBeLost;
};
// MyGameInstance.cpp — 왕복은 소유자 한 곳에서만
#include "Kismet/GameplayStatics.h"   // SaveGameToSlot 등 저장/로드 정적 함수가 여기 있음

// 슬롯 이름은 상수 한 곳에서만 — 문자열을 여기저기 직접 쓰면 오타 하나로 슬롯이 갈라짐
static const TCHAR* kSlot = TEXT("MySlot");

// 게임 시작 시 1회 호출(예: GameInstance::Init에서) —
// 흐름: 1) 슬롯 파일 존재 확인 → 2) 로드+캐스팅+마이그레이션 / 3) 없으면 새 객체 생성
void UMyGameInstance::LoadOrCreate()
{
    if (UGameplayStatics::DoesSaveGameExist(kSlot, 0))   // 1) .sav 파일 존재 확인 — 두 번째 인자 UserIndex는 PC 단일 사용자면 0 고정
    {
        // 2) LoadGameFromSlot의 반환 타입은 부모(USaveGame*) — 내 타입으로 캐스팅 필수.
        //    파일이 손상되었거나 다른 클래스로 저장된 경우 Cast가 nullptr를 반환
        CurrentSave = Cast<UMySaveGame>(
            UGameplayStatics::LoadGameFromSlot(kSlot, 0));
        MigrateIfNeeded(CurrentSave);   // 버전 분기
    }
    else
    {
        // 3) 첫 실행 — 메모리에만 빈 세이브 객체를 만든다 (이 시점엔 디스크에 아무것도 없음)
        CurrentSave = Cast<UMySaveGame>(
            UGameplayStatics::CreateSaveGameObject(UMySaveGame::StaticClass()));
        // 첫 실행 기본값은 여기서 채운다 (Settings 상수 권장)
    }
}

// 값이 바뀔 때마다 호출("변경 즉시 저장" 정책) — 저장 시점을 이 함수 하나로 모은다
void UMyGameInstance::SaveNow()
{
    if (!CurrentSave) return;   // LoadOrCreate 전에 불리면 nullptr — 이 방어가 없으면 아래에서 크래시
    // 객체의 UPROPERTY 필드 전부를 .sav 파일로 직렬화 — 동기 호출(끝날 때까지 블로킹)
    const bool bOk = UGameplayStatics::SaveGameToSlot(CurrentSave, kSlot, 0);
    if (!bOk) { /* 로그 — 디스크 부족/권한 등 */ }   // 실패해도 예외가 없으므로 반환값 확인이 유일한 감지 수단
}

// 로드 직후 1회 호출 — 구버전 세이브를 최신 구조로 끌어올린다.
// 이후 코드는 항상 최신 구조만 상대하면 되므로 분기가 여기 한 곳에 격리됨
void UMyGameInstance::MigrateIfNeeded(UMySaveGame* S)
{
    if (S->SaveVersion < 2)   // 버전 비교로 구버전만 분기 — 최신 세이브는 그대로 통과
    {
        // 예: v1의 int32 골드를 v2의 int64 필드로 이전
        // S->Gold = S->OldGold_DEPRECATED;
        S->SaveVersion = 2;
        SaveNow();   // 이전 결과를 즉시 굳힌다
    }
}
06 — 확장 & 주의

말없이 데이터를 잃는 함정들

⚠️
① UPROPERTY 누락 — 증상: 저장·로드 모두 "성공"하는데 특정 필드만 항상 기본값. 새 필드를 추가할 때 매크로를 빠뜨린 경우가 대부분입니다. 필드 추가 = UPROPERTY()까지가 한 세트라고 기억하세요.
⚠️
② 구버전 필드 삭제 — 필드를 지우면 구버전 .sav의 그 값은 로드 시 조용히 버려집니다. 마이그레이션이 끝나기 전까지는 옛 필드를 _DEPRECATED로 남겨 두고, 이전 완료 후에 지우세요.
⚠️
③ 테스트 세이브 오염 — 개발 중 구조를 바꾸면 옛 .sav가 이상 동작을 만듭니다. 프로젝트의 Saved/SaveGames/ 폴더에서 파일을 지우면 첫 실행 상태로 돌아갑니다 — "세이브 지우고 재현되나?"를 디버깅 1순위 질문으로.
💡
비동기 저장 — 데이터가 커져 프레임이 튀면 AsyncSaveGameToSlot(완료 콜백 델리게이트 제공)으로 바꿉니다. 단, "변경 즉시 저장" 정책에서는 대부분 동기 저장으로 충분합니다.
💡
슬롯 전략 — 슬롯 이름을 바꾸면 사실상 별도 세이브가 됩니다(수동 슬롯 여러 개, 자동 저장 슬롯 분리 등). 슬롯 이름은 한 곳(상수)에서만 정의하세요.
