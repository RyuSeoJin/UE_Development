DeveloperSettings
코드 밖에서 조정하는 밸런스 상수

"가챠 비용을 1,000에서 800으로 바꿔볼까?"가 코드 수정 + 리컴파일이 되는 순간 밸런싱은 멈춥니다. UDeveloperSettings는 C++ 클래스 하나로 Project Settings 메뉴에 자동 노출되는 설정 페이지를 만들어, 상수를 에디터에서 조정하고 .ini 파일로 관리하게 해 줍니다. 행 하나짜리 값을 DataTable로 만들지 않아도 되는 이유이기도 합니다.

엔진 자체 기능 UE4 / UE5 공통 C++ · 에디터
01 — 개념 · 역할

ini 계층 위에 얹힌 자동 설정 페이지

언리얼의 모든 설정은 결국 텍스트 파일(.ini)에 삽니다. UDeveloperSettings는 "내 클래스의 Config 프로퍼티들을 특정 ini 파일과 자동으로 동기화하고, 편집 UI(Project Settings 페이지)까지 만들어 주는" 편의 기반 클래스입니다.

핵심 용어

📖

Config 시스템 — 프로퍼티 값을 .ini 파일에서 읽고 쓰는 엔진 장치. UPROPERTY(Config)가 붙은 프로퍼티만 참여합니다.

ini 계층 — 같은 설정 파일이 여러 층으로 겹칩니다: 엔진 기본(Base*.ini) → 프로젝트 기본(Config/Default*.ini, 버전 관리 대상) → 로컬 사용자(Saved/Config/). 나중 층이 앞 층을 덮어씁니다.

CDO(Class Default Object) — 클래스마다 엔진이 만들어 두는 기본값 원본 객체. Settings 값을 읽는 GetDefault<T>()는 이 CDO를 돌려주는 것입니다 — 인스턴스를 새로 만드는 것이 아닙니다.

UDeveloperSettings — Config 시스템 + Project Settings UI 자동 등록을 묶은 기반 클래스. 파생 클래스를 만들기만 하면 에디터 메뉴에 페이지가 생깁니다.

03 — 보관처 경계

Settings인가, SaveGame인가, DataTable인가

값 하나가 생겼을 때 세 보관처 중 어디로 보낼지 — 이 경계가 흐려지면 데이터 설계 전체가 무너집니다.

질문보관처예
기획자가 정하는 게임 전체 공통 상수인가? (유저마다 다르지 않음)DeveloperSettings가챠 비용, 보상 배율, 타이머 기본값
유저마다 다르고 껐다 켜도 남아야 하는가?SaveGame보유 재화, 유저가 바꾼 설정값
같은 구조의 행이 여러 개인 콘텐츠인가?DataTable캐릭터 목록, 레벨 테이블
💡
"기본값은 Settings, 유저의 변경본은 SaveGame" — 새 유저의 타이머 기본 25분은 Settings 상수, 유저가 30분으로 바꾼 값은 SaveGame 필드입니다. 첫 실행 때 Settings → SaveGame으로 복사하는 한 방향 흐름만 허용하면 두 보관처가 충돌하지 않습니다.
04 — 사용 흐름

클래스 작성 → 에디터 조정 → 코드에서 읽기

1
클래스 선언
UCLASS(Config=Game, DefaultConfig)로 파생 클래스를 만듭니다. Config=Game = "DefaultGame.ini에 저장", DefaultConfig = "로컬 층이 아니라 프로젝트 기본 층에 기록"(팀 공유에 필수).
2
프로퍼티 추가
저장할 값마다 UPROPERTY(Config, EditAnywhere, Category="..."). Config가 빠지면 에디터에 보여도 ini에 저장되지 않습니다(§06 함정 ①). 결과: 컴파일 후 Project Settings ▸ Game 아래 페이지가 자동 생성.
3
에디터에서 조정
Project Settings에서 값을 바꾸면 Config/DefaultGame.ini에 즉시 기록됩니다. 이 파일을 커밋하면 팀 전체가 같은 밸런스를 받습니다.
4
코드에서 읽기
const UMyGameSettings* S = GetDefault<UMyGameSettings>(); — 읽기 전용 CDO입니다. 어느 시점(Initialize 포함)에서도 안전하게 읽을 수 있습니다 — World가 필요 없기 때문입니다.
05 — 구현 스켈레톤

설정 클래스와 읽는 쪽

// MyGameSettings.h — 컴파일하면 Project Settings ▸ Game 에 자동 노출
#pragma once   // 같은 헤더의 중복 include 방지 — 언리얼 헤더 공통 첫 줄
#include "Engine/DeveloperSettings.h"   // 부모 클래스 UDeveloperSettings의 선언이 여기 있음
#include "MyGameSettings.generated.h"   // UHT 자동 생성 헤더 — 반드시 마지막 #include

// Config=Game: 값의 저장처가 Config/DefaultGame.ini 라는 지정 (Engine이면 DefaultEngine.ini)
// DefaultConfig: 에디터에서 고친 값을 사용자별 파일이 아닌 프로젝트의 Default*.ini에 기록 — 팀 전체 공유(버전 관리 대상)
// meta=(DisplayName=...): Project Settings 목록에 보일 사람용 이름
UCLASS(Config=Game, DefaultConfig, meta=(DisplayName="My Game Balance"))
class MYGAME_API UMyGameSettings : public UDeveloperSettings   // UDeveloperSettings 상속만으로 설정 UI에 자동 등록
{
    GENERATED_BODY()   // UHT가 만든 리플렉션 코드 삽입 지점 — UCLASS 본문 필수
public:
    // Config가 빠지면 UI에 보여도 ini에 저장되지 않는다
    // (EditAnywhere는 "에디터에서 편집 가능", Config는 "ini에 기록" — 역할이 다르므로 둘 다 필요)
    UPROPERTY(Config, EditAnywhere, Category="Economy")
    int32 HireCost = 1000;          // 코드의 값은 "최초 기본값"일 뿐

    // ClampMin/Max: 에디터 입력을 0~100000으로 제한 — 음수 골드 같은 잘못된 값을 입력 단계에서 차단
    UPROPERTY(Config, EditAnywhere, Category="Economy",
              meta=(ClampMin="0", ClampMax="100000"))
    int32 StartingGold = 5000;     // meta로 에디터 입력 범위 제한

    UPROPERTY(Config, EditAnywhere, Category="Timer")
    int32 DefaultFocusMinutes = 25;   // 단위: 분 — Category="Timer"로 설정 UI에서 별도 그룹으로 묶임
};
// .cpp 는 비어 있어도 된다 (GENERATED_BODY 구현은 UHT가 생성)
// 어느 시스템에서든 — World가 필요 없어 Initialize()에서도 안전
#include "MyGameSettings.h"   // 설정 클래스 선언 — 읽는 쪽은 이 include 하나면 충분

// 게임플레이 코드(고용 버튼 등)에서 호출되는 함수 — 설정 읽기는 이렇게 한 줄
void UGachaSubsystem::Hire()
{
    // GetDefault: 엔진이 시작 시 ini 값을 얹어 만들어 둔 CDO(기본 객체)를 돌려줌 —
    // 우리가 생성·해제할 필요 없고, 읽기 전용이므로 const로 받는다
    const UMyGameSettings* S = GetDefault<UMyGameSettings>();

    if (!Economy->TrySpendGold(S->HireCost))   // 상수를 하드코딩하지 않는다
        return;   // 골드 부족 — 고용 실패로 조용히 종료
    // ...
}

// 결과 확인: Config/DefaultGame.ini 에 이렇게 기록된다
// [/Script/MyGame.MyGameSettings]
// HireCost=800
// StartingGold=5000
06 — 확장 & 주의

ini가 말을 안 듣는 순간들

⚠️
① Config 지정자 누락 — EditAnywhere만 붙이면 Project Settings에 보이고 편집도 되지만 에디터를 껐다 켜면 값이 사라집니다(ini에 기록이 안 되므로). 프로퍼티마다 Config를 확인하세요.
⚠️
② DefaultConfig 누락 — 클래스에 DefaultConfig가 없으면 값이 Saved/Config/(로컬 층)에 저장돼 내 PC에서만 적용됩니다. "내 에디터에선 800인데 팀원은 1000"의 원인입니다.
⚠️
③ 런타임 변경의 착각 — GetMutableDefault<T>()로 값을 바꿔도 메모리의 CDO만 바뀝니다. 유저별로 저장하고 싶은 값이라면 Settings가 아니라 SaveGame이 맞는 자리입니다(§03 경계).
💡
페이지 위치 바꾸기 — 기본은 Project Settings ▸ Game 아래입니다. virtual FName GetCategoryName() const override { return TEXT("Project"); }로 다른 분류로 옮길 수 있고, meta=(DisplayName)으로 페이지 이름을 정합니다.
