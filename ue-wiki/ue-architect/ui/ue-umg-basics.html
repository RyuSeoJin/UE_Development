Basics

체력바·메인 메뉴·인벤토리 같은 게임 화면 UI를 위젯 블루프린트로 만드는 언리얼의 표준 UI 시스템입니다. 이 문서는 UMG의 뼈대 — 위젯 트리·슬롯·앵커 구조, 제작부터 화면 표시까지의 절차, C++ 연동 — 를 다룹니다. 위젯 종류별 레퍼런스는 별도 문서(UMG 위젯 카탈로그), 하부 C++ 프레임워크는 별도 문서(Slate 개요)에서 다룹니다.

01 — 개념 · 역할

UMG란 — 화면 UI를 조립하는 시스템

UMG(Unreal Motion Graphics)는 게임 화면에 표시되는 UI — 체력바, 메인 메뉴, 인벤토리 창, 대화 상자 — 를 만드는 언리얼의 UI 저작 시스템입니다. 핵심 재료는 위젯 블루프린트(Widget Blueprint)라는 에셋으로, 에셋 1개 = 화면 조각 1개(메뉴 하나, HUD 하나)에 대응합니다. 위젯 블루프린트를 열면 두 개의 탭이 보입니다: 디자이너(Designer) 탭에서 버튼·텍스트 같은 부품을 드래그해 배치하고, 그래프(Graph) 탭에서 "버튼을 누르면 무엇을 할지" 로직을 블루프린트 노드로 작성합니다.

UMG 아래에는 Slate라는 C++ UI 프레임워크가 있습니다 — 언리얼 에디터 화면 전체가 Slate로 그려집니다. UMG는 그 Slate를 블루프린트에서 편집할 수 있게 감싼 저작 레이어이므로, 게임 UI는 UMG만으로 충분합니다. Slate가 무엇이고 언제 직접 쓰는지는 별도 문서(Slate 개요)에서 다룹니다.

🔑

핵심 용어 — 위젯(Widget): UI를 이루는 부품 하나(버튼, 텍스트, 이미지…). · 위젯 블루프린트: 위젯들을 조립해 만든 화면 조각 에셋(관례 접두사 WBP_). · 슬롯(Slot): 부모가 자식의 배치 속성을 담는 자리 — 02 섹션의 주인공. · 앵커(Anchor): 해상도가 바뀔 때 위젯이 붙어 있을 기준점. · 뷰포트(Viewport): 게임이 렌더링되는 화면 — 위젯을 여기에 "Add"해야 보입니다.

02 — 구성 요소 관계

위젯 트리와 슬롯 — 배치는 부모가 결정한다

위젯 블루프린트 안의 위젯들은 트리(계층 구조)를 이룹니다. 디자이너 탭 좌측의 계층(Hierarchy) 패널이 이 트리를 보여줍니다. 부모가 되는 위젯(패널)이 자식들을 담고, 자식이 다시 자식을 담는 구조입니다.

// 일시정지 버튼이 있는 HUD의 위젯 트리 예 WBP_PlayerHUD // 위젯 블루프린트 (화면 조각 1개) └─ Canvas Panel // 루트 패널 — 자유 배치 + 앵커 ├─ Vertical Box // 좌상단: 자식을 위→아래로 쌓는 패널 │ ├─ Text Block "PlayerName" │ └─ Progress Bar "HealthBar" └─ Button "PauseButton" // Button은 자식을 1개만 담을 수 있음 └─ Text Block "II"

슬롯 — UMG 이해의 최대 고비

슬롯(Slot)은 "부모 패널이 자식 하나를 담는 자리"입니다. 여기서 UMG의 가장 중요한 규칙이 나옵니다: 자식 위젯의 배치 속성(위치·크기·정렬·여백)은 자식이 아니라 부모의 슬롯이 결정합니다. 같은 Button이라도 어느 패널 밑에 있느냐에 따라 배치 옵션이 완전히 달라집니다.

⚠️
"같은 위젯인데 디테일 패널이 다르게 보여요"의 정체가 바로 슬롯입니다. 디테일 패널 최상단의 Slot (○○ Slot) 카테고리는 위젯 자신의 속성이 아니라 부모가 준 배치 속성입니다. 위젯을 다른 패널로 옮기면 이 카테고리가 통째로 바뀝니다. "앵커 설정이 안 보여요" = 부모가 Canvas Panel이 아니라는 뜻입니다.

앵커 — 해상도가 바뀌어도 제자리에

앵커(Anchor)는 Canvas Panel 슬롯 전용 개념으로, "화면 크기가 바뀔 때 이 위젯이 어느 기준점에 붙어 있을 것인가"를 정합니다. 위젯의 Position은 절대 좌표가 아니라 앵커로부터의 오프셋입니다. 디테일 패널의 Anchors 드롭다운에서 프리셋(좌상단·중앙·우하단·상하좌우 늘이기 등)을 고를 수 있습니다.

⚠️
앵커를 기본값(좌상단)으로 방치하면 다른 해상도에서 UI가 틀어집니다. 에디터의 미리보기 해상도에서는 멀쩡해 보여도, 화면 우측에 배치한 위젯이 좌상단 앵커면 넓은 화면에서 안쪽으로 밀려 들어옵니다. 위젯을 배치한 직후 앵커부터 지정하는 습관을 들이세요.
03 — 책임 배치

UI 로직을 어디에 두나

UMG를 쓰기 시작하면 "이 코드를 위젯에 쓸까, 컨트롤러에 쓸까"라는 결정이 계속 생깁니다. 기준은 하나입니다 — 위젯은 화면일 뿐, 데이터의 주인이 아니다.

일두는 곳왜
시각 표현 — 레이아웃·스타일·애니메이션위젯 BP 디자이너보이는 것은 전부 디자이너 탭에서. 로직과 분리
UI 내부 반응 — 버튼 클릭 시 다른 패널 열기, 탭 전환위젯 BP 그래프그 화면 안에서 끝나는 로직은 위젯이 스스로 처리
UI 열기/닫기, 입력 모드 전환PlayerController"어떤 화면을 띄울지"는 플레이어 입력·상태를 아는 쪽의 일 — 위젯이 스스로를 만들 수 없음
표시할 데이터의 원본 — 체력·점수·아이템 목록캐릭터 · GameState 등 게임 로직위젯은 읽어서 보여줄 뿐. 체력의 진실이 위젯에 있으면 게임 로직이 UI에 종속됨
여러 위젯 BP가 공유할 골격·성능 민감 로직C++ (UUserWidget 파생)BP 부모 클래스로 제공 — 05 섹션의 주제
💡

위젯 → 게임 로직 방향은 "이벤트 디스패처"로. 버튼이 눌렸을 때 위젯이 게임 로직을 직접 부르면(직접 참조) 위젯을 재사용할 수 없게 됩니다. 위젯은 "눌렸다"는 신호(이벤트 디스패처)만 쏘고, 그 신호를 컨트롤러나 소유자가 받아 처리하게 하면 위젯이 어디서든 재사용됩니다. (기초 단계에서는 직접 참조로 시작해도 되지만, 위젯을 두 곳 이상에서 쓰게 되는 순간 이 패턴이 필요해집니다.)

04 — 제작 · 표시 절차

위젯 블루프린트 생성부터 화면 표시까지 5단계

전제: 에디터 사용 가능. 예시로 메인 메뉴(WBP_MainMenu)를 만들어 표시합니다. 각 단계 끝의 확인 결과를 함께 적었습니다.

01
위젯 블루프린트 생성
Content Browser에서 우클릭 ▸ User Interface ▸ Widget Blueprint → 부모 클래스 선택 창에서 UserWidget 선택 → 이름을 WBP_MainMenu로 지정. 결과 확인: 더블클릭하면 디자이너/그래프 탭이 있는 UMG 에디터가 열립니다.
02
디자이너 탭에서 배치
좌측 팔레트(Palette)에서 위젯을 캔버스로 드래그합니다. 루트에 Canvas Panel이 기본으로 있고, 그 안에 Button·Text Block 등을 배치합니다. 배치 직후 디테일 패널의 Anchors부터 지정(02 섹션 함정 참조)하고, 각 위젯의 이름을 계층 패널에서 의미 있게 바꿉니다(예: StartButton — 05 섹션 C++ 연동에서 이 이름이 중요해집니다). 결과 확인: 계층 패널의 트리가 의도한 구조와 일치.
03
그래프 탭에서 이벤트 연결
디자이너에서 버튼을 선택 → 디테일 패널 맨 아래 Events ▸ On Clicked의 + 버튼 → 그래프 탭에 On Clicked (StartButton) 이벤트 노드가 생성됩니다. 여기에 원하는 로직(레벨 열기, 메뉴 닫기 등)을 연결합니다. 결과 확인: 컴파일(좌상단 Compile) 시 에러 없음.
04
화면에 표시 — Create Widget + Add to Viewport
위젯 BP는 만들어 두기만 하면 절대 저절로 표시되지 않습니다. 표시하는 쪽(보통 PlayerController나 레벨 블루프린트)의 그래프에서: Create Widget 노드(Class = WBP_MainMenu) → 반환값을 Add to Viewport 노드에 연결. BeginPlay 이벤트에서 호출하면 시작 시 표시됩니다. 결과 확인: 플레이(PIE) 시 메뉴가 화면에 나타남.
05
입력 모드 전환 · 닫기
메뉴가 떠도 마우스가 안 보이면 조작할 수 없습니다. PlayerController에서 Set Input Mode UI Only(또는 Game and UI) 노드 + Set Show Mouse Cursor = true. 닫을 때는 위젯의 Remove from Parent 노드로 화면에서 떼고, Set Input Mode Game Only + 커서 숨김으로 복원합니다. 결과 확인: 메뉴 조작 가능, 닫으면 게임 조작 복귀.
⚠️
Create Widget을 반복 호출하면 위젯이 계속 새로 생깁니다. 메뉴 열기 입력마다 Create Widget을 부르면 안 보이는 뒤편에 위젯이 겹겹이 쌓입니다. 한 번 만들어 변수에 저장해 두고, 이후에는 Add to Viewport / Remove from Parent(또는 Set Visibility)로 껐다 켜는 것이 기본 패턴입니다.
⚠️
Add to Viewport 했는데 안 보인다면 — ① Create Widget의 Class가 비어 있지 않은지, ② 다른 전체 화면 위젯이 위를 덮고 있지 않은지(ZOrder), ③ 루트 위젯의 Visibility가 Hidden/Collapsed가 아닌지 순서로 확인하세요.
05 — 구현 스켈레톤

C++ 연동 — BindWidget으로 디자이너와 코드 잇기

🚀 이 문서로 작업 시작하기

🧭

전제 조건 — WBP 제작 절차(본문)가 본체이고, 이 블록은 C++ 연동 스켈레톤의 착수입니다. 위저드·Build.cs·컴파일의 공통 절차는 위키 『C++ 클래스 생성 · 빌드 워크플로우』(core-cpp)를 따릅니다.

만들 파일 / 작업 대상부모 클래스 (위저드 검색어)비고
MainMenuWidget.h/.cppUserWidget (All Classes)WBP의 C++ 부모
표시 코드새 파일 없음기존 PlayerController에 Create Widget·Add to Viewport 추가
💡

코드 외 작업 — ① Build.cs에 "UMG"(+"Slate", "SlateCore") ② WBP 생성 후 Class Settings ▸ Parent Class를 MainMenuWidget로 재지정(Reparent) ③ BindWidget 대상 위젯의 이름을 C++ 변수명과 정확히 일치시키기.

⚠️
컴파일 주의 — 새 클래스·Build.cs → 재빌드. 이름 불일치는 컴파일이 아니라 런타임 nullptr로 나타나므로 재빌드 성공 ≠ 완료입니다.

BP만으로 UI가 완성되지만, 여러 위젯이 공유할 골격이나 성능 민감 로직은 C++로 내리는 것이 정석입니다. 패턴: C++ 클래스(UUserWidget 파생)가 뼈대를 정의하고, 위젯 BP가 그 클래스를 부모로 삼아 시각을 채웁니다. MYGAME_API는 자기 모듈 이름으로 바꾸세요. 

#pragma once   // 이 헤더가 한 번만 포함되도록 하는 중복 방지 지시문

#include "CoreMinimal.h"
#include "Blueprint/UserWidget.h"      // UUserWidget이 여기 있음
#include "MainMenuWidget.generated.h"  // 반드시 마지막 #include

class UButton;      // 전방 선언 — 포인터만 쓰므로 헤더 include 불필요
class UTextBlock;

// 위젯 BP의 "부모 클래스"가 될 C++ 골격
// Abstract: 이 클래스 자체로는 위젯을 만들지 않고 BP 자식만 사용
UCLASS(Abstract)
// MYGAME_API: DLL 경계에서 이 클래스를 다른 모듈이 쓸 수 있게 내보내는 매크로(모듈명_API)
class MYGAME_API UMainMenuWidget : public UUserWidget
{
    // UHT(리플렉션)가 생성한 보일러플레이트를 주입 — U클래스는 반드시 필요
    GENERATED_BODY()

public:
    // BindWidget: 디자이너의 "같은 이름" 위젯을 이 변수에 자동 연결
    // → 위젯 BP 계층 패널에 StartButton 이라는 이름의 Button이 반드시 있어야 함
    UPROPERTY(meta = (BindWidget))
    UButton* StartButton;

    UPROPERTY(meta = (BindWidget))
    UTextBlock* TitleText;

protected:
    // BP의 Event Construct에 해당하는 C++ 초기화 지점
    // 엔진이 위젯을 뷰포트에 추가(Add to Viewport)해 화면에 나타나는 시점에 호출 — 여기서 버튼 이벤트를 연결
    virtual void NativeConstruct() override;

private:
    // 동적 델리게이트(OnClicked)에 바인딩할 함수는 UFUNCTION 필수
    UFUNCTION()
    void HandleStartClicked();
};
#include "MainMenuWidget.h"
#include "Components/Button.h"      // UButton
#include "Components/TextBlock.h"   // UTextBlock

void UMainMenuWidget::NativeConstruct()
{
    Super::NativeConstruct();   // 부모 초기화 먼저 (관례)

    if (StartButton)   // BindWidget 연결 실패 대비 방어
    {
        // 버튼 클릭 델리게이트에 함수 연결
        // AddDynamic은 이름으로 함수를 찾는 방식이라 대상이 UFUNCTION()이어야 동작 — 클릭 시 HandleStartClicked 호출
        StartButton->OnClicked.AddDynamic(this, &UMainMenuWidget::HandleStartClicked);
    }
}

void UMainMenuWidget::HandleStartClicked()
{
    // TODO: 레벨 이동, 설정 열기 등
    RemoveFromParent();   // 자신을 화면에서 제거
}
⚠️
BindWidget은 "이름 완전 일치" 계약입니다. meta = (BindWidget)이 붙은 변수는, 그 클래스를 부모로 삼은 위젯 BP의 계층 패널에 변수와 정확히 같은 이름·호환 타입의 위젯이 있어야 컴파일됩니다. 없으면 위젯 BP 컴파일 시 "필수 위젯 바인딩이 없다"는 에러가 나고, 이름 오타가 가장 흔한 원인입니다. 선택 사항으로 만들려면 meta = (BindWidgetOptional)을 쓰되 사용 전 null 체크가 필수입니다.

표시하는 쪽(PlayerController) 코드입니다. BP의 Create Widget + Add to Viewport(04 섹션 4단계)와 같은 일을 C++로 한 것입니다.

// 헤더에 선언해 둘 위젯 클래스 참조:
// UPROPERTY(EditDefaultsOnly, Category="UI")
// TSubclassOf<UMainMenuWidget> MenuWidgetClass;
// → 에디터에서 BP 디폴트로 WBP_MainMenu를 지정해 둡니다.

// 메뉴를 띄우고 싶은 시점(예: BeginPlay나 입력 처리)에서 직접 호출하는 함수
void AMyPlayerController::ShowMainMenu()
{
    if (!MenuWidgetClass) { return; }   // 클래스 미지정 방어

    // Create Widget 노드의 C++판 — 소유자로 this(컨트롤러)를 넘긴다
    UMainMenuWidget* Menu = CreateWidget<UMainMenuWidget>(this, MenuWidgetClass);
    if (Menu)
    {
        Menu->AddToViewport();          // Add to Viewport 노드의 C++판

        // UI 전용 입력 모드 설정값 — 게임 입력을 막고 UI 위젯만 입력을 받게 함
        FInputModeUIOnly InputMode;
        SetInputMode(InputMode);        // 입력을 UI로
        bShowMouseCursor = true;       // 마우스 커서 표시
    }
}
⚠️

준비물 2가지를 잊지 마세요. ① 모듈 의존성: 프로젝트의 <모듈이름>.Build.cs에서 PublicDependencyModuleNames에 "UMG"를 추가해야 UUserWidget·UButton이 링크됩니다. ② 부모 클래스 재지정: 위젯 BP를 열고 File ▸ Reparent Blueprint(또는 그래프 탭 우측 Class Settings ▸ Parent Class)에서 부모를 MainMenuWidget으로 바꿔야 BindWidget 연결이 작동합니다.

⚠️
속성 바인딩(Bind)은 매 프레임 실행됩니다. 디자이너에서 Text·Percent 옆의 Bind 드롭다운으로 함수를 연결하면 편하지만, 그 함수는 매 프레임 호출됩니다. 값이 가끔만 바뀌는 데이터(체력·점수)는 바인딩 대신 값이 바뀌는 순간 Set Text / Set Percent를 호출하는 이벤트 구동 방식이 성능상 옳습니다.
⚠️
위젯 Tick에서 무거운 일 금지 — 위젯 BP의 Event Tick도 매 프레임입니다. 검색·정렬·문자열 조립을 Tick에 두면 UI가 곧 프레임 저하의 주범이 됩니다. 갱신은 "데이터가 바뀔 때만"이 원칙입니다.
💡
어떤 위젯이 있는지 궁금하다면 — 팔레트의 전 위젯(패널·일반·입력·리스트·최적화)의 역할과 활용은 별도 문서 "UMG 위젯 카탈로그"에서 다룹니다(중앙 허브에서 열 수 있습니다). Border·Button·Vertical Box가 각각 언제 쓰이는지, 어느 패널을 골라야 하는지는 그쪽이 전담합니다.
