UMG 팔레트의 위젯들을 분류별로 정리한 레퍼런스입니다. "이 UI를 만들려면 어떤 위젯을 골라야 하나"에 답하는 문서로, 처음부터 끝까지 읽기보다 필요한 분류로 점프해 찾아보도록 구성했습니다. 위젯 블루프린트가 무엇이고 슬롯·앵커가 어떻게 동작하는지는 별도 문서(UMG 기초)를 먼저 읽는 것을 권장합니다.

01 — 팔레트 · 공통 속성

팔레트 분류 체계와 모든 위젯의 공통 속성

팔레트(Palette)는 위젯 블루프린트 디자이너 탭의 좌측 패널로, 배치할 수 있는 모든 위젯이 분류별로 정리되어 있습니다. 검색창에 이름을 치면 바로 찾을 수 있습니다. 이 문서는 팔레트의 분류를 그대로 따라갑니다.

02 — 패널 위젯 (Panel)

어느 패널에 담을 것인가 — UI 골조 결정

패널은 자식을 담아 배치하는 컨테이너입니다. UI 제작의 첫 결정이 "어느 패널로 골조를 짜나"이므로, 상황 → 패널 결정표부터 보세요.

만들려는 배치패널왜
HUD처럼 화면 곳곳에 자유 배치 + 해상도 대응Canvas Panel유일하게 앵커·절대 좌표 지원 — 최상위 루트의 단골
메뉴 항목처럼 위→아래로 차곡차곡Vertical Box순서대로 자동 배치 — 항목 추가·삭제에 강함

비율을 유지하며 맞춰 늘이기Scale Box이미지·미니맵을 왜곡 없이 확대/축소

Canvas Panel — 자유 배치 + 앵커

자식마다 앵커·Position·Size·ZOrder(겹침 순서)를 개별 지정하는 자유 배치 패널입니다. 화면 전체를 기준으로 곳곳에 요소를 놓는 최상위 루트로 주로 쓰입니다. 반면 "항목이 늘어나면 자동으로 밀려나는" 배치는 못 하므로, 목록·메뉴는 Canvas 안에 Box를 넣어 조립합니다.

⚠️
모든 것을 Canvas로 조립하는 습관은 버리세요. Canvas 자식은 각각 위치 계산 비용이 들고, 항목 수가 변하는 UI(메뉴·목록)에서는 위치를 일일이 다시 잡아야 합니다. 원칙: 루트만 Canvas, 내부 골조는 Box·Overlay 조합. 위젯 BP를 부품으로 쓸 때(User Created)는 루트조차 Canvas가 아니어도 됩니다(Size Box나 Border 루트가 가벼움).

Vertical Box · Horizontal Box — 차곡차곡 쌓기

자식을 추가한 순서대로 세로/가로로 배치합니다. 핵심은 슬롯의 Size 규칙 — Auto vs Fill입니다.

슬롯 속성효과예
Size: Auto자식이 원하는 크기만큼만 차지텍스트 라벨, 아이콘
Size: Fill (비율)남는 공간을 비율대로 나눠 차지Fill 1 : Fill 2 = 남은 공간을 1:2로
Padding슬롯 바깥 여백 — 항목 사이 간격메뉴 항목 간 8px
Horizontal/Vertical Alignment슬롯 안에서의 정렬(Left/Center/Right/Fill)버튼을 슬롯 중앙에
⚠️
"Box 안의 위젯이 이상하게 늘어나요" — Alignment 기본값 Fill 때문입니다. Box 슬롯의 Alignment가 Fill이면 자식이 슬롯 폭/높이를 꽉 채우도록 늘어납니다. 고유 크기를 유지하려면 Alignment를 Left/Center/Right(또는 Top/Center/Bottom)로 바꾸세요.

Overlay — 겹치기

자식들을 같은 영역에 층층이 포갭니다(뒤에 추가한 것이 위). 배경 이미지 + 어둡게 덮는 틴트 + 제목 텍스트 같은 "겹침 스택"의 표준 도구입니다. 각 층은 슬롯 Alignment로 상하좌우 정렬만 지정합니다.

Scroll Box — 넘치면 스크롤

자식들이 패널 크기를 넘으면 스크롤바가 생깁니다. Orientation(세로/가로), Scroll Bar Visibility 정도만 만지면 됩니다. 그래프에서 Scroll to End 노드로 채팅창처럼 맨 아래로 보낼 수 있습니다. 주의: Scroll Box의 자식은 화면 밖에 있어도 전부 생성되어 있습니다 — 항목이 수백 개라면 05 섹션의 List View가 옳습니다.

Widget Switcher — 탭 전환

자식 여러 개 중 Active Widget Index로 지정한 하나만 표시합니다. 설정 창의 "그래픽/사운드/조작" 탭 내용부처럼, 같은 자리에서 화면을 전환하는 UI의 표준입니다. 탭 버튼(Horizontal Box + Button들)의 OnClicked에서 Set Active Widget Index를 호출하는 조합으로 씁니다.

Text Block — 표시 전용 텍스트

모든 UI의 기본입니다. 폰트·크기·색은 디테일 패널의 Font/Color, 런타임 갱신은 Set Text 노드로 합니다. 값이 자주 바뀌는 텍스트라도 바인딩(Bind) 대신 바뀌는 순간 Set Text가 성능 원칙입니다(UMG 기초 06 참조). 여러 스타일이 섞인 문장(예: 데미지 숫자만 빨갛게)이 필요해지면 Rich Text Block으로 넘어갑니다 — 스타일 표를 DataTable(행 구조체 RichTextStyleRow)로 만들어 연결하는 방식이라, DataTable 문서에서 배운 구조가 그대로 재등장합니다.

Button — 클릭의 표준

Button 자체는 모양 없는 클릭 영역 + 상태별 스타일이고, 내용물(글자·아이콘)은 자식으로 넣습니다(자식 1개 — 여러 개를 넣으려면 자식으로 Horizontal Box를 먼저). 이벤트는 용도별로 구분하세요:

이벤트발생 시점쓰는 상황
OnClicked누르고 같은 버튼 위에서 뗐을 때일반 클릭 — 기본 선택지
OnPressed / OnReleased누르는 순간 / 떼는 순간꾹 누르는 동안 지속되는 동작(게이지 충전)
OnHovered / OnUnhovered마우스가 올라옴 / 벗어남하이라이트·툴팁·사운드

Border — 배경과 여백의 프레임

자식 1개를 감싸 배경(색·이미지)과 안쪽 여백(Padding)을 주는 위젯입니다. 패널 카드·말풍선·목록 항목의 배경이 전부 Border입니다. Button과의 차이: Border는 상태별 스타일·클릭 이벤트가 없는 "장식 프레임"입니다(마우스 이벤트를 받을 수는 있지만 버튼 대용은 아님). "배경 있는 버튼"은 Button 안에 Border가 아니라, Button의 Style 자체에 배경 브러시를 지정하는 것이 정석입니다.

Image · Progress Bar — HUD의 단골 2종

Image는 Brush에 텍스처를 지정해 아이콘·배경을 표시하고, 머티리얼도 지정할 수 있어 회전하는 미니맵·쿨다운 소용돌이 같은 효과도 가능합니다. Progress Bar는 Percent(0.0~1.0)만 갱신하면 되는 게이지로, 체력바의 표준입니다 — 체력이 바뀌는 이벤트에서 현재HP / 최대HP를 Set Percent에 넣으면 끝. 좌→우 외에 우→좌·차오르는 방향(Bar Fill Type)도 지원합니다.

04 — 입력 위젯 (Input)

키보드 입력과 값 선택

위젯역할구분 포인트
Editable Text한 줄 텍스트 입력 — 배경 없음모양은 직접 Border 등으로 감싸서
Editable Text Box한 줄 텍스트 입력 — 배경·스타일 포함일반적인 "입력창"은 이것
Editable Text (Multi-Line) / Box여러 줄 입력채팅 입력·메모
Combo Box (String)드롭다운에서 문자열 하나 선택해상도·언어 선택. Options 배열 + OnSelectionChanged
Spin Box숫자 입력 + 드래그 증감Min/Max·Delta 지정 — 수량 선택
⚠️
OnTextChanged와 OnTextCommitted를 구분하세요. OnTextChanged는 한 글자 칠 때마다, OnTextCommitted는 Enter를 치거나 포커스가 떠날 때 발생합니다. 닉네임 검증·채팅 전송은 Committed에서(Commit Method로 Enter인지 포커스 이탈인지 분기), 실시간 검색 필터만 Changed에서 처리합니다. Changed에 무거운 로직을 달면 타이핑이 버벅입니다.
💡

입력창이 클릭해도 반응이 없다면 입력 모드를 확인하세요 — 게임이 Input Mode Game Only 상태면 UI가 키보드 포커스를 받지 못합니다. UMG 기초 04 섹션 5단계(Set Input Mode UI Only / Game and UI)가 전제 조건입니다.

05 — 리스트 위젯 (Lists)

수백 개 항목을 다루는 가상화 목록

Lists 분류의 위젯들은 가상화(Virtualization)로 동작합니다: 항목이 1,000개라도 화면에 보이는 개수만큼만 위젯을 실제 생성하고, 스크롤되면 그 위젯들을 재활용해 다른 항목의 데이터를 다시 채웁니다. "항목 = 데이터(UObject)"와 "항목을 그리는 위젯(Entry Widget)"을 분리하는 구조라서, Scroll Box에 위젯을 수백 개 직접 만들어 넣는 방식과 성능이 근본적으로 다릅니다.

Scroll Box + 직접 생성

간단수십 개까지

루프에서 Create Widget → Add Child. 구조가 단순해 항목 수십 개 이하(설정 메뉴·짧은 목록)면 충분하고 빠르게 만듭니다.

List View (가상화)

수백~수천 개데이터-위젯 분리

Add Item으로 데이터 오브젝트만 넘기면 보이는 만큼만 위젯 생성. 인벤토리·순위표·상점처럼 항목이 많거나 수가 유동적이면 이쪽.

위젯배치쓰는 상황
List View세로 1열순위표·퀘스트 목록·채팅 로그
Tile View격자(타일)인벤토리·상점 아이템 그리드
Tree View접고 펴는 계층제작 트리·폴더형 목록

사용 골격(3단계): ① 항목 데이터용 오브젝트(BP면 Object 상속 BP, C++면 UObject 파생)를 만들고 → ② 항목을 그릴 위젯 BP를 만들어 Interfaces에 UserObjectListEntry 인터페이스를 추가(클래스 세팅 ▸ Implemented Interfaces), On List Item Object Set 이벤트에서 받은 데이터로 텍스트·아이콘을 채우고 → ③ List View의 Entry Widget Class에 그 위젯을 지정한 뒤 그래프에서 Add Item으로 데이터를 넘깁니다.

⚠️
Entry Widget은 재활용됩니다 — 상태를 위젯에 저장하지 마세요. 스크롤하면 같은 위젯 인스턴스가 다른 항목의 데이터로 다시 채워집니다. "선택됨" 같은 상태를 Entry Widget의 변수에 저장하면 스크롤할 때 엉뚱한 항목이 선택된 것처럼 보입니다. 상태는 데이터 오브젝트 쪽에 두고, 위젯은 On List Item Object Set에서 매번 데이터로부터 다시 그리는 것이 원칙입니다.
