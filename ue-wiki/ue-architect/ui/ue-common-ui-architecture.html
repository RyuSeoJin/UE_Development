Common
UI

포트나이트 수준의 다층 UI(메뉴 위에 팝업, 그 위에 확인창)를 안전하게 쌓고 입력을 자동으로 배분하는 엔진 플러그인입니다. 이 문서는 Common UI의 핵심 부품(Activatable Widget·스택·입력 라우팅)을 이해하고, 공식 샘플 Lyra가 이 부품들을 조립한 4레이어 아키텍처를 분석한 뒤, 같은 구조를 내 게임에 최소 구현으로 이식하는 것까지 다룹니다. 전제 지식: UMG 기초 문서(위젯 BP·슬롯·BindWidget).

엔진 플러그인 UE5 (Lyra 검증 구조) C++ · Blueprint
01 — 개념 · 역할

Common UI가 해결하는 세 가지 문제

Common UI는 언리얼에 내장된 UI 프레임워크 플러그인입니다(Edit ▸ Plugins에서 활성화). UMG를 대체하는 것이 아니라 UMG 위에 얹혀서, UMG만으로 만들다 보면 반드시 부딪히는 세 가지 문제를 구조적으로 해결합니다. UMG 기초 문서에서 손으로 했던 일들을 떠올리면 왜 필요한지 바로 보입니다.

문제 (UMG만 쓸 때)Common UI의 답
화면이 겹칠수록 지옥 — 일시정지 위에 설정, 그 위에 "저장할까요?" 팝업… 어느 화면이 입력을 받아야 하나를 매번 수동 관리Activatable Widget + 스택 — 맨 위 화면만 활성·입력 수신, 닫히면 이전 화면이 자동 복귀
Set Input Mode·Show Mouse Cursor를 화면 열고 닫을 때마다 수동 호출 (하나만 빠뜨려도 조작 불능)위젯이 자기가 원하는 입력 상태를 선언하면(GetDesiredInputConfig), 활성/비활성 시 자동 적용·복원
게임패드 포커스·플랫폼별 버튼 아이콘(A/×)을 전부 직접 구현입력 라우팅 + 플랫폼별 컨트롤러 데이터·스타일 에셋 내장
🔑

핵심 용어 — Activatable Widget: "활성(보이고 입력받음)/비활성" 상태를 가진 Common UI의 화면 단위 위젯 — 02 섹션의 주인공. · 스택(Stack): Activatable Widget을 쌓는 컨테이너. 맨 위만 활성. · 레이어(Layer): 용도별로 분리한 스택(게임 HUD층 / 메뉴층 / 팝업층…). · 입력 라우팅(Input Routing): 입력을 활성 위젯에게만 전달하는 교통정리. · GameplayTag: UI.Layer.Menu처럼 점으로 구분하는 계층형 이름표 — 레이어를 식별하는 열쇠.

02 — 구조 · Lyra 분석

핵심 부품, 그리고 Lyra가 조립한 방법

부품 ① Activatable Widget — "활성"이라는 새 수명

Activatable Widget(UCommonActivatableWidget)은 Common UI 화면 단위의 기본 위젯입니다. 일반 UserWidget과의 차이는 Construct(생성)와 별개로 Activate(활성)/Deactivate(비활성) 상태를 가진다는 것 — 위젯이 만들어져 있어도 비활성이면 입력을 받지 않고(설정에 따라 숨겨지고), 활성화되면 입력 라우팅의 대상이 됩니다. "화면을 닫는다"는 Remove from Parent가 아니라 DeactivateWidget이 됩니다.

부품 ② 스택 — 맨 위만 산다

UCommonActivatableWidgetStack은 Activatable Widget을 쌓는 컨테이너입니다. 규칙은 단순하고 강력합니다: 맨 위 위젯만 활성이고, 아래 위젯들은 전부 비활성. 맨 위 위젯이 비활성화되면(뒤로가기·닫기) 자동으로 스택에서 빠지고, 그 아래 위젯이 다시 활성화됩니다. "설정을 닫으면 일시정지 메뉴가 다시 살아나는" 흐름이 코드 한 줄 없이 스택 규칙만으로 성립합니다. (Queue 컨테이너도 있지만 — 순서대로 하나씩 보여주는 알림용 — 기본은 스택입니다.)

부품 ③ 입력 라우팅 — 입력의 교통정리

입력 라우팅은 마우스·키보드·게임패드 입력을 활성 위젯에게만 전달하는 시스템입니다. 관문은 프로젝트 설정의 CommonGameViewportClient(04 섹션 2단계)이며, 이것이 입력을 가로채 활성 스택의 맨 위 위젯으로 보냅니다. 여기에 각 위젯이 선언하는 입력 설정(GetDesiredInputConfig — 게임만/UI만/둘 다)이 합쳐져, UMG 기초 문서의 "Set Input Mode + Show Mouse" 수동 관리가 통째로 사라집니다.

Common UI는 그 외에도 전용 위젯 라이브러리를 제공합니다 — UMG 위젯의 Common 판이며, 스타일 에셋 연동과 게임패드 대응이 내장된 것이 차이입니다.

03 — 레이어 책임 배치

어떤 화면을 어느 레이어에 올리나

레이어를 나누는 기준은 "겹쳤을 때 누가 위인가 + 입력을 누가 받아야 하나"입니다. Lyra의 4층은 대부분의 게임에 그대로 통용됩니다.

화면레이어왜
HUD (체력바·미니맵·조준점)UI.Layer.Game상시 표시 + 입력은 게임이 계속 받아야 함 — 이 층의 위젯은 입력 설정을 Game으로 선언
일시정지 메뉴, 인벤토리, 지도 화면UI.Layer.GameMenu게임 도중 잠깐 덮는 화면 — 열리면 게임 입력을 뺏고, 닫히면 HUD 층으로 자동 복귀
설정, 타이틀/프론트엔드, 크레딧UI.Layer.Menu게임 플레이와 분리된 전체 화면 흐름 — GameMenu 위에 겹쳐도 순서가 보장됨
"정말 삭제할까요?", 에러 알림, 연결 끊김UI.Layer.Modal무조건 최상단 + 응답 전까지 아래 모든 층의 입력 차단

기존 UMG 수동 방식

수동 관리소규모엔 충분

Create Widget → Add to Viewport(ZOrder 직접) → Set Input Mode/커서 수동 전환 → 닫을 때 Remove from Parent + 이전 상태 수동 복원. 화면이 3개만 겹쳐도 상태 꼬임이 시작됩니다.

Common UI 레이어 방식

자동 복귀다층 UI에 강함

Push to Layer(태그 지정) → 입력·포커스·커서는 위젯의 선언대로 자동 → 닫기는 DeactivateWidget 하나 → 이전 화면 자동 활성 + 입력 상태 자동 복원.

수명 구분 — Construct와 Activate는 다르다

이벤트시점여기서 할 일
Construct / NativeOnInitialized위젯이 만들어질 때 (1회)바인딩·참조 준비 등 한 번만 하는 초기화
On Activated스택 맨 위가 되어 살아날 때 (여러 번 가능)데이터 새로고침, 첫 포커스 지정 — "화면이 다시 보일 때마다" 할 일
On Deactivated덮이거나 닫혀 잠들 때일시 상태 정리 (파괴 아님!)
⚠️
"뒤로 갔다 돌아왔더니 화면이 옛날 데이터예요" — 갱신 로직을 Construct에만 두면 생기는 전형적 증상입니다. 스택에서 아래에 깔렸다 다시 올라온 위젯은 Construct가 다시 불리지 않습니다. 화면이 보일 때마다 필요한 갱신은 On Activated에 두세요.
04 — 적용 절차

내 게임에 이식하는 7단계 (직접 최소 구현)

전제: C++ 프로젝트(05 스켈레톤 사용), 에디터 사용 가능. 1~3단계는 Common UI 공식 퀵스타트의 필수 설정이고, 4~7단계가 Lyra식 레이어 시스템의 최소 구현입니다. 각 단계에 붙은 🔧 커스텀 포인트 카드는 그 단계에서 프로젝트에 맞게 바꿔도 되는 항목입니다 — 절차 본문은 "규격", 카드는 "자유 영역"으로 읽으세요. 예로 든 4개 레이어 태그조차 엔진 규칙이 아니라 Lyra의 관례입니다.

01
플러그인 활성화
Edit ▸ Plugins에서 Common UI Plugin 검색 → 체크 → 에디터 재시작. C++ 모듈 의존성은 05 섹션에서. 결과 확인: 팔레트에 Common 위젯들(CommonButtonBase 등)이 나타남.
02
Viewport Client 교체 — 입력 라우팅의 관문
Project Settings ▸ Engine ▸ General Settings ▸ Game Viewport Client Class를 CommonGameViewportClient로 변경. 이 클래스가 입력을 가로채 활성 위젯으로 보내는 라우팅의 시작점입니다. 결과 확인: 설정값이 저장됨(이 단계만으로 눈에 보이는 변화는 없음 — 정상).

🔧 커스텀 포인트

  • CommonGameViewportClient를 상속한 내 클래스로 지정 가능 (Lyra도 LyraGameViewportClient 사용)
  • 입력이 UI에 닿기 전의 전역 처리 — 로딩 중 전체 입력 차단, 특정 입력 필터링
  • 초반엔 기본 클래스 그대로 쓰다가, 필요해질 때 상속으로 전환
03
기본 입력 데이터 — 클릭·뒤로가기 정의
① DataTable을 만들되 행 구조체로 CommonInputActionDataBase 선택(DataTable 문서의 절차 그대로) → 클릭·뒤로가기 행 추가. ② CommonUIInputData를 부모로 BP 생성 → Click/Back에 방금 만든 행 지정. ③ Project Settings ▸ Game ▸ Common Input Settings ▸ Input Data에 그 BP 지정. (게임패드까지 갈 경우: CommonInputBaseControllerData 기반 플랫폼별 데이터를 Platform Input에 추가 — 처음엔 생략 가능) 결과 확인: 이후 Activatable 위젯에서 뒤로가기 입력이 인식됨.

🔧 커스텀 포인트

  • DataTable에 액션 행 추가 자유 — Click/Back은 최소 요건. "다음/이전 탭(Q/E)", "정렬 변경" 등
  • 행마다 키보드·게임패드 키 별도 지정 → 버튼의 Triggering Input Action에 연결하면 키로 눌리는 버튼이 됨
  • 추가한 액션은 CommonBoundActionBar가 하단 조작 안내에 자동 표시
  • 플랫폼별 ControllerData를 늘릴수록 버튼 아이콘(Ⓐ/×) 자동 전환 · Hold(꾹 누르기) 발동도 설정 가능
04
레이어 태그 등록
Project Settings ▸ Project ▸ GameplayTags ▸ Manage Gameplay Tags에서 4개 태그 추가: UI.Layer.Game · UI.Layer.GameMenu · UI.Layer.Menu · UI.Layer.Modal. 결과 확인: 태그 선택 드롭다운에 UI.Layer 계층이 보임.

🔧 커스텀 포인트

  • 4개 태그는 Lyra 관례일 뿐 — UI.Layer.Loading·Notification·Tutorial·Debug 등 필요한 만큼 추가
  • 단, 태그는 스스로 동작하지 않음 — 태그 등록 + 스택 배치(5단계) + Layers 맵 1줄(05 스켈레톤)이 한 세트
  • UI.Layer.* 계층을 유지하면 태그 드롭다운 정리·일괄 조회에 유리
05
PrimaryLayout 위젯 제작 — 유일한 루트
05 섹션의 UMyPrimaryLayout를 컴파일한 뒤, 그것을 부모로 위젯 BP(WBP_PrimaryLayout) 생성. 디자이너에서 루트 Overlay 아래에 CommonActivatableWidgetStack 4개를 배치하고 이름을 정확히 GameStack·GameMenuStack·MenuStack·ModalStack으로(BindWidget 이름 계약 — UMG 기초 참조). 각 스택 슬롯은 화면 전체를 덮게(Overlay 정렬 Fill). 결과 확인: 위젯 BP 컴파일 성공(이름이 틀리면 바인딩 에러).

🔧 커스텀 포인트

  • 층의 개수·겹침 순서가 전부 여기서 결정 — Overlay에서 아래쪽에 배치한 스택일수록 위에 그려짐
  • 층마다 컨테이너 선택: Stack(쌓고 복귀) vs Queue(한 번에 하나씩 — 획득 알림·연속 팝업용)
  • 스택이 아닌 상시 요소도 배치 가능 — 공통 배경, 버전 워터마크, 페이드 오버레이
  • ClearLayer(태그)·GetTopWidget(태그) 같은 편의 함수 확장
06
화면들을 Activatable로 제작
각 화면(HUD·일시정지·설정…)을 CommonActivatableWidget(또는 05의 UMenuScreenWidget) 파생 위젯 BP로 제작. 메뉴류는 디테일 패널에서 Is Back Handler·Is Back Action Displayed in Action Bar를 켜면 뒤로가기(Esc/Ⓑ)로 DeactivateWidget이 자동 호출됩니다. 결과 확인: 각 위젯 단독 컴파일 성공.

🔧 커스텀 포인트

  • GetDesiredInputConfig를 화면마다 다르게 — HUD는 Game, 인벤토리는 All(게임+UI 동시), 메뉴는 Menu
  • OnHandleBackAction 오버라이드 — 뒤로가기를 "종료하시겠습니까?" 확인 팝업으로 교체 가능
  • GetDesiredFocusTarget — 게임패드 첫 포커스를 화면별 지정 (06 섹션 함정 참조)
  • 활성/비활성 시 Visibility 처리 방식, On Activated에서 등장 애니메이션 재생
07
UI 매니저로 올리고 검증
PlayerController BeginPlay에서 UI 매니저 서브시스템(05 섹션)의 CreateLayout 호출(레이아웃 클래스 = WBP_PrimaryLayout) → 이후 어디서든 Push(UI.Layer.Game, WBP_HUD)처럼 올립니다. 결과 확인(핵심): 일시정지를 GameMenu 층에 Push하면 마우스가 자동으로 나타나고, 뒤로가기로 닫으면 게임 입력이 자동 복원되어야 합니다 — 수동 Set Input Mode 없이.

🔧 커스텀 포인트

  • API를 내 게임의 어휘로 확장 — PopTopScreen, 결과 델리게이트를 받는 ShowConfirmDialog(Modal 전용 헬퍼)
  • 위젯 클래스를 소프트 참조로 받아 비동기 로드 후 Push (Lyra의 PushWidgetToLayerStackAsync에 해당)
  • 분할 화면(로컬 멀티)이면 플레이어별 레이아웃 관리로 확장
  • 규모가 커지면 Policy 층 재도입 — 플랫폼별로 다른 레이아웃 BP 선택
⚠️
Activatable Widget을 스택 밖에서 쓰면 "죽은 화면"이 됩니다. CommonActivatableWidget을 그냥 Add to Viewport 하면 활성화 관리자(스택)가 없어 활성 상태·입력 라우팅·자동 복귀가 전부 동작하지 않습니다. 이 구조에서 Add to Viewport는 PrimaryLayout 단 한 번이고, 나머지 모든 화면은 반드시 레이어 스택에 Push합니다.
⚠️
반대로, 바꾸면 구조가 무너지는 고정 규격은 두 가지입니다. ① Viewport Client는 CommonGameViewportClient 계열을 유지해야 합니다 — 이것을 빼면 입력 라우팅 전체가 무효가 됩니다. ② Activatable Widget은 반드시 스택/큐에 올려야 합니다(직접 Add to Viewport 금지 — 위 함정 참조). 그 외에 BindWidget 스택 이름을 바꿀 때는 C++ 변수와 디자이너 이름을 동시에 바꿔야 한다는 이름 계약도 기억하세요.
05 — 구현 스켈레톤

최소 구현 — 레이아웃 · UI 매니저 · 화면 부모

🚀 이 문서로 작업 시작하기

🧭

전제 조건 — Common UI는 플러그인입니다 — Edit ▸ Plugins ▸ Common UI 활성화 + 에디터 재시작이 0순위 작업입니다. 위저드·Build.cs·컴파일의 공통 절차는 위키 『C++ 클래스 생성 · 빌드 워크플로우』(core-cpp)를 따릅니다.

만들 파일 / 작업 대상부모 클래스 (위저드 검색어)비고
UIManagerSubsystem.h/.cppGameInstanceSubsystem레이어 푸시/팝 창구
MyPrimaryLayout.h/.cppCommonUserWidget레이어 스택을 담는 최상위 레이아웃
MenuScreenWidget.h/.cppCommonActivatableWidget화면 단위 위젯의 공통 부모
💡

코드 외 작업 — ① Build.cs에 "CommonUI", "CommonInput" 추가 ② Project Settings ▸ General Settings ▸ Game Viewport Client Class = CommonGameViewportClient(입력 라우팅의 전제) ③ 각 C++ 위젯의 WBP 자식 생성 — 시각 레이아웃은 WBP에서.

⚠️
컴파일 주의 — 플러그인·Build.cs·새 클래스 전부 재빌드 코스입니다.

Lyra의 CommonGame 플러그인(GameUIManagerSubsystem·GameUIPolicy·PrimaryGameLayout)을 1인 프로젝트 규모로 축소한 3파일입니다. 정책(Policy) 층은 "레이아웃 클래스를 골라주는" 역할뿐이므로 생략하고, 서브시스템이 직접 레이아웃을 만듭니다. MYGAME_API는 자기 모듈 이름으로. 먼저 <모듈이름>.Build.cs의 의존성에 "CommonUI", "CommonInput", "GameplayTags", "UMG"를 추가하세요. 

#pragma once

#include "CoreMinimal.h"
#include "CommonUserWidget.h"
#include "GameplayTagContainer.h"    // FGameplayTag
#include "MyPrimaryLayout.generated.h"

// 전방 선언(forward declaration) — 헤더에서는 포인터/참조만 쓰므로 전체 정의 include 없이 이름만 알린다(컴파일 시간 단축·순환 include 방지)
class UCommonActivatableWidget;
class UCommonActivatableWidgetStack;

// 화면 전체를 덮는 "UI의 뿌리" — 레이어 스택 4개를 담는다.
// 게임에서 AddToViewport 되는 유일한 위젯 (Lyra PrimaryGameLayout의 축소판)
// UCLASS(Abstract): 이 클래스는 직접 인스턴스화 금지 — 반드시 위젯 BP(WBP_PrimaryLayout)로 파생해 디자이너에서 스택을 배치한 뒤 사용한다
UCLASS(Abstract)
class MYGAME_API UMyPrimaryLayout : public UCommonUserWidget
{
    // GENERATED_BODY(): 언리얼 리플렉션 시스템이 요구하는 생성 코드 자리 — UCLASS를 쓰면 필수
    GENERATED_BODY()

public:
    // 태그가 가리키는 레이어 스택에 위젯을 올린다
    // UFUNCTION(BlueprintCallable): 블루프린트 그래프에서 이 함수를 노드로 호출할 수 있게 노출
    UFUNCTION(BlueprintCallable, Category = "UI")
    UCommonActivatableWidget* PushToLayer(FGameplayTag LayerTag,
        TSubclassOf<UCommonActivatableWidget> WidgetClass);

protected:
    // NativeOnInitialized: 위젯 트리 생성 직후 엔진이 1회 호출 — BindWidget 포인터가 채워진 뒤라 스택→맵 등록에 적합한 시점(Construct보다 이르다)
    virtual void NativeOnInitialized() override;

    // 디자이너의 스택 4개와 "같은 이름"으로 연결 (BindWidget — UMG 기초 참조)
    // BindWidget: 디자이너에 반드시 이 변수명과 똑같은 이름의 위젯이 있어야 함 — 없으면 컴파일 에러(이름 계약)
    UPROPERTY(meta = (BindWidget))
    UCommonActivatableWidgetStack* GameStack;

    UPROPERTY(meta = (BindWidget))
    UCommonActivatableWidgetStack* GameMenuStack;

    UPROPERTY(meta = (BindWidget))
    UCommonActivatableWidgetStack* MenuStack;

    UPROPERTY(meta = (BindWidget))
    UCommonActivatableWidgetStack* ModalStack;

private:
    // 태그 → 스택 조회용 맵 (NativeOnInitialized에서 채움)
    // UPROPERTY(): 인자가 없어도 붙이는 이유는 GC 추적 — 이게 있어야 담긴 UObject(스택)가 가비지 컬렉터에 수집되지 않는다. TObjectPtr는 UE5의 GC 안전 스마트 포인터
    UPROPERTY()
    TMap<FGameplayTag, TObjectPtr<UCommonActivatableWidgetStack>> Layers;
};
#include "MyPrimaryLayout.h"
#include "Widgets/CommonActivatableWidgetContainer.h"  // 스택 클래스

void UMyPrimaryLayout::NativeOnInitialized()
{
    // 부모 클래스의 초기화를 먼저 실행 — 오버라이드 시 Super 호출을 빠뜨리면 엔진 기본 준비가 누락된다
    Super::NativeOnInitialized();

    // 태그는 Project Settings ▸ GameplayTags에 미리 등록 (04 섹션 4단계)
    // RequestGameplayTag: 문자열을 실제 등록된 FGameplayTag로 변환 — 미등록 문자열이면 경고. 이렇게 태그↔스택을 짝지어 Push 때 태그로 스택을 찾는다
    Layers.Add(FGameplayTag::RequestGameplayTag("UI.Layer.Game"),     GameStack);
    Layers.Add(FGameplayTag::RequestGameplayTag("UI.Layer.GameMenu"), GameMenuStack);
    Layers.Add(FGameplayTag::RequestGameplayTag("UI.Layer.Menu"),     MenuStack);
    Layers.Add(FGameplayTag::RequestGameplayTag("UI.Layer.Modal"),    ModalStack);
}

UCommonActivatableWidget* UMyPrimaryLayout::PushToLayer(
    FGameplayTag LayerTag, TSubclassOf<UCommonActivatableWidget> WidgetClass)
{
    // FindRef: 맵에서 태그로 스택을 찾음 — 없으면 nullptr 반환(예외 없음)이라 바로 아래에서 방어 검사
    UCommonActivatableWidgetStack* Stack = Layers.FindRef(LayerTag);
    if (!Stack || !WidgetClass) { return nullptr; }   // 미등록 태그 방어

    // AddWidget: 생성 + 스택 맨 위에 올려 활성화. 이전 맨 위는 자동 비활성
    return Stack->AddWidget(WidgetClass);
}
#pragma once

#include "CoreMinimal.h"
#include "Subsystems/GameInstanceSubsystem.h"
#include "GameplayTagContainer.h"
#include "UIManagerSubsystem.generated.h"

// 전방 선언 — 헤더는 포인터만 다루므로 정의 include 불필요
class UMyPrimaryLayout;
class UCommonActivatableWidget;

// 어디서든 UI를 올릴 수 있는 전역 창구 —
// GameInstanceSubsystem이므로 별도 등록 없이 자동 생성 (해당 문서 참조)
// 수명은 GameInstance와 동일 — 레벨(맵)을 옮겨도 파괴되지 않아 UI 창구로 적합
UCLASS()
class MYGAME_API UUIManagerSubsystem : public UGameInstanceSubsystem
{
    // 리플렉션 생성 코드 자리 (UCLASS 필수)
    GENERATED_BODY()

public:
    // 게임 시작 시 PlayerController가 1회 호출 — 루트 레이아웃 생성
    UFUNCTION(BlueprintCallable, Category = "UI")
    void CreateLayout(APlayerController* OwningPlayer,
                      TSubclassOf<UMyPrimaryLayout> LayoutClass);

    // 화면 올리기 — 예: Push(UI.Layer.GameMenu, WBP_PauseMenu)
    UFUNCTION(BlueprintCallable, Category = "UI")
    UCommonActivatableWidget* Push(FGameplayTag LayerTag,
        TSubclassOf<UCommonActivatableWidget> WidgetClass);

private:
    // 생성한 루트 레이아웃을 보관 — UPROPERTY로 GC가 수집하지 못하게 붙든다(없으면 화면이 갑자기 사라질 수 있음)
    UPROPERTY()
    TObjectPtr<UMyPrimaryLayout> RootLayout;
};
#include "UIManagerSubsystem.h"
#include "MyPrimaryLayout.h"
#include "Blueprint/UserWidget.h"   // CreateWidget

void UUIManagerSubsystem::CreateLayout(
    APlayerController* OwningPlayer, TSubclassOf<UMyPrimaryLayout> LayoutClass)
{
    if (RootLayout || !OwningPlayer || !LayoutClass) { return; }   // 중복 생성 방어

    // CreateWidget: LayoutClass(WBP_PrimaryLayout)의 인스턴스를 생성 — OwningPlayer를 소유자로 지정해 그 플레이어 화면에 속하게 한다
    RootLayout = CreateWidget<UMyPrimaryLayout>(OwningPlayer, LayoutClass);
    RootLayout->AddToViewport();   // 이 프로젝트의 처음이자 마지막 AddToViewport
}

UCommonActivatableWidget* UUIManagerSubsystem::Push(
    FGameplayTag LayerTag, TSubclassOf<UCommonActivatableWidget> WidgetClass)
{
    // 레이아웃이 아직 안 만들어졌으면(CreateLayout 전) nullptr — 크래시 대신 안전하게 실패시킨 뒤, 있으면 실제 Push를 레이아웃에 위임
    return RootLayout ? RootLayout->PushToLayer(LayerTag, WidgetClass) : nullptr;
}

마지막으로 화면 위젯들의 공통 부모입니다 — 입력 상태를 위젯이 "선언"하는 지점이 Common UI 자동화의 핵심입니다.

#pragma once

#include "CoreMinimal.h"
#include "CommonActivatableWidget.h"
#include "MenuScreenWidget.generated.h"

// 메뉴류 화면의 공통 부모 — 활성인 동안 원하는 입력 상태를 "선언"
// Abstract: 이 부모를 직접 쓰지 말고, 설정·인벤토리 등 실제 화면 BP를 이 클래스에서 파생시켜 쓰라는 뜻
UCLASS(Abstract)
class MYGAME_API UMenuScreenWidget : public UCommonActivatableWidget
{
    // 리플렉션 생성 코드 자리 (UCLASS 필수)
    GENERATED_BODY()

public:
    // UMG 기초에서 손으로 하던 Set Input Mode + Show Mouse가 자동화되는 지점.
    // 이 위젯이 활성이면 적용되고, 비활성화되면 이전 상태로 자동 복원된다
    // GetDesiredInputConfig: 이 위젯이 스택 맨 위(활성)가 될 때 엔진(입력 라우팅)이 호출해 원하는 입력 모드를 물어본다 — 반환값대로 Input Mode·커서가 자동 설정된다
    virtual TOptional<FUIInputConfig> GetDesiredInputConfig() const override
    {
        // Menu: UI만 입력받음 · Game: 게임만 · All: 둘 다
        // 2번째 인자 NoCapture: 마우스를 뷰포트에 가두지 않음(커서 자유 이동) — 메뉴 조작에 알맞다
        return FUIInputConfig(ECommonInputMode::Menu, EMouseCaptureMode::NoCapture);
    }
};
💡

닫기는 어디서나 한 줄입니다 — 화면 자신이 DeactivateWidget()(BP: Deactivate Widget 노드)을 호출하면, 스택이 자동으로 pop하고 아래 화면을 활성화하며 입력 상태를 복원합니다. 04 섹션 6단계처럼 Is Back Handler를 켜 두면 Esc/게임패드 Ⓑ가 이 호출을 대신해 줍니다. HUD처럼 상시 표시 화면은 입력 설정을 ECommonInputMode::Game으로 선언하면 됩니다.

06 — 확장 & 주의

더 나아가기와 흔한 함정

🎨
스타일 에셋
CommonTextStyle · ButtonStyle
글꼴·색·버튼 상태를 에셋으로 분리해 전 화면이 공유. Common Style 클래스로 BP를 만들어 위젯 Style 필드에 지정하고, Template Styles로 전역 기본값 설정 — 게임 전체의 룩을 에셋 몇 개로 통제.
🧩
UIExtension
Lyra 동봉 플러그인
HUD에 "확장 포인트"를 선언하고, 게임 기능(GameFeature)이 그 지점에 위젯을 꽂는 방식 — HUD를 고치지 않고 기능별 UI를 추가. 모듈식 게임을 지향할 때 다음 단계.
🎮
게임패드 대응
ControllerData · ActionBar
플랫폼별 CommonInputBaseControllerData를 등록하면 버튼 아이콘(Ⓐ/×)이 자동 교체되고, CommonBoundActionBar가 화면 하단 조작 안내를 자동 표시.
⚠️
게임패드에서 "포커스가 어디 갔는지 모르는" 문제 — 화면이 활성화될 때 첫 포커스 대상을 정해줘야 게임패드 내비게이션이 시작됩니다. Activatable Widget의 GetDesiredFocusTarget을 오버라이드해(BP: Get Desired Focus Target) 첫 버튼을 반환하세요. 마우스로만 테스트하면 출시 직전까지 못 알아채는 함정입니다.
⚠️
Common UI 도입 후에도 수동 Set Input Mode를 섞으면 꼬입니다. 라우팅 시스템이 관리하는 입력 상태를 밖에서 덮어쓰는 셈이라, 이후 자동 복원이 엉킵니다. 도입했다면 입력 상태는 전부 위젯의 선언(GetDesiredInputConfig)으로 일원화하세요 — 반쯤 도입이 가장 나쁜 상태입니다.
💡
더 큰 규모가 필요해지면 Lyra의 원본을 가져올 수 있습니다 — 이 문서의 최소 구현은 Lyra CommonGame 플러그인의 축소판이므로, 프로젝트가 커져 정책(Policy) 층·플레이어별 레이아웃·비동기 Push가 필요해지면 Lyra에서 해당 플러그인을 복사해 갈아타는 경로가 자연스럽습니다. 구조가 같아서 이 문서의 개념이 그대로 통용됩니다.
