한글 폰트 파이프라인
UMG에서 한글이 □로 나올 때

첫 위젯에 "안녕하세요"를 넣는 순간 □□□□□가 뜹니다. 코드도 에셋도 멀쩡한데 글자만 깨져서, 초보자는 인코딩·FText·언어 설정을 헤매다 하루를 잃습니다. 진짜 원인은 하나 — UMG 기본 폰트(Roboto)에 한글 글리프가 없기 때문입니다. 이 문서는 TTF 임포트부터 프로젝트 전역 폰트 통일까지, 한 번 세팅하면 다시는 만질 일 없는 파이프라인을 다룹니다.

엔진 자체 기능 UE4 / UE5 공통 에디터 · C++
01 — 개념 · 원인

폰트는 그림 사전이다 — 없는 글자는 못 그린다

폰트 파일은 "문자 코드 → 글자 모양"의 사전입니다. 텍스트 데이터 자체(FString/FText)는 유니코드라 한글을 완벽히 담고 있지만, 그릴 때 쓰는 사전에 한글 항목이 없으면 렌더러는 대체 기호(□)를 그립니다. 즉 □는 데이터 문제가 아니라 그리기 재료 문제입니다.

핵심 용어

📖

글리프(Glyph) — 폰트 안에 든 글자 모양 하나. 한글을 다 담으려면 완성형 11,172자(실사용 2,350자+)의 글리프가 필요해, 영문 폰트(수백 글리프)와는 파일 크기부터 다릅니다.

두부(Tofu) 현상 — 글리프가 없는 문자가 □로 나오는 것. 생김새 때문에 업계에서 tofu라 부릅니다. 구글의 한글 폰트 이름 "Noto"가 No Tofu의 줄임말일 정도로 보편적인 문제입니다.

Font Face / Font(컴포짓) — 언리얼의 폰트 2계층. Font Face는 TTF/OTF 파일 하나를 감싼 에셋이고, Font(컴포짓 폰트)는 Font Face 여러 개를 "기본 서체 + 폴백 목록"으로 묶은 패밀리입니다. 위젯이 실제로 가리키는 것은 Font 쪽입니다.

폴백(Fallback) — 기본 서체에 글리프가 없을 때 다음 후보 폰트에서 찾아 그리는 체인. 영문은 A폰트, 한글은 B폰트로 자동 분담되는 원리입니다.

💡
에디터에서는 한글이 보였는데? — 언리얼 에디터 UI는 자체 CJK 폴백을 갖고 있어 디테일 패널 등에선 한글이 잘 보입니다. 하지만 게임 화면(UMG)의 기본 폰트는 Roboto라 별개입니다. "에디터는 되는데 게임은 □"가 헷갈림의 근원입니다.

창구 하나 원칙, 폰트에도 — 위젯들이 Font Face를 직접 가리키게 하지 말고, Font 에셋(F_Main) 하나만 가리키게 하세요. 나중에 폰트를 교체할 때 F_Main의 Face만 바꾸면 전체가 따라옵니다.

03 — 폰트 선택 책임

어떤 폰트를 쓸 것인가 — 라이선스부터

스팀 판매작에 넣을 폰트는 상업적 이용·임베딩(파일 동봉)이 허용돼야 합니다. 아래는 검증된 무료 선택지입니다.

폰트라이선스성격비고
PretendardOFL (SIL Open Font License)현대적 고딕 — UI 본문에 최적9굵기 제공. 이 디자인 시스템의 본문 폰트이기도 함
Noto Sans KROFL구글 표준 고딕 — 무난·안전글리프 커버리지 가장 넓음
나눔고딕/나눔스퀘어OFL부드러운 인상네이버 배포
시스템 폰트(맑은 고딕 등)동봉 불가—OS 폰트는 게임에 포함해 배포하면 안 됩니다 — 반드시 임포트용 폰트를 따로
⚠️
OFL도 조건이 있습니다 — 폰트 자체를 단독 판매하지 않는 한 게임 동봉·상업 이용이 자유지만, 라이선스 전문을 배포물(크레딧 등)에 포함하는 것이 안전합니다. 폰트 zip 안의 OFL.txt를 프로젝트에 함께 보관하세요.
04 — 임포트 → 적용 절차

다섯 걸음 — 한 번만 하면 되는 세팅

1
TTF 임포트 → Font Face 생성
Content Browser의 Content/UI/Fonts/(권장 위치)에 TTF/OTF를 드래그하면 임포트 대화상자가 뜹니다. 결과: Font Face 에셋(예: FF_Pretendard)과, 옵션에 따라 Font 에셋까지 함께 생성됩니다. Bold를 쓸 거면 Pretendard-Bold.ttf도 같은 방법으로.
2
Font(컴포짓) 에셋 구성
함께 생성된 Font 에셋(없으면 우클릭 ▸ User Interface ▸ Font)을 열고 — Default Font Family에 Typeface 항목을 추가합니다: Regular → FF_Pretendard, Bold → FF_Pretendard_Bold. 이름을 F_Main처럼 창구답게 지어 두세요.
3
폴백 체인 등록 (선택이지만 권장)
Font 에셋의 Fallback Font Families에 보조 폰트(예: Noto Sans KR Face)를 추가합니다. 결과: 기본 서체에 없는 문자(한자 이름, 특수 이모지 등)가 □ 대신 보조 폰트로 그려집니다. 전제: 보조 폰트도 Face로 임포트돼 있어야 함.
4
위젯에 적용
위젯 디자이너에서 TextBlock 선택 ▸ Details ▸ Appearance ▸ Font ▸ Font Family = F_Main, Typeface = Regular, Size 지정. 미리보기에서 한글이 즉시 살아나면 성공입니다.
5
전역 통일 — 위젯마다 반복하지 않기
전 위젯의 폰트를 한 방에 바꾸는 공식 스위치는 없습니다. 정석은 공통 텍스트 위젯: C++ 부모(예: UAppTextBlock : UTextBlock)가 생성 시점에 F_Main을 지정하게 하고(아래 스켈레톤), 모든 화면이 TextBlock 대신 이것을 쓰는 규칙을 세웁니다. 결과: 폰트·크기 정책이 클래스 한 곳에 모입니다.
05 — 구현 스켈레톤

공통 텍스트 위젯 — 폰트 정책의 단일 창구

🚀 이 문서로 작업 시작하기

🧭

전제 조건 — 폰트 파이프라인(임포트→Font Face→폴백 체인)이 본체인 문서로, C++는 선택 보강입니다. 위저드·Build.cs·컴파일의 공통 절차는 위키 『C++ 클래스 생성 · 빌드 워크플로우』(core-cpp)를 따릅니다.

만들 파일 / 작업 대상부모 클래스 (위저드 검색어)비고
AppTextBlock.h/.cpp (선택)TextBlock (All Classes)프로젝트 전역 서체 강제용 커스텀 텍스트 블록
💡

코드 외 작업 — 이 문서의 본체 — 한글 .ttf/.otf 임포트 → Font Face·Composite Font 폴백 등록 → 위젯 기본 폰트 교체(본문 절차). C++ 없이도 완결됩니다.

⚠️
컴파일 주의 — AppTextBlock을 만들 때만 재빌드(+Build.cs "UMG" 계열 확인).
#pragma once
#include "Components/TextBlock.h"
#include "AppTextBlock.generated.h"

// 프로젝트의 모든 텍스트가 상속하는 공통 위젯 — 폰트 정책은 여기 한 곳에만 산다.
UCLASS()   // 리플렉션 등록: 이 클래스를 UMG 팔레트·BP에서 위젯으로 배치 가능하게 함
// UTextBlock을 상속 → 기존 TextBlock의 모든 기능은 그대로 두고 폰트 기본값만 덮어씀
class MYGAME_API UAppTextBlock : public UTextBlock
{
    GENERATED_BODY()   // 리플렉션 코드 자동 생성 자리 — UCLASS 클래스에 필수
public:
    UAppTextBlock();
};
#include "AppTextBlock.h"
#include "Engine/Font.h"

UAppTextBlock::UAppTextBlock()
{
    // 생성자에서 로드해도 되는 이유: 위젯 CDO 구성 시점 — ConstructorHelpers 허용 구간
    // static: 클래스별로 딱 한 번만 폰트를 찾도록(생성자가 여러 번 불려도 조회는 1회)
    // 1단계 — F_Main 폰트 에셋을 경로로 찾는다. 경로 형식은 "/Game/폴더/에셋.에셋"(에셋 이름을 두 번)
    static ConstructorHelpers::FObjectFinder<UFont>
        MainFont(TEXT("/Game/UI/Fonts/F_Main.F_Main"));

    if (MainFont.Succeeded())   // 2단계 — 경로가 틀리면 못 찾음. 실패 시 손대지 않고 기본 Roboto 유지
    {
        // 3단계 — 현재 폰트 설정을 복사해와 필요한 필드만 바꾼 뒤 되돌려준다(부분 수정)
        FSlateFontInfo Info = GetFont();
        Info.FontObject   = MainFont.Object;   // 창구는 F_Main 하나
        Info.TypefaceFontName = TEXT("Regular");   // F_Main의 Typeface 목록에 있는 이름과 정확히 일치해야 함
        if (Info.Size <= 0) Info.Size = 15;   // 디자이너 지정이 없을 때의 기본(단위: 포인트)
        SetFont(Info);   // 4단계 — 수정한 폰트 정보를 이 위젯의 기본값으로 확정
    }
}
// 사용 규칙: 위젯 디자이너 팔레트에서 TextBlock 대신 AppTextBlock을 놓는다.
// 이미 만든 위젯은 우클릭 ▸ Replace With ▸ AppTextBlock 으로 일괄 교체.
💡

BP만으로 하고 싶다면 — C++ 부모 없이도, "폰트가 지정된 TextBlock을 품은 위젯 BP"를 만들어 재사용하는 방식이 가능합니다. 다만 크기·색까지 정책이 늘어나면 결국 클래스 한 곳이 편해집니다 — UMG+C++ 연동 문서의 BindWidget 구조와 같은 철학입니다.

06 — 확장 & 주의

폰트가 다시 □로 돌아오는 순간들

⚠️
① 패키징 후에만 □ — 위젯이 F_Main을 정상 참조하면 쿠킹에 자동 포함되지만, C++ 문자열 경로로만 참조하면(위 ConstructorHelpers 포함) 쿠커가 추적 못 할 수 있습니다. Project Settings ▸ Packaging ▸ Additional Asset Directories to Cook에 /Game/UI/Fonts를 등록해 두면 확실합니다 — 소프트 레퍼런스 문서의 쿠킹 누락과 같은 계열입니다.
⚠️
② Bold를 지정했는데 얇게 나옴 — Font 에셋의 Typeface 목록에 "Bold" 항목이 없으면 이름 매칭에 실패해 기본 서체로 그려집니다. 굵기마다 별도 TTF를 임포트해 Typeface로 등록해야 합니다(프로그램이 가짜 굵기를 만들어주지 않습니다).
⚠️
③ 숫자·영문이 촌스럽게 섞임 — 한글 폰트의 라틴 글리프가 마음에 안 들면, Font 에셋에서 기본 서체 = 영문 폰트, 폴백 = 한글 폰트 순서로 뒤집으세요. 라틴은 기본에서, 한글은 폴백에서 그려져 두 폰트가 자연스럽게 분담합니다.
💡
파일 크기 관리 — 한글 폰트는 굵기당 1~2MB대입니다. 쓸 굵기(Regular·Bold 정도)만 임포트하고, 장식용 디스플레이 폰트가 필요하면 제목 전용 Face 하나만 추가하는 식으로 절제하세요.
